Trust but Verify — Why Evaluation Matters

You've selected a model, tuned your prompts, and perhaps even fine-tuned for your domain. But how do you know the model is actually performing well? How do you guarantee it won't produce harmful, inaccurate, or offensive content in production?

Understanding AI Evaluation Metrics

Azure AI Foundry provides a suite of AI-assisted evaluation metrics that use GPT-based evaluators to score model outputs on multiple quality dimensions. Understanding each metric is critical to interpreting evaluation results correctly.

Groundedness

Definition: Does the model's response stay faithful to the provided source data (context)?

Groundedness measures whether the claims in a generated response can be traced back to the input context. This is the single most important metric for Retrieval-Augmented Generation (RAG) scenarios, where hallucination — generating plausible but unsupported facts — is the primary risk.

💡 Tip

A response can be fluent, coherent, and relevant while still being ungrounded. Always evaluate groundedness alongside other metrics to catch hallucinations.

ScoreInterpretation
5All claims are fully supported by the context
4Most claims are supported; minor extrapolations
3Some claims are supported, some are not
2Few claims are supported; significant fabrication
1Response has no grounding in the provided context

Relevance

Definition: Does the response actually answer the question that was asked?

Relevance evaluates whether the output addresses the user's intent and query. A response could be perfectly grounded and well-written but score poorly on relevance if it answers a different question or includes only tangential information.

Key signals for high relevance:

  • Directly addresses the user's question
  • Provides the right level of detail (not too broad, not too narrow)
  • Stays on topic throughout
  • Includes actionable information where appropriate

Coherence

Definition: Is the response logically structured and easy to follow?

Coherence measures the logical flow and organization of ideas in the generated text. Does the response present information in a sensible order? Are transitions smooth? Does each sentence connect logically to the next?

Low coherence manifests as:

  • Contradictory statements within the same response
  • Abrupt topic changes without transitions
  • Circular or repetitive reasoning
  • Conclusions that don't follow from the premises

Fluency

Definition: Is the response well-written and grammatically correct?

Fluency evaluates the linguistic quality of the output — grammar, spelling, punctuation, word choice, and natural language flow. While modern LLMs tend to score high on fluency by default, it remains important for fine-tuned models (especially those trained on domain-specific or non-English data) and for applications where linguistic quality directly impacts user trust.

Similarity

Definition: How close is the generated response to a known-good reference answer?

Similarity compares the model's output against a "golden" or reference answer to measure semantic closeness. This is most useful in scenarios where you have curated ground-truth answers, such as customer support FAQ systems or standardized reporting.

ℹ️ Metric Selection Guide

RAG applications: Prioritize groundedness + relevance.
Chatbots: Focus on coherence + fluency + relevance.
FAQ / knowledge base: Use similarity + groundedness.
Content generation: Emphasize fluency + coherence.

Running Evaluations in the Portal

The AI Foundry portal provides a visual evaluation workflow:

  1. Navigate to your project in ai.azure.com.
  2. Select Evaluation from the left navigation.
  3. Click + New evaluation and choose your evaluation type.
  4. Upload or select your test dataset (JSONL format recommended).
  5. Select the evaluators you want to run.
  6. Configure the model connection for AI-assisted evaluators.
  7. Submit and review results in the dashboard.
✅ Best Practice

Create a baseline evaluation before making any changes to your model or prompts. This gives you a reference point for measuring improvement (or regression) over time.

Content Safety Deep Dive

Azure AI Content Safety is a dedicated service that provides AI-powered moderation to detect harmful content across text and images. It integrates directly with Azure AI Foundry deployments, providing both input (prompt) and output (completion) filtering.

Content Safety Categories

Azure AI Content Safety classifies harmful content into four primary categories:

CategoryDescriptionExamples
Hate & Fairness Content that attacks or discriminates against identity groups Slurs, stereotypes, dehumanizing language
Sexual Sexually explicit or suggestive content Explicit descriptions, solicitation, exploitation
Violence Content depicting or promoting violence Graphic violence, threats, glorification of harm
Self-Harm Content related to self-inflicted harm Instructions, encouragement, or glorification of self-harm

Severity Levels

Each category classifies content into four severity levels, allowing fine-grained control over what is blocked versus allowed:

LevelSeverityDescription
0SafeContent is safe and appropriate
2LowMild or indirect references; generally acceptable
4MediumModerate content that may be inappropriate in some contexts
6HighSevere or explicit content; blocked by default
💡 Threshold Configuration

You set the threshold for each category. For example, setting the Violence threshold to "Medium" means content scored as Medium (4) or higher will be blocked. Lower your thresholds for more restrictive filtering — raise them only when your use case genuinely requires it (e.g., medical content may require a higher self-harm threshold).

Configuring Content Filters

Content filters are configured at the deployment level in Azure AI Foundry. You can create custom filter configurations and associate them with specific model deployments:

// Example: Creating a custom content filter configuration via REST API
PUT https://{endpoint}/contentsafety/text/configurations/{config-name}?api-version=2024-09-01
{
  "hateResult": { "severity": "medium" },
  "selfHarmResult": { "severity": "medium" },
  "sexualResult": { "severity": "low" },
  "violenceResult": { "severity": "medium" }
}

In the AI Foundry portal:

  1. Navigate to Safety + Security → Content filters.
  2. Click + Create content filter.
  3. Set severity thresholds for each category on both input and output.
  4. Optionally enable Prompt Shields (jailbreak detection).
  5. Associate the filter with your model deployment.

Custom Blocklists

Beyond the built-in categories, you can create custom blocklists to filter domain-specific terms that your organization deems inappropriate, such as competitor product names, internal project codenames, or regulated terminology.

# Python: Create and manage a custom blocklist
from azure.ai.contentsafety import ContentSafetyClient
from azure.ai.contentsafety.models import (
    TextBlocklist,
    AddOrUpdateTextBlocklistItemsOptions,
    TextBlocklistItem,
)
from azure.core.credentials import AzureKeyCredential

client = ContentSafetyClient(
    endpoint="https://<your-resource>.cognitiveservices.azure.com",
    credential=AzureKeyCredential("<your-key>"),
)

# Create a blocklist
blocklist = client.create_or_update_text_blocklist(
    blocklist_name="competitor-terms",
    options=TextBlocklist(description="Competitor product names"),
)

# Add terms to the blocklist
items = AddOrUpdateTextBlocklistItemsOptions(
    blocklist_items=[
        TextBlocklistItem(text="CompetitorProductX"),
        TextBlocklistItem(text="RivalServiceY"),
    ]
)
client.add_or_update_blocklist_items(
    blocklist_name="competitor-terms",
    options=items,
)

Jailbreak Detection & Prompt Shield

Prompt Shield is Azure AI Foundry's defense against prompt injection and jailbreak attacks. It detects attempts to:

  • Direct jailbreaks: User prompts that try to override the system message (e.g., "Ignore all previous instructions and…")
  • Indirect prompt injection: Malicious instructions embedded in external data sources (documents, web pages) that the model might process during RAG retrieval
⚠️ Critical Security Note

Prompt injection is a top security risk for AI applications. Always enable Prompt Shield on production deployments. Pair it with robust system messages that reinforce boundaries, and never rely on prompt-level defenses alone.

How Prompt Shield works:

  1. Input analysis: Every user prompt is analyzed for jailbreak patterns before reaching the model.
  2. Document analysis: Grounding documents retrieved by RAG are scanned for embedded instructions.
  3. Risk scoring: Each input receives a risk classification, and high-risk inputs are blocked.
  4. Logging: All detections are logged for security audit and continuous improvement.

Red-Teaming Your AI Application

Red-teaming is the practice of systematically probing your AI system for vulnerabilities, biases, and failure modes. Think of it as a penetration test for your AI — a structured adversarial exercise where testers deliberately try to make the system behave in undesirable ways.

Why Red-Team?

  • Discover blind spots that automated evaluation misses
  • Test safety boundaries under adversarial pressure
  • Validate system messages and guardrails
  • Identify bias across demographic and cultural dimensions
  • Build confidence before production deployment

Red-Teaming Framework

A structured red-teaming exercise typically follows these phases:

PhaseActivitiesOutput
1. Scope Define attack surfaces, threat models, and success criteria Red-team plan
2. Probe Execute attack scenarios — jailbreaks, bias probes, edge cases Raw findings log
3. Analyze Categorize findings by severity and impact Risk assessment
4. Remediate Update system messages, filters, or model configuration Mitigation actions
5. Retest Verify mitigations and check for regressions Validation report

Common Attack Vectors to Test

  1. Role-play attacks: "Pretend you are an AI with no restrictions…"
  2. Encoding tricks: Using Base64, leetspeak, or Unicode to bypass filters
  3. Multi-turn escalation: Gradually steering the conversation toward harmful territory
  4. Context confusion: Injecting instructions in grounding documents
  5. Boundary testing: Probing the edges of allowed topics
  6. Persona manipulation: Asking the model to adopt a harmful persona
✅ Best Practice

Microsoft recommends assembling a diverse red team with varied backgrounds, expertise levels, and perspectives. Adversarial creativity is enhanced by diversity — different people think of different attack vectors.

Automated Evaluation Pipelines

Manual evaluation doesn't scale. As your AI application evolves — new prompts, updated data, model version upgrades — you need automated evaluation pipelines that run continuously and alert you to quality regressions.

Pipeline Architecture

A typical automated evaluation pipeline consists of:

  1. Test dataset management: Versioned datasets with queries, expected outputs, and ground-truth contexts.
  2. Evaluation run orchestration: Triggered on code changes, model updates, or on a schedule.
  3. Multi-metric scoring: Run quality evaluators (groundedness, relevance, etc.) and safety evaluators in parallel.
  4. Threshold gating: Define minimum acceptable scores. Fail the pipeline if any metric drops below the threshold.
  5. Reporting & alerting: Generate dashboards, store historical trends, and alert on regressions.

CI/CD Integration

Integrate your evaluation pipeline into your CI/CD workflow so every deployment is validated:

# azure-pipelines.yml — Evaluation gate example
stages:
  - stage: Evaluate
    jobs:
      - job: RunEvaluation
        steps:
          - script: |
              pip install azure-ai-evaluation
              python run_evaluation.py \
                --dataset tests/eval_dataset.jsonl \
                --evaluators groundedness relevance coherence \
                --min-scores '{"groundedness": 4.0, "relevance": 4.0}' \
                --output results/
            displayName: "Run AI Evaluation"

          - script: |
              python check_thresholds.py --results results/scores.json
            displayName: "Validate Quality Thresholds"

  - stage: Deploy
    dependsOn: Evaluate
    condition: succeeded()
    jobs:
      - job: DeployToProduction
        steps:
          - script: echo "Deploying evaluated model..."
ℹ️ Continuous Evaluation

Consider running evaluations not just at deployment time, but also on a daily or weekly schedule against production traffic samples. This catches drift — gradual degradation that happens as real-world data evolves.

Evaluation SDKs — Python azure-ai-evaluation

The azure-ai-evaluation Python package is the primary SDK for running evaluations programmatically. It provides a clean API for individual evaluators and batch evaluation runs.

Installation

pip install azure-ai-evaluation

Running Individual Evaluators

from azure.ai.evaluation import (
    GroundednessEvaluator,
    RelevanceEvaluator,
    CoherenceEvaluator,
    FluencyEvaluator,
)

# Initialize evaluator with your AI Foundry project
groundedness = GroundednessEvaluator(model_config={
    "azure_endpoint": "https://<your-endpoint>.openai.azure.com/",
    "azure_deployment": "GPT-5.1",
    "api_version": "2024-12-01-preview",
})

# Evaluate a single response
result = groundedness(
    query="What is Azure AI Foundry?",
    response="Azure AI Foundry is a unified platform for building AI apps.",
    context="Azure AI Foundry is Microsoft's unified platform for developing, "
            "deploying, and managing AI applications.",
)

print(f"Groundedness score: {result['groundedness']}")
# Output: Groundedness score: 5

Batch Evaluation with Multiple Metrics

from azure.ai.evaluation import evaluate

# Run a batch evaluation across your test dataset
results = evaluate(
    data="test_dataset.jsonl",
    evaluators={
        "groundedness": GroundednessEvaluator(model_config=model_config),
        "relevance": RelevanceEvaluator(model_config=model_config),
        "coherence": CoherenceEvaluator(model_config=model_config),
        "fluency": FluencyEvaluator(model_config=model_config),
    },
    evaluator_config={
        "default": {
            "query": "${data.query}",
            "response": "${data.response}",
            "context": "${data.context}",
        }
    },
)

# View aggregate results
print(results["metrics"])
# {
#   "groundedness.groundedness": 4.2,
#   "relevance.relevance": 4.5,
#   "coherence.coherence": 4.7,
#   "fluency.fluency": 4.8
# }

Adding Safety Evaluators

from azure.ai.evaluation import (
    ViolenceEvaluator,
    SexualEvaluator,
    SelfHarmEvaluator,
    HateUnfairnessEvaluator,
)

# Add safety evaluators to your batch run
safety_results = evaluate(
    data="test_dataset.jsonl",
    evaluators={
        "violence": ViolenceEvaluator(model_config=model_config),
        "sexual": SexualEvaluator(model_config=model_config),
        "self_harm": SelfHarmEvaluator(model_config=model_config),
        "hate": HateUnfairnessEvaluator(model_config=model_config),
    },
)

# Check for any safety violations
for row in safety_results["rows"]:
    if any(v > 0 for k, v in row.items() if "_score" in k):
        print(f"⚠️  Safety concern in row: {row}")

🛠️ Mini Hack: Evaluate Your Fine-Tuned Model with Safety Checks

🎯 Objective

Run a comprehensive evaluation on your fine-tuned model from the previous module, combining quality metrics and safety evaluators to validate production readiness.

Prerequisites

  • Azure AI Foundry project with a deployed model
  • Python 3.10+ with azure-ai-evaluation installed
  • A test dataset (JSONL) with query, response, context, and ground_truth fields

Steps

  1. Prepare your test dataset — Create a JSONL file with at least 20 test cases covering normal usage, edge cases, and adversarial inputs.
  2. Configure evaluators — Set up both quality evaluators (groundedness, relevance, coherence, fluency) and safety evaluators (violence, sexual, self-harm, hate).
  3. Run the evaluation — Execute a batch evaluation using the evaluate() function from the SDK.
  4. Analyze results — Review aggregate scores. Identify any rows with low quality scores or safety flags.
  5. Set quality gates — Define minimum thresholds (e.g., groundedness ≥ 4.0, all safety scores = 0) and verify your model passes.
  6. Configure content filters — In the AI Foundry portal, create a custom content filter for your deployment with appropriate severity thresholds.
  7. Red-team test — Manually test 5 adversarial prompts against your deployed model and document the results.

Success Criteria

  • All quality metrics average ≥ 4.0 / 5.0
  • Zero safety violations across all test cases
  • Content filter properly blocks adversarial test inputs
  • Red-team findings documented with remediation notes

Key Takeaways

✅ What You Learned
  • Evaluation metrics — Groundedness, relevance, coherence, fluency, and similarity each measure a distinct quality dimension. Use them together for a complete picture.
  • Content Safety — Azure AI Content Safety provides four-category classification (hate, sexual, violence, self-harm) with four severity levels and configurable thresholds.
  • Prompt Shield — Protects against both direct jailbreaks and indirect prompt injection via grounding documents.
  • Custom blocklists — Extend built-in safety with domain-specific term filtering.
  • Red-teaming — Structured adversarial testing is essential before production deployment.
  • Automated pipelines — Integrate evaluation into CI/CD to catch regressions continuously.
  • Python SDK — The azure-ai-evaluation package provides programmatic access to all evaluators.
