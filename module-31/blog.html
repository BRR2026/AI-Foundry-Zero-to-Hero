🔐 Why IAM Matters for AI Foundry

Identity and Access Management (IAM) is the cornerstone of every secure cloud system. In Azure AI Foundry, IAM governs who can access your AI resources, what they can do, and how services authenticate to each other — all without hardcoded credentials.

⚠️
Real-World Impact

Over 80% of cloud security breaches involve misconfigured access. A single over-permissioned identity can expose model endpoints, training data, and inference pipelines. IAM done right is your first — and most critical — line of defense.

Azure AI Foundry's IAM model rests on three pillars:

🛡️

Role-Based Access Control

Assign precise permissions via built-in or custom RBAC roles scoped to subscriptions, resource groups, or individual resources.

🤖

Managed Identities

Eliminate credential management entirely. Azure handles token issuance and rotation for service-to-service authentication.

🔑

Service Principals

Non-interactive identities for CI/CD pipelines, automation scripts, and external service integrations.

👥 RBAC Roles for Azure AI Foundry

Azure Role-Based Access Control (RBAC) provides fine-grained access management for Azure AI Foundry resources. Roles are assigned to security principals (users, groups, service principals, or managed identities) at a specific scope (management group, subscription, resource group, or resource).

Built-in Roles

👑

Owner

Full access to all resources. Can assign roles to others. Use sparingly — typically only for subscription admins.

🔧

Contributor

Create and manage all resource types. Cannot grant access to others. Ideal for developers building AI Foundry projects.

👁️

Reader

View existing resources only. Perfect for auditors, stakeholders, or monitoring dashboards.

⚙️

Custom Roles

Tailor permissions precisely. Combine specific actions from data and control planes for granular access.

AI Foundry-Specific Roles

Role Scope Key Permissions Use Case
Azure AI Developer Resource / RG Create deployments, manage endpoints, run experiments Data scientists, ML engineers
Azure AI Inference Deployment Operator Resource Deploy models, manage inference endpoints MLOps teams deploying to production
Cognitive Services User Resource / RG Call AI services APIs, list keys Applications consuming AI endpoints
Cognitive Services Contributor Resource / RG Create and manage Cognitive Services resources Teams provisioning AI services
Search Index Data Reader Search Resource Query search indexes RAG applications reading index data
Storage Blob Data Contributor Storage Account Read, write, delete blob data AI Foundry accessing training data

Assigning Roles via Azure CLI

Azure CLI — Role Assignment Bash 
# Assign Azure AI Developer role to a user at the resource group scope
az role assignment create \
  --assignee "user@contoso.com" \
  --role "Azure AI Developer" \
  --scope "/subscriptions/{sub-id}/resourceGroups/{rg-name}"

# Assign Cognitive Services User to a managed identity at resource scope
az role assignment create \
  --assignee-object-id "{managed-identity-principal-id}" \
  --assignee-principal-type "ServicePrincipal" \
  --role "Cognitive Services User" \
  --scope "/subscriptions/{sub-id}/resourceGroups/{rg-name}/providers/Microsoft.CognitiveServices/accounts/{account-name}"

# List all role assignments for a resource group
az role assignment list \
  --resource-group "{rg-name}" \
  --output table

Creating Custom Roles

When built-in roles are too broad or too narrow, define custom roles with exactly the permissions needed. Custom roles are defined as JSON and can be scoped to specific subscriptions or management groups.

custom-ai-foundry-role.json JSON 
{
  "Name": "AI Foundry Model Deployer",
  "Description": "Can deploy and manage model endpoints but cannot modify project settings",
  "Actions": [
    "Microsoft.CognitiveServices/accounts/deployments/*",
    "Microsoft.CognitiveServices/accounts/models/read",
    "Microsoft.Resources/subscriptions/resourceGroups/read"
  ],
  "NotActions": [
    "Microsoft.CognitiveServices/accounts/delete",
    "Microsoft.CognitiveServices/accounts/write"
  ],
  "DataActions": [
    "Microsoft.CognitiveServices/accounts/OpenAI/deployments/completions/action",
    "Microsoft.CognitiveServices/accounts/OpenAI/deployments/chat/completions/action"
  ],
  "AssignableScopes": [
    "/subscriptions/{subscription-id}"
  ]
}
Create the custom role Azure CLI 
az role definition create --role-definition @custom-ai-foundry-role.json
💡
Best Practice

Always start with the narrowest built-in role that meets your needs. Only create custom roles when built-in roles genuinely cannot satisfy your requirements. Custom roles add maintenance overhead.

🤖 Managed Identity for Service-to-Service Auth

Managed Identities eliminate the need to store credentials in code, configuration files, or key vaults for service-to-service authentication. Azure automatically provisions and rotates the underlying credentials.

System-Assigned vs. User-Assigned

Feature System-Assigned User-Assigned
Lifecycle Tied to the resource — deleted when resource is deleted Independent resource — persists until explicitly deleted
Sharing Cannot be shared across resources Can be shared across multiple resources
Use Case Single resource needing its own identity Multiple resources needing the same identity/permissions
Management Automatic — no extra resources to manage Requires creating/managing the identity resource
Recommendation Simple scenarios, single-service workloads Production environments, multi-service architectures

Enabling Managed Identity on AI Foundry Resources

Enable System-Assigned Managed Identity Azure CLI 
# Enable system-assigned managed identity on an AI Foundry resource
az cognitiveservices account identity assign \
  --name "my-ai-foundry-account" \
  --resource-group "rg-ai-foundry-prod"

# Verify the identity was created
az cognitiveservices account show \
  --name "my-ai-foundry-account" \
  --resource-group "rg-ai-foundry-prod" \
  --query "identity"
Create and Assign User-Assigned Managed Identity Azure CLI 
# Create a user-assigned managed identity
az identity create \
  --name "id-ai-foundry-prod" \
  --resource-group "rg-ai-foundry-prod" \
  --location "eastus2"

# Get the identity's resource ID and principal ID
IDENTITY_ID=$(az identity show \
  --name "id-ai-foundry-prod" \
  --resource-group "rg-ai-foundry-prod" \
  --query "id" -o tsv)

PRINCIPAL_ID=$(az identity show \
  --name "id-ai-foundry-prod" \
  --resource-group "rg-ai-foundry-prod" \
  --query "principalId" -o tsv)

# Grant Storage Blob Data Contributor on the storage account
az role assignment create \
  --assignee-object-id "$PRINCIPAL_ID" \
  --assignee-principal-type "ServicePrincipal" \
  --role "Storage Blob Data Contributor" \
  --scope "/subscriptions/{sub-id}/resourceGroups/rg-ai-foundry-prod/providers/Microsoft.Storage/storageAccounts/staidata"

Using Managed Identity in Python Code

managed_identity_client.py Python 
from azure.identity import DefaultAzureCredential, ManagedIdentityCredential
from azure.ai.projects import AIProjectClient
from azure.storage.blob import BlobServiceClient

# DefaultAzureCredential automatically uses managed identity in Azure,
# falls back to CLI/env credentials locally
credential = DefaultAzureCredential()

# Connect to AI Foundry project — no keys or connection strings needed
ai_client = AIProjectClient(
    credential=credential,
    subscription_id="your-subscription-id",
    resource_group_name="rg-ai-foundry-prod",
    project_name="my-ai-project",
)

# Access blob storage using the same credential
blob_client = BlobServiceClient(
    account_url="https://staidata.blob.core.windows.net",
    credential=credential,
)

# List containers — no storage keys needed!
for container in blob_client.list_containers():
    print(f"Container: {container['name']}")
✅
Zero Secrets in Code

With DefaultAzureCredential, your code works seamlessly both locally (using Azure CLI auth) and in production (using managed identity) — without changing a single line.

🔑 Service Principals & CI/CD Authentication

While managed identities are ideal for Azure-hosted services, service principals are required for scenarios where code runs outside Azure — such as CI/CD pipelines in GitHub Actions, Azure DevOps, or on-premises automation.

Creating a Service Principal for CI/CD

Create Service Principal with Scoped Permissions Azure CLI 
# Create a service principal with Contributor role on the AI Foundry resource group
az ad sp create-for-rbac \
  --name "sp-ai-foundry-cicd" \
  --role "Contributor" \
  --scopes "/subscriptions/{sub-id}/resourceGroups/rg-ai-foundry-prod" \
  --sdk-auth

Security: Prefer Federated Credentials

Instead of storing client secrets, use Workload Identity Federation with GitHub Actions or Azure DevOps. This eliminates long-lived secrets entirely by exchanging OIDC tokens.

Workload Identity Federation with GitHub Actions

Configure Federated Credential Azure CLI 
# Create federated credential for GitHub Actions
az ad app federated-credential create \
  --id "{app-object-id}" \
  --parameters '{
    "name": "github-actions-deploy",
    "issuer": "https://token.actions.githubusercontent.com",
    "subject": "repo:contoso/ai-foundry-app:ref:refs/heads/main",
    "audiences": ["api://AzureADTokenExchange"]
  }'
.github/workflows/deploy.yml YAML 
name: Deploy AI Foundry Model
on:
  push:
    branches: [main]

permissions:
  id-token: write    # Required for OIDC
  contents: read

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Azure Login (Federated)
        uses: azure/login@v2
        with:
          client-id: ${{ secrets.AZURE_CLIENT_ID }}
          tenant-id: ${{ secrets.AZURE_TENANT_ID }}
          subscription-id: ${{ secrets.AZURE_SUBSCRIPTION_ID }}

      - name: Deploy Model Endpoint
        run: |
          az cognitiveservices account deployment create \
            --name my-ai-foundry-account \
            --resource-group rg-ai-foundry-prod \
            --deployment-name gpt-4o-prod \
            --model-name gpt-4o \
            --model-version "2024-11-20" \
            --model-format OpenAI \
            --sku-capacity 10 \
            --sku-name GlobalStandard

🎯 Least-Privilege Access Patterns

The principle of least privilege states that every identity should have only the minimum permissions required to perform its designated function — nothing more, nothing less. This limits blast radius when credentials are compromised.

🏗️ Layered Access Architecture

Apply permissions at the narrowest scope possible. A developer who needs to deploy a model should get Azure AI Inference Deployment Operator on that specific resource, not Contributor on the entire subscription.

Access Pattern Matrix

Persona Recommended Roles Scope Auth Method
Platform Admin Owner (with PIM/JIT) Subscription Entra ID + MFA
AI Engineer Azure AI Developer Resource Group Entra ID + MFA
Data Scientist Cognitive Services User + Storage Blob Data Reader Resource Entra ID + MFA
Application (Prod) Cognitive Services User Resource Managed Identity
CI/CD Pipeline Custom "Model Deployer" Resource Group Workload Identity Federation
Monitoring / Audit Reader + Log Analytics Reader Resource Group Entra ID

Security Hardening Checklist

  • Enable Privileged Identity Management (PIM) for standing admin access
  • Use Conditional Access policies requiring MFA for all human identities
  • Replace API keys with Managed Identity or Workload Identity Federation
  • Scope role assignments to the narrowest resource possible
  • Audit role assignments quarterly using Azure Policy and Access Reviews
  • Set expiration on role assignments using PIM time-bound access
  • Enable diagnostic logging for all identity operations
  • Use separate identities for dev, staging, and production environments

🧪 Mini Hack

🛠️ Configure RBAC & Managed Identity for Your Project

Apply what you've learned by securing an Azure AI Foundry project with proper RBAC and managed identity configuration. Estimated time: 30–40 minutes.

Step 1 — Create a Resource Group and AI Foundry Account

Provision a new resource group and AI Foundry account (or use an existing one from previous modules). Ensure you have Owner or User Access Administrator at the resource group scope.

Step 2 — Create a User-Assigned Managed Identity

Create a managed identity named id-ai-foundry-hack. This identity will represent your application.

Step 3 — Assign RBAC Roles with Least Privilege

Grant Cognitive Services User on the AI Foundry account and Storage Blob Data Reader on the storage account to the managed identity.

Step 4 — Create a Custom Role

Define a custom role called "AI Foundry Model Reader" that allows listing and reading model deployments but not creating or deleting them.

Step 5 — Validate with Python

Write a Python script using DefaultAzureCredential to authenticate as the managed identity, list available models, and read from blob storage. Verify no excess permissions exist.

Step 6 — Audit Your Configuration

Use az role assignment list to review all role assignments. Verify no identity has broader permissions than needed. Document your findings.

📋
Success Criteria

✅ Managed identity can call AI Foundry endpoints and read blobs
✅ Custom role restricts deployment creation
✅ No identity has Owner or Contributor at subscription scope
✅ All authentication uses managed identity (zero API keys in code)
