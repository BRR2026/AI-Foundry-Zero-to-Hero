1 Why CI/CD for AI Matters

Traditional software CI/CD pipelines ensure that code changes flow predictably from development to production. AI workloads add unique challenges: model artifacts, evaluation datasets, prompt templates, and infrastructure provisioning all need to be versioned, tested, and deployed together. Azure AI Foundry provides a unified platform for AI development — and integrating it with CI/CD pipelines transforms ad-hoc experimentation into production-grade MLOps.

⚠️ Without CI/CD

  • Manual model deployment via portal clicks
  • No reproducibility across environments
  • Missing quality gates before production
  • Configuration drift between dev/staging/prod
  • No audit trail for who deployed what

✅ With CI/CD

  • Automated deployment on every merge
  • Identical infrastructure via IaC templates
  • Evaluation gates block bad deployments
  • Environment parity guaranteed
  • Full traceability via Git history

💡 Azure AI Foundry & CI/CD

Azure AI Foundry supports CLI-driven and SDK-driven operations for projects, deployments, connections, and evaluations. This makes it a first-class citizen in any CI/CD pipeline — no portal interactions required for production workflows.

2 GitHub Actions for Azure AI Foundry Deployments

GitHub Actions provides a flexible, YAML-based CI/CD platform that integrates natively with Azure through workload identity federation (OIDC). This eliminates the need for stored credentials and enables secure, automated deployments to Azure AI Foundry.

2.1 Authentication: Workload Identity Federation

The recommended approach is OIDC-based authentication using azure/login@v2. This creates a short-lived token for each workflow run — no secrets stored in GitHub.

YAML
# .github/workflows/ai-foundry-deploy.yml
name: Deploy AI Foundry Resources

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

permissions:
  id-token: write
  contents: read

env:
  AZURE_SUBSCRIPTION_ID: ${{ secrets.AZURE_SUBSCRIPTION_ID }}
  RESOURCE_GROUP: rg-ai-foundry-prod
  AI_PROJECT_NAME: my-ai-project
  LOCATION: eastus2

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Azure Login (OIDC)
        uses: azure/login@v2
        with:
          client-id: ${{ secrets.AZURE_CLIENT_ID }}
          tenant-id: ${{ secrets.AZURE_TENANT_ID }}
          subscription-id: ${{ secrets.AZURE_SUBSCRIPTION_ID }}

2.2 Deploying AI Foundry Resources

After authentication, use the Azure CLI or AI Foundry SDK to manage deployments programmatically:

2.3 Python Deployment Script

Python
# scripts/deploy_model.py
import argparse
from azure.identity import DefaultAzureCredential
from azure.ai.projects import AIProjectClient

def deploy_model(args):
    credential = DefaultAzureCredential()
    client = AIProjectClient(
        credential=credential,
        subscription_id=args.subscription_id,
        resource_group_name=args.resource_group,
        project_name=args.project_name,
    )

    # Create or update the model deployment
    deployment = client.deployments.begin_create_or_update(
        deployment_name=args.deployment_name,
        deployment={
            "model": {"name": args.model_name, "version": "latest"},
            "sku": {
                "name": "GlobalStandard",
                "capacity": args.sku_capacity,
            },
        },
    ).result()

    print(f"✅ Deployed {deployment.name} successfully")
    return deployment

if __name__ == "__main__":
    parser = argparse.ArgumentParser()
    parser.add_argument("--project-name", required=True)
    parser.add_argument("--resource-group", required=True)
    parser.add_argument("--subscription-id", default=None)
    parser.add_argument("--model-name", default="gpt-4o")
    parser.add_argument("--deployment-name", required=True)
    parser.add_argument("--sku-capacity", type=int, default=30)
    deploy_model(parser.parse_args())

3 Azure DevOps Pipeline Integration

For organizations using Azure DevOps, YAML-based pipelines provide the same automation capabilities with tight Azure integration through service connections.

3.1 Service Connection Setup

Create a service connection in Azure DevOps that uses workload identity federation for secure authentication:

  1. Navigate to Project Settings → Service connections → New
  2. Select Azure Resource Manager → Workload identity federation (automatic)
  3. Select the subscription and resource group for your AI Foundry project
  4. Name the connection (e.g., ai-foundry-prod)

3.2 Multi-Stage Pipeline

YAML
# azure-pipelines.yml
trigger:
  branches:
    include:
      - main
  paths:
    include:
      - infra/**
      - src/**
      - prompts/**

variables:
  - name: resourceGroup
    value: rg-ai-foundry
  - name: projectName
    value: contoso-ai-project

stages:
  # ── Stage 1: Validate ──
  - stage: Validate
    jobs:
      - job: ValidateInfra
        pool:
          vmImage: ubuntu-latest
        steps:
          - task: AzureCLI@2
            displayName: Validate Bicep templates
            inputs:
              azureSubscription: ai-foundry-prod
              scriptType: bash
              scriptLocation: inlineScript
              inlineScript: |
                az bicep build --file infra/main.bicep
                az deployment group validate \
                  --resource-group $(resourceGroup) \
                  --template-file infra/main.bicep

  # ── Stage 2: Deploy to Staging ──
  - stage: DeployStaging
    dependsOn: Validate
    jobs:
      - deployment: DeployToStaging
        environment: staging
        strategy:
          runOnce:
            deploy:
              steps:
                - task: AzureCLI@2
                  displayName: Deploy AI Foundry Resources
                  inputs:
                    azureSubscription: ai-foundry-prod
                    scriptType: bash
                    inlineScript: |
                      az deployment group create \
                        --resource-group $(resourceGroup)-staging \
                        --template-file infra/main.bicep \
                        --parameters env=staging

  # ── Stage 3: Evaluate ──
  - stage: Evaluate
    dependsOn: DeployStaging
    jobs:
      - job: RunEvaluations
        steps:
          - script: |
              pip install azure-ai-evaluation azure-ai-projects
              python scripts/run_evaluation.py --env staging
            displayName: Run AI quality evaluations

  # ── Stage 4: Deploy to Production ──
  - stage: DeployProd
    dependsOn: Evaluate
    jobs:
      - deployment: DeployToProduction
        environment: production
        strategy:
          runOnce:
            deploy:
              steps:
                - task: AzureCLI@2
                  displayName: Deploy to Production
                  inputs:
                    azureSubscription: ai-foundry-prod
                    scriptType: bash
                    inlineScript: |
                      az deployment group create \
                        --resource-group $(resourceGroup)-prod \
                        --template-file infra/main.bicep \
                        --parameters env=production

⚠️ Environments & Approvals

Always configure manual approval gates on your production environment in Azure DevOps. The evaluation stage should pass before any human approves the production deployment — creating a two-layer quality gate.

4 Infrastructure as Code — Bicep & Terraform

All Azure AI Foundry resources — hubs, projects, model deployments, connections, and evaluation jobs — can and should be defined as Infrastructure as Code. This ensures reproducibility, version control, and automated provisioning across environments.

AI Foundry Hub & Project (Bicep)

Bicep
// infra/main.bicep
@description('Environment name (dev, staging, production)')
param env string = 'dev'

@description('Azure region for deployment')
param location string = resourceGroup().location

@description('Base name for all resources')
param baseName string = 'contoso-ai'

var uniqueSuffix = uniqueString(resourceGroup().id)
var hubName = 'hub-${baseName}-${env}-${uniqueSuffix}'
var projectName = 'proj-${baseName}-${env}'

// ── AI Services Account ──
resource aiServices 'Microsoft.CognitiveServices/accounts@2024-10-01' = {
  name: 'ais-${baseName}-${env}-${uniqueSuffix}'
  location: location
  kind: 'AIServices'
  sku: { name: 'S0' }
  properties: {
    customSubDomainName: 'ais-${baseName}-${env}-${uniqueSuffix}'
    publicNetworkAccess: 'Enabled'
  }
}

// ── AI Foundry Hub ──
resource hub 'Microsoft.MachineLearningServices/workspaces@2024-10-01' = {
  name: hubName
  location: location
  kind: 'Hub'
  sku: { name: 'Basic'; tier: 'Basic' }
  identity: { type: 'SystemAssigned' }
  properties: {
    friendlyName: 'AI Foundry Hub (${env})'
    description: 'Deployed via CI/CD pipeline'
  }
}

// ── AI Foundry Project ──
resource project 'Microsoft.MachineLearningServices/workspaces@2024-10-01' = {
  name: projectName
  location: location
  kind: 'Project'
  sku: { name: 'Basic'; tier: 'Basic' }
  identity: { type: 'SystemAssigned' }
  properties: {
    friendlyName: 'AI Project (${env})'
    hubResourceId: hub.id
  }
}

// ── Outputs for pipeline consumption ──
output hubId string = hub.id
output projectId string = project.id
output aiServicesEndpoint string = aiServices.properties.endpoint

5 Automated Testing in CI/CD Pipelines

AI pipelines demand testing at multiple layers: unit tests for code logic, integration tests for API contracts, and AI-specific evaluations for model quality. Azure AI Foundry's evaluation SDK integrates directly into your pipeline to enforce quality gates.

5.1 Testing Layers

🧩 Unit Tests

  • Prompt template rendering
  • Input/output schema validation
  • Data preprocessing functions
  • Configuration loading logic

🔗 Integration Tests

  • API endpoint connectivity
  • Model deployment health checks
  • Connection string validation
  • End-to-end request flow

📊 AI Evaluations

  • Groundedness scoring
  • Relevance & coherence metrics
  • Safety & content filtering
  • Latency / throughput benchmarks

5.2 AI Evaluation as a Pipeline Gate

Python
# scripts/run_evaluation.py
import sys
import json
from azure.identity import DefaultAzureCredential
from azure.ai.evaluation import evaluate, GroundednessEvaluator, RelevanceEvaluator
from azure.ai.projects import AIProjectClient

def run_evaluation():
    credential = DefaultAzureCredential()

    # Initialize evaluators
    model_config = {
        "azure_deployment": "gpt-4o-eval",
        "azure_endpoint": "https://ais-contoso.openai.azure.com/",
    }

    groundedness = GroundednessEvaluator(model_config=model_config)
    relevance = RelevanceEvaluator(model_config=model_config)

    # Run evaluation against test dataset
    result = evaluate(
        data="data/evaluation_dataset.jsonl",
        evaluators={
            "groundedness": groundedness,
            "relevance": relevance,
        },
        evaluator_config={
            "groundedness": {
                "query": "${data.query}",
                "context": "${data.context}",
                "response": "${data.response}",
            },
            "relevance": {
                "query": "${data.query}",
                "response": "${data.response}",
            },
        },
    )

    # Enforce quality thresholds
    metrics = result["metrics"]
    thresholds = {"groundedness.groundedness": 4.0, "relevance.relevance": 4.0}

    failures = []
    for metric, threshold in thresholds.items():
        score = metrics.get(metric, 0)
        if score < threshold:
            failures.append(f"❌ {metric}: {score:.2f} < {threshold}")
        else:
            print(f"✅ {metric}: {score:.2f} >= {threshold}")

    if failures:
        print("\n🚫 Evaluation gate FAILED:")
        for f in failures:
            print(f"  {f}")
        sys.exit(1)
    else:
        print("\n🎉 All evaluation gates PASSED")

if __name__ == "__main__":
    run_evaluation()

6.1 Complete GitHub Actions Workflow

YAML
# .github/workflows/ai-foundry-complete.yml
name: AI Foundry — Full CI/CD Pipeline

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

permissions:
  id-token: write
  contents: read

jobs:
  # ── Job 1: Lint & Unit Test ──
  lint-and-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with: { python-version: "3.12" }
      - run: |
          pip install -r requirements.txt
          pip install ruff pytest
          ruff check src/
          pytest tests/ -v --tb=short

  # ── Job 2: Validate IaC ──
  validate-iac:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: azure/login@v2
        with:
          client-id: ${{ secrets.AZURE_CLIENT_ID }}
          tenant-id: ${{ secrets.AZURE_TENANT_ID }}
          subscription-id: ${{ secrets.AZURE_SUBSCRIPTION_ID }}
      - run: |
          az bicep build --file infra/main.bicep
          az deployment group what-if \
            --resource-group rg-ai-foundry-prod \
            --template-file infra/main.bicep

  # ── Job 3: Deploy Staging ──
  deploy-staging:
    needs: [lint-and-test, validate-iac]
    if: github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    environment: staging
    steps:
      - uses: actions/checkout@v4
      - uses: azure/login@v2
        with:
          client-id: ${{ secrets.AZURE_CLIENT_ID }}
          tenant-id: ${{ secrets.AZURE_TENANT_ID }}
          subscription-id: ${{ secrets.AZURE_SUBSCRIPTION_ID }}
      - name: Deploy infrastructure & model
        run: |
          az deployment group create \
            --resource-group rg-ai-foundry-staging \
            --template-file infra/main.bicep \
            --parameters env=staging

          pip install azure-ai-projects azure-identity
          python scripts/deploy_model.py \
            --project-name proj-contoso-staging \
            --resource-group rg-ai-foundry-staging \
            --deployment-name gpt-4o-staging \
            --sku-capacity 10

  # ── Job 4: Evaluate ──
  evaluate:
    needs: deploy-staging
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: azure/login@v2
        with:
          client-id: ${{ secrets.AZURE_CLIENT_ID }}
          tenant-id: ${{ secrets.AZURE_TENANT_ID }}
          subscription-id: ${{ secrets.AZURE_SUBSCRIPTION_ID }}
      - name: Run AI evaluations (quality gate)
        run: |
          pip install azure-ai-evaluation azure-ai-projects azure-identity
          python scripts/run_evaluation.py --env staging

  # ── Job 5: Deploy Production ──
  deploy-production:
    needs: evaluate
    runs-on: ubuntu-latest
    environment: production
    steps:
      - uses: actions/checkout@v4
      - uses: azure/login@v2
        with:
          client-id: ${{ secrets.AZURE_CLIENT_ID }}
          tenant-id: ${{ secrets.AZURE_TENANT_ID }}
          subscription-id: ${{ secrets.AZURE_SUBSCRIPTION_ID }}
      - name: Deploy to production
        run: |
          az deployment group create \
            --resource-group rg-ai-foundry-prod \
            --template-file infra/main.bicep \
            --parameters env=production

          python scripts/deploy_model.py \
            --project-name proj-contoso-prod \
            --resource-group rg-ai-foundry-prod \
            --deployment-name gpt-4o-prod \
            --sku-capacity 50

8 Mini Hack: Deploy & Evaluate with GitHub Actions

🏆 Mini Hack Challenge

Goal: Set up a GitHub Actions pipeline that deploys an Azure AI Foundry project and runs automated evaluations as a quality gate.

Prerequisites

  • Azure subscription with AI Foundry access
  • GitHub repository with Actions enabled
  • Azure AD app registration with federated credentials for OIDC
  • An evaluation dataset (minimum 20 test samples)

Step-by-Step

  1. Repository Setup: Create the directory structure: 
    Directory
your-repo/
├── .github/workflows/
│   └── ai-foundry-deploy.yml
├── infra/
│   └── main.bicep
├── scripts/
│   ├── deploy_model.py
│   └── run_evaluation.py
├── data/
│   └── evaluation_dataset.jsonl
├── tests/
│   └── test_prompts.py
└── requirements.txt
  2. Configure OIDC: In Azure, create a federated credential on your app registration: 
    Azure CLI
# Create federated credential for GitHub Actions
az ad app federated-credential create \
  --id <APP_OBJECT_ID> \
  --parameters '{
    "name": "github-actions-main",
    "issuer": "https://token.actions.githubusercontent.com",
    "subject": "repo:YOUR_ORG/YOUR_REPO:ref:refs/heads/main",
    "audiences": ["api://AzureADTokenExchange"]
  }'
  3. Store Secrets: In your GitHub repo → Settings → Secrets, add:
    • AZURE_CLIENT_ID — App registration client ID
    • AZURE_TENANT_ID — Azure AD tenant ID
    • AZURE_SUBSCRIPTION_ID — Target subscription ID
  4. Create the Workflow: Use the complete workflow from Section 6 as your starting template.
  5. Write Evaluation Tests: Create at least 20 query/context/response triples in your evaluation dataset.
  6. Push & Verify: Push to main and monitor the Actions tab. The pipeline should:
    • Pass lint and unit tests
    • Validate Bicep templates
    • Deploy to staging
    • Run evaluations and pass all quality thresholds
    • Wait for manual approval
    • Deploy to production

🎯 Success Criteria

  • Pipeline runs end-to-end without manual intervention (except approval gate)
  • Evaluation scores meet thresholds (groundedness ≥ 4.0, relevance ≥ 4.0)
  • Infrastructure is consistently created across staging and production
  • Failed evaluations correctly block production deployment

9 Best Practices & Anti-Patterns

✅ Best Practices

  • Use OIDC / workload identity — no stored secrets
  • Separate IaC from application code in repo
  • Run what-if / plan on every PR
  • Enforce evaluation gates before production
  • Version prompt templates alongside code
  • Use GitHub Environments for approval gates
  • Cache pip dependencies between runs
  • Store evaluation results as pipeline artifacts
  • Tag deployments with commit SHA for traceability

❌ Anti-Patterns

  • Storing service principal secrets in workflow files
  • Skipping the staging environment
  • Deploying without evaluation gates
  • Hardcoding resource names instead of parameterizing
  • Running IaC changes without preview first
  • Using portal deployments for production resources
  • No rollback strategy for failed deployments
  • Ignoring pipeline run duration (costs money!)
  • Mixing AI Foundry SDK versions across pipeline steps
