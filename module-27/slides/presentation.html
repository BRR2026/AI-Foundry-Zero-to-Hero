🤔 Why CI/CD for AI?

⚠️ Without CI/CD

  • Manual portal deployments
  • No reproducibility
  • Configuration drift
  • No quality gates
  • "Works on my machine"

✅ With CI/CD

  • Automated on every merge
  • IaC ensures consistency
  • Environment parity
  • Evaluation gates block bad deploys
  • Full audit trail via Git

🔐 GitHub Actions — OIDC Authentication

Workload Identity Federation: no stored secrets, short-lived tokens per run.

# Permissions for OIDC token
permissions:
  id-token: write
  contents: read

steps:
  - name: Azure Login (OIDC)
    uses: azure/login@v2
    with:
      client-id: ${{ secrets.AZURE_CLIENT_ID }}
      tenant-id: ${{ secrets.AZURE_TENANT_ID }}
      subscription-id: ${{ secrets.AZURE_SUBSCRIPTION_ID }}

💡 No client secrets or certificates needed — GitHub generates a JWT that Azure trusts via federated credentials.

🚀 GitHub Actions — Deploy AI Foundry

# Deploy infrastructure + model
- name: Deploy Infrastructure
  run: |
    az deployment group create \
      --resource-group rg-ai-foundry-staging \
      --template-file infra/main.bicep \
      --parameters env=staging

- name: Deploy Model
  run: |
    pip install azure-ai-projects azure-identity
    python scripts/deploy_model.py \
      --project-name proj-contoso-staging \
      --resource-group rg-ai-foundry-staging \
      --deployment-name GPT-5.1-staging

Two-phase deploy: Infrastructure (Bicep) then Model (Python SDK).

🔧 Azure DevOps — Multi-Stage Pipeline

stages:
  - stage: Validate       # Bicep lint + what-if
  - stage: DeployStaging  # Deploy to staging env
    dependsOn: Validate
  - stage: Evaluate       # Run AI quality evals
    dependsOn: DeployStaging
  - stage: DeployProd     # Production (approval gate)
    dependsOn: Evaluate

Key difference: Azure DevOps uses service connections for auth and environments for approval gates.

📐 Infrastructure as Code — Bicep

// AI Foundry Hub
resource hub 'Microsoft.MachineLearningServices/workspaces@2024-10-01' = {
  name: hubName
  location: location
  kind: 'Hub'
  sku: { name: 'Basic'; tier: 'Basic' }
  identity: { type: 'SystemAssigned' }
}

// AI Foundry Project (child of Hub)
resource project 'Microsoft.MachineLearningServices/workspaces@2024-10-01' = {
  name: projectName
  location: location
  kind: 'Project'
  properties: { hubResourceId: hub.id }
}

Parameterize with env for dev / staging / production parity.

📐 Infrastructure as Code — Terraform

resource "azurerm_ai_foundry" "hub" {
  name                = "hub-contoso-${var.environment}"
  location            = azurerm_resource_group.main.location
  resource_group_name = azurerm_resource_group.main.name
  sku_name            = "Basic"
  identity { type = "SystemAssigned" }
}

resource "azurerm_ai_foundry_project" "project" {
  name               = "proj-contoso-${var.environment}"
  ai_services_hub_id = azurerm_ai_foundry.hub.id
  identity { type = "SystemAssigned" }
}

Bicep vs. Terraform: Bicep is Azure-native (no state file). Terraform works multi-cloud with remote state.

🚦 Evaluation as a Quality Gate

from azure.ai.evaluation import evaluate, GroundednessEvaluator

result = evaluate(
    data="data/evaluation_dataset.jsonl",
    evaluators={"groundedness": GroundednessEvaluator(model_config)},
)

# Enforce threshold — fail pipeline if below 4.0
score = result["metrics"]["groundedness.groundedness"]
if score < 4.0:
    sys.exit(1)  # ❌ Pipeline fails

The evaluation gate acts as an automated quality checkpoint — bad models never reach production.

✅ Best Practices

Do ✅

  • Use OIDC — no stored secrets
  • Run what-if / plan on PRs
  • Enforce evaluation gates
  • Version prompts with code
  • Cache dependencies
  • Tag deploys with commit SHA
  • Use environments + approvals

Don't ❌

  • Store secrets in workflow files
  • Skip staging environment
  • Deploy without evaluation
  • Hardcode resource names
  • Use portal for production
  • Mix SDK versions
  • Ignore pipeline costs

🏆 Mini Hack: Deploy & Evaluate

Build a complete GitHub Actions pipeline:

  1. Create repo structure: .github/workflows/, infra/, scripts/, data/
  2. Configure OIDC auth with federated credentials
  3. Write Bicep template for AI Foundry Hub + Project
  4. Create deployment script using Python SDK
  5. Add evaluation gate with quality thresholds
  6. Push to main and watch pipeline run end-to-end

Success: Evaluation passes ≥ 4.0 → staging deploys → manual approval → production deploys 🚀
