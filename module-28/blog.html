Why Observability Matters for AI

Traditional software has predictable outputs for given inputs. AI systems are stochastic — the same prompt can produce different results, quality can silently degrade as data distributions shift, and failures often manifest as subtly wrong answers rather than crashes. This makes monitoring AI fundamentally different from monitoring conventional applications.

⚠️

The Silent Failure Problem: Unlike a 500 error, a model that starts giving lower-quality answers won't trigger traditional alerting. Without AI-specific observability, you won't know until users complain — or worse, leave.

Three Pillars of AI Observability

📊
Metrics
Latency, throughput, token counts, error rates, cost-per-request
📜
Traces
End-to-end request flow, agent step tracing, tool invocations
📝
Logs
Structured events, prompt/completion pairs, guardrail triggers
🎯
Evaluations
Quality scores, drift indicators, safety checks, groundedness

Azure AI Foundry integrates deeply with Azure Monitor, Application Insights, and Log Analytics to provide a unified observability stack purpose-built for AI workloads.

Configure diagnostic settings on your AI Foundry project to route telemetry to a Log Analytics workspace:

# Enable diagnostic settings for your AI Foundry project
az monitor diagnostic-settings create \
  --name "ai-foundry-diagnostics" \
  --resource "/subscriptions/{sub-id}/resourceGroups/{rg}/providers/Microsoft.MachineLearningServices/workspaces/{project}" \
  --workspace "/subscriptions/{sub-id}/resourceGroups/{rg}/providers/Microsoft.OperationalInsights/workspaces/{workspace}" \
  --logs '[
    {"category": "AmlComputeClusterEvent", "enabled": true},
    {"category": "AmlRunStatusChangedEvent", "enabled": true},
    {"category": "AmlModelsEvent", "enabled": true}
  ]' \
  --metrics '[{"category": "AllMetrics", "enabled": true}]'

Custom Telemetry Wrapper

Wrap your model calls to automatically capture key metrics:

import time
from opentelemetry import trace, metrics
from azure.ai.projects import AIProjectClient
from azure.identity import DefaultAzureCredential

tracer = trace.get_tracer("ai-monitor")
meter = metrics.get_meter("ai-monitor")

# Define counters and histograms
request_counter = meter.create_counter("ai.requests.total")
error_counter = meter.create_counter("ai.errors.total")
token_counter = meter.create_counter("ai.tokens.consumed")
latency_hist = meter.create_histogram("ai.request.latency_ms")

class MonitoredModelClient:
    def __init__(self, project_client: AIProjectClient):
        self.client = project_client

    def complete(self, messages, model="gpt-4o", **kwargs):
        with tracer.start_as_current_span("model.completion") as span:
            span.set_attribute("ai.model", model)
            span.set_attribute("ai.message_count", len(messages))

            start = time.perf_counter()
            request_counter.add(1, {"model": model})

            try:
                response = self.client.inference.get_chat_completions(
                    model=model, messages=messages, **kwargs
                )

                # Record latency
                latency = (time.perf_counter() - start) * 1000
                latency_hist.record(latency, {"model": model})

                # Record token usage
                usage = response.usage
                token_counter.add(
                    usage.prompt_tokens + usage.completion_tokens,
                    {"model": model, "type": "total"}
                )

                span.set_attribute("ai.latency_ms", latency)
                span.set_attribute("ai.tokens.prompt", usage.prompt_tokens)
                span.set_attribute("ai.tokens.completion", usage.completion_tokens)

                return response

            except Exception as e:
                error_counter.add(1, {"model": model, "error": type(e).__name__})
                span.set_status(trace.StatusCode.ERROR, str(e))
                raise

Detecting Model Drift & Quality Degradation

Model drift occurs when the statistical properties of input data or the relationship between inputs and outputs changes over time, causing model performance to degrade. In the context of generative AI, drift can manifest as:

  • Data drift: User prompts shift to topics or phrasing the model wasn't optimized for
  • Concept drift: The "correct" answer changes (e.g., regulations update, product info changes)
  • Quality drift: Model outputs gradually become less relevant, accurate, or grounded
  • Behavioral drift: Model updates from the provider change response patterns

Implementing Drift Detection with Azure AI Foundry Evaluations

from azure.ai.projects import AIProjectClient
from azure.ai.evaluation import (
    GroundednessEvaluator,
    RelevanceEvaluator,
    CoherenceEvaluator,
    FluencyEvaluator,
)
from azure.identity import DefaultAzureCredential
import json, statistics

# Initialize the project client
project = AIProjectClient(
    credential=DefaultAzureCredential(),
    project_name="my-ai-project",
    resource_group_name="my-rg",
    subscription_id="<sub-id>"
)

# Define baseline thresholds from initial evaluation
BASELINES = {
    "groundedness": 0.80,
    "relevance":     0.75,
    "coherence":     0.80,
    "fluency":       0.85,
}

# Drift detection threshold (percentage drop from baseline)
DRIFT_THRESHOLD = 0.10  # 10% drop triggers alert

def evaluate_sample_batch(samples: list[dict]) -> dict:
    """Evaluate a batch of recent production samples for drift."""
    evaluators = {
        "groundedness": GroundednessEvaluator(project),
        "relevance":     RelevanceEvaluator(project),
        "coherence":     CoherenceEvaluator(project),
        "fluency":       FluencyEvaluator(project),
    }

    results = {name: [] for name in evaluators}

    for sample in samples:
        for name, evaluator in evaluators.items():
            score = evaluator(
                query=sample["query"],
                response=sample["response"],
                context=sample.get("context", "")
            )
            results[name].append(score.score)

    # Compute averages and detect drift
    drift_report = {}
    for metric, scores in results.items():
        avg_score = statistics.mean(scores)
        baseline = BASELINES[metric]
        drift_pct = (baseline - avg_score) / baseline

        drift_report[metric] = {
            "current_score": round(avg_score, 3),
            "baseline": baseline,
            "drift_percentage": round(drift_pct * 100, 1),
            "drifted": drift_pct > DRIFT_THRESHOLD,
        }

    return drift_report

Alerting & Dashboards

Effective alerting transforms raw metrics into actionable signals. The goal is to catch problems early while minimizing alert fatigue.

Alert Strategy for AI Workloads

🔔 Recommended Alert Rules
Alert Condition Severity Action
High Latency P95 > 5s for 5 min Sev 2 Page on-call, check capacity
Error Spike Error rate > 5% for 3 min Sev 1 Page on-call, check model health
Token Budget Daily tokens > 80% of budget Sev 3 Notify team, review usage
Drift Detected Quality score < baseline - 10% Sev 2 Trigger evaluation pipeline
Rate Limiting 429 count > 10 in 5 min Sev 2 Scale up, check retry logic
Safety Trigger Content filter rate > 5% Sev 1 Review inputs, check for abuse

Creating Alert Rules with Azure CLI

# Create a metric alert for high latency
az monitor metrics alert create \
  --name "ai-high-latency" \
  --resource-group "my-rg" \
  --scopes "/subscriptions/{sub}/resourceGroups/{rg}/providers/Microsoft.MachineLearningServices/workspaces/{project}" \
  --condition "avg RequestLatency > 5000" \
  --window-size "5m" \
  --evaluation-frequency "1m" \
  --severity 2 \
  --action "/subscriptions/{sub}/resourceGroups/{rg}/providers/Microsoft.Insights/actionGroups/ai-ops-team" \
  --description "AI model P95 latency exceeds 5 seconds"

# Create a log-based alert for error spikes
az monitor scheduled-query create \
  --name "ai-error-spike" \
  --resource-group "my-rg" \
  --scopes "/subscriptions/{sub}/resourceGroups/{rg}/providers/Microsoft.OperationalInsights/workspaces/{workspace}" \
  --condition "count > 50" \
  --condition-query "AzureDiagnostics
    | where Category == 'AmlModelsEvent'
    | where ResultType != 'Success'
    | where TimeGenerated > ago(5m)
    | summarize ErrorCount = count() by bin(TimeGenerated, 5m)" \
  --severity 1 \
  --action-groups "/subscriptions/{sub}/resourceGroups/{rg}/providers/Microsoft.Insights/actionGroups/ai-ops-team"

Application Insights for AI Agents

Querying Agent Traces in Application Insights

// Find slowest agent invocations in the last hour
dependencies
| where timestamp > ago(1h)
| where type == "AI Agent"
| summarize
    AvgDuration = avg(duration),
    P95Duration = percentile(duration, 95),
    FailureRate = countif(success == false) * 100.0 / count()
    by target
| order by P95Duration desc

// Trace a specific agent invocation end-to-end
union requests, dependencies, customEvents, traces
| where operation_Id == "abc123"
| project timestamp, itemType, name, duration, success,
         customDimensions
| order by timestamp asc

Best Practices & Troubleshooting

Observability Best Practices

  • Start with the four golden signals: latency, traffic, errors, and saturation
  • Use structured logging: Emit JSON logs with consistent fields like model, trace_id, token_count
  • Sample wisely: Log 100% of errors, but sample successful requests (e.g., 10%) to manage costs
  • Establish baselines early: Capture quality and performance baselines before going to production
  • Separate model metrics from infra metrics: Track model quality separately from compute health
  • Set up anomaly detection: Use Azure Monitor's dynamic thresholds to catch unusual patterns
  • Tag everything: Include deployment version, model version, and environment in all telemetry
  • Automate drift detection: Run scheduled evaluations, don't rely on manual checks

Common Troubleshooting Scenarios

🔧 Troubleshooting Guide

Symptom: Latency suddenly increased

  1. Check token usage — longer prompts cause higher latency
  2. Check rate limiting — 429 responses indicate throttling
  3. Check region health via Azure Status page
  4. Review recent prompt changes that may have increased complexity

Symptom: Quality scores dropping

  1. Analyze input distribution — are users asking new types of questions?
  2. Check if the knowledge base is stale or missing recent information
  3. Verify if a model version update changed behavior
  4. Review system prompt — recent changes may have degraded quality

Symptom: High error rates

  1. Check content filter blocks — review triggered categories
  2. Inspect token limits — requests exceeding context window
  3. Verify authentication — expired keys or managed identity issues
  4. Check endpoint health — model deployment status
