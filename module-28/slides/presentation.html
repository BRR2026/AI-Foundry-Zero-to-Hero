⚠️ The Silent Failure Problem

Traditional Software

  • Fails with clear error codes (500, timeout)
  • Same input → same output (deterministic)
  • Standard monitoring catches failures

AI Systems

  • Returns plausible but wrong answers
  • Same input → different outputs (stochastic)
  • Quality degrades silently

💀 Without AI-specific observability, you won't know your model is degrading until users complain — or leave.

📊 Four Pillars of AI Observability

📈
Metrics
Latency, throughput, tokens, cost-per-request
🔗
Traces
End-to-end request flow, agent steps, tool calls
📝
Logs
Structured events, guardrail triggers, prompt pairs
🎯
Evaluations
Quality scores, drift indicators, safety checks

Evaluations are unique to AI — traditional monitoring doesn't have this pillar.

🔍 KQL for AI Telemetry

Latency Percentiles

AzureDiagnostics
| where Category == "AmlModelsEvent"
| where TimeGenerated > ago(24h)
| summarize
    P50=percentile(DurationMs, 50),
    P95=percentile(DurationMs, 95),
    P99=percentile(DurationMs, 99)
  by bin(TimeGenerated, 1h)
| render timechart

Error Rate by Hour

AzureDiagnostics
| where Category == "AmlModelsEvent"
| summarize
    ErrorRate = round(
      countif(ResultType != "Success") * 100.0 / count(), 2)
  by bin(TimeGenerated, 1h)
| render timechart

🔭 OpenTelemetry Instrumentation

from azure.monitor.opentelemetry import configure_azure_monitor
from opentelemetry import trace, metrics

configure_azure_monitor(
    connection_string="InstrumentationKey=<key>;..."
)

tracer = trace.get_tracer("ai-foundry-app")
meter  = metrics.get_meter("ai-foundry-app")

# Custom AI metrics
token_counter = meter.create_counter("ai.tokens.total")
latency_hist  = meter.create_histogram("ai.request.duration",
                                        unit="ms")

Key packages: azure-monitor-opentelemetry, opentelemetry-api, opentelemetry-sdk

🌊 Types of Model Drift

📊
Data Drift
User prompts shift to new topics or phrasing not seen during optimization
🔄
Concept Drift
The "correct" answer changes — regulations update, products change
📉
Quality Drift
Outputs gradually become less relevant, accurate, or grounded
🔧
Behavioral Drift
Provider-side model updates silently change response patterns

🎯 Key Takeaways

  • AI systems fail silently — you need AI-specific observability
  • Track the four pillars: metrics, traces, logs, and evaluations
  • Azure Monitor + Application Insights = unified AI observability
  • Drift is inevitable — automate detection, don't rely on manual checks
  • Every alert needs a runbook — actionable, not just noise
  • Establish baselines before production — you can't detect drift without them
