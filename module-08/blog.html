1. Beyond Text — The Multimodal Era

Until recently, deploying an AI model meant spinning up a text-in-text-out endpoint. You sent a prompt, you got words back. That was powerful — but it was also limiting. The real world is not made of text alone; it's a rich tapestry of images, sounds, documents, and videos. Customers send photos of defective products. Doctors share X-rays. Field engineers snap images of damaged equipment.

With GPT-4o and the broader model catalog in Azure AI Foundry, you now have the ability to deploy a single model that understands text, images, and audio natively — a truly multimodal AI endpoint. This module dives deep into how to deploy these models, how to send multimodal payloads, and how to compare different models head-to-head so you can make data-driven decisions about which model best fits your workload.

  • How to deploy GPT-4o for multimodal use (text + vision + audio)

  • How to benchmark and A/B test models for quality, latency, and cost

By the end of this module, you will have deployed your own multimodal model and built a Mini Hack application that analyzes images using GPT-4o's vision capabilities.

3. Understanding Deployment Types

Azure AI Foundry provides three distinct deployment strategies. Each serves different scenarios based on latency, throughput, cost, and control requirements. Choosing the right one is the first critical decision you'll make when operationalizing a model.

🖥️ Managed Compute

What: A dedicated VM-backed endpoint that runs your model exclusively.

When: You need full control over infrastructure, want to deploy custom or fine-tuned models, or have compliance requirements for dedicated compute.

Key traits:

  • Dedicated VM (no shared resources)
  • You choose the SKU
  • Auto-scaling available
  • Pay for compute time

☁️ Serverless API

What: A pay-per-token endpoint with no infrastructure to manage.

When: You want to get started quickly, your traffic is unpredictable, or you're experimenting with different models.

Key traits:

  • No VM management
  • Pay per token consumed
  • Instant provisioning
  • Built-in rate limits

⚡ Provisioned Throughput

What: Reserved capacity measured in PTUs (Provisioned Throughput Units).

When: You have predictable, high-volume workloads and need guaranteed latency and throughput.

Key traits:

  • Guaranteed tokens/minute
  • Lowest and most consistent latency
  • Monthly commitment
  • Best unit economics at scale

Comparison Table

Factor Managed Compute Serverless API Provisioned Throughput
Provisioning Speed Minutes (VM spin-up) Seconds Minutes
Billing Model Per-hour compute Per-token Monthly PTU reservation
Latency Low (dedicated) Variable (shared) Lowest & guaranteed
Scaling Auto-scale policies Automatic Fixed capacity
Custom Models ✅ Full support ❌ Catalog only ✅ Select models
Best For Custom/fine-tuned models, compliance Experimentation, variable traffic Production at scale

4. Step-by-Step: Deploying a Model in AI Foundry

Let's walk through deploying GPT-4o as a serverless API endpoint — the fastest way to get a multimodal model running.

  1. Open AI Foundry Portal
    Navigate to ai.azure.com and sign in with your Azure credentials. Select or create a project.
  2. Navigate to the Model Catalog
    In the left sidebar, click Model catalog. Use the search bar or filters to find GPT-4o. Click on the model card.
  3. Click "Deploy"
    On the model detail page, click the Deploy button. Select Serverless API as the deployment type.
  4. Configure the Deployment
    Provide a deployment name (e.g., gpt-4o-multimodal). Set the API version and any content filters. Review the pricing information.
  5. Deploy and Get Your Endpoint
    Click Deploy. After a few seconds, you'll see your endpoint URL and API key. Copy both — you'll need them to send requests.
  6. Test in the Playground
    Navigate to the Playground in AI Foundry. Select your new deployment. Send a test message to verify it's working.
✅ Checkpoint

At this point, you should have a running GPT-4o endpoint. Try sending it a simple text prompt in the Playground to confirm it responds correctly before moving on to multimodal payloads.

5. Multimodal Capabilities with GPT-4o

GPT-4o is a natively multimodal model, meaning it was trained from the ground up to understand text, images, and audio within a single architecture. This is fundamentally different from systems that pipeline separate models together — GPT-4o processes all modalities in a unified latent space, yielding better cross-modal understanding and faster responses.

What Makes GPT-4o Multimodal?

  • Text — Standard chat and completion, instruction following, reasoning, and code generation
  • Vision — Analyze images, read text in photos (OCR), describe scenes, interpret charts and diagrams
  • Audio — Process spoken language, understand tone and context (via paired Whisper/TTS models)

The Unified API Pattern

All modalities flow through the same Chat Completions API. You use the messages array with content blocks that specify the type — text, image_url, or input_audio:

POST {endpoint}/openai/deployments/{deployment}/chat/completions?api-version=2024-12-01-preview

{
  "messages": [
    {
      "role": "user",
      "content": [
        { "type": "text", "text": "What do you see in this image?" },
        {
          "type": "image_url",
          "image_url": {
            "url": "https://example.com/photo.jpg",
            "detail": "high"
          }
        }
      ]
    }
  ],
  "max_tokens": 1000
}
💡 The detail Parameter

Set "detail": "high" for complex images (diagrams, small text). Use "detail": "low" for simple images or when you want to reduce token consumption. The "auto" setting lets the model decide.

6. Working with Images

There are two primary ways to send images to GPT-4o: via a publicly accessible URL or as base64-encoded data embedded directly in the request.

Method 1: URL Reference

The simplest approach — pass a URL that the model can fetch at inference time:

{
  "type": "image_url",
  "image_url": {
    "url": "https://storage.blob.core.windows.net/images/product-defect.jpg"
  }
}

Pros: Small request size, easy to implement.
Cons: URL must be publicly accessible (or SAS-token protected). Adds network latency if the image is large.

Method 2: Base64 Encoding

Embed the image directly in the JSON payload as a base64-encoded data URI:

import base64

with open("product-defect.jpg", "rb") as f:
    image_data = base64.b64encode(f.read()).decode("utf-8")

message = {
    "type": "image_url",
    "image_url": {
        "url": f"data:image/jpeg;base64,{image_data}"
    }
}

Pros: Works with private images, no external URL needed.
Cons: Increases request payload size significantly (~33% larger than binary).

Image Use Cases

Use Case Description Recommended Detail
Product Defect Detection Identify scratches, dents, or anomalies in product images high
Receipt / Invoice OCR Extract line items, totals, and vendor info high
Chart Interpretation Summarize trends from bar charts, line graphs, pie charts high
Scene Description Describe what's happening in a photograph low or auto
Thumbnail Classification Classify small images into categories low

Multiple Images in One Request

GPT-4o can process multiple images in a single turn. Include several image_url content blocks in the content array to compare images, detect differences, or analyze a batch:

{
  "messages": [
    {
      "role": "user",
      "content": [
        { "type": "text", "text": "Compare these two product images. Are there any defects in the second one?" },
        { "type": "image_url", "image_url": { "url": "https://example.com/reference.jpg" } },
        { "type": "image_url", "image_url": { "url": "https://example.com/sample.jpg" } }
      ]
    }
  ]
}

7. Working with Audio & Speech

While GPT-4o handles text and vision natively in a single call, audio workflows in AI Foundry typically use specialized models deployed alongside your primary model.

Speech-to-Text with Whisper

Deploy the Whisper model from the AI Foundry Model Catalog for transcription and translation:

import requests

url = f"{endpoint}/openai/deployments/whisper-1/audio/transcriptions?api-version=2024-12-01-preview"

headers = { "api-key": api_key }

files = {
    "file": ("meeting.mp3", open("meeting.mp3", "rb")),
}

data = {
    "response_format": "verbose_json",
    "language": "en",
    "timestamp_granularities": "word"
}

response = requests.post(url, headers=headers, files=files, data=data)
transcript = response.json()

Text-to-Speech (TTS)

Use the TTS model to generate natural-sounding speech from text:

url = f"{endpoint}/openai/deployments/tts-1/audio/speech?api-version=2024-12-01-preview"

payload = {
    "model": "tts-1",
    "input": "Your insurance claim has been approved. The total reimbursement is $1,250.",
    "voice": "nova"
}

response = requests.post(url, headers={"api-key": api_key, "Content-Type": "application/json"}, json=payload)

with open("output.mp3", "wb") as f:
    f.write(response.content)
ℹ️ Audio Pipeline Pattern

A common pattern is to chain Whisper → GPT-4o → TTS to build voice-driven AI assistants. Audio comes in via Whisper, GPT-4o reasons over the transcript (plus optional images or documents), and TTS speaks the response back.

8. Document Understanding

GPT-4o's vision capabilities extend beyond photographs — it excels at understanding structured documents like PDFs rendered as images, scanned forms, and multi-page reports.

Processing PDFs as Images

Convert each page of a PDF to an image, then send the images to GPT-4o:

from pdf2image import convert_from_path
import base64, io

pages = convert_from_path("contract.pdf", dpi=200)

image_messages = []
for page in pages:
    buffer = io.BytesIO()
    page.save(buffer, format="PNG")
    b64 = base64.b64encode(buffer.getvalue()).decode("utf-8")
    image_messages.append({
        "type": "image_url",
        "image_url": { "url": f"data:image/png;base64,{b64}", "detail": "high" }
    })

messages = [{
    "role": "user",
    "content": [
        { "type": "text", "text": "Extract key terms, dates, and obligations from this contract." },
        *image_messages
    ]
}]

Form Data Extraction

For structured form extraction (insurance forms, tax documents, invoices), combine GPT-4o vision with a structured output schema to get consistent JSON responses:

{
  "messages": [...],
  "response_format": {
    "type": "json_schema",
    "json_schema": {
      "name": "invoice_data",
      "schema": {
        "type": "object",
        "properties": {
          "vendor_name": { "type": "string" },
          "invoice_number": { "type": "string" },
          "date": { "type": "string" },
          "line_items": {
            "type": "array",
            "items": {
              "type": "object",
              "properties": {
                "description": { "type": "string" },
                "quantity": { "type": "number" },
                "unit_price": { "type": "number" }
              }
            }
          },
          "total": { "type": "number" }
        }
      }
    }
  }
}
💡 Tip: Combine with Azure AI Document Intelligence

For high-volume document processing with pre-built extraction models, consider pairing GPT-4o with Azure AI Document Intelligence. Use Document Intelligence for reliable field extraction and GPT-4o for reasoning, summarization, and answering questions about the content.

9. Model Benchmarking & A/B Testing

Deploying a model is only half the job. You need to verify that the model you chose delivers the best results for your specific use case. AI Foundry provides tools for model evaluation and comparison.

Why Benchmark?

  • Quality — Does Model A produce more accurate, relevant, complete answers than Model B?
  • Latency — How fast is the first token? What's the end-to-end response time?
  • Cost — What's the per-request cost at your expected token usage?
  • Safety — Does the model produce harmful or off-topic content with your prompts?

Benchmarking Approach

  1. Build an evaluation dataset — Curate 50-200 representative queries with expected outputs or quality criteria
  2. Deploy candidate models — Deploy 2-3 models (e.g., GPT-4o, GPT-4o-mini, Llama 3) as serverless endpoints
  3. Run batch evaluations — Use AI Foundry's evaluation framework to score each model's responses
  4. Compare metrics — Review scores for groundedness, relevance, coherence, fluency, and safety
  5. Measure latency & cost — Record response times and token consumption per model

Using AI Foundry Evaluation

from azure.ai.evaluation import evaluate
from azure.ai.evaluation.metrics import (
    GroundednessEvaluator,
    RelevanceEvaluator,
    CoherenceEvaluator,
    FluencyEvaluator
)

# Define evaluators
evaluators = {
    "groundedness": GroundednessEvaluator(model_config),
    "relevance": RelevanceEvaluator(model_config),
    "coherence": CoherenceEvaluator(model_config),
    "fluency": FluencyEvaluator(model_config),
}

# Run evaluation on your dataset
results = evaluate(
    data="eval-dataset.jsonl",
    evaluators=evaluators,
    output_path="eval-results/"
)

A/B Testing in Production

For live A/B testing, deploy two models behind a traffic-splitting layer. Route a percentage of requests to each model and track:

Metric How to Measure Tool
User satisfaction Thumbs up/down feedback App UI + logging
Task completion rate Did the user accomplish their goal? Application analytics
Response latency P50, P95, P99 response times Azure Monitor
Token consumption Avg tokens per request AI Foundry metrics
Error rate HTTP 4xx/5xx responses Azure Monitor
✅ Best Practice

Always benchmark with your data, not just public benchmarks. A model that scores highest on MMLU may not be the best for your specific domain. Run evaluations on a curated dataset that represents your actual workload.

10. Cost Considerations for Different Deployment Types

Understanding cost is critical when choosing a deployment type. Here's how costs compare across the three deployment strategies:

Serverless API (Pay-Per-Token)

  • Best for: Low to moderate traffic, experimentation, prototyping
  • You pay for input tokens + output tokens consumed
  • No idle cost — if nobody calls your endpoint, you pay nothing
  • Image tokens are calculated based on image size and detail level

Managed Compute (Pay-Per-Hour)

  • Best for: Custom models, compliance-heavy workloads, steady traffic
  • You pay for VM uptime regardless of usage
  • More cost-effective than serverless at ~60-70% utilization
  • Consider auto-scale-to-zero if supported for your SKU

Provisioned Throughput (PTU Reservation)

  • Best for: High-volume production workloads with predictable demand
  • Monthly commitment gives you reserved throughput capacity
  • Most cost-effective at scale — typically 30-50% cheaper than pay-per-token at high volume
  • Use the AI Foundry pricing calculator to estimate PTU requirements

Cost Calculation Example

# Estimating monthly cost for a serverless GPT-4o deployment
# Scenario: Customer support chatbot, 100K conversations/month

avg_input_tokens = 500
avg_output_tokens = 300
conversations_per_month = 100_000

input_cost_per_1k = 0.005    # $/1K input tokens (example rate)
output_cost_per_1k = 0.015   # $/1K output tokens (example rate)

monthly_input_cost = (avg_input_tokens * conversations_per_month / 1000) * input_cost_per_1k
monthly_output_cost = (avg_output_tokens * conversations_per_month / 1000) * output_cost_per_1k

total_monthly = monthly_input_cost + monthly_output_cost
print(f"Estimated monthly cost: ${total_monthly:,.2f}")
# Output: Estimated monthly cost: $700.00
⚠️ Vision Token Costs

Images consume additional tokens based on their resolution and the detail setting. A single high-detail image can consume 1,000+ tokens. Always factor image token costs into your estimates for multimodal workloads.

11. 🧪 Mini Hack — Build an Image Analysis App

🛠️ Mini Hack Challenge

Build a Python application that accepts an image (URL or file path), sends it to GPT-4o's vision endpoint, and returns a structured analysis including: description, detected objects, any text found (OCR), and a sentiment assessment.

Requirements

  1. Accept an image from the user (URL or local file)
  2. If local file, encode as base64
  3. Send to your deployed GPT-4o endpoint with a structured analysis prompt
  4. Return a JSON response with: description, objects, extracted_text, sentiment
  5. Handle errors gracefully (invalid image, API errors, etc.)

Starter Code

import os, base64, sys, json
from openai import AzureOpenAI

client = AzureOpenAI(
    azure_endpoint=os.environ["AZURE_OPENAI_ENDPOINT"],
    api_key=os.environ["AZURE_OPENAI_API_KEY"],
    api_version="2024-12-01-preview"
)

DEPLOYMENT = "gpt-4o-multimodal"

ANALYSIS_PROMPT = """Analyze this image and return a JSON object with:
- "description": A 2-3 sentence description of the image
- "objects": An array of objects/items detected in the image
- "extracted_text": Any text visible in the image (empty string if none)
- "sentiment": The overall mood/sentiment of the image (positive/neutral/negative)
Return ONLY valid JSON, no markdown."""


def encode_local_image(path: str) -> str:
    """Read a local image file and return base64 data URI."""
    ext = path.rsplit(".", 1)[-1].lower()
    mime = {"jpg": "jpeg", "jpeg": "jpeg", "png": "png", "gif": "gif", "webp": "webp"}.get(ext, "jpeg")
    with open(path, "rb") as f:
        b64 = base64.b64encode(f.read()).decode("utf-8")
    return f"data:image/{mime};base64,{b64}"


def analyze_image(image_source: str) -> dict:
    """Send an image to GPT-4o and return structured analysis."""
    if image_source.startswith(("http://", "https://")):
        image_url = image_source
    else:
        image_url = encode_local_image(image_source)

    response = client.chat.completions.create(
        model=DEPLOYMENT,
        messages=[{
            "role": "user",
            "content": [
                { "type": "text", "text": ANALYSIS_PROMPT },
                { "type": "image_url", "image_url": { "url": image_url, "detail": "high" } }
            ]
        }],
        max_tokens=1000,
        temperature=0.2
    )

    return json.loads(response.choices[0].message.content)


if __name__ == "__main__":
    source = sys.argv[1] if len(sys.argv) > 1 else input("Image URL or path: ")
    try:
        result = analyze_image(source)
        print(json.dumps(result, indent=2))
    except Exception as e:
        print(f"Error: {e}", file=sys.stderr)

Stretch Goals

  • Add a Streamlit or Gradio web UI for drag-and-drop image upload
  • Support multiple images in a single analysis
  • Compare GPT-4o results against GPT-4o-mini for the same image
  • Add cost tracking — calculate and display the token cost of each analysis
✅ Success Criteria

Your Mini Hack is complete when you can provide a local image or URL, receive a valid JSON response with all four fields (description, objects, extracted_text, sentiment), and handle at least one error case (e.g., invalid file path).

12. Key Takeaways

  1. Three deployment types — Managed Compute for control, Serverless API for agility, Provisioned Throughput for guaranteed performance at scale.
  2. GPT-4o is natively multimodal — text, vision, and audio in one model, one API, one call.
  3. Images can be sent via URL or base64 — choose based on your security and latency requirements.
  4. Audio workflows use specialized Whisper and TTS deployments that chain with GPT-4o for end-to-end voice AI.
  5. Document understanding — render PDFs as images and leverage GPT-4o's vision for extraction and reasoning.
  6. Always benchmark with your own data — public leaderboards don't reflect your workload; use AI Foundry's evaluation framework.
  7. Cost varies significantly by deployment type — model your expected traffic to choose the most cost-effective strategy.
  8. A/B testing in production gives you real-world signal that offline benchmarks can't provide.
