Overview

AI agents are powerful but inherently unpredictable. Unlike deterministic code, agents make autonomous decisions about tool usage, reasoning paths, and response generation. Without proper evaluation and observability, failures become invisible, quality degrades silently, and debugging is nearly impossible.

In this module, you'll learn to build a comprehensive evaluation and observability stack for your Azure AI Foundry agents — from measuring task completion accuracy to tracing every LLM call, tool invocation, and reasoning step through Application Insights and OpenTelemetry.

Why Observability Matters for Agents

Traditional software observability focuses on request/response cycles. AI agents introduce entirely new challenges because each "request" can spawn dozens of internal steps — tool calls, model invocations, retries, and branching decisions — all hidden from the caller.

🔍

Visibility

See every LLM call, tool invocation, and reasoning step in a single trace.

📊

Quality Metrics

Measure task completion rate, accuracy, latency, and cost per interaction.

🐛

Root Cause Analysis

Pinpoint exactly where and why an agent produced incorrect results.

⚡

Performance

Identify bottlenecks in tool calls, token usage, and round-trip latency.

The Agent Observability Gap

Without dedicated observability tooling, you face these critical blind spots:

  • Silent failures — the agent returns a plausible-sounding but incorrect answer
  • Tool call loops — the agent calls the same tool repeatedly without progress
  • Token budget exhaustion — complex reasoning paths consume excessive tokens
  • Hallucinated actions — the agent fabricates tool outputs or skips required steps
  • Latency spikes — some conversations take 10× longer without visible cause

Evaluating Agent Quality

Agent evaluation goes beyond simple accuracy metrics. You need to assess whether the agent completes tasks correctly, follows instructions, uses tools appropriately, and handles edge cases gracefully.

Evaluation Dimensions

Dimension What It Measures Example Metric
Task Completion Did the agent achieve the user's goal? Completion rate (%)
Accuracy Were the agent's outputs factually correct? Groundedness score (0–5)
Tool Usage Did the agent call the right tools with correct parameters? Tool precision/recall
Efficiency Did the agent use resources wisely? Tokens per task, latency
Safety Did the agent stay within safety guardrails? Violation rate
Instruction Adherence Did the agent follow its configured persona and rules? Adherence score (0–5)

Built-in Evaluation with Azure AI Foundry

Azure AI Foundry provides built-in evaluators for agent quality assessment. Use the azure-ai-evaluation SDK to run automated evaluations against ground-truth datasets.

Python
from azure.ai.evaluation import AgentEvaluator
from azure.identity import DefaultAzureCredential

# Initialize the evaluator
evaluator = AgentEvaluator(
    azure_ai_project={
        "subscription_id": "<subscription-id>",
        "resource_group_name": "<resource-group>",
        "project_name": "<project-name>",
    },
    credential=DefaultAzureCredential()
)

# Define evaluation dataset
eval_dataset = [
    {
        "query": "What is the current stock price of MSFT?",
        "expected_tool_calls": ["bing_search"],
        "ground_truth": "The current stock price should be retrieved from a live source."
    },
    {
        "query": "Summarize Q3 earnings from the uploaded PDF.",
        "expected_tool_calls": ["file_search"],
        "ground_truth": "Revenue was $56.5B, up 13% YoY."
    }
]

# Run evaluation
results = evaluator.evaluate(
    data=eval_dataset,
    evaluators={
        "task_completion": "task_completion",
        "groundedness": "groundedness",
        "tool_call_accuracy": "tool_call_accuracy",
    }
)

print(f"Task Completion: {results.metrics['task_completion']:.2%}")
print(f"Groundedness:    {results.metrics['groundedness']:.2f}/5")
print(f"Tool Accuracy:   {results.metrics['tool_call_accuracy']:.2%}")

Creating Evaluation Datasets

A high-quality evaluation dataset should cover:

  • Happy-path scenarios — common tasks the agent handles daily
  • Edge cases — ambiguous queries, missing data, conflicting instructions
  • Multi-step tasks — workflows requiring multiple tool calls in sequence
  • Adversarial inputs — prompt injection attempts, off-topic requests
  • Boundary conditions — very long inputs, empty inputs, special characters
Pro Tip: Dataset from Traces

Use production traces to build evaluation datasets. Export real user interactions from Application Insights, add ground-truth annotations, and create a living test suite that evolves with your agent's usage patterns.

Application Insights Integration

Application Insights is the primary telemetry backend for Azure AI Foundry agents. When connected to your AI Foundry project, it automatically captures traces for every agent run, including LLM calls, tool invocations, and token usage.

Connecting App Insights to Your Project

  1. Create an Application Insights Resource

    If you don't have one, create an Application Insights resource in the Azure portal or use the Azure CLI:

    Azure CLI
az monitor app-insights component create \
  --app my-agent-insights \
  --location eastus2 \
  --resource-group my-rg \
  --kind web \
  --application-type web

  2. Link to Your AI Foundry Project

    Connect the Application Insights resource to your Azure AI Foundry project through the project settings in the Azure portal or programmatically.

  3. Enable Tracing in Your Agent Code

    Configure the Azure AI Inference SDK to export telemetry:

    Python
from azure.ai.projects import AIProjectClient
from azure.identity import DefaultAzureCredential
from azure.monitor.opentelemetry import configure_azure_monitor

# Enable Azure Monitor / App Insights export
configure_azure_monitor(
    connection_string="InstrumentationKey=<key>;..."
)

# Create project client — traces are now auto-collected
client = AIProjectClient(
    credential=DefaultAzureCredential(),
    endpoint="https://<project>.services.ai.azure.com",
)

# Enable additional instrumentation for agent internals
client.telemetry.enable()

  4. Verify Traces in the Portal

    Open your Application Insights resource → Transaction search to view captured traces. Each agent run appears as a distributed trace with child spans for tool calls and LLM invocations.

What Gets Traced Automatically

Span Type Information Captured Attributes
Agent Run Top-level agent execution agent.id, thread.id, run.id, status, duration
LLM Call Each model invocation model, prompt_tokens, completion_tokens, finish_reason
Tool Call Each tool invocation tool.name, tool.input, tool.output, duration
Token Usage Token consumption per call prompt_tokens, completion_tokens, total_tokens
Exceptions Errors and failures exception.type, exception.message, stack_trace

OpenTelemetry Semantic Conventions

Azure AI Foundry follows the OpenTelemetry Semantic Conventions for Generative AI. Understanding these conventions lets you query traces effectively and build custom dashboards.

Key Semantic Attributes

Attributes
# GenAI Span Attributes (OpenTelemetry Semantic Conventions)

# Model-level attributes
gen_ai.system          = "az.ai.agents"
gen_ai.request.model   = "gpt-4o"
gen_ai.response.model  = "gpt-4o-2024-08-06"

# Token usage attributes
gen_ai.usage.input_tokens    = 1250
gen_ai.usage.output_tokens   = 340

# Operation-level attributes
gen_ai.operation.name  = "chat"
gen_ai.request.temperature = 0.7
gen_ai.request.top_p       = 0.95
gen_ai.response.finish_reasons = ["stop"]

# Agent-specific attributes
gen_ai.agent.id     = "asst_abc123"
gen_ai.thread.id    = "thread_xyz789"
gen_ai.thread.run.id = "run_def456"

Custom Span Enrichment

Add your own attributes to traces for business-specific observability:

Python
from opentelemetry import trace

tracer = trace.get_tracer("my-agent-app")

with tracer.start_as_current_span("process_customer_request") as span:
    # Add business context to the trace
    span.set_attribute("customer.id", customer_id)
    span.set_attribute("request.type", "refund_inquiry")
    span.set_attribute("request.priority", "high")

    # Run the agent — child spans auto-captured
    response = agent_client.run(
        agent_id=agent_id,
        thread_id=thread_id
    )

    # Record outcome
    span.set_attribute("agent.task_completed", True)
    span.set_attribute("agent.tools_used", 3)
    span.set_attribute("agent.total_tokens", response.usage.total_tokens)

Debugging Agent Failures

When an agent fails or produces incorrect results, a systematic debugging approach is essential. Use traces as your primary investigation tool.

Common Failure Patterns

Failure Pattern Symptoms Root Cause Fix
Tool Call Loop Agent calls same tool 5+ times, high latency Ambiguous instructions or tool output not actionable Clarify instructions, improve tool error messages
Wrong Tool Selected Agent uses File Search instead of Bing Search Tool descriptions too similar or vague Make tool descriptions distinct and specific
Hallucinated Output Response contains fabricated data Tool output was empty or agent skipped tool call Add guardrails requiring tool verification
Token Exhaustion Response truncated or incomplete Thread history too long, excessive tool outputs Implement conversation summarization, truncate tool outputs
Timeout Agent run times out after max duration External tool call hangs or model is slow Add timeout configs, implement circuit breakers

Step-by-Step Debugging Workflow

  1. Identify the Failed Run

    Find the specific run ID from your logs or user report. Query Application Insights:

    KQL
// Find failed agent runs in the last 24 hours
dependencies
| where timestamp > ago(24h)
| where type == "az.ai.agents"
| where success == false
| project timestamp, id, name, duration, resultCode
| order by timestamp desc

  2. Inspect the Full Trace

    Open the distributed trace to see the complete execution flow: which tools were called, what the model generated at each step, and where the failure occurred.

    KQL
// Get the complete trace for a specific operation
union dependencies, requests, traces, exceptions
| where operation_Id == "<operation-id>"
| order by timestamp asc
| project timestamp, itemType, name,
         duration, success,
         customDimensions

  3. Analyze Tool Call Sequences

    Check whether the agent called tools in the expected order with correct parameters.

  4. Review Token Usage

    Compare actual token consumption against expected ranges to detect reasoning loops or context overflow.

  5. Test the Fix

    After modifying instructions or tool configurations, replay the same conversation to verify the fix.

Programmatic Debugging

Python
from azure.ai.projects import AIProjectClient
from azure.identity import DefaultAzureCredential

client = AIProjectClient(
    credential=DefaultAzureCredential(),
    endpoint="https://<project>.services.ai.azure.com",
)

# Retrieve run steps to inspect agent's reasoning
run_steps = client.agents.list_run_steps(
    thread_id="thread_xyz789",
    run_id="run_def456"
)

for step in run_steps.data:
    print(f"Step: {step.type} | Status: {step.status}")
    if step.type == "tool_calls":
        for tool_call in step.step_details.tool_calls:
            print(f"  Tool: {tool_call.type}")
            print(f"  Input: {tool_call.function.arguments}")
    elif step.type == "message_creation":
        msg_id = step.step_details.message_creation.message_id
        msg = client.agents.get_message(
            thread_id="thread_xyz789",
            message_id=msg_id
        )
        print(f"  Response: {msg.content[0].text.value[:200]}")

Logging, Observability & Trace Analysis

Beyond debugging individual failures, trace analysis enables you to identify systemic patterns, optimize agent performance, and build dashboards that give stakeholders confidence in your agent's reliability.

Key KQL Queries for Agent Monitoring

Agent Success Rate Over Time

KQL
// Agent success rate by hour
dependencies
| where timestamp > ago(7d)
| where type == "az.ai.agents"
| summarize
    total = count(),
    succeeded = countif(success == true),
    failed = countif(success == false)
    by bin(timestamp, 1h)
| extend success_rate = round(100.0 * succeeded / total, 2)
| render timechart

Token Usage Analysis

KQL
// Average token usage per agent run
dependencies
| where timestamp > ago(7d)
| where type == "az.ai.agents"
| extend input_tokens = toint(customDimensions["gen_ai.usage.input_tokens"])
| extend output_tokens = toint(customDimensions["gen_ai.usage.output_tokens"])
| summarize
    avg_input = avg(input_tokens),
    avg_output = avg(output_tokens),
    p95_total = percentile(input_tokens + output_tokens, 95)
    by bin(timestamp, 1d)
| render timechart

Slowest Agent Runs

KQL
// Top 20 slowest agent runs
dependencies
| where timestamp > ago(24h)
| where type == "az.ai.agents"
| top 20 by duration desc
| project timestamp, operation_Id, duration,
         customDimensions["gen_ai.agent.id"],
         customDimensions["gen_ai.usage.input_tokens"]

Setting Up Alerts

Configure Azure Monitor alerts to catch agent issues before users report them:

  • Success rate drops below 95% in any 1-hour window
  • P95 latency exceeds 30 seconds for agent runs
  • Token usage spikes above 2× the daily average
  • Error rate for a specific tool exceeds 10%
Cost Consideration

Tracing generates telemetry data that is ingested by Application Insights. For high-volume agents, configure sampling to control costs. Use OTEL_TRACES_SAMPLER=parentbased_traceidratio and OTEL_TRACES_SAMPLER_ARG=0.1 to sample 10% of traces in production.

Mini Hack: End-to-End Agent Tracing

In this hands-on exercise, you'll set up complete observability for an agent — from OpenTelemetry instrumentation through Application Insights to a live dashboard.

Challenge

Instrument an existing Azure AI Foundry agent with Application Insights tracing, add custom business-context attributes, run a batch of test conversations, then build a KQL dashboard showing success rates, latency, and token usage.

Implementation

Python
import os
from azure.ai.projects import AIProjectClient
from azure.identity import DefaultAzureCredential
from azure.monitor.opentelemetry import configure_azure_monitor
from opentelemetry import trace

# ── Step 1: Configure Azure Monitor telemetry export ──
configure_azure_monitor(
    connection_string=os.environ["APPLICATIONINSIGHTS_CONNECTION_STRING"]
)

# ── Step 2: Initialize the project client with telemetry ──
client = AIProjectClient(
    credential=DefaultAzureCredential(),
    endpoint=os.environ["PROJECT_ENDPOINT"],
)
client.telemetry.enable()

# ── Step 3: Set up custom tracer ──
tracer = trace.get_tracer("agent-observability-hack")

# ── Step 4: Run agent with full tracing ──
test_queries = [
    "What are the top 3 trending stocks today?",
    "Summarize the attached earnings report.",
    "Calculate the compound interest on $10,000 at 5% for 10 years.",
    "Compare revenue growth between MSFT and AAPL.",
    "What was the S&P 500 performance last quarter?",
]

agent = client.agents.get_agent(agent_id=os.environ["AGENT_ID"])

for i, query in enumerate(test_queries):
    with tracer.start_as_current_span(f"test_query_{i}") as span:
        span.set_attribute("test.query_index", i)
        span.set_attribute("test.query_text", query)

        # Create a fresh thread for each test
        thread = client.agents.create_thread()

        # Send the test query
        client.agents.create_message(
            thread_id=thread.id,
            role="user",
            content=query
        )

        # Execute the agent run
        run = client.agents.create_and_process_run(
            thread_id=thread.id,
            agent_id=agent.id
        )

        # Record results
        span.set_attribute("agent.run.status", run.status)
        span.set_attribute("agent.run.id", run.id)
        if run.usage:
            span.set_attribute("agent.tokens.total", run.usage.total_tokens)

        print(f"Query {i+1}: {run.status} "
              f"(tokens: {run.usage.total_tokens if run.usage else 'N/A'})")

print("\n✅ All test queries complete. Check Application Insights for traces.")

Best Practices

Evaluation Best Practices

  • Automate evaluations in CI/CD — run your evaluation suite on every agent configuration change
  • Use production data — curate evaluation datasets from real user interactions, not just synthetic examples
  • Track metrics over time — establish baselines and detect regressions early
  • Combine AI and human evaluation — use AI evaluators for scale, human reviewers for nuanced quality

Tracing Best Practices

  • Always enable tracing in production — the cost of missed insights far exceeds telemetry costs
  • Use sampling for high-volume agents — 10–25% sampling captures enough data for analysis
  • Add business context — enrich spans with customer ID, request type, and priority
  • Set up alerts from day one — don't wait for the first production incident
  • Implement log correlation — use trace IDs to connect agent traces with your application's logs

Debugging Best Practices

  • Reproduce before fixing — use the thread ID to replay conversations in the playground
  • Check tool outputs first — most agent failures stem from unexpected tool responses
  • Compare fast vs. slow runs — use trace analysis to identify what makes some runs take 10× longer
  • Version your agent instructions — track changes to instructions alongside performance metrics
