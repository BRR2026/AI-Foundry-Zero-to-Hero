Introduction

You've built intelligent agents, grounded them in enterprise knowledge, secured them with private networking, and optimized their costs. Now the question becomes: how do you do this for the next 50 customers without rebuilding from scratch each time?

Module 38 is about the transition from project delivery to product delivery. Whether you're an ISV building an AI-powered SaaS product, a consulting partner accelerating customer engagements, or an enterprise team creating internal platforms, the principles are the same: parameterize, isolate, automate, and monetize.

We'll explore how to package Azure AI Foundry solutions as distributable products, implement white-label and multi-tenant architectures, build solution templates that provision complete AI systems in minutes, and define pricing strategies that ensure healthy margins.

💡 Why This Matters

The AI market is rapidly moving from custom consulting toward productized offerings. Partners and ISVs who can deliver repeatable, scalable AI solutions will capture far more market share than those building one-off projects. Azure AI Foundry provides the building blocks — this module teaches you how to assemble them into a product.

From Projects to Products — The Productization Mindset

Every AI engagement starts as a project. A customer needs a chatbot, a document processor, or an intelligent assistant. You build it, deploy it, and move on. But what happens when the next customer needs something nearly identical?

The Productization Spectrum

Custom Consulting Configurable Template Self-Service SaaS │ │ │ High Touch Medium Touch Low Touch High Margin Good Margin Scale Margin Low Scale Medium Scale High Scale │ │ │ ◄──────────────────────────────────────────────────────────────► Per-customer build Parameterized deploy Customer self-serve

The goal is to move your AI solution rightward on this spectrum — from bespoke to repeatable — without losing the ability to customize for each customer's unique needs.

What Makes an AI Solution "Repeatable"?

  • Configuration over code — Customer differences are captured in config files, not code branches
  • Infrastructure as Code — All Azure resources can be provisioned programmatically
  • Parameterized prompts — System prompts accept customer-specific variables (name, tone, guardrails)
  • Pluggable knowledge — Each customer brings their own data; the pipeline handles any content
  • Automated evaluation — Quality gates run automatically for every deployment
  • Self-service onboarding — New customers can be provisioned without developer intervention
⚠️ Common Pitfall

Don't try to productize too early. Build the solution for 2–3 customers first, observe what actually varies between them, and then extract the parameterization. Premature abstraction leads to over-engineered templates that don't fit real customer needs.

Packaging AI Solutions in Azure AI Foundry

Azure AI Foundry provides several natural packaging points for distributing AI solutions. The right choice depends on your target audience and distribution model.

Packaging Options

📦

Project Template

Pre-configured Foundry project with agents, knowledge connections, evaluation pipelines, and documentation. Best for partners deploying into customer tenants.

🏗️

Bicep / Terraform Module

Infrastructure as Code that provisions all Azure resources and Foundry assets. Best for enterprise IT teams who manage their own deployments.

🐳

Container Image

Dockerized agent with agent.yaml published to Azure Container Registry. Best for ISVs distributing via ACR or Azure Marketplace.

📋

ARM Template Spec

Versioned ARM template stored in Azure with role-based access. Best for managed service providers deploying across multiple customer subscriptions.

Solution Template Structure

Regardless of the packaging format, every repeatable AI offering should follow a consistent structure:

White-Label Patterns — One Solution, Many Brands

A white-label AI offering allows customers to present the solution under their own brand. End users interact with "Contoso Assistant" or "Fabrikam Help Desk" — never knowing the underlying platform or provider.

What to Make Configurable

LayerConfigurable ElementsImplementation
Visual brandingLogo, colors, bot name, avatarsCSS variables + config file
ConversationGreeting, tone, persona nameSystem prompt template
DomainCustom domain or subdomainAzure Front Door / App Service
KnowledgeFAQ documents, product catalogPer-tenant AI Search indexes
ModelGPT-4o vs. GPT-4o-mini, temperatureDeployment parameters
GuardrailsBlocked topics, response limitsSystem prompt + content filters

Customer Configuration File

The entire customer identity is captured in a single YAML file:

YAML
# configs/contoso.yaml tenant_id: "contoso" display_name: "Contoso AI Assistant" branding: primary_color: "#0078D4" logo_url: "https://contoso.com/logo.png" greeting: "Hello! I'm the Contoso assistant. How can I help?" knowledge: index_name: "contoso-knowledge-index" data_source: "contoso-blob-container" model: deployment: "gpt-4o" temperature: 0.3 guardrails: blocked_topics: ["competitor-products", "pricing-negotiation"] max_response_length: 500
💡 Design Principle

A new customer should be onboarded by creating a single config file — not by modifying any code. If you need a code change for a new customer, your template isn't parameterized enough.

Multi-Tenant Architecture for AI

Multi-tenancy determines how you share (or isolate) infrastructure across customers. For AI offerings, the critical isolation points are data (knowledge bases), compute (model inference), and configuration (prompts, guardrails).

Isolation Strategies

StrategyIsolationCostComplexityBest For
Shared everythingLowLowestLowDev/test environments
Shared infra, separate dataMediumMediumMediumMost commercial offerings
Separate Foundry projectsHighHighMediumRegulated industries
Separate subscriptionsMaximumHighestHighGovernment, defense

Recommended Pattern: Shared Infra, Separate Data

For most AI offerings, we recommend the "shared infra, separate data" pattern:

  • Single AI Foundry hub with multiple projects (one per tenant)
  • Shared model deployments with per-tenant rate limiting
  • Separate Azure AI Search indexes per tenant for knowledge isolation
  • Tenant routing layer that directs requests to the correct project and index
  • Centralized monitoring with tenant-level dashboards and cost attribution

Multi-Tenant Request Flow

Customer Request │ ▼ ┌─────────────────┐ │ API Gateway / │ │ Tenant Router │ └───────┬─────────┘ │ (extract tenant ID from header / token) ▼ ┌─────────────────┐ ┌──────────────────┐ │ Config Lookup │───────▶│ Tenant Config DB │ └───────┬─────────┘ └──────────────────┘ │ ▼ ┌─────────────────┐ ┌──────────────────┐ │ AI Foundry │───────▶│ Tenant Project │ │ Agent Endpoint │ └──────────────────┘ └───────┬─────────┘ │ ▼ ┌─────────────────┐ ┌──────────────────┐ │ Knowledge │───────▶│ Tenant Search │ │ Retrieval │ │ Index │ └───────┬─────────┘ └──────────────────┘ │ ▼ Response
🔐 Critical: Data Isolation

Never co-mingle customer data in a shared search index, even with field-level filtering. Use separate indexes per tenant as the minimum isolation boundary. This prevents data leakage, simplifies compliance, and allows per-tenant index management (rebuild, delete, scale).

Solution Templates & Accelerators

A solution accelerator is a production-ready, parameterized starting point that reduces time-to-value from months to days. Unlike a demo or proof-of-concept, an accelerator is designed for production workloads from day one.

Demo vs. Accelerator

AttributeDemo / SampleSolution Accelerator
PurposeShow what's possibleGet to production fast
InfrastructureManual setupFull IaC (Bicep / Terraform)
SecurityNone or basicPrivate endpoints, RBAC, managed identity
MonitoringConsole logsApp Insights, dashboards, alerts
EvaluationAd-hoc testingAutomated eval pipelines with baselines
DocumentationREADME onlyArchitecture diagram, onboarding guide, runbook
Time to deployHours to daysMinutes

Building an Accelerator: Step by Step

  1. Start with a working solution — Build the solution for one real customer first
  2. Identify variation points — What changes between customers? (model, data, branding, scale)
  3. Extract parameters — Move all variable values into config files and IaC parameters
  4. Create IaC templates — Wrap all resources in Bicep or Terraform modules
  5. Add deployment automation — Scripts or GitHub Actions for one-click provisioning
  6. Embed evaluation — Include eval datasets and automated quality gates
  7. Document everything — Onboarding guide, architecture diagram, troubleshooting guide

Parameterized Bicep Example

Bicep
@description('Customer tenant identifier') param tenantId string @description('Azure region for deployment') param location string = resourceGroup().location @description('AI model to deploy') @allowed(['gpt-4o', 'gpt-4o-mini', 'gpt-4.1']) param modelName string = 'gpt-4o' @description('Tokens-per-minute quota') param tpmQuota int = 30000 resource aiHub 'Microsoft.MachineLearningServices/workspaces@2024-10-01' = { name: '${tenantId}-ai-hub' location: location kind: 'Hub' identity: { type: 'SystemAssigned' } properties: { friendlyName: '${tenantId} AI Hub' } } resource aiProject 'Microsoft.MachineLearningServices/workspaces@2024-10-01' = { name: '${tenantId}-faq-project' location: location kind: 'Project' properties: { hubResourceId: aiHub.id } }

Pricing & Licensing Strategies

Pricing an AI offering requires understanding both your cost structure and your customer's willingness to pay. The right pricing model aligns your revenue with the value your solution delivers.

Pricing Models

ModelHow It WorksBest ForRisk
Per-conversationCharge per completed sessionFAQ bots, support agentsLow — aligned with usage
Per-queryCharge per API callSearch, analytics toolsLow — predictable cost
Per-seat / per-userFixed monthly fee per userEnterprise copilotsMedium — usage may vary
Tiered consumptionVolume-based pricing tiersHigh-volume applicationsLow — scales with value
Flat subscriptionFixed monthly/annual feeSMB solutionsHigh — unpredictable cost
Outcome-basedBased on measurable resultsPremium offeringsHigh — hard to measure

Cost Components to Include

When pricing your AI offering, account for all cost layers:

🧠

Model Inference

Tokens consumed (input + output) across all customer conversations

🔍

AI Search

Index storage, query volume, and semantic ranking operations

💾

Storage

Knowledge base documents, conversation logs, evaluation data

🌐

Networking

Private endpoints, VNet integration, data transfer

📊

Monitoring

Application Insights, Log Analytics, custom dashboards

🛟

Support

Your team's time for onboarding, maintenance, and incident response

Margin Calculation Example

Pricing
Per-conversation pricing example (FAQ bot): Average tokens per conversation: 2,500 GPT-4o cost per 1K tokens: ~$0.005 (blended in/out) Azure AI Search cost per query: ~$0.001 Infrastructure overhead: ~$0.002 ───────────────────────────────────────────────── Total cost per conversation: ~$0.020 Your price per conversation: $0.10 Gross margin: 80%

Licensing Considerations

  • Model licensing — Ensure your usage of Azure AI models complies with Microsoft's terms for redistribution and embedding in commercial products
  • Data ownership — Customer data must stay in the customer's tenant; never co-mingle across customers
  • IP ownership — Clearly define in your agreements who owns fine-tuned models, evaluation datasets, custom prompts, and derived insights
  • Usage transparency — Provide customers with dashboards showing their actual consumption, not just invoiced amounts
  • SLA commitments — Define uptime guarantees, response latency targets, and remediation procedures in your customer agreements

Revenue Growth Strategies

Once your base offering is established, explore these strategies for growing revenue per customer:

StrategyDescriptionRevenue Impact
Tier upgradesOffer premium tiers with faster models, higher limits, or advanced features+30–50% per customer
Add-on modulesSell optional capabilities like sentiment analytics, multilingual support, or custom reporting+20–40% per customer
Professional servicesOffer knowledge base curation, prompt engineering, and custom integration servicesVariable, high-margin
Usage-based expansionAs customers see value, they naturally increase usage — make this easy to scaleOrganic growth
Multi-department expansionExpand from one department (e.g., support) to others (HR, sales, IT) within the same customer2–5x per customer
⚠️ Pricing Anti-Pattern

Don't price based on Azure costs alone. Your value includes the solution design, evaluation quality, domain expertise, and time-to-value. A customer paying $0.10 per conversation for a solution that took you one day to deploy (instead of three months to build from scratch) is getting exceptional value.

Best Practices

Template Design

  • Start concrete, then abstract — Build for real customers first, extract patterns second
  • Use semantic versioning — v1.2.3 for templates, agents, and evaluation datasets
  • Include sample data — Every template ships with example knowledge and evaluation sets
  • Test the onboarding flow — Have someone outside your team deploy from scratch using only your docs

Multi-Tenant Operations

  • Tag all resources — Use tenant_id tags for cost attribution and resource discovery
  • Implement per-tenant rate limiting — Prevent one customer's spike from affecting others
  • Plan for tenant offboarding — Automate the cleanup of indexes, storage, and projects when a customer leaves
  • Monitor per-tenant quality — Track evaluation metrics (groundedness, coherence) per tenant, not just globally
  • Automate tenant provisioning — Use deployment scripts that create all resources from a single config file
  • Implement health checks — Scheduled probes that verify each tenant's agent is responding correctly

Pricing & Business

  • Build cost dashboards early — Understand your per-tenant cost before you set prices
  • Offer a free tier or trial — Lower the barrier to entry, then convert to paid
  • Review margins quarterly — Azure pricing changes, model costs decrease, usage patterns shift
  • Document IP boundaries clearly — Prevent disputes over who owns what after the engagement ends
