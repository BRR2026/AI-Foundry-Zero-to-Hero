Why End-to-End Security Matters

AI workloads process some of the most sensitive data in any organization — customer records, proprietary models, inference results, and training datasets. A single misconfigured endpoint can expose this data to the public internet, creating regulatory, financial, and reputational risk.

Azure AI Foundry provides enterprise-grade security capabilities that let you lock down every layer of the AI stack. In this module, you will learn how to build a defense-in-depth architecture using four pillars:

🌐
VNet Integration

Isolate AI services inside your own virtual network with no public internet exposure.

🔗
Private Endpoints

Connect to AI Foundry resources over private IP addresses using Azure Private Link.

🛡️
NSGs & Firewall

Control inbound and outbound traffic with granular network security rules.

🔑
Key Vault

Centralize secrets, keys, and certificates with RBAC and audit logging.

📘 Prerequisites
This module assumes you have completed Module 34: Responsible AI Governance and have a working Azure subscription with Contributor access. Familiarity with basic networking concepts (IP addressing, subnets, DNS) is recommended.

Understanding the AI Threat Landscape

Before diving into implementation, it's essential to understand what you're defending against. AI workloads face a unique blend of traditional infrastructure threats and AI-specific attack vectors.

Traditional Network Threats

  • Data exfiltration — Unauthorized transfer of training data or model weights to external endpoints
  • Man-in-the-middle attacks — Intercepting API calls between your application and AI services
  • Lateral movement — Attackers pivoting from a compromised VM to AI service endpoints
  • Credential theft — Harvesting API keys stored in code, environment variables, or config files

AI-Specific Threats

  • Model extraction — Repeated inference calls to reverse-engineer your proprietary model
  • Training data poisoning — Injecting malicious data through unsecured data pipelines
  • Prompt injection — Manipulating input to override model safety guardrails
  • Inference endpoint abuse — Unauthorized usage driving up costs or degrading performance
⚠️ Real-World Impact
According to Microsoft's 2026 AI Security Report, 73% of organizations running AI workloads experienced at least one security incident in the prior 12 months. The most common root cause was misconfigured network access — exactly what this module teaches you to prevent.

Virtual Network (VNet) Integration

A Virtual Network (VNet) is the fundamental building block for private networking in Azure. By integrating Azure AI Foundry resources into a VNet, you ensure that all traffic stays within Microsoft's backbone network — never traversing the public internet.

How VNet Integration Works

When you deploy Azure AI Foundry into a VNet, each component receives a private IP address from a subnet you define. Resources communicate over these private IPs, and DNS resolution is configured to point to internal addresses rather than public endpoints.

Creating the VNet with Azure CLI

Bash # Define variables RG="rg-ai-foundry-secure" LOCATION="eastus2" VNET_NAME="vnet-ai-foundry" # Create resource group az group create --name $RG --location $LOCATION # Create VNet with address space az network vnet create \ --resource-group $RG \ --name $VNET_NAME \ --address-prefix "10.1.0.0/16" \ --location $LOCATION # Create subnets az network vnet subnet create \ --resource-group $RG \ --vnet-name $VNET_NAME \ --name "snet-ai-hub" \ --address-prefixes "10.1.1.0/24" az network vnet subnet create \ --resource-group $RG \ --vnet-name $VNET_NAME \ --name "snet-compute" \ --address-prefixes "10.1.2.0/23" az network vnet subnet create \ --resource-group $RG \ --vnet-name $VNET_NAME \ --name "snet-storage" \ --address-prefixes "10.1.4.0/26" az network vnet subnet create \ --resource-group $RG \ --vnet-name $VNET_NAME \ --name "snet-keyvault" \ --address-prefixes "10.1.4.64/27"

Bicep Template for VNet Provisioning

Bicep // vnet-ai-foundry.bicep param location string = resourceGroup().location param vnetName string = 'vnet-ai-foundry' resource vnet 'Microsoft.Network/virtualNetworks@2023-09-01' = { name: vnetName location: location properties: { addressSpace: { addressPrefixes: [ '10.1.0.0/16' ] } subnets: [ { name: 'snet-ai-hub' properties: { addressPrefix: '10.1.1.0/24' privateEndpointNetworkPolicies: 'Disabled' } } { name: 'snet-compute' properties: { addressPrefix: '10.1.2.0/23' } } { name: 'snet-storage' properties: { addressPrefix: '10.1.4.0/26' privateEndpointNetworkPolicies: 'Disabled' } } { name: 'snet-keyvault' properties: { addressPrefix: '10.1.4.64/27' privateEndpointNetworkPolicies: 'Disabled' } } ] } } output vnetId string = vnet.id output aiHubSubnetId string = vnet.properties.subnets[0].id
⚡ Important: Subnet Delegation
AI Foundry compute instances require the subnet property privateEndpointNetworkPolicies to be set to 'Disabled'. Without this, private endpoint creation will fail. This is a common deployment blocker.

Private Endpoints for AI Services

Azure Private Link enables you to access Azure AI Foundry, Azure Storage, Azure Key Vault, and Azure Container Registry over a private endpoint — a network interface with a private IP address in your VNet. This eliminates exposure to the public internet entirely.

How Private Endpoints Work

🔒 Private Link Data Flow

When a private endpoint is created for an Azure service:

  • A network interface (NIC) is provisioned in the specified subnet with a private IP
  • DNS is configured (via Private DNS Zone) to resolve the service FQDN to the private IP
  • The service's public endpoint can be disabled, ensuring all traffic is private
  • Traffic flows over the Microsoft backbone network — never the internet

Services Requiring Private Endpoints

Service Group ID Private DNS Zone
AI Foundry Hub amlworkspace privatelink.api.azureml.ms
Storage (Blob) blob privatelink.blob.core.windows.net
Storage (File) file privatelink.file.core.windows.net
Key Vault vault privatelink.vaultcore.azure.net
Container Registry registry privatelink.azurecr.io
Azure AI Services account privatelink.cognitiveservices.azure.com

Creating a Private Endpoint for AI Foundry

Bash # Get the AI Foundry Hub resource ID HUB_ID=$(az resource show \ --resource-group $RG \ --name "hub-ai-foundry" \ --resource-type "Microsoft.MachineLearningServices/workspaces" \ --query "id" -o tsv) # Create private endpoint az network private-endpoint create \ --resource-group $RG \ --name "pe-ai-hub" \ --vnet-name $VNET_NAME \ --subnet "snet-ai-hub" \ --private-connection-resource-id $HUB_ID \ --group-id "amlworkspace" \ --connection-name "conn-ai-hub" # Configure Private DNS Zone az network private-dns zone create \ --resource-group $RG \ --name "privatelink.api.azureml.ms" # Link DNS zone to VNet az network private-dns link vnet create \ --resource-group $RG \ --zone-name "privatelink.api.azureml.ms" \ --name "link-ai-foundry" \ --virtual-network $VNET_NAME \ --registration-enabled false # Create DNS zone group for automatic DNS record management az network private-endpoint dns-zone-group create \ --resource-group $RG \ --endpoint-name "pe-ai-hub" \ --name "dzg-ai-hub" \ --private-dns-zone "privatelink.api.azureml.ms" \ --zone-name "azureml"

Bicep: Private Endpoint Module

Disabling Public Network Access

After creating private endpoints, you should disable public access to ensure all traffic flows privately:

Bash # Disable public access on the AI Foundry Hub az resource update \ --ids $HUB_ID \ --set properties.publicNetworkAccess="Disabled" # Disable public access on Storage account az storage account update \ --resource-group $RG \ --name "staifoundrysecure" \ --default-action "Deny" \ --public-network-access "Disabled" # Disable public access on Key Vault az keyvault update \ --resource-group $RG \ --name "kv-ai-foundry-secure" \ --public-network-access "Disabled"
✅ Validation Check
After creating private endpoints and disabling public access, verify connectivity by running nslookup hub-ai-foundry.api.azureml.ms from a VM inside the VNet. The resolved IP should be a private IP (e.g., 10.1.1.x), not a public IP.

Network Security Groups & Firewall Rules

Network Security Groups (NSGs) act as virtual firewalls for your subnets and individual network interfaces. They contain inbound and outbound security rules that filter traffic based on source, destination, port, and protocol.

NSG Design Principles for AI Foundry

🛡️ Defense-in-Depth NSG Strategy

  • Default deny — Start with all traffic blocked, then allow only what's needed
  • Subnet-level NSGs — Apply rules at the subnet level for broad protection
  • NIC-level NSGs — Apply additional rules at the NIC level for critical resources
  • Service Tags — Use Azure service tags instead of hardcoded IP ranges
  • Application Security Groups — Group VMs by function for simplified rules

Essential NSG Rules for AI Foundry

Priority Name Direction Source Destination Port Action

110 AllowAzureRM Outbound VNet AzureResourceManager 443 Allow
120 AllowStorage Outbound VNet Storage.EastUS2 443 Allow
130 AllowKeyVault Outbound VNet AzureKeyVault 443 Allow
140 AllowACR Outbound VNet AzureContainerRegistry 443 Allow
150 AllowMonitor Outbound VNet AzureMonitor 443 Allow
4096 DenyAllOutbound Outbound * * * Deny

Creating NSGs with Azure CLI

Bash # Create NSG for compute subnet az network nsg create \ --resource-group $RG \ --name "nsg-ai-compute" # Allow outbound to Azure AD for authentication az network nsg rule create \ --resource-group $RG \ --nsg-name "nsg-ai-compute" \ --name "AllowAzureAD" \ --priority 100 \ --direction "Outbound" \ --access "Allow" \ --protocol "Tcp" \ --source-address-prefixes "VirtualNetwork" \ --destination-address-prefixes "AzureActiveDirectory" \ --destination-port-ranges "443" # Allow outbound to Azure Resource Manager az network nsg rule create \ --resource-group $RG \ --nsg-name "nsg-ai-compute" \ --name "AllowARM" \ --priority 110 \ --direction "Outbound" \ --access "Allow" \ --protocol "Tcp" \ --source-address-prefixes "VirtualNetwork" \ --destination-address-prefixes "AzureResourceManager" \ --destination-port-ranges "443" # Deny all other outbound traffic az network nsg rule create \ --resource-group $RG \ --nsg-name "nsg-ai-compute" \ --name "DenyAllOutbound" \ --priority 4096 \ --direction "Outbound" \ --access "Deny" \ --protocol "*" \ --source-address-prefixes "*" \ --destination-address-prefixes "*" \ --destination-port-ranges "*" # Associate NSG with subnet az network vnet subnet update \ --resource-group $RG \ --vnet-name $VNET_NAME \ --name "snet-compute" \ --network-security-group "nsg-ai-compute"

Azure Firewall for Centralized Egress

For enterprise deployments, Azure Firewall provides centralized egress filtering with FQDN-based rules, threat intelligence, and TLS inspection. This is the recommended approach for production AI Foundry deployments.

Bash # Create Azure Firewall subnet (must be named AzureFirewallSubnet) az network vnet subnet create \ --resource-group $RG \ --vnet-name $VNET_NAME \ --name "AzureFirewallSubnet" \ --address-prefixes "10.1.5.0/26" # Create public IP for firewall az network public-ip create \ --resource-group $RG \ --name "pip-fw-ai" \ --sku "Standard" \ --allocation-method "Static" # Create Azure Firewall az network firewall create \ --resource-group $RG \ --name "fw-ai-foundry" \ --location $LOCATION \ --sku "AZFW_VNet" \ --tier "Standard" # Add application rule for AI Foundry dependencies az network firewall application-rule create \ --resource-group $RG \ --firewall-name "fw-ai-foundry" \ --collection-name "AllowAIFoundry" \ --priority 200 \ --action "Allow" \ --name "ai-foundry-deps" \ --source-addresses "10.1.0.0/16" \ --protocols "Https=443" \ --target-fqdns \ "*.api.azureml.ms" \ "*.notebooks.azure.net" \ "*.instances.azureml.ms" \ "*.blob.core.windows.net"
📘 NSG Flow Logs
Enable NSG Flow Logs for audit and compliance. Flow logs capture all traffic allowed or denied by NSG rules and can be sent to Azure Monitor, Log Analytics, or Storage for analysis. Use the Traffic Analytics feature for visual insights.

Key Vault for Secrets Management

Azure Key Vault is the centralized secrets management service used by AI Foundry to store API keys, connection strings, certificates, and cryptographic keys. Proper Key Vault configuration is essential for any secure AI deployment.

Key Vault Integration with AI Foundry

When you create an AI Foundry Hub, a Key Vault is automatically provisioned as a dependent resource. AI Foundry uses this vault to securely store:

  • Data store connection strings — credentials for accessing Azure Storage, SQL databases, and external data sources
  • API keys — keys for Azure AI Services (OpenAI, Speech, Vision, etc.)
  • Custom secrets — any secrets your training or inference code needs at runtime
  • SSL/TLS certificates — certificates for custom domains on endpoints

Configuring Key Vault with RBAC

The modern approach uses Azure RBAC instead of access policies for fine-grained, identity-based access control:

Bash # Create Key Vault with RBAC authorization az keyvault create \ --resource-group $RG \ --name "kv-ai-foundry-secure" \ --location $LOCATION \ --enable-rbac-authorization true \ --enable-purge-protection true \ --enable-soft-delete true \ --soft-delete-retention-days 90 \ --public-network-access "Disabled" # Assign "Key Vault Secrets Officer" to AI Foundry managed identity HUB_PRINCIPAL_ID=$(az resource show \ --ids $HUB_ID \ --query "identity.principalId" -o tsv) az role assignment create \ --role "Key Vault Secrets Officer" \ --assignee-object-id $HUB_PRINCIPAL_ID \ --assignee-principal-type "ServicePrincipal" \ --scope "/subscriptions/{sub}/resourceGroups/$RG/providers/Microsoft.KeyVault/vaults/kv-ai-foundry-secure" # Assign "Key Vault Secrets User" for read-only access by compute az role assignment create \ --role "Key Vault Secrets User" \ --assignee-object-id $COMPUTE_PRINCIPAL_ID \ --assignee-principal-type "ServicePrincipal" \ --scope "/subscriptions/{sub}/resourceGroups/$RG/providers/Microsoft.KeyVault/vaults/kv-ai-foundry-secure"

Storing and Retrieving Secrets

Python from azure.identity import DefaultAzureCredential from azure.keyvault.secrets import SecretClient # Authenticate using managed identity (no keys in code!) credential = DefaultAzureCredential() vault_url = "https://kv-ai-foundry-secure.vault.azure.net" client = SecretClient(vault_url=vault_url, credential=credential) # Store an API key client.set_secret("openai-api-key", "sk-your-key-here") # Retrieve the secret at runtime secret = client.get_secret("openai-api-key") print(f"Secret value: {secret.value}")

Key Vault Monitoring & Auditing

Enable diagnostic settings to track every secret access, modification, and deletion:

Bash # Enable diagnostic logs for Key Vault az monitor diagnostic-settings create \ --resource "/subscriptions/{sub}/resourceGroups/$RG/providers/Microsoft.KeyVault/vaults/kv-ai-foundry-secure" \ --name "kv-diagnostics" \ --workspace "law-ai-foundry" \ --logs '[{"category":"AuditEvent","enabled":true,"retentionPolicy":{"enabled":true,"days":365}}]'
✅ Best Practice: No Keys in Code
Never embed API keys, connection strings, or passwords directly in application code, environment variables, or configuration files. Always use managed identities with Key Vault references. This eliminates the need to rotate credentials manually and prevents accidental exposure in source control.

Key Architectural Decisions

  • Hub-spoke topology — Centralized firewall in the hub VNet with AI Foundry in spoke VNets connected via peering

  • User-Defined Routes (UDRs) — Force all egress traffic through Azure Firewall for inspection

Mini Hack: Deploy AI Foundry with Private Endpoints

In this hands-on exercise, you will deploy a fully secured AI Foundry Hub inside a Virtual Network with Private Endpoints for all dependent services. This is the same pattern used by enterprise customers in production.

⏱️ Estimated Time: 45–60 minutes
This mini hack deploys multiple Azure resources. Expect provisioning to take 15–20 minutes. Use the wait time to review the security configuration of each resource.

Step 1: Set Up the Foundation

Bash # Variables for the entire deployment RG="rg-ai-foundry-secure-hack" LOCATION="eastus2" VNET_NAME="vnet-ai-secure" HUB_NAME="hub-ai-secure-$RANDOM" KV_NAME="kv-aisec-$RANDOM" STORAGE_NAME="staisec$RANDOM" ACR_NAME="acraisec$RANDOM" # Create resource group az group create -n $RG -l $LOCATION # Create VNet with subnets az network vnet create \ -g $RG -n $VNET_NAME \ --address-prefix "10.0.0.0/16" for subnet in "snet-hub:10.0.1.0/24" "snet-compute:10.0.2.0/23" "snet-pe:10.0.4.0/24"; do name=${subnet%%:*} prefix=${subnet##*:} az network vnet subnet create \ -g $RG --vnet-name $VNET_NAME \ -n $name --address-prefixes $prefix \ --private-endpoint-network-policies Disabled done

Step 2: Deploy Supporting Services

Bash # Create Key Vault with RBAC az keyvault create \ -g $RG -n $KV_NAME -l $LOCATION \ --enable-rbac-authorization true \ --public-network-access Disabled # Create Storage account az storage account create \ -g $RG -n $STORAGE_NAME -l $LOCATION \ --sku Standard_LRS \ --default-action Deny \ --public-network-access Disabled # Create Container Registry az acr create \ -g $RG -n $ACR_NAME -l $LOCATION \ --sku Premium \ --public-network-access Disabled

Step 3: Create Private Endpoints

Bash # Create private endpoints for each service KV_ID=$(az keyvault show -g $RG -n $KV_NAME --query id -o tsv) ST_ID=$(az storage account show -g $RG -n $STORAGE_NAME --query id -o tsv) ACR_ID=$(az acr show -g $RG -n $ACR_NAME --query id -o tsv) # Key Vault PE az network private-endpoint create \ -g $RG -n "pe-keyvault" \ --vnet-name $VNET_NAME --subnet "snet-pe" \ --private-connection-resource-id $KV_ID \ --group-id "vault" \ --connection-name "conn-kv" # Storage Blob PE az network private-endpoint create \ -g $RG -n "pe-storage-blob" \ --vnet-name $VNET_NAME --subnet "snet-pe" \ --private-connection-resource-id $ST_ID \ --group-id "blob" \ --connection-name "conn-blob" # ACR PE az network private-endpoint create \ -g $RG -n "pe-acr" \ --vnet-name $VNET_NAME --subnet "snet-pe" \ --private-connection-resource-id $ACR_ID \ --group-id "registry" \ --connection-name "conn-acr"

Security Best Practices Checklist

Use this checklist to validate that your AI Foundry deployment meets enterprise security standards:

Network Security

  • Deploy AI Foundry inside a dedicated VNet with planned subnet topology
  • Create Private Endpoints for all dependent services (Storage, Key Vault, ACR, AI Services)
  • Disable public network access on all resources
  • Configure Private DNS Zones for automatic DNS resolution
  • Apply NSGs with default-deny outbound rules on all subnets
  • Use Azure Firewall for centralized egress control with FQDN filtering
  • Enable NSG Flow Logs and Traffic Analytics
  • Implement DDoS Protection Standard on the VNet

Identity & Secrets

  • Use Azure RBAC authorization for Key Vault (not access policies)
  • Enable soft-delete and purge protection on Key Vault
  • Use managed identities instead of API keys wherever possible
  • Enable diagnostic logging on Key Vault for audit compliance
  • Rotate secrets automatically using Key Vault rotation policies
  • Use Conditional Access policies for user access to AI Foundry portal

Data Protection

  • Enable customer-managed keys (CMK) for data encryption at rest
  • Enforce HTTPS/TLS 1.2+ for all data in transit
  • Enable Azure Defender for Storage for threat detection
  • Implement data classification and sensitivity labels

Common Issues & Troubleshooting

Problem: "Unable to connect to workspace"

🔧 Resolution

After disabling public access, users must connect from within the VNet (via VPN, ExpressRoute, or Azure Bastion). Verify DNS resolution returns a private IP:

nslookup {hub-name}.api.azureml.ms

If it returns a public IP, check that the Private DNS Zone is linked to your VNet.

Problem: "Private endpoint creation fails with policy error"

🔧 Resolution

Ensure the subnet has privateEndpointNetworkPolicies set to Disabled. Update it with:

az network vnet subnet update --disable-private-endpoint-network-policies true

Problem: "Compute instance cannot pull container images"

🔧 Resolution

Container Registry requires a Private Endpoint. Also verify that NSG rules allow outbound traffic to the AzureContainerRegistry service tag on port 443.

Problem: "Key Vault access denied (403)"

🔧 Resolution

When using RBAC authorization, ensure the calling identity has the appropriate role (Key Vault Secrets User or Key Vault Secrets Officer) assigned at the correct scope. Role assignments can take up to 10 minutes to propagate.
