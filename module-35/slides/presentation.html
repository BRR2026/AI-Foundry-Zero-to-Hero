Virtual Network Integration

Key Concepts

  • Private IP addresses for every AI Foundry component
  • Subnet isolation between hub, compute, and storage
  • DNS resolution pointing to internal addresses
  • No public internet traversal — Microsoft backbone only

Private Endpoints for AI Services

How Private Link Works

  • NIC provisioned in your subnet with a private IP
  • Private DNS Zone resolves FQDN → private IP
  • Public endpoint can be disabled
  • Traffic stays on Microsoft backbone

🔑 Critical Difference

Service Endpoints = traffic from your VNet goes via optimized route but service still has a public IP.
Private Endpoints = service gets a private IP in your VNet. No public IP at all.

Creating Private Endpoints

# Create Private Endpoint for AI Foundry Hub az network private-endpoint create \ --resource-group $RG \ --name "pe-ai-hub" \ --vnet-name $VNET_NAME \ --subnet "snet-ai-hub" \ --private-connection-resource-id $HUB_ID \ --group-id "amlworkspace" \ --connection-name "conn-ai-hub" # Configure Private DNS Zone az network private-dns zone create \ --resource-group $RG \ --name "privatelink.api.azureml.ms" # Link DNS zone to VNet az network private-dns link vnet create \ --resource-group $RG \ --zone-name "privatelink.api.azureml.ms" \ --name "link-ai-foundry" \ --virtual-network $VNET_NAME \ --registration-enabled false

After creating the PE, validate with nslookup — expect a private IP (10.x.x.x)

Azure Firewall for Centralized Egress

Why Use Azure Firewall?

  • FQDN-based rules — allow *.api.azureml.ms
  • Threat intelligence — block known malicious IPs
  • TLS inspection — decrypt and inspect traffic
  • Centralized logging — all egress in one place

🌐 Required FQDNs

  • *.api.azureml.ms
  • *.notebooks.azure.net
  • *.instances.azureml.ms
  • *.blob.core.windows.net

Key Vault for Secrets Management

What AI Foundry Stores in Key Vault

  • Data store connection strings
  • API keys for AI Services
  • Custom secrets for training/inference
  • SSL/TLS certificates

🔒 Configuration

  • RBAC authorization (not access policies)
  • Soft-delete + purge protection
  • Private Endpoint for access
  • Diagnostic logging enabled

RBAC Roles

No Keys in Code — Managed Identities

# Python — authenticate with managed identity from azure.identity import DefaultAzureCredential from azure.keyvault.secrets import SecretClient # No API keys needed — uses managed identity automatically credential = DefaultAzureCredential() vault_url = "https://kv-ai-foundry-secure.vault.azure.net" client = SecretClient(vault_url=vault_url, credential=credential) # Retrieve secret at runtime secret = client.get_secret("openai-api-key") print(f"Retrieved: {secret.name}")

Mini Hack: Secure Deployment

Deploy a fully network-isolated AI Foundry environment in 5 steps:

Step 1–2

Create VNet with subnets. Deploy Key Vault, Storage, ACR with public access disabled.

Step 3–4

Create Private Endpoints for each service. Configure Private DNS Zones and link to VNet.

Step 5

Deploy test VM inside VNet. Validate DNS resolves to private IPs. Clean up resources.

⏱️ Time Estimate

45–60 minutes. Resource provisioning takes 15–20 min — use wait time to review configurations.
