Why Evaluation Datasets Matter

Without evaluation datasets, AI development is guesswork. You're testing with ad hoc queries, relying on subjective impressions, and discovering quality regressions from user complaints. Evaluation datasets transform this into a systematic, measurable discipline.

The Cost of Not Measuring

  • Silent regressions: A model update quietly degrades answers for 20% of queries — nobody notices until users churn.
  • Prompt drift: Well-intentioned prompt tweaks fix one scenario while breaking three others.
  • Data source rot: An index update introduces stale or conflicting information, but there's no automated check.
  • Anecdotal quality: "It seems better" doesn't hold up in production reviews or compliance audits.

What Good Evaluation Provides

Eval Dataset → Automated Pipeline → Quality Metrics → Trend Dashboard → Regression Alerts
  • Confidence in deployments: Every release is validated against a known set of test cases.
  • Data-driven decisions: Compare prompts, models, and configurations with hard numbers.
  • Compliance evidence: Provide auditors with documented quality metrics over time.
  • Team alignment: Everyone agrees on what "good" looks like.

Building Golden Datasets

A golden dataset (also called a "gold set" or "benchmark set") is a curated collection of input-output pairs that define the expected behavior of your AI application. Think of it as unit tests for your AI.

Sources for Golden Dataset Questions

  1. Real user conversations: Mine production logs for common queries, edge cases, and failure modes.
  2. Subject matter experts: Have domain experts craft questions that cover critical knowledge areas.
  3. Historical support tickets: Extract Q&A pairs from resolved support cases.
  4. Adversarial testing: Deliberately include tricky, ambiguous, and out-of-scope queries.
  5. Synthetic generation: Use LLMs to generate diverse question variants (with human review).

Dataset Structure

A well-structured evaluation dataset in Azure AI Foundry typically follows this format:

eval_dataset.jsonl 
{"query": "What is the SLA for Azure OpenAI Service?", "context": "Azure OpenAI Service offers a 99.9% uptime SLA for standard deployments...", "ground_truth": "Azure OpenAI Service provides a 99.9% uptime SLA for standard deployments.", "category": "factual", "difficulty": "easy"}

How Many Examples Do You Need?

Stage Dataset Size Purpose
Prototyping 20–50 examples Quick sanity checks during development
Pre-production 100–300 examples Coverage of major scenarios and edge cases
Production 500+ examples Comprehensive regression testing with statistical significance
Enterprise 1,000+ examples Multi-domain coverage, compliance, A/B testing baselines
⚠️ Quality Over Quantity

50 carefully curated examples with expert-validated ground truth will outperform 5,000 auto-generated examples with sloppy answers. Invest in quality annotations — they're the foundation of your entire evaluation system.

Ground Truth & Reference Answers

Ground truth is the "correct" or "ideal" answer against which your AI's output is compared. Establishing reliable ground truth is one of the most challenging — and most important — steps in building an evaluation system.

Types of Ground Truth

  • Exact match: Used for factual lookups — "What is the capital of France?" → "Paris"
  • Semantic equivalence: The answer must convey the same meaning, even if worded differently.
  • Key-point coverage: The answer must address specific points (e.g., "mentions three security benefits").
  • Rubric-based: A scoring guide defines levels of quality (excellent, adequate, insufficient).
  • Constraint-based: The answer must satisfy conditions (e.g., "under 200 words," "includes a code example").

Creating High-Quality Ground Truth

1

Expert Annotation

Have 2–3 domain experts independently write reference answers. Use consensus or adjudication for disagreements. This is the gold standard for quality.

2

Inter-Annotator Agreement

Measure agreement between annotators using metrics like Cohen's Kappa or Krippendorff's Alpha. Low agreement indicates ambiguous questions that need refinement.

3

Version Control

Ground truth evolves as products change. Maintain versioned datasets and track when ground truth is updated, by whom, and why.

4

Coverage Mapping

Map evaluation examples to knowledge domains, question types, and difficulty levels. Identify gaps and over-represented areas in your dataset.

Automated Evaluation Pipelines

Manual evaluation doesn't scale. Azure AI Foundry provides the tools to build automated evaluation pipelines that run on every deployment, every prompt change, or on a schedule.

Pipeline Architecture

Trigger (CI/CD, Schedule) → Load Dataset → Run Inference → Score Outputs → Report & Alert

Setting Up an Evaluation Pipeline

Python — Automated Evaluation Pipeline 
from azure.ai.evaluation import evaluate
from azure.ai.evaluation import (
    GroundednessEvaluator,
    RelevanceEvaluator,
    CoherenceEvaluator,
    FluencyEvaluator,
    SimilarityEvaluator
)
from azure.identity import DefaultAzureCredential

# Initialize evaluators with your Azure AI Foundry project
credential = DefaultAzureCredential()
azure_ai_project = {
    "subscription_id": "your-sub-id",
    "resource_group_name": "your-rg",
    "project_name": "your-foundry-project"
}

# Configure evaluators
groundedness_eval = GroundednessEvaluator(
    credential=credential,
    azure_ai_project=azure_ai_project
)
relevance_eval = RelevanceEvaluator(
    credential=credential,
    azure_ai_project=azure_ai_project
)
coherence_eval = CoherenceEvaluator(
    credential=credential,
    azure_ai_project=azure_ai_project
)
fluency_eval = FluencyEvaluator(
    credential=credential,
    azure_ai_project=azure_ai_project
)

# Run evaluation against your dataset
results = evaluate(
    data="eval_dataset.jsonl",
    evaluators={
        "groundedness": groundedness_eval,
        "relevance": relevance_eval,
        "coherence": coherence_eval,
        "fluency": fluency_eval
    },
    evaluator_config={
        "groundedness": {
            "query": "${data.query}",
            "context": "${data.context}",
            "response": "${data.response}"
        },
        "relevance": {
            "query": "${data.query}",
            "response": "${data.response}"
        },
        "coherence": {
            "response": "${data.response}"
        },
        "fluency": {
            "response": "${data.response}"
        }
    },
    azure_ai_project=azure_ai_project
)

print(f"Groundedness: {results['metrics']['groundedness.groundedness']:.2f}")
print(f"Relevance:    {results['metrics']['relevance.relevance']:.2f}")
print(f"Coherence:    {results['metrics']['coherence.coherence']:.2f}")
print(f"Fluency:      {results['metrics']['fluency.fluency']:.2f}")

LLM-as-a-Judge Patterns

Traditional NLP metrics like BLEU and ROUGE fail to capture the nuances of generative AI output. LLM-as-a-Judge uses a powerful language model to evaluate another model's output, providing human-like quality assessments at scale.

How LLM-as-a-Judge Works

  1. Define criteria: What aspects of quality matter? (accuracy, completeness, tone, format)
  2. Craft a judging prompt: Instruct the judge model on how to evaluate, with a clear rubric.
  3. Present the evidence: Provide the query, context, response, and optionally ground truth.
  4. Collect structured scores: The judge returns a numeric score and reasoning.

Best Practices for LLM-as-a-Judge

✅ Recommended Practices
  • Use a stronger model as judge: GPT-4o judging GPT-4o-mini output is more reliable than self-evaluation.
  • Calibrate with human labels: Compare judge scores with human ratings on 50–100 examples to validate alignment.
  • Use structured output: Request JSON responses from the judge to ensure parseable results.
  • Include reasoning: Ask the judge to explain its score — this helps debug evaluation failures.
  • Test for position bias: Swap response order to check if the judge is biased toward first/last responses.
⚠️ Common Pitfalls
  • Self-evaluation bias: Models tend to rate their own outputs more favorably.
  • Verbosity bias: Longer answers often get higher scores regardless of quality.
  • Judge model drift: Updates to the judge model can shift scores — pin model versions.
  • Circular reasoning: If both the application and the judge use the same model, errors may be invisible.

Tracking Quality Over Time

A single evaluation snapshot tells you where you are. A quality trend tells you where you're going. Tracking metrics over time is essential for detecting gradual degradation and proving continuous improvement.

Building a Quality Timeline

Python — Track Evaluation History 
import json
from datetime import datetime
from pathlib import Path

class EvalTracker:
    """Track evaluation results over time for trend analysis."""

    def __init__(self, history_file="eval_history.jsonl"):
        self.history_file = Path(history_file)

    def record_run(self, run_name, metrics, metadata=None):
        """Record an evaluation run with its metrics."""
        entry = {
            "timestamp": datetime.utcnow().isoformat(),
            "run_name": run_name,
            "metrics": metrics,
            "metadata": metadata or {}
        }
        with open(self.history_file, "a") as f:
            f.write(json.dumps(entry) + "\n")

    def get_trend(self, metric_name, last_n=10):
        """Get the trend for a specific metric."""
        entries = []
        with open(self.history_file) as f:
            for line in f:
                entry = json.loads(line)
                if metric_name in entry["metrics"]:
                    entries.append({
                        "timestamp": entry["timestamp"],
                        "value": entry["metrics"][metric_name],
                        "run": entry["run_name"]
                    })
        return entries[-last_n:]

    def detect_regression(self, metric_name, threshold=0.1):
        """Detect if latest run regressed beyond threshold."""
        trend = self.get_trend(metric_name, last_n=2)
        if len(trend) < 2:
            return False, 0
        delta = trend[-1]["value"] - trend[-2]["value"]
        return delta < -threshold, delta

# Usage
tracker = EvalTracker()
tracker.record_run(
    run_name="v2.3-prompt-update",
    metrics={
        "groundedness": 4.2,
        "relevance": 4.5,
        "coherence": 4.7,
        "fluency": 4.8,
        "similarity": 0.85
    },
    metadata={
        "model": "gpt-4o",
        "prompt_version": "v2.3",
        "dataset_version": "gold-v5"
    }
)

# Check for regressions
regressed, delta = tracker.detect_regression("groundedness")
if regressed:
    print(f"⚠️ Groundedness regressed by {abs(delta):.2f}!")

When to Re-evaluate

  • Model changes: Switching models or model versions
  • Prompt updates: Any modification to system prompts or templates
  • Data source changes: New documents indexed, schema changes, data refreshes
  • Configuration changes: Temperature, top-p, max tokens, chunk size
  • Scheduled cadence: Weekly or bi-weekly even without changes (to catch external drift)

Regression Detection

Regression detection is the practice of automatically flagging when quality metrics drop below acceptable thresholds or decline significantly from baseline.

Regression Detection Strategies

1

Absolute Thresholds

Set minimum acceptable scores. If groundedness drops below 3.5, block deployment. Simple and predictable.

2

Relative Delta

Flag if any metric drops more than 0.3 points from the previous run. Catches sudden degradation regardless of absolute level.

3

Rolling Averages

Compare against the 5-run rolling average. This smooths out noise and catches gradual decline that individual comparisons might miss.

4

Category-Level Analysis

Break down regressions by question category. A model might improve overall but regress on "how-to" questions specifically — aggregate metrics would hide this.

Python — CI/CD Gate with Regression Check 
import sys

def evaluation_gate(results, thresholds, previous_results=None):
    """
    CI/CD gate that blocks deployment on quality regressions.
    Returns exit code 0 (pass) or 1 (fail).
    """
    failures = []

    # Check absolute thresholds
    for metric, min_score in thresholds.items():
        score = results["metrics"].get(metric, 0)
        if score < min_score:
            failures.append(
                f"❌ {metric}: {score:.2f} < {min_score} (threshold)"
            )

    # Check relative regression
    if previous_results:
        for metric in results["metrics"]:
            current = results["metrics"][metric]
            previous = previous_results["metrics"].get(metric, current)
            delta = current - previous
            if delta < -0.3:
                failures.append(
                    f"📉 {metric}: dropped {abs(delta):.2f} from previous"
                )

    if failures:
        print("🚫 EVALUATION GATE: FAILED")
        for f in failures:
            print(f"   {f}")
        return 1
    else:
        print("✅ EVALUATION GATE: PASSED")
        return 0

# Usage in CI/CD
thresholds = {
    "groundedness.groundedness": 3.5,
    "relevance.relevance": 3.5,
    "coherence.coherence": 3.5,
    "fluency.fluency": 3.5
}

exit_code = evaluation_gate(results, thresholds, previous_results)
sys.exit(exit_code)

azure-ai-evaluation SDK

The azure-ai-evaluation Python package is your primary tool for programmatic evaluation in Azure AI Foundry. It provides pre-built evaluators, custom evaluator support, and integration with the AI Foundry portal.

Installation & Setup

Terminal 
pip install azure-ai-evaluation azure-identity

Core Concepts

  • Evaluators: Classes that score model outputs on specific criteria (groundedness, relevance, etc.).
  • evaluate() function: Orchestrates running multiple evaluators against a dataset.
  • Evaluation data: JSONL files or pandas DataFrames with query, context, response, and optional ground truth columns.
  • Results: Returned as dictionaries with per-row scores and aggregate metrics.

Running a Complete Evaluation

Python — Full Evaluation Workflow 
from azure.ai.evaluation import evaluate
from azure.ai.evaluation import (
    GroundednessEvaluator,
    RelevanceEvaluator,
    CoherenceEvaluator,
    FluencyEvaluator,
    SimilarityEvaluator,
    F1ScoreEvaluator,
    ContentSafetyEvaluator

from azure.identity import DefaultAzureCredential
import pandas as pd

credential = DefaultAzureCredential()
project = {
    "subscription_id": "...",
    "resource_group_name": "...",
    "project_name": "my-foundry-project"
}

# Run the evaluation
results = evaluate(
    data="golden_dataset.jsonl",
    evaluators={
        "groundedness": GroundednessEvaluator(
            credential=credential, azure_ai_project=project
        ),
        "relevance": RelevanceEvaluator(
            credential=credential, azure_ai_project=project
        ),
        "coherence": CoherenceEvaluator(
            credential=credential, azure_ai_project=project
        ),
        "fluency": FluencyEvaluator(
            credential=credential, azure_ai_project=project
        ),
        "similarity": SimilarityEvaluator(
            credential=credential, azure_ai_project=project
        ),
        "f1": F1ScoreEvaluator()
    },
    evaluator_config={
        "groundedness": {
            "query": "${data.query}",
            "context": "${data.context}",
            "response": "${data.response}"
        },
        "relevance": {
            "query": "${data.query}",
            "response": "${data.response}"
        },
        "coherence": {
            "response": "${data.response}"
        },
        "fluency": {
            "response": "${data.response}"
        },
        "similarity": {
            "query": "${data.query}",
            "response": "${data.response}",
            "ground_truth": "${data.ground_truth}"

🧪 Mini Hack: Create an Eval Dataset & Run a Quality Benchmark

In this hands-on exercise, you'll create a golden evaluation dataset from scratch and run a quality benchmark using the azure-ai-evaluation SDK.

Challenge Steps

1

Create Your Dataset (20 min)

2

  • Created a JSONL dataset with 20+ examples across 4 categories
  • Successfully ran 4 evaluators against the dataset

Key Takeaways

What You've Learned

  • Golden datasets are the foundation of systematic AI quality measurement — invest in expert-curated examples with validated ground truth.
  • Automated evaluation pipelines eliminate manual testing and provide consistent, reproducible quality assessments on every change.
  • LLM-as-a-Judge enables nuanced evaluation at scale, but requires careful calibration against human labels and awareness of biases.
  • The azure-ai-evaluation SDK provides pre-built evaluators for groundedness, relevance, coherence, fluency, and similarity.
  • Quality tracking over time catches gradual degradation and provides evidence of continuous improvement.
  • Regression detection gates in CI/CD pipelines prevent quality regressions from reaching production.
  • Category-level analysis reveals hidden regressions that aggregate metrics would mask.
  • Evaluation is not a one-time activity — it's a continuous practice that should run on every prompt change, model update, and data refresh.
