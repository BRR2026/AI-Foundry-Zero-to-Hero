2. Azure AI Foundry Pricing Models

Azure AI Foundry offers two primary pricing models for model deployments. Choosing the right one is the single biggest cost decision you'll make. Each model is suited to different workload patterns, and many organizations use both simultaneously for different use cases.

🔒 PTU

Provisioned Throughput Units

  • Reserved capacity with guaranteed throughput
  • Predictable monthly cost (fixed hourly rate)
  • Best for steady-state, high-volume workloads
  • SLA-backed latency guarantees
  • 1-month minimum commitment
  • No per-token charges — unlimited tokens within capacity

📊 Pay-as-You-Go

Token-Based Consumption

  • Pay per token consumed (input + output)
  • No upfront commitment or reservation
  • Best for variable, bursty, or dev workloads
  • Scales automatically with demand
  • Subject to regional TPM/RPM quotas
  • Simple to start — no capacity planning needed

PTU (Provisioned Throughput Units) — Deep Dive

Provisioned Throughput Units represent dedicated model processing capacity reserved for your exclusive use. Each PTU provides a consistent, guaranteed number of tokens per minute, regardless of overall platform load. This eliminates the "noisy neighbor" problem and provides predictable performance for production workloads.

CharacteristicDetails
Billing ModelHourly rate per PTU, billed per second of usage
Minimum Commitment1 month (with monthly, quarterly, or annual options)

✅ When to Choose PTU: If your AI application processes more than ~1 million tokens per hour consistently, PTU pricing often beats pay-as-you-go by 40–70%. Run the break-even analysis in the Mini Hack section to confirm for your specific workload.

PTU Break-Even Calculator

Python
import math def calculate_ptu_breakeven( avg_input_tokens: int, avg_output_tokens: int, requests_per_hour: int, payg_input_price: float, # per 1K tokens payg_output_price: float, # per 1K tokens ptu_hourly_rate: float # per PTU per hour ) -> dict: """Calculate when PTU becomes cheaper than pay-as-you-go.""" # Pay-as-you-go hourly cost payg_hourly = ( (avg_input_tokens * requests_per_hour / 1000) * payg_input_price + (avg_output_tokens * requests_per_hour / 1000) * payg_output_price ) # Break-even PTU count breakeven_ptus = math.ceil(payg_hourly / ptu_hourly_rate) return { "payg_hourly_cost": round(payg_hourly, 2), "payg_monthly_cost": round(payg_hourly * 730, 2), "breakeven_ptus": breakeven_ptus, "ptu_monthly_cost": round(breakeven_ptus * ptu_hourly_rate * 730, 2), "monthly_savings": round( (payg_hourly * 730) - (breakeven_ptus * ptu_hourly_rate * 730), 2 ) } # Example: GPT-5.1 production workload result = calculate_ptu_breakeven( avg_input_tokens=800, avg_output_tokens=400, requests_per_hour=5000, payg_input_price=0.0025, payg_output_price=0.01, ptu_hourly_rate=2.0 ) print(result)

Pay-as-You-Go — Deep Dive

The pay-as-you-go model charges per token processed — separately for input (prompt) and output (completion) tokens. This is the default pricing model and the simplest to get started with. There's no commitment, no reservation, and no capacity planning required.

3. Token Usage Optimization Strategies

Token optimization is the art of achieving the same (or better) results with fewer tokens. This involves techniques at every stage: prompt design, response management, model selection, and intelligent caching.

Understanding Token Consumption

Before optimizing, you need to measure. The tiktoken library is the standard tool for counting tokens in Azure OpenAI models.

Python — Token Counting with tiktoken
import tiktoken def count_tokens(text: str, model: str = "GPT-5.1") -> int: """Count tokens for a given text and model.""" encoding = tiktoken.encoding_for_model(model) return len(encoding.encode(text)) def estimate_cost(input_text: str, output_tokens: int, model: str = "GPT-5.1") -> dict: """Estimate API call cost based on token counts.""" pricing = { "GPT-5.1": {"input": 2.50, "output": 10.00}, "GPT-5-mini": {"input": 0.15, "output": 0.60}, "gpt-4.1": {"input": 2.00, "output": 8.00}, "gpt-4.1-mini": {"input": 0.40, "output": 1.60}, "gpt-4.1-nano": {"input": 0.10, "output": 0.40}, } input_tokens = count_tokens(input_text, model) rates = pricing[model] input_cost = (input_tokens / 1_000_000) * rates["input"] output_cost = (output_tokens / 1_000_000) * rates["output"] return { "input_tokens": input_tokens, "output_tokens": output_tokens, "input_cost": round(input_cost, 6), "output_cost": round(output_cost, 6), "total_cost": round(input_cost + output_cost, 6), "model": model } prompt = "Explain the benefits of cloud computing in three sentences." result = estimate_cost(prompt, output_tokens=100, model="GPT-5.1") print(f"Input: {result['input_tokens']} tokens (${result['input_cost']})") print(f"Output: {result['output_tokens']} tokens (${result['output_cost']})") print(f"Total: ${result['total_cost']}")

Prompt Engineering for Cost Efficiency

The same task can be accomplished with dramatically different token counts depending on how you structure your prompt. Here are three proven techniques:

🔧 Technique 1: System Prompt Compression

Replace verbose system prompts with concise, structured instructions. Use role markers and bullet points instead of paragraphs. A well-compressed system prompt can reduce token usage by 40–60% while maintaining output quality.

💡 Before: 500-token verbose system prompt → After: 200-token structured prompt with identical behavior

🎯 Technique 2: Response Constraints

Explicitly control output length with max_tokens, format instructions ("respond in JSON", "limit to 3 sentences"), and structured output schemas. Every unnecessary token in the response costs 3–5× the input rate.

💡 Example: Adding "Respond in under 50 words" can reduce output from 200 to 40 tokens — an 80% cost reduction on the expensive output side

📦 Technique 3: Few-Shot Optimization

Use minimal, high-quality examples instead of many. Often 1–2 carefully chosen few-shot examples outperform 5–6 mediocre ones — at a fraction of the token cost. Consider moving examples to a fine-tuned model for zero-shot performance.

💡 Savings: Reducing from 5 examples (1,500 tokens) to 2 examples (600 tokens) saves 900 tokens per call

Practical Before/After Comparison

❌ Before Optimization (847 tokens)

  • Verbose system prompt with backstory
  • 5 few-shot examples
  • No output constraints
  • Paragraph-style instructions
  • Average output: 350 tokens
  • Cost per call: $0.0056

✅ After Optimization (312 tokens)

  • Compressed system prompt
  • 2 high-quality examples
  • max_tokens=100, JSON output
  • Bullet-point instructions
  • Average output: 80 tokens
  • Cost per call: $0.0016

That's a 71% cost reduction with equivalent output quality. At 100,000 calls per day, that's $12,000/month in savings.

Python — Smart Model Router
from openai import AzureOpenAI class CostAwareModelRouter: """Route requests to the cheapest capable model.""" ROUTING_RULES = { "classify": "gpt-4.1-nano", "extract": "gpt-4.1-nano", "summarize": "gpt-4.1-mini", "translate": "gpt-4.1-mini", "analyze": "GPT-5.1", "reason": "o3-mini", "code": "gpt-4.1", "default": "GPT-5-mini", } def __init__(self, client: AzureOpenAI): self.client = client def route(self, task_type: str, messages: list) -> dict: model = self.ROUTING_RULES.get( task_type, self.ROUTING_RULES["default"] ) response = self.client.chat.completions.create( model=model, messages=messages, max_tokens=500 ) return { "model_used": model, "response": response.choices[0].message.content, "tokens_used": response.usage.total_tokens, } # Usage router = CostAwareModelRouter(client) result = router.route("classify", [ {"role": "user", "content": "Is this email spam? 'You won $1M!'"} ]) # Uses gpt-4.1-nano instead of GPT-5.1 → 95% cheaper!

Caching Strategies

Caching identical or semantically similar requests can eliminate redundant API calls entirely. Azure AI Foundry supports multiple caching approaches:

🗄️ Exact-Match Caching

Cache responses keyed by the exact prompt hash. Simple to implement and provides 100% cache hit accuracy. Best for applications with predictable, repeated queries (e.g., FAQ bots, template-based generation).

🧠 Semantic Caching with Azure Cache for Redis

Use vector similarity search to find semantically similar past queries. Even if the exact wording differs, a cached answer for a sufficiently similar question can be returned. Requires embeddings and Azure Cache for Redis with vector search capabilities.

⚡ Azure OpenAI Prompt Caching (Built-In)

Azure OpenAI's built-in prompt caching automatically discounts repeated prompt prefixes. If the first N tokens of your prompt match a recent request, those tokens are billed at 50% of the normal input rate. This works automatically — no code changes needed.

Python — Semantic Cache with Redis
import hashlib, json, redis from openai import AzureOpenAI class SemanticCache: """Cache LLM responses with semantic similarity matching.""" def __init__(self, client: AzureOpenAI, redis_client: redis.Redis, similarity_threshold: float = 0.95): self.client = client self.redis = redis_client self.threshold = similarity_threshold def get_or_create(self, prompt: str, **kwargs) -> dict: # Check exact cache first (cheapest check) exact_key = hashlib.sha256(prompt.encode()).hexdigest() cached = self.redis.get(f"exact:{exact_key}") if cached: return {"response": json.loads(cached), "cache": "exact_hit"} # Check semantic cache via embedding similarity query_embedding = self.client.embeddings.create( model="text-embedding-3-small", input=prompt ).data[0].embedding # → Search Redis vector index for similar embeddings (FT.SEARCH) # Cache miss — call the API response = self.client.chat.completions.create( messages=[{"role": "user", "content": prompt}], **kwargs ) result = response.choices[0].message.content # Store in cache with 1-hour TTL self.redis.setex(f"exact:{exact_key}", 3600, json.dumps(result)) return {"response": result, "cache": "miss"}

Python — Capacity Planning Calculator
def plan_capacity( avg_requests_per_min: int, avg_input_tokens: int, avg_output_tokens: int, peak_multiplier: float = 2.5, growth_rate_monthly: float = 0.15, forecast_months: int = 6 ) -> dict: """Generate a capacity plan for AI Foundry deployment.""" avg_tokens = avg_input_tokens + avg_output_tokens base_tpm = avg_requests_per_min * avg_tokens peak_tpm = base_tpm * peak_multiplier buffered_tpm = int(peak_tpm * 1.2) forecasted_tpm = int( buffered_tpm * (1 + growth_rate_monthly) ** forecast_months ) return { "current_base_tpm": f"{base_tpm:,}", "current_peak_tpm": f"{int(peak_tpm):,}", "recommended_quota_tpm": f"{buffered_tpm:,}", f"{forecast_months}mo_forecast_tpm": f"{forecasted_tpm:,}", "recommendation": ( "PTU" if base_tpm > 500_000 else "Pay-as-you-go" if base_tpm < 100_000 else "Hybrid (PTU base + Pay-as-you-go burst)" ) } plan = plan_capacity( avg_requests_per_min=200, avg_input_tokens=600, avg_output_tokens=300, peak_multiplier=3.0 ) for k, v in plan.items(): print(f" {k}: {v}")

7. Mini Hack — Analyze Token Usage & Implement Cost Savings

🏆 Challenge: Optimize an AI Application's Costs by 40%+

In this hands-on exercise, you'll analyze a sample AI application's token usage, identify optimization opportunities, and implement strategies to reduce costs.

Step 1: Set Up Token Usage Tracking

Create a middleware that logs every API call's token usage and estimated cost.

Python — Usage Tracker Middleware
import json, time from datetime import datetime from openai import AzureOpenAI class UsageTracker: """Track and log all API usage for cost analysis.""" PRICING = { "GPT-5.1": {"input": 2.50, "output": 10.00}, "GPT-5-mini": {"input": 0.15, "output": 0.60}, "gpt-4.1-mini": {"input": 0.40, "output": 1.60}, "gpt-4.1-nano": {"input": 0.10, "output": 0.40}, } def __init__(self, client: AzureOpenAI): self.client = client self.log = [] self.total_cost = 0.0 def chat(self, model: str, messages: list, **kwargs): start = time.time() response = self.client.chat.completions.create( model=model, messages=messages, **kwargs ) latency = time.time() - start usage = response.usage rates = self.PRICING.get(model, {"input": 0, "output": 0}) cost = ( (usage.prompt_tokens / 1e6) * rates["input"] + (usage.completion_tokens / 1e6) * rates["output"] ) entry = { "timestamp": datetime.now().isoformat(), "model": model, "input_tokens": usage.prompt_tokens, "output_tokens": usage.completion_tokens, "cost_usd": round(cost, 6), "latency_ms": round(latency * 1000, 1) } self.log.append(entry) self.total_cost += cost return response def summary(self): print(f"Total calls: {len(self.log):,}") print(f"Total cost: ${self.total_cost:.4f}")

8. Key Takeaways

Key Takeaways

  1. Choose the right pricing model: PTU for steady high-volume workloads (40–70% savings), pay-as-you-go for variable or development workloads. Many organizations use both.
  2. Output tokens are the cost driver: At 3–5× the input price, controlling response length via max_tokens and structured output formats yields the highest cost-saving ROI.
  3. Route tasks to the cheapest capable model: A smart model router using GPT-4.1-nano for simple tasks can reduce per-call costs by 95% compared to GPT-5.1.
  4. Implement caching strategically: Exact-match caching for deterministic queries, semantic caching for FAQ patterns, and leverage built-in prompt caching for automatic 50% input discounts.
  5. Monitor proactively with budget alerts: Set multiple alert thresholds (50%, 75%, 90%, 100%) with escalating notification channels and automated remediation.
  6. Enforce governance with tags and policies: Consistent tagging enables chargeback, cost allocation, and FinOps practices. Use Azure Policy to enforce requirements organization-wide.
  7. Plan capacity ahead of demand: Use the capacity planning framework to forecast growth, select quota levels, and avoid costly HTTP 429 throttling in production.
