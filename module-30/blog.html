MLOps Maturity Model

The MLOps Maturity Model describes the progression from manual, ad-hoc AI development to fully automated, self-healing production systems. Understanding where your organization sits on this spectrum is the first step toward building better operational practices.

The Five Levels of MLOps Maturity

0

No MLOps — Manual Everything

Models trained in notebooks, manually copied to production. No version control for data or models. Deployments are "works on my machine" affairs. No monitoring, no rollback capability.

1

DevOps but No MLOps

Application code is in Git with CI/CD, but the AI/ML components are still manually managed. Model updates require manual intervention. Basic monitoring exists but doesn't track model-specific metrics like groundedness or relevance.

2

Automated Training & Deployment

CI/CD pipelines handle model packaging and deployment. Infrastructure is defined as code. Evaluation runs automatically on each commit. However, retraining and prompt optimization are still manual processes.

3

Automated Retraining & Monitoring

Data drift detection triggers automated retraining pipelines. Evaluation gates prevent regressions from reaching production. A/B testing validates changes against live traffic. Comprehensive observability across all environments.

4

Full MLOps — Self-Healing Systems

The system automatically detects quality degradation, triggers prompt re-optimization, evaluates candidates, and promotes the best version — all without human intervention. Humans review audit logs, not deployment decisions.

⚠️ Where Most Teams Sit

Most organizations are between Level 1 and Level 2. The goal of this capstone is to get you solidly to Level 2 with the foundations for Level 3. Don't try to jump to Level 4 — it requires significant organizational maturity and is overkill for most projects.

Maturity Assessment Checklist

Capability Level 0 Level 2 Level 4
Version Control None / local files Git for code + prompts Git + data versioning
Deployment Manual copy CI/CD automated Self-healing auto-deploy
Testing Manual spot checks Automated eval gates Continuous evaluation
Infrastructure Portal click-ops Bicep / Terraform GitOps + drift detection
Monitoring None Metrics + alerts Auto-remediation
Rollback Redeploy previous One-click rollback Automatic rollback

Complete GitHub Actions Workflow

This workflow implements the full pipeline — from code commit through evaluation gates to production deployment with canary rollout:

YAML — .github/workflows/mlops-pipeline.yml name: MLOps Pipeline — AI Agent on: push: branches: [main] pull_request: branches: [main] env: ACR_NAME: acragentprod IMAGE_NAME: ai-agent RESOURCE_GROUP: rg-ai-foundry-prod permissions: id-token: write contents: read jobs: # ── Stage 1: Code Quality ── lint-and-test: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - uses: actions/setup-python@v5 with: python-version: '3.12' - name: Install dependencies run: pip install -r requirements.txt -r requirements-dev.txt - name: Lint with ruff run: ruff check src/ tests/ - name: Type check with mypy run: mypy src/ --strict - name: Unit tests run: pytest tests/unit/ -v --tb=short - name: Security scan run: bandit -r src/ -f json -o security-report.json # ── Stage 2: Build & Push ── build: needs: lint-and-test runs-on: ubuntu-latest outputs: image-tag: ${{ github.sha }} steps: - uses: actions/checkout@v4 - name: Login to Azure uses: azure/login@v2 with: client-id: ${{ secrets.AZURE_CLIENT_ID }} tenant-id: ${{ secrets.AZURE_TENANT_ID }} subscription-id: ${{ secrets.AZURE_SUBSCRIPTION_ID }} - name: Build & push to ACR run: | az acr build --registry $ACR_NAME \ --image $IMAGE_NAME:${{ github.sha }} \ --image $IMAGE_NAME:latest . # ── Stage 3: Deploy to Staging ── deploy-staging: needs: build runs-on: ubuntu-latest environment: staging steps: - uses: actions/checkout@v4 - name: Login to Azure uses: azure/login@v2 with: client-id: ${{ secrets.AZURE_CLIENT_ID }} tenant-id: ${{ secrets.AZURE_TENANT_ID }} subscription-id: ${{ secrets.AZURE_SUBSCRIPTION_ID }} - name: Deploy to staging run: | az containerapp update \ --name agent-staging \ --resource-group rg-ai-staging \ --image $ACR_NAME.azurecr.io/$IMAGE_NAME:${{ github.sha }} # ── Stage 4: Evaluation Gate ── evaluate: needs: deploy-staging runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - uses: actions/setup-python@v5 with: python-version: '3.12' - name: Run AI evaluation suite run: | python evaluations/run_eval.py \ --endpoint ${{ vars.STAGING_ENDPOINT }} \ --dataset evaluations/golden-set.jsonl \ --min-groundedness 0.85 \ --min-relevance 0.80 \ --min-coherence 0.80 \ --max-latency-p95 5000 - name: Upload eval results uses: actions/upload-artifact@v4 with: name: evaluation-results path: evaluations/results/ # ── Stage 5: Production Canary ── deploy-prod: needs: evaluate runs-on: ubuntu-latest environment: production # Requires approval steps: - uses: actions/checkout@v4 - name: Login to Azure uses: azure/login@v2 with: client-id: ${{ secrets.AZURE_CLIENT_ID }} tenant-id: ${{ secrets.AZURE_TENANT_ID }} subscription-id: ${{ secrets.AZURE_SUBSCRIPTION_ID }} - name: Canary deploy (10% traffic) run: | az containerapp revision copy \ --name agent-prod \ --resource-group $RESOURCE_GROUP \ --image $ACR_NAME.azurecr.io/$IMAGE_NAME:${{ github.sha }} az containerapp ingress traffic set \ --name agent-prod \ --resource-group $RESOURCE_GROUP \ --revision-weight latest=10 - name: Validate canary (5 min) run: | python tests/canary_validation.py \ --endpoint ${{ vars.PROD_ENDPOINT }} \ --duration 300 \ --error-threshold 0.02 - name: Promote to 100% run: | az containerapp ingress traffic set \ --name agent-prod \ --resource-group $RESOURCE_GROUP \ --revision-weight latest=100
🚨 Critical: Federated Identity

This pipeline uses workload identity federation (id-token: write) instead of storing Azure credentials as secrets. This is the recommended approach — no passwords or client secrets to rotate. Configure an Entra ID app registration with a federated credential tied to your GitHub repository.

Branch Protection Rules

Your pipeline is only as strong as your branch protections. Configure these rules on the main branch:

  • Require pull request reviews — at least 1 approval before merge
  • Require status checks — lint-and-test and evaluate must pass
  • Require conversation resolution — all review comments must be resolved
  • Restrict direct pushes — no one bypasses the pipeline, not even admins
  • Require signed commits — verify commit authorship

Infrastructure as Code with Bicep

Infrastructure as Code (IaC) ensures that every environment — dev, staging, and production — is provisioned identically and repeatably. Bicep is the native IaC language for Azure, offering concise syntax, type safety, and first-class support for all Azure AI Foundry resources.

Project Structure

File Structure infra/ ├── main.bicep # Orchestrator — calls modules ├── main.bicepparam # Parameter values per environment ├── modules/ │ ├── ai-foundry.bicep # AI Foundry project + hub │ ├── container-app.bicep # Container Apps environment │ ├── acr.bicep # Azure Container Registry │ ├── monitoring.bicep # App Insights + Log Analytics │ ├── keyvault.bicep # Key Vault for secrets │ └── networking.bicep # VNet + private endpoints └── environments/ ├── dev.bicepparam ├── staging.bicepparam └── prod.bicepparam

Deploying Infrastructure per Environment

Azure CLI # Deploy dev environment az deployment group create \ --resource-group rg-ai-dev \ --template-file infra/main.bicep \ --parameters infra/environments/dev.bicepparam \ --parameters imageTag="abc123" # Deploy production environment az deployment group create \ --resource-group rg-ai-prod \ --template-file infra/main.bicep \ --parameters infra/environments/prod.bicepparam \ --parameters imageTag="abc123" # Preview changes before deploying (what-if) az deployment group what-if \ --resource-group rg-ai-prod \ --template-file infra/main.bicep \ --parameters infra/environments/prod.bicepparam
✅ Best Practice: What-If Before Deploy

Always run az deployment group what-if before applying infrastructure changes to production. This shows exactly what resources will be created, modified, or deleted — preventing accidental destruction of production resources.

Automated Evaluation Gates

Evaluation gates are the AI equivalent of unit tests — they prevent bad code (or in this case, bad model behavior) from reaching production. Unlike traditional software where tests are binary (pass/fail), AI evaluation uses quality thresholds on continuous metrics.

Core Evaluation Metrics

🎯

Groundedness

Does the response stick to the provided context? Measures hallucination rate. Threshold: ≥ 0.85 for production.

🔗

Relevance

Does the response actually answer the question asked? Measures topical accuracy. Threshold: ≥ 0.80 for production.

📐

Coherence

Is the response logically structured and easy to follow? Measures linguistic quality. Threshold: ≥ 0.80 for production.

⏱️

Latency (P95)

95th percentile response time. Ensures the agent is fast enough for real users. Threshold: < 5 seconds.

Evaluation Script

This script runs against a golden dataset and enforces quality thresholds as a CI/CD gate:

Python — evaluations/run_eval.py import json, sys, argparse, time from pathlib import Path from azure.ai.evaluation import evaluate from azure.ai.evaluation import ( GroundednessEvaluator, RelevanceEvaluator, CoherenceEvaluator, ) from azure.identity import DefaultAzureCredential import requests def run_evaluation(args): """Run evaluation suite against the staging endpoint.""" # Load golden dataset dataset = [] with open(args.dataset) as f: for line in f: dataset.append(json.loads(line)) # Collect responses from the agent results = [] latencies = [] for item in dataset: start = time.time() resp = requests.post( f"{args.endpoint}/api/chat", json={"query": item["query"], "context": item.get("context", "")} ) elapsed = (time.time() - start) * 1000 latencies.append(elapsed) results.append({ "query": item["query"], "context": item.get("context", ""), "response": resp.json()["answer"], "ground_truth": item["expected"], }) # Run AI-powered evaluators credential = DefaultAzureCredential() eval_result = evaluate( data=results, evaluators={ "groundedness": GroundednessEvaluator(credential), "relevance": RelevanceEvaluator(credential), "coherence": CoherenceEvaluator(credential), }, ) # Check thresholds metrics = eval_result.metrics p95_latency = sorted(latencies)[int(len(latencies) * 0.95)] gates = { "groundedness": (metrics["groundedness"], args.min_groundedness), "relevance": (metrics["relevance"], args.min_relevance), "coherence": (metrics["coherence"], args.min_coherence), "latency_p95_ms": (args.max_latency_p95 - p95_latency, 0), } all_passed = True for name, (actual, threshold) in gates.items(): passed = actual >= threshold status = "✅ PASS" if passed else "❌ FAIL" print(f"{status} {name}: {actual:.3f} (min: {threshold})") if not passed: all_passed = False if not all_passed: print("\n🚫 Evaluation gate FAILED — blocking deployment.") sys.exit(1) print("\n🎉 All evaluation gates passed!") if __name__ == "__main__": parser = argparse.ArgumentParser() parser.add_argument("--endpoint", required=True) parser.add_argument("--dataset", required=True) parser.add_argument("--min-groundedness", type=float, default=0.85) parser.add_argument("--min-relevance", type=float, default=0.80) parser.add_argument("--min-coherence", type=float, default=0.80) parser.add_argument("--max-latency-p95", type=float, default=5000) args = parser.parse_args() run_evaluation(args)

Golden Dataset Format

The golden dataset is a JSONL file containing representative queries, expected context, and ground truth answers:

JSONL — evaluations/golden-set.jsonl {"query": "How do I create a new AI Foundry project?", "context": "AI Foundry projects are created within hubs...", "expected": "Navigate to ai.azure.com, select your hub, and click Create Project..."} {"query": "What models are available for agents?", "context": "Azure AI Foundry supports GPT-4o, GPT-4o-mini...", "expected": "AI Foundry agents support GPT-4o, GPT-4o-mini, and other OpenAI models..."} {"query": "How do I add file search to my agent?", "context": "File search is a built-in tool in AI Foundry agents...", "expected": "Use the file_search tool type when creating your agent, and attach a vector store..."}
⚠️ Golden Dataset Maintenance

Your golden dataset should be version-controlled alongside your code. Review and update it quarterly — stale test cases lead to false confidence. Aim for 50–100 representative examples that cover edge cases, failure modes, and critical business scenarios.

A/B Testing & Canary Deployments

Even with evaluation gates, production traffic can behave differently from test datasets. A/B testing and canary deployments let you validate changes against real user traffic before full rollout.

Canary Deployment Strategy

┌─────────────────────────────────────────────────┐ │ Production Traffic │ │ │ │ │ ┌────────┴────────┐ │ │ │ Load Balancer │ │ │ └────┬───────┬───┘ │ │ │ │ │ │ ┌─────┴──┐ ┌──┴─────┐ │ │ │ v2.1 │ │ v2.2 │ │ │ │ 90% │ │ 10% │ │ │ │(stable)│ │(canary)│ │ │ └────────┘ └────────┘ │ │ │ │ Monitor error rate, latency, user feedback │ │ If canary healthy → promote to 100% │ │ If canary failing → rollback to 0% │ └─────────────────────────────────────────────────┘

Traffic Shifting with Container Apps

Azure CLI # Phase 1: Deploy canary with 10% traffic az containerapp ingress traffic set \ --name agent-prod \ --resource-group rg-ai-prod \ --revision-weight latest=10 # Phase 2: Monitor for 10 minutes, then increase to 50% az containerapp ingress traffic set \ --name agent-prod \ --resource-group rg-ai-prod \ --revision-weight latest=50 # Phase 3: Full rollout az containerapp ingress traffic set \ --name agent-prod \ --resource-group rg-ai-prod \ --revision-weight latest=100 # Emergency rollback — shift all traffic to previous revision az containerapp ingress traffic set \ --name agent-prod \ --resource-group rg-ai-prod \ --revision-weight <previous-revision-name>=100

A/B Testing for Prompt Variations

A/B testing is particularly valuable when optimizing agent prompts. Deploy two versions of your agent with different system prompts and measure which performs better on real user traffic:

Python — A/B test routing import hashlib from enum import Enum class Variant(Enum): CONTROL = "v1-baseline" TREATMENT = "v2-optimized" def assign_variant(user_id: str, experiment: str, traffic_pct: float = 0.1) -> Variant: """Deterministic A/B assignment based on user ID.""" hash_input = f"{user_id}:{experiment}" hash_val = int(hashlib.sha256( hash_input.encode() ).hexdigest(), 16) bucket = (hash_val % 100) / 100.0 if bucket < traffic_pct: return Variant.TREATMENT return Variant.CONTROL # Log which variant was served for analysis def handle_request(user_id, query): variant = assign_variant(user_id, "prompt-opt-2026-04") if variant == Variant.TREATMENT: response = call_agent_v2(query) else: response = call_agent_v1(query) log_experiment(user_id, variant.value, query, response) return response
💡 Statistical Significance

Don't declare a winner too early. Use a minimum of 1,000 requests per variant and check for statistical significance (p < 0.05) before promoting the treatment. Tools like Evan Miller's calculator help determine the required sample size for your expected effect size.

🚀 MLOps Pipeline Challenge

Design and implement a complete MLOps pipeline that takes an AI agent from code commit to production with automated evaluation gates.

Challenge Requirements

  1. GitHub Actions workflow — multi-stage pipeline with lint, build, evaluate, and deploy jobs
  2. Bicep infrastructure — at least 3 modules (AI Foundry, Container App, monitoring)
  3. Evaluation gate — script that enforces quality thresholds and blocks deployment on failure
  4. Golden dataset — minimum 20 test cases covering happy paths and edge cases
  5. Canary deployment — traffic splitting strategy with automatic rollback on failure

Bonus Objectives

✅ Success Criteria

Your pipeline should take a code change from PR to production in under 30 minutes (excluding approval wait time), with zero manual steps after the initial PR approval. A failed evaluation should automatically block the deployment and post results as a PR comment.
