Introduction & Learning Objectives

Welcome to Module 3 of our Zero to Hero journey! In the first two modules, you got oriented with Azure AI and set up your first environment. Now we pull back the curtain and examine how AI Foundry is actually built — the resource hierarchy, the networking, the identity model, and the deployment options that determine how your AI solutions behave in production.

🏠 Building Analogy

Think of AI Foundry like a modern office building. The Hub is the building itself — it has the lobby, security desk, shared conference rooms, and central utilities. Projects are the individual office suites inside the building — each team has their own space, but they share the building's infrastructure.

By the end of this module, you will be able to:

  • Diagram the complete AI Foundry resource hierarchy from Subscription down to individual model deployments
  • Explain the difference between a Hub and a Project and articulate when to use multiple of each
  • Identify all underlying Azure resources that AI Foundry provisions and manages
  • Describe networking options including public access, private endpoints, and VNet integration
  • Configure RBAC roles and Managed Identity for secure access
  • Compare Managed Compute, Serverless API (MaaS), and Provisioned deployment types
  • Connect AI Foundry projects to external data sources
  • Estimate costs for different AI Foundry configurations

💡 Technical Detail

The Hub is the ARM resource type Microsoft.MachineLearningServices/workspaces with kind Hub. Projects are the same resource type but with kind Project. Under the hood, AI Foundry builds on Azure Machine Learning workspace architecture.

Hub vs Project: Deep Dive

What Each Contains

🏢

Hub (AI Foundry Account)

Shared infrastructure & governance:

  • Storage Account
  • Key Vault
  • Application Insights
  • Container Registry
  • Networking configuration
  • Managed Identity
  • Shared connections
  • Compute instances (optional)
📁

Project

Team workspace for development:

  • Model deployments
  • Project-specific connections
  • Prompt flows
  • Evaluations
  • Data assets & indexes
  • Fine-tuning jobs
  • Playground access

When to Use Multiple Hubs vs Multiple Projects

ScenarioSolutionRationale
Different teams, same org, same regionMultiple Projects under one HubShared infra, separate workspaces
Different compliance domains (HIPAA, GDPR)Multiple HubsSeparate networking, separate Key Vault
Dev/Staging/Prod environmentsMultiple Projects OR HubsProjects for light isolation; Hubs for strict
Multi-region deploymentMultiple Hubs (one per region)Hubs are region-specific
Different billing departmentsMultiple SubscriptionsSubscription is the billing boundary

⚠️ Important

A Project can only belong to one Hub. You cannot move a Project between Hubs after creation.

Underlying Azure Resources

When you create an AI Foundry Hub, Azure automatically provisions several underlying resources. Understanding these is critical for troubleshooting, cost management, and security.

💾

Storage Account

Data, model artifacts, logs, prompt flow files. The most critical underlying resource.

🔐

Key Vault

API keys, connection strings, certificates, managed secrets.

📊

Application Insights

Telemetry, performance monitoring, error tracking, usage analytics.

📦

Container Registry

Custom container images, prompt flow environments, fine-tuned model packages.

🧠

Azure OpenAI

LLM hosting and inference. Connected via Connection, not always auto-provisioned.

📋

Log Analytics

Log aggregation workspace backing Application Insights.

⚠️ Critical

The Storage Account stores everything — uploaded data, prompt flow definitions, evaluation results, and fine-tuning datasets. If you accidentally delete the storage account, you lose all project data.

Authentication & Identity

AI Foundry uses Azure's identity platform (Microsoft Entra ID) and supports multiple authentication methods:

🔑 Managed Identity

✅ Preferred. No secrets to manage. Azure handles credential rotation automatically.

🤖 Service Principal

For CI/CD pipelines and automation. Uses client secret or certificate.

👤 User Identity

For interactive / portal access. Browser-based SSO via Entra ID.

💡 Best Practice

Use Azure AI Developer for team members who need to build and experiment. Use Contributor sparingly — only for those who need to create or modify infrastructure. Always prefer Managed Identity over API keys.

Azure OpenAI Deployment Sub-Types

Global Standard

Lowest cost. Best-effort global routing. Data may leave region. Ideal for prototyping and dev/test.

Standard

Balanced. Regional deployment. Data stays in region. Pay-per-token. Production with compliance needs.

Provisioned (PTU)

Guaranteed. Reserved capacity. Consistent latency. Monthly commitment. High-volume production.

⚠️ Data Residency

When using Global Standard deployments, your data may be processed in any Azure region globally. If you have data residency requirements (GDPR, HIPAA, etc.), use Standard or Provisioned deployments.

💡 Hub vs Project Connections

Hub-level connections are inherited by all projects — use for shared services like Azure OpenAI or AI Search. Project-level connections are isolated to one project — use for team-specific data sources.

Quick Code Example: Chat Completion

from azure.ai.inference import ChatCompletionsClient
from azure.identity import DefaultAzureCredential

client = ChatCompletionsClient(
    endpoint="https://my-project.eastus.api.azureml.ms/",
    credential=DefaultAzureCredential()
)

response = client.complete(
    model="gpt-4o",
    messages=[
        {"role": "system", "content": "You are a helpful assistant."},
        {"role": "user", "content": "What is Azure AI Foundry?"}
    ]
)
print(response.choices[0].message.content)

💡 Region Selection Tips

1) Check compliance requirements first (GDPR → West Europe). 2) Verify model availability in your target region. 3) Choose the closest region to your users for latency. 4) Compare costs — some regions are more expensive than others.

Real-World Architecture Examples

🏢 Example 1: Enterprise RAG Chatbot

Scenario: Customer support chatbot using company knowledge base.

  • Hub: hub-contoso-support → Project: proj-support-chatbot
  • Models: GPT-4o (Standard) + text-embedding-3-large
  • Connections: Azure AI Search (knowledge base), Blob Storage (documents), CosmosDB (chat history)
  • Networking: Private endpoints (production)
  • Identity: Managed Identity + RBAC
  • Cost: ~$2,000/month

🏗️ Example 2: Multi-Team AI Platform

Scenario: Central AI platform serving multiple business units.

  • Hub: hub-enterprise-ai → 4 Projects (marketing, legal, analytics, HR)
  • Shared: GPT-4o via Hub-level connection
  • Per-team: Separate data connections, flows, and evaluations
  • Networking: Managed VNet (fully private)
  • Cost: ~$8,000/month (mix of Standard + Provisioned)

🌍 Example 3: Multi-Region Deployment

Scenario: Global application with data residency requirements.

  • US Hub: hub-us-east → proj-us-app (US data stays in US)
  • EU Hub: hub-eu-west → proj-eu-app (EU data stays in EU)
  • Front Door: Azure Front Door routes users to nearest region
  • Cost: ~$6,000/month (two regions)

Summary & Key Takeaways

1️⃣ Hierarchy: Subscription → RG → Hub → Project

2️⃣ Hub = Shared infra & governance

3️⃣ Project = Team workspace for development

4️⃣ Networking: Public → PE → Managed VNet

5️⃣ Identity: Managed Identity > SP > API Keys

6️⃣ Deployments: Global → Standard → Provisioned

7️⃣ Connections bridge projects to external data

8️⃣ Cost: Hub/Project free; pay for compute + tokens + storage
