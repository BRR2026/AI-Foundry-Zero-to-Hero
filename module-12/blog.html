What is Azure AI Search?

Azure AI Search (formerly known as Azure Cognitive Search) is a fully managed cloud search service that provides rich search experiences over your content. It's designed to handle the complete search lifecycle — from ingesting and enriching data to serving fast, relevant search results at scale.

Core Capabilities

📝

Full-Text Search

Industry-standard BM25 scoring with language analyzers, phonetic matching, synonyms, and autocomplete suggestions for traditional keyword retrieval.

🧠

Semantic Ranking

Transformer-based re-ranking that understands language context and meaning, providing AI-quality relevance with extractive captions and answers.

🔢

Vector Search

Nearest-neighbor retrieval using HNSW or exhaustive KNN algorithms on embedding vectors, enabling similarity search across any content type.

🔗

AI Foundry Integration

Deep integration with Azure AI Foundry projects, enabling seamless connection between your search indexes and AI workflows like RAG and agents.

Key Architecture Concepts

Before building indexes, it helps to understand the key building blocks of Azure AI Search:

⚠️ Azure AI Search vs. Other Search Services

Azure AI Search is not a general-purpose database or a data warehouse. It's a purpose-built search engine optimized for fast retrieval. Use it as a read-optimized index fed from your primary data stores.

How Azure AI Search Fits into AI Foundry

Within Azure AI Foundry, Azure AI Search serves as the primary retrieval service for building RAG-based applications. When you create a project in the AI Foundry portal (ai.azure.com), you can connect a search index directly to your project. This allows you to:

  • Ground AI models in your own data by connecting search indexes to prompt flow and AI agents
  • Use the "Add your data" feature to quickly connect an index to a chat deployment
  • Evaluate retrieval quality using built-in evaluation tools that measure groundedness, relevance, and coherence
  • Build end-to-end RAG pipelines from data ingestion through search to generated responses

Understanding Embeddings & Vectors

At the heart of vector search lies a deceptively simple idea: represent text (or images, audio, etc.) as numerical vectors, then find similar items by comparing their vectors. These numerical representations are called embeddings.

What Are Embeddings?

An embedding is a dense numerical vector — typically hundreds or thousands of floating-point numbers — that captures the semantic meaning of a piece of content. Embedding models like Azure OpenAI's text-embedding-ada-002 (1536 dimensions) or text-embedding-3-large (up to 3072 dimensions) are trained on vast amounts of text to learn these representations.

The key insight is that semantically similar content produces similar vectors. "How to make a cake" and "baking recipe instructions" would have embeddings that are close together in vector space, even though they share very few keywords.

"Azure AI Search is great" → [0.023, -0.041, 0.087, ..., 0.012] (1536 dims) "Search service in Azure" → [0.021, -0.038, 0.091, ..., 0.015] (1536 dims) "I like chocolate cake" → [0.156, 0.203, -0.044, ..., -0.089] (1536 dims) Cosine similarity: "AI Search is great" ↔ "Search service" = 0.94 ✅ Similar "AI Search is great" ↔ "chocolate cake" = 0.23 ❌ Different

Similarity Metrics

When comparing vectors, Azure AI Search supports several distance/similarity metrics:

MetricDescriptionBest For
CosineMeasures the angle between vectors. Normalized, so vector magnitude doesn't matter.Text embeddings (default)
Dot ProductMeasures both direction and magnitude. Faster to compute.Pre-normalized vectors
EuclideanMeasures straight-line distance between points in vector space.When magnitude matters
✅ Recommendation

For most text-based retrieval scenarios with Azure OpenAI embeddings, cosine similarity is the recommended metric. It's the default in Azure AI Search's HNSW configuration and works well with all major embedding models.

Choosing an Embedding Model

Azure OpenAI provides several embedding models optimized for different use cases:

ModelDimensionsMax TokensUse Case
text-embedding-ada-00215368,191General-purpose text embedding
text-embedding-3-small512–15368,191Cost-optimized, flexible dimensions
text-embedding-3-large256–30728,191Highest quality, configurable dimensions

The text-embedding-3 family supports Matryoshka Representation Learning (MRL), allowing you to truncate vectors to lower dimensions while retaining most of the semantic information. This gives you a tunable trade-off between quality and cost/performance.

Creating Vector Indexes

A vector index in Azure AI Search is simply a search index that includes one or more fields of type Collection(Edm.Single) configured for vector search. Let's walk through the process of creating one.

Step 1: Configure Vector Search Algorithms

Azure AI Search supports two vector search algorithms:

AlgorithmTypeSpeedRecallBest For
HNSWApproximate (ANN)FastHighProduction workloads (recommended)
Exhaustive KNNExactSlowerPerfectSmall datasets, ground truth testing

HNSW (Hierarchical Navigable Small World) is the recommended algorithm for production use. It builds a multi-layered graph structure that enables fast approximate nearest-neighbor search with high recall.

HNSW Parameters

ParameterDefaultDescription
m4Bi-directional links per node. Higher = better recall but more memory.
efConstruction400Neighbors evaluated during index building. Higher = better quality but slower indexing.
efSearch500Neighbors evaluated during search. Higher = better recall but slower queries.
metriccosineSimilarity metric: cosine, dotProduct, or euclidean.

Step 2: Define the Index Schema with Python SDK

Python # Install: pip install azure-search-documents azure-identity from azure.search.documents.indexes import SearchIndexClient from azure.search.documents.indexes.models import ( SearchIndex, SearchField, SearchFieldDataType, SimpleField, SearchableField, VectorSearch, HnswAlgorithmConfiguration, VectorSearchProfile, SemanticConfiguration, SemanticSearch, SemanticPrioritizedFields, SemanticField, ) from azure.core.credentials import AzureKeyCredential # Define fields fields = [ SimpleField(name="id", type=SearchFieldDataType.String, key=True, filterable=True), SearchableField(name="title", type=SearchFieldDataType.String, analyzer_name="en.microsoft"), SearchableField(name="content", type=SearchFieldDataType.String, analyzer_name="en.microsoft"), SimpleField(name="category", type=SearchFieldDataType.String, filterable=True, facetable=True), SearchField( name="content_vector", type=SearchFieldDataType.Collection(SearchFieldDataType.Single), searchable=True, vector_search_dimensions=1536, vector_search_profile_name="hnsw-profile", ), ] # Configure vector search vector_search = VectorSearch( algorithms=[HnswAlgorithmConfiguration(name="hnsw-config")], profiles=[ VectorSearchProfile( name="hnsw-profile", algorithm_configuration_name="hnsw-config", ) ], ) # Configure semantic search semantic_config = SemanticConfiguration( name="semantic-config", prioritized_fields=SemanticPrioritizedFields( title_field=SemanticField(field_name="title"), content_fields=[SemanticField(field_name="content")], ), ) # Create the index index = SearchIndex( name="my-vector-index", fields=fields, vector_search=vector_search, semantic_search=SemanticSearch(configurations=[semantic_config]), ) client = SearchIndexClient( endpoint="https://<your-service>.search.windows.net", credential=AzureKeyCredential("<your-admin-key>"), ) result = client.create_or_update_index(index) print(f"Created index: {result.name}")

Index Schema Design Best Practices

A well-designed index schema is the foundation of effective retrieval. The schema determines not just what data you store, but how effectively it can be searched, filtered, and ranked.

Field Types Reference

Field TypePurposeSearchableFilterableVector
SearchableFieldFull-text search with BM25✅Optional❌
SimpleFieldFilters, sorting, faceting❌✅❌
SearchField (vector)Vector similarity search✅❌✅
ComplexFieldNested/structured objectsVariesVaries❌

Design Principles

✅

Do: Separate Text & Vectors

Keep your content text field and content_vector as separate fields. This enables hybrid search where both BM25 and vector retrieval contribute to ranking.

✅

Do: Use Language Analyzers

Apply language-specific analyzers like en.microsoft to text fields. They handle stemming, stop words, and linguistic normalization for better recall.

⚠️

Consider: Multiple Vector Fields

For complex scenarios, you can have multiple vector fields — one for title embeddings, another for content embeddings, or different embedding models entirely.

❌

Don't: Over-index Fields

Making every field searchable, filterable, sortable, and facetable wastes storage and slows indexing. Only enable attributes you'll actually use in queries.

Chunking Considerations

When your source documents are long (pages, PDFs, articles), you need to split them into smaller chunks before embedding. Each chunk becomes a separate document in your index. Key considerations:

  • Chunk size: 200–800 tokens is typical. Too small loses context; too large dilutes relevance.
  • Overlap: Add 50–100 tokens of overlap between chunks to preserve context at boundaries.
  • Parent tracking: Include a parent_id or source_url field so you can trace chunks back to their source document.
  • Metadata: Store page numbers, section headers, or document titles as filterable fields on each chunk.

Hybrid Search: Keyword + Semantic + Vector

Hybrid search is where Azure AI Search truly shines. Instead of choosing between keyword search and vector search, you use both simultaneously — and optionally add semantic ranking as a third layer of intelligence.

Why Hybrid?

Each search approach has strengths and weaknesses:

ApproachStrengthsWeaknesses
Keyword (BM25)Exact term matching, great for names/codes/IDsMisses synonyms and semantic meaning
VectorUnderstands meaning, handles synonyms naturallyCan miss exact terms, needs embedding model
Semantic RankingDeep language understanding, extractive answersAdds latency, only re-ranks top results

By combining them, you get the best of all worlds: exact matching when it matters, semantic understanding when keywords fail, and intelligent re-ranking for the final results.

How It Works: Reciprocal Rank Fusion (RRF)

User Query: "How does RAG reduce hallucinations?" │ ┌───────────┼───────────────┐ ▼ ▼ ▼ ┌─────────┐ ┌──────────┐ ┌────────────┐ │ Keyword │ │ Vector │ │ (Optional) │ │ BM25 │ │ HNSW │ │ Semantic │ │ Ranking │ │ Nearest │ │ Reranker │ │ │ │ Neighbor │ │ │ └────┬─────┘ └─────┬─────┘ └──────┬──────┘ │ │ │ └──────┬───────┘ │ ▼ │ ┌──────────────────┐ │ │ Reciprocal Rank │◄────────────┘ │ Fusion (RRF) │ └────────┬──────────┘ ▼ ┌──────────────────┐ │ Merged, Ranked │ │ Results 🎯 │ └──────────────────┘

Reciprocal Rank Fusion (RRF) merges ranked lists from different retrieval methods. For each document, RRF calculates a score based on its position in each list: score = Σ 1/(k + rank), where k is a constant (default 60). Documents appearing highly in multiple lists get boosted.

Hybrid Search Query Example

Python from azure.search.documents import SearchClient from azure.search.documents.models import VectorizedQuery # Assume query_embedding is already generated results = search_client.search( # 1️⃣ Keyword search component search_text="How does RAG reduce hallucinations?", # 2️⃣ Vector search component vector_queries=[ VectorizedQuery( vector=query_embedding, k_nearest_neighbors=5, fields="content_vector", ) ], # 3️⃣ Semantic ranking (re-ranks merged results) query_type="semantic", semantic_configuration_name="semantic-config", # Standard parameters select="title,content,category", top=10, ) for r in results: print(f"{r['title']} — Score: {r['@search.score']:.4f}") print(f" Reranker: {r.get('@search.reranker_score', 'N/A')}")
💡 Pro Tip: Use Filters with Hybrid Search

Add filter="category eq 'Documentation'" to narrow results before ranking. Pre-filtering reduces the search space and improves both performance and relevance. Filters are applied before the BM25 and vector scoring.

Integrated Vectorization

Traditionally, building a vector search pipeline requires you to manage the embedding process yourself: call the embedding API for each document during indexing, store the resulting vectors, and embed every query at search time. Integrated vectorization simplifies this by letting Azure AI Search handle embedding generation automatically.

How Integrated Vectorization Works

When you configure integrated vectorization, you define a vectorizer within the index definition. This vectorizer specifies which embedding model to use (e.g., an Azure OpenAI deployment). Azure AI Search then:

  1. At index time: Automatically calls the embedding model to generate vectors for text content as documents are indexed (via skillsets).
  2. At query time: Automatically embeds the user's search text before performing vector similarity search.

Manual vs. Integrated Comparison

AspectManual PipelineIntegrated Vectorization
Index-time embeddingYour code calls embedding APISkillset calls embedding API
Query-time embeddingYour code embeds the queryVectorizer embeds automatically
Code complexityMore code to write/maintainLess code, simpler architecture
ControlFull control over embedding processLess control, managed by service
Embedding modelAny model you chooseMust be Azure OpenAI or supported model
Cost transparencyYou see each API callCalls are made on your behalf

Configuring a Vectorizer

Python from azure.search.documents.indexes.models import ( AzureOpenAIVectorizer, AzureOpenAIVectorizerParameters, ) # Define the vectorizer vectorizer = AzureOpenAIVectorizer( vectorizer_name="my-openai-vectorizer", parameters=AzureOpenAIVectorizerParameters( resource_url="https://<your-openai>.openai.azure.com", deployment_name="text-embedding-ada-002", model_name="text-embedding-ada-002", ), ) # Add to vector search config vector_search = VectorSearch( algorithms=[HnswAlgorithmConfiguration(name="hnsw-config")], profiles=[ VectorSearchProfile( name="hnsw-profile", algorithm_configuration_name="hnsw-config", vectorizer_name="my-openai-vectorizer", # Link the vectorizer ) ], vectorizers=[vectorizer], )

With this configuration, you can send plain text queries and Azure AI Search will automatically embed them before performing vector search:

Python from azure.search.documents.models import VectorizableTextQuery # No need to call embedding API yourself! results = search_client.search( search_text="How does RAG work?", vector_queries=[ VectorizableTextQuery( text="How does RAG work?", # Plain text, not a vector k_nearest_neighbors=5, fields="content_vector", ) ], query_type="semantic", semantic_configuration_name="semantic-config", )
⚠️ When to Use Each Approach

Use integrated vectorization when you want a simpler architecture and your embedding model is hosted on Azure OpenAI. Use manual pipelines when you need custom preprocessing, non-Azure embedding models, or fine-grained control over the embedding process.

Python # Hybrid search with category filter and scoring profile results = search_client.search( search_text="best practices for chunking documents", vector_queries=[ VectorizedQuery( vector=query_embedding, k_nearest_neighbors=5, fields="content_vector", ) ], filter="category eq 'Best Practices'", query_type="semantic", semantic_configuration_name="semantic-config", top=10, include_total_count=True, ) print(f"Total matching documents: {results.get_count()}") for result in results: print(f" {result['title']}") print(f" Score: {result['@search.score']}") # Semantic captions (if available) captions = result.get("@search.captions") if captions: print(f" Caption: {captions[0].text}")

Performance & Scaling

As your vector index grows and query volume increases, you'll need to think about performance and scaling. Azure AI Search provides several mechanisms to handle production workloads.

Replicas and Partitions

ResourcePurposeScaling Effect
ReplicasCopies of your index for concurrent queriesMore replicas = higher query throughput
PartitionsDivisions of your index across storage unitsMore partitions = larger index capacity
✅ HA Recommendations

2 replicas for read high-availability. 3 replicas for read/write high-availability (required for SLA). Max replicas and partitions vary by tier.

Vector Index Size Estimation

Each vector in your index consumes memory. Estimate your index size:

Formula Vector Memory = num_documents × dimensions × 4 bytes (float32) Example: 1 million docs × 1536 dims × 4 bytes = 6,144,000,000 bytes ≈ 5.7 GB of vector data alone HNSW graph overhead adds ~10-30% depending on m parameter Total estimate: ~6.5-7.5 GB for vectors + HNSW graph

Performance Optimization Tips

  • Use filters aggressively — Pre-filter narrows the vector search space, reducing latency significantly.
  • Right-size your dimensions — If using text-embedding-3-small, consider 512 dims instead of 1536 when quality is acceptable.
  • Tune HNSW parameters — Lower efSearch for faster queries (at slight recall cost). Higher for better recall.
  • Monitor with Azure Monitor — Track query latency, throttling, document count, and storage usage.
  • Use caching — For repeated queries, cache results at the application layer.
  • Optimize chunk sizes — Fewer, larger chunks mean fewer vectors, but potentially lower retrieval precision.

Key Takeaways

🔍

Azure AI Search is the Retrieval Backbone

A unified platform for keyword, semantic, and vector search — deeply integrated with Azure AI Foundry for end-to-end AI applications.

📐

Schema Design Matters

Pair SearchableFields with vector fields. Use language analyzers, semantic configs, and filterable metadata for optimal retrieval.

⚡

Hybrid Search is the Gold Standard

Combine BM25 keyword scoring + vector similarity + semantic re-ranking with RRF for the best retrieval quality.

🔧

Integrated Vectorization Simplifies Architecture

Let Azure AI Search handle embedding at index and query time. Less code, fewer moving parts, easier maintenance.

📊

Scale with Replicas & Partitions

Replicas for throughput, partitions for capacity. Monitor with Azure Monitor and size your tier based on vector storage needs.

🎯

Embeddings Are the Foundation

Choose the right model and dimensions. Cosine similarity with text-embedding models is the default go-to for text retrieval.
