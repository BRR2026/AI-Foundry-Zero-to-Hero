Introduction

Single-agent systems are powerful, but they hit a ceiling when tasks require diverse expertise, parallel processing, or iterative quality improvement. Multi-agent systems break through that ceiling by coordinating multiple specialized agents — each with its own instructions, tools, and domain focus — into a cohesive workflow.

In this module, you'll learn how to design, build, and orchestrate multi-agent architectures using Azure AI Foundry. We'll cover the three foundational orchestration patterns — Supervisor, Router, and Swarm — and dive deep into agent-to-agent communication, state management, and fault tolerance strategies that keep complex workflows running smoothly in production.

💡 What You'll Learn:
By the end of this module, you will be able to design multi-agent topologies, implement agent-to-agent handoffs, build supervisor orchestration logic, handle errors across distributed agent workflows, and construct a complete 3-agent system from scratch.

Why Multi-Agent Systems?

Modern enterprise scenarios demand capabilities that stretch beyond what a single agent can deliver. Multi-agent systems address these challenges through specialization, parallelism, and quality control loops.

🎯 Specialization

Each agent focuses on a narrow domain — research, writing, code review, data analysis — with tailored system prompts, tool access, and model configurations. Specialists outperform generalists on complex tasks.

⚡ Parallelism

Independent subtasks run simultaneously across multiple agents, dramatically reducing end-to-end latency. A research agent can gather data while a planning agent outlines the approach.

🔍 Quality Gates

Reviewer agents evaluate outputs from producer agents, creating iterative feedback loops that systematically improve quality before results reach the user.

🔧 Modularity

Swap, upgrade, or retrain individual agents without disrupting the entire system. Each agent is an independent unit with a clean interface contract.

Dimension Single Agent Multi-Agent System
Complexity Ceiling Limited by context window & prompt size Scales with number of specialized agents
Latency Sequential processing only Parallel execution of independent tasks
Error Recovery Retry entire workflow on failure Isolate and retry failed agent only
Quality Assurance Self-review (limited effectiveness) Independent reviewer agents
Maintainability Monolithic prompt engineering Modular agent definitions
Cost Optimization One model for all tasks Right-size models per agent role
⚠️ When NOT to Use Multi-Agent:
Not every problem needs multiple agents. If your task is well-defined, can be handled by a single prompt, or doesn't benefit from specialization — a single agent with good tools is simpler, cheaper, and faster to maintain. Multi-agent architectures add coordination overhead that must be justified by the complexity of the problem.

Multi-Agent Architecture Patterns

Azure AI Foundry supports three fundamental orchestration topologies, each suited to different coordination requirements. Understanding these patterns is the first step to designing effective multi-agent systems.

🎛️ Supervisor

A central orchestrator delegates tasks to worker agents, collects their results, and makes routing decisions. Best for structured workflows with predictable stages.

🔀 Router

An intelligent classifier analyzes incoming requests and routes each to the most appropriate specialist agent. Best for intent-based dispatching with diverse request types.

🐝 Swarm

Agents communicate peer-to-peer, passing context and handoff signals to each other without a central controller. Best for dynamic, emergent collaboration.

Agent-to-Agent Communication

The foundation of any multi-agent system is how agents share information. Azure AI Foundry supports several communication patterns, each with distinct tradeoffs.

Message Passing

Agents communicate by sending structured messages through a shared message bus or direct invocation. Each message carries a payload, sender identity, and optional metadata.

# Structured message format for agent-to-agent communication from dataclasses import dataclass, field from typing import Any, Optional from datetime import datetime @dataclass class AgentMessage: sender: str # Agent ID of the sender recipient: str # Target agent ID or "broadcast" content: str # The message payload msg_type: str # "task", "result", "feedback", "handoff" metadata: dict = field(default_factory=dict) timestamp: str = field( default_factory=lambda: datetime.utcnow().isoformat() ) # Example: Supervisor assigns a research task task_msg = AgentMessage( sender="supervisor", recipient="researcher", content="Research the latest advances in multi-modal AI agents.", msg_type="task", metadata={"priority": "high", "max_sources": 5} )

Shared State

For workflows where multiple agents need to read and update the same context, a shared state store provides a centralized workspace. This is particularly useful for iterative refinement cycles.

# Shared state store for multi-agent collaboration class WorkflowState: """Thread-safe shared state for multi-agent workflows.""" def __init__(self): self.data = { "research_results": None, "draft_content": None, "review_feedback": [], "revision_count": 0, "status": "initialized", "agent_logs": [], } def update(self, key: str, value: Any, agent_id: str): self.data[key] = value self.data["agent_logs"].append({ "agent": agent_id, "action": f"updated {key}", "timestamp": datetime.utcnow().isoformat() }) def get(self, key: str) -> Any: return self.data.get(key)

Handoff Protocol

Agent handoffs transfer control and context from one agent to another. A well-designed handoff includes the current state, the reason for the transfer, and any constraints the receiving agent should respect.

# Handoff protocol between agents @dataclass class AgentHandoff: from_agent: str to_agent: str context: dict # Accumulated state to pass along reason: str # Why the handoff is happening constraints: list = field(default_factory=list) # Example: Researcher hands off to Writer handoff = AgentHandoff( from_agent="researcher", to_agent="writer", context={ "topic": "Multi-modal AI agents", "sources": ["arxiv:2401.1234", "arxiv:2402.5678"], "key_findings": ["Finding 1...", "Finding 2..."], }, reason="Research complete. Ready for content drafting.", constraints=["Keep under 1500 words", "Include citations"] )

Supervisor Pattern Deep Dive

The Supervisor pattern places a central orchestrator agent in charge of the workflow. The supervisor breaks tasks into subtasks, dispatches them to specialized worker agents, evaluates their outputs, and decides the next step. This is the most common pattern for structured, multi-step workflows.

Pythonfrom azure.ai.projects import AIProjectClient from azure.identity import DefaultAzureCredential from azure.ai.agents import Agent # Initialize the Azure AI Foundry project client client = AIProjectClient( credential=DefaultAzureCredential(), project_endpoint="https://<your-project>.services.ai.azure.com/api" ) class SupervisorOrchestrator: """Coordinates a team of specialized agents.""" def __init__(self, client: AIProjectClient): self.client = client self.agents = {} self.state = WorkflowState() def register_agent(self, role: str, agent: Agent): """Register a specialist agent under a role name.""" self.agents[role] = agent def dispatch(self, role: str, task: str) -> str: """Send a task to a specific agent and get the response.""" agent = self.agents[role] thread = self.client.agents.threads.create() # Send the task as a user message self.client.agents.messages.create( thread_id=thread.id, role="user", content=task ) # Run the agent and wait for completion run = self.client.agents.runs.create_and_process( thread_id=thread.id, agent_id=agent.id ) # Extract the response messages = self.client.agents.messages.list(thread_id=thread.id) return messages.data[0].content[0].text.value async def run_workflow(self, user_request: str): """Execute the full supervisor workflow.""" # Step 1: Research self.state.update("status", "researching", "supervisor") research = self.dispatch( "researcher", f"Research the following topic thoroughly:\n{user_request}" ) self.state.update("research_results", research, "researcher") # Step 2: Write draft based on research self.state.update("status", "drafting", "supervisor") draft = self.dispatch( "writer", f"Write a comprehensive article based on:\n{research}" ) self.state.update("draft_content", draft, "writer") # Step 3: Review and iterate self.state.update("status", "reviewing", "supervisor") review = self.dispatch( "reviewer", f"Review this draft for accuracy and quality:\n{draft}" ) self.state.update("review_feedback", review, "reviewer") # Step 4: Revise if needed if "APPROVED" not in review: revised = self.dispatch( "writer", f"Revise this draft based on feedback:\n" f"Draft:\n{draft}\n\nFeedback:\n{review}" ) self.state.update("draft_content", revised, "writer") self.state.update("revision_count", self.state.get("revision_count") + 1, "supervisor") self.state.update("status", "complete", "supervisor") return self.state.get("draft_content")
✅ Supervisor Pattern Best Practices:
• Keep the supervisor's system prompt focused on routing and evaluation, not domain tasks
• Set a maximum revision count to avoid infinite loops
• Log every dispatch and result for debugging and tracing
• Use a smaller, faster model for the supervisor (e.g., GPT-5-mini) since it primarily makes routing decisions

Router Pattern Deep Dive

The Router pattern uses an intelligent classifier to direct incoming requests to the most appropriate specialist agent. Unlike the Supervisor, which orchestrates a multi-step workflow, the Router performs a single-step dispatch based on intent classification.

Pythonclass RouterOrchestrator: """Routes requests to the best-fit specialist agent.""" def __init__(self, client: AIProjectClient): self.client = client self.routes = {} self.router_agent = self._create_router() def _create_router(self) -> Agent: """Create the routing classifier agent.""" return self.client.agents.create_agent( model="GPT-5-mini", name="RequestRouter", instructions="""You are a request classifier. Analyze the user's message and respond with EXACTLY one of these category names: - BILLING: payment, invoice, subscription questions - TECH_SUPPORT: technical issues, errors, bugs - GENERAL: everything else Respond with ONLY the category name, nothing else.""" ) def register_route(self, category: str, agent: Agent): self.routes[category] = agent def classify_and_route(self, user_message: str) -> str: """Classify the request and route to the right agent.""" # Step 1: Classify the intent category = self._classify(user_message).strip() # Step 2: Route to specialist (with fallback) if category in self.routes: return self._invoke_agent( self.routes[category], user_message ) else: # Fallback to general agent return self._invoke_agent( self.routes["GENERAL"], user_message )
💡 Router vs. Supervisor:
Use a Router when requests are independent and each one maps to a single specialist. Use a Supervisor when a single request requires coordination across multiple agents in sequence. In practice, many systems combine both — a Router at the entry point and Supervisors within each specialist domain.

Swarm Pattern Deep Dive

The Swarm pattern enables peer-to-peer agent collaboration without a central controller. Each agent decides independently when to hand off control to another agent, based on its own assessment of the task. This pattern excels in dynamic scenarios where the optimal workflow isn't known in advance.

Pythonclass SwarmAgent: """An agent that can autonomously hand off to peers.""" def __init__(self, name: str, agent: Agent, handoff_targets: list): self.name = name self.agent = agent self.handoff_targets = handoff_targets class SwarmOrchestrator: """Decentralized agent swarm with handoff support.""" def __init__(self, client: AIProjectClient, max_handoffs: int = 10): self.client = client self.swarm_agents = {} self.max_handoffs = max_handoffs def register(self, swarm_agent: SwarmAgent): self.swarm_agents[swarm_agent.name] = swarm_agent def run(self, start_agent: str, message: str) -> str: current = start_agent context = message handoff_count = 0 while handoff_count < self.max_handoffs: agent = self.swarm_agents[current] response = self._invoke(agent, context) # Check for handoff signal in response next_agent = self._extract_handoff(response) if next_agent and next_agent in agent.handoff_targets: context = response current = next_agent handoff_count += 1 else: return response # No handoff = final answer return "Max handoffs reached. Returning last response."
⚠️ Swarm Safety:
Always set a max_handoffs limit to prevent infinite loops where agents keep passing tasks back and forth. Monitor handoff chains in production to detect unhealthy cycling patterns.

Error Handling in Multi-Agent Workflows

Multi-agent systems introduce failure modes that don't exist in single-agent applications. Each agent can fail independently, and a failure in one agent may cascade through the entire workflow. Robust error handling is essential for production multi-agent systems.

Failure Categories

Failure Type Description Mitigation Strategy
Agent Timeout Agent takes too long to respond Per-agent timeouts with fallback agent
Model Error API rate limit, service unavailable Exponential backoff with jitter, retry queue
Quality Failure Agent produces low-quality or off-topic output Validation checks, reviewer agents, auto-retry
Handoff Failure Agent hands off to non-existent or unavailable agent Handoff validation, fallback routing table
State Corruption Shared state becomes inconsistent Immutable state snapshots, transaction logs
Cascade Failure One agent's failure breaks downstream agents Circuit breakers, graceful degradation paths

Resilient Workflow Wrapper

Pythonimport asyncio import logging from tenacity import retry, stop_after_attempt, wait_exponential logger = logging.getLogger("multi_agent") class ResilientOrchestrator: """Multi-agent orchestrator with comprehensive error handling.""" def __init__(self, client, max_retries=3, timeout_seconds=120): self.client = client self.max_retries = max_retries self.timeout = timeout_seconds self.circuit_breakers = {} @retry( stop=stop_after_attempt(3), wait=wait_exponential(multiplier=1, max=30) ) async def safe_dispatch(self, agent_id: str, task: str) -> str: """Dispatch with timeout, retry, and circuit breaker.""" # Check circuit breaker if self._is_circuit_open(agent_id): logger.warning(f"Circuit open for {agent_id}, using fallback") return await self._fallback_response(agent_id, task) try: result = await asyncio.wait_for( self._dispatch_to_agent(agent_id, task), timeout=self.timeout ) # Validate response quality if not self._validate_response(result): raise ValueError("Response failed quality check") self._record_success(agent_id) return result except asyncio.TimeoutError: logger.error(f"Agent {agent_id} timed out") self._record_failure(agent_id) raise except Exception as e: logger.error(f"Agent {agent_id} failed: {e}") self._record_failure(agent_id) raise def _is_circuit_open(self, agent_id: str) -> bool: """Check if an agent's circuit breaker is tripped.""" failures = self.circuit_breakers.get(agent_id, 0) return failures >= 5 # Open after 5 consecutive failures
❌ Common Anti-Patterns:
• No timeout: An agent hangs indefinitely, blocking the entire workflow
• Silent failures: Swallowing exceptions without logging or fallback
• Retry storms: All agents retrying simultaneously, overwhelming the API
• Missing circuit breakers: Continuously dispatching to a failed agent

Implementation Guide: Agent Definitions

Let's define the three specialist agents for our Researcher → Writer → Reviewer pipeline using the Azure AI Foundry Agent SDK.

Creating Specialist Agents

Pythonfrom azure.ai.projects import AIProjectClient from azure.identity import DefaultAzureCredential client = AIProjectClient( credential=DefaultAzureCredential(), project_endpoint="https://<your-project>.services.ai.azure.com/api" ) # 🔬 Researcher Agent — Gathers and synthesizes information researcher = client.agents.create_agent( model="GPT-5.1", name="ResearchAgent", instructions="""You are a meticulous research analyst. Your job is to: 1. Break down the research topic into key questions 2. Gather comprehensive information from available sources 3. Synthesize findings into a structured research brief 4. Cite sources and flag uncertainty levels Output format: ## Key Findings - Finding 1 (confidence: high/medium/low) - Finding 2 ... ## Sources - Source 1: [description] ## Open Questions - Questions that need further investigation""", tools=[{"type": "bing_grounding"}] ) # ✍️ Writer Agent — Creates polished content from research writer = client.agents.create_agent( model="GPT-5.1", name="WriterAgent", instructions="""You are a professional technical writer. Your job is to: 1. Transform research findings into clear, engaging content 2. Structure the article with logical flow and headings 3. Use concrete examples and analogies 4. Maintain accuracy — never embellish beyond the research Style guidelines: - Professional but accessible tone - Short paragraphs (3-4 sentences max) - Use bullet points for lists of 3+ items - Include a TL;DR summary at the top""" ) # 📝 Reviewer Agent — Quality assurance and feedback reviewer = client.agents.create_agent( model="GPT-5.1", name="ReviewerAgent", instructions="""You are a senior content reviewer. Evaluate the submitted draft on these criteria: 1. **Accuracy**: Are claims supported by the research? 2. **Clarity**: Is the writing clear and jargon-free? 3. **Completeness**: Are key topics adequately covered? 4. **Structure**: Is the flow logical and well-organized? Scoring: Rate each criterion 1-5. If ALL scores are 4+, respond with "APPROVED" at the end. Otherwise, provide specific, actionable revision suggestions. Format: ## Review Scores - Accuracy: X/5 - Clarity: X/5 - Completeness: X/5 - Structure: X/5 ## Feedback [Specific suggestions for improvement] ## Verdict APPROVED or REVISION_NEEDED""" )

Mini Hack: Build a 3-Agent System

🛠️ Objective

Build a complete Researcher → Writer → Reviewer multi-agent pipeline using Azure AI Foundry. The system should research a topic, produce a polished article, and iterate based on reviewer feedback until the article is approved.

Prerequisites

  • Azure AI Foundry project with GPT-5.1 deployed
  • Python 3.10+ with azure-ai-projects and azure-identity packages
  • Azure CLI authenticated (az login)
  • Bing Grounding connection configured in your project (for the researcher)

Steps

  1. Set up the project: Create a new Python file and install dependencies:
    pip install azure-ai-projects azure-identity
  2. Define the three agents: Create Researcher (with Bing Grounding tool), Writer, and Reviewer agents with the system prompts shown in the Implementation Guide above
  3. Build the orchestrator: Implement the SupervisorOrchestrator class with dispatch, state tracking, and revision loop logic
  4. Add error handling: Wrap each dispatch call with timeout (120s), retry (3 attempts), and quality validation
  5. Run the pipeline: Test with a topic like "Impact of multi-agent systems on enterprise software development"
  6. Observe the revision loop: Log each agent interaction and track how many revision cycles occur before the reviewer approves
  7. Clean up: Delete all created agents after the run completes

Success Criteria

  • ✅ All three agents are created and registered correctly
  • ✅ The research agent produces structured findings with sources
  • ✅ The writer transforms research into a well-formatted article
  • ✅ The reviewer provides scored feedback and drives at least one revision
  • ✅ The workflow completes within 3 revision cycles
  • ✅ All agents are cleaned up after the run

Bonus Challenges

  • 🌟 Add a 4th agent — an Editor that polishes the final approved draft
  • 🌟 Implement parallel research with two researcher agents (breadth vs. depth)
  • 🌟 Add observability by logging all messages to Application Insights
  • 🌟 Convert the supervisor to a swarm pattern where agents decide handoffs autonomously

Key Takeaways

🏗️ Three Core Patterns

Supervisor for structured workflows, Router for intent-based dispatch, and Swarm for dynamic peer-to-peer collaboration. Each pattern suits different coordination requirements.

💬 Communication Matters

Choose between message passing (loose coupling), shared state (rich context), and handoff protocols (explicit transfers) based on your workflow's needs.

🛡️ Design for Failure

Multi-agent systems need timeouts, circuit breakers, retry policies, and fallback paths. Every agent is a potential failure point — plan accordingly.

📐 Right-Size Your Agents

Use smaller models for routing/classification and larger models for generation. Specialize agent prompts narrowly — each agent should do one thing exceptionally well.
