### Question 1

**When integrating an Azure AI Foundry agent with Microsoft Teams, which Azure service acts as the messaging relay between Teams and your bot application?**

- A) Azure API Management
- B) Azure Event Grid
- C) Azure Bot Service
- D) Azure SignalR Service

<details>
<summary>✅ Answer</summary>

**C) Azure Bot Service**

Azure Bot Service serves as the messaging relay between Microsoft Teams and your bot application. It handles the Bot Framework protocol, manages channel registrations (including Teams), and routes activities (messages, card actions) between the client and your bot backend.

</details>

### Question 2

**Why is it critical to map Teams conversation IDs to AI Foundry thread IDs in a persistent store?**

- A) To reduce Azure costs by reusing API connections
- B) To maintain multi-turn conversation context across messages
- C) To comply with Microsoft Teams data residency requirements
- D) To enable real-time typing indicators in the chat

<details>
<summary>✅ Answer</summary>

**B) To maintain multi-turn conversation context across messages**

AI Foundry agents use threads to maintain conversation state. Without mapping the Teams conversation ID to a persistent AI Foundry thread ID, each message would start a new thread and the agent would lose all previous context, making follow-up questions impossible.

</details>

### Question 3

**You are building a Power Automate flow that invokes an AI Foundry agent. What is the recommended architecture?**

- A) Power Automate → Direct AI Foundry API call with connection string in the flow
- B) Power Automate → Custom Connector → Azure Function (proxy) → AI Foundry Agent
- C) Power Automate → Azure Logic App → AI Foundry Agent
- D) Power Automate → Power Virtual Agent → AI Foundry Agent

<details>
<summary>✅ Answer</summary>

**B) Power Automate → Custom Connector → Azure Function (proxy) → AI Foundry Agent**

The recommended pattern uses a custom connector backed by an Azure Function proxy. The Function authenticates with AI Foundry using managed identity, keeping credentials server-side. Embedding connection strings directly in Power Automate flows (option A) is a security risk.

</details>

### Question 4

**Which authentication method should you prefer for service-to-service communication between an Azure Function and Azure AI Foundry?**

- A) API Key stored in Azure Key Vault
- B) OAuth 2.0 client credentials flow
- C) Managed Identity with `DefaultAzureCredential`
- D) Shared Access Signature (SAS) token

<details>
<summary>✅ Answer</summary>

**C) Managed Identity with `DefaultAzureCredential`**

Managed Identity eliminates the need to manage any secrets in configuration. `DefaultAzureCredential` automatically uses the Azure Function's system-assigned or user-assigned managed identity to authenticate with AI Foundry — no keys, passwords, or connection strings required.

</details>

### Question 5

**A client application needs to display AI Foundry agent responses token-by-token as they are generated. Which REST API pattern is most appropriate?**

- A) Synchronous request-response
- B) Asynchronous polling with 202 Accepted
- C) Server-Sent Events (SSE) streaming
- D) GraphQL subscription

<details>
<summary>✅ Answer</summary>

**C) Server-Sent Events (SSE) streaming**

SSE streaming delivers tokens as they are generated by the agent, enabling progressive rendering in the client UI. This provides the fastest time-to-first-token (typically under 1 second) compared to synchronous (waits for complete response) or async polling (adds polling overhead).

</details>

### Question 6

**You want to automatically summarize documents when they are uploaded to Azure Blob Storage using an AI Foundry agent. Which event-driven architecture is most appropriate?**

- A) Azure Functions Timer Trigger → polling Blob Storage → AI Foundry Agent
- B) Azure Event Grid (Blob Created event) → Azure Function → AI Foundry Agent
- C) Azure Logic App → Blob Storage Connector → AI Foundry REST API
- D) Power Automate scheduled flow → SharePoint check → AI Foundry Custom Connector

<details>
<summary>✅ Answer</summary>

**B) Azure Event Grid (Blob Created event) → Azure Function → AI Foundry Agent**

Event Grid provides near-real-time, push-based event routing for Azure Blob Storage events. When a blob is created, Event Grid immediately triggers the Azure Function, which invokes the AI Foundry agent. This is more efficient and responsive than timer-based polling (option A).

</details>

### Question 7

**When receiving an incoming webhook from a third-party system (e.g., GitHub), what is the FIRST step your Azure Function should perform before invoking the AI Foundry agent?**

- A) Parse the JSON payload and extract the event data
- B) Validate the webhook signature to verify authenticity
- C) Create a new AI Foundry thread for the event
- D) Send a typing indicator to the downstream notification channel

<details>
<summary>✅ Answer</summary>

**B) Validate the webhook signature to verify authenticity**

Security first. Before processing any webhook payload, you must validate the cryptographic signature (e.g., `X-Hub-Signature-256` for GitHub) to ensure the request genuinely came from the expected source. Processing unvalidated webhooks exposes your system to spoofed events and potential abuse.

</details>

### Question 8

**An AI Foundry agent has a function tool defined called `create_support_ticket`. In the context of integration patterns, what type of pattern does this represent?**

- A) Incoming webhook — the agent receives external events
- B) Outgoing webhook — the agent initiates calls to external systems
- C) Synchronous REST API — the agent exposes an endpoint
- D) Event Grid subscription — the agent subscribes to Azure events

<details>
<summary>✅ Answer</summary>

**B) Outgoing webhook — the agent initiates calls to external systems**

Function tools defined on an AI Foundry agent enable the agent to call external APIs as part of its reasoning process. When the agent determines it needs to create a support ticket, it invokes the function tool, which calls the external ticketing system. This is an outgoing action initiated by the agent.

</details>

### Question 9

**A developer wants to expose their AI Foundry agent to external partners with metered access, rate limiting, and usage analytics. Which Azure service should front the API?**

- A) Azure Front Door
- B) Azure Application Gateway
- C) Azure API Management (APIM)
- D) Azure Traffic Manager

<details>
<summary>✅ Answer</summary>

**C) Azure API Management (APIM)**

Azure API Management provides subscription keys for partner access, built-in rate limiting and throttling policies, usage analytics dashboards, and request/response transformation. It is purpose-built for managing, securing, and monitoring APIs exposed to external consumers.

</details>

### Question 10

**You are designing an integration that must handle scenarios where the AI Foundry agent is temporarily unavailable. In an event-driven architecture, which combination of patterns provides the most resilient design?**

- A) Retry immediately three times, then drop the event
- B) Dead-letter queue + exponential backoff + circuit breaker
- C) Log the error to Application Insights and notify the user
- D) Switch to a secondary AI Foundry project in another region

<details>
<summary>✅ Answer</summary>

**B) Dead-letter queue + exponential backoff + circuit breaker**

This combination provides comprehensive resilience:
- **Exponential backoff** prevents overwhelming the service during transient failures.
- **Dead-letter queue** preserves events that cannot be processed after all retries, enabling later reprocessing.
- **Circuit breaker** prevents cascading failures by temporarily stopping agent calls when the failure rate exceeds a threshold.

Option D (failover) is valid for high-availability but is not about *handling* failures gracefully — it's about *avoiding* them.

</details>
