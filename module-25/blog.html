1. Microsoft Teams Bot Integration

Microsoft Teams is the default collaboration surface for millions of enterprise users. Integrating your AI Foundry agent as a Teams bot puts AI capabilities directly into the flow of work — no context switching required.

Architecture Overview

A Teams bot integration uses the Azure Bot Service as the messaging relay between Teams and your agent. The bot receives activities (messages, card actions, adaptive card submissions), forwards them to your Azure AI Foundry agent endpoint, and returns the response to the user.

Setting Up the Bot Registration

  1. Navigate to the Azure Portal → Create an Azure Bot resource.
  2. Configure the Messaging Endpoint to point to your Azure Function or App Service.
  3. Enable the Microsoft Teams channel in the Bot Service configuration.
  4. Register the bot in Teams Admin Center or side-load for development.

Bot Framework Handler — Forwarding to AI Foundry

The bot handler intercepts each incoming message, constructs a request to the AI Foundry agent API, and returns the response as a Teams message. Below is a simplified implementation: 

Python
import os
from botbuilder.core import ActivityHandler, TurnContext
from azure.identity import DefaultAzureCredential
from azure.ai.projects import AIProjectClient
from azure.ai.projects.models import AgentThread

class FoundryBot(ActivityHandler):
    def __init__(self):
        self.client = AIProjectClient(
            credential=DefaultAzureCredential(),
            project=os.environ["FOUNDRY_PROJECT_CONN_STR"]
        )
        self.agent_id = os.environ["FOUNDRY_AGENT_ID"]

    async def on_message_activity(self, turn_context: TurnContext):
        # Create a thread and send the user's message
        thread: AgentThread = self.client.agents.create_thread()

        self.client.agents.create_message(
            thread_id=thread.id,
            role="user",
            content=turn_context.activity.text
        )

        # Run the agent and wait for completion
        run = self.client.agents.create_and_process_run(
            thread_id=thread.id,
            agent_id=self.agent_id
        )

        # Retrieve the latest assistant response
        messages = self.client.agents.list_messages(thread_id=thread.id)
        reply = messages.data[0].content[0].text.value

        await turn_context.send_activity(reply)
⚠️ Production Considerations
  • Thread management: Map each Teams conversation ID to a persistent AI Foundry thread ID using Azure Table Storage or Cosmos DB to preserve context across messages.
  • Typing indicators: Show a typing indicator while the agent processes to avoid user frustration. Use turn_context.send_activities() with a typing activity.
  • Rate limiting: Implement token bucket or sliding window rate limiting per user to prevent abuse of your agent quota.

Building the Azure Function Proxy

Custom connectors communicate via REST. An Azure Function acts as a lightweight proxy that authenticates with AI Foundry using managed identity and forwards requests: 

C#
using Azure.AI.Projects;
using Azure.Identity;

[Function("InvokeAgent")]
public async Task<IActionResult> Run(
    [HttpTrigger(AuthorizationLevel.Function, "post")] HttpRequest req)
{
    var body = await req.ReadFromJsonAsync<AgentRequest>();

    var client = new AIProjectClient(
        Environment.GetEnvironmentVariable("FOUNDRY_PROJECT_CONN_STR"),
        new DefaultAzureCredential()
    );

    // Create thread, message, and run the agent
    AgentThread thread = await client.Agents.CreateThreadAsync();

    await client.Agents.CreateMessageAsync(
        thread.Id, "user", body.Message);

    ThreadRun run = await client.Agents.CreateAndProcessRunAsync(
        thread.Id, Environment.GetEnvironmentVariable("AGENT_ID"));

    var messages = await client.Agents.GetMessagesAsync(thread.Id);
    var reply = messages.Value.Data[0].Content[0].Text;

    return new OkObjectResult(new { response = reply });
}

Registering the Custom Connector

  1. In Power Automate, go to Data → Custom Connectors → New Custom Connector.

  3. Set Authentication to API Key — pass the Function key in the header.
  4. Define the Actions with request/response schemas matching your Function's contract.
  5. Test the connector, then publish it for your organization.

3. REST API Patterns for External Applications

When you need to integrate AI Foundry agents with applications outside the Microsoft ecosystem — mobile apps, third-party SaaS platforms, or custom web applications — REST APIs are the universal integration language.

API Design Patterns

Synchronous Request-Response

Best for short interactions (under 30 seconds). The client sends a request, waits for the full response, and renders it.

HTTP
POST /api/agent/invoke HTTP/1.1
Host: your-function-app.azurewebsites.net
Content-Type: application/json
x-functions-key: {{functionKey}}

{
  "message": "Summarize Q4 sales trends",
  "thread_id": "thread_abc123"  // optional: continue conversation
}

// Response
{
  "response": "Q4 sales showed a 12% increase...",
  "thread_id": "thread_abc123",
  "usage": { "prompt_tokens": 340, "completion_tokens": 215 }
}

Asynchronous Polling

Ideal for longer agent tasks. Submit a job, receive a 202 Accepted with a polling URL, and check for completion.

HTTP
// 1. Submit
POST /api/agent/jobs  →  202 Accepted
Location: /api/agent/jobs/job_xyz789

// 2. Poll
GET /api/agent/jobs/job_xyz789
{
  "status": "running",
  "progress": 65
}

// 3. Completed
GET /api/agent/jobs/job_xyz789
{
  "status": "completed",
  "result": { "response": "Analysis complete..." }
}

Server-Sent Events (SSE) Streaming

Best for real-time UX. The agent streams tokens as they are generated, providing immediate feedback.

Python
from flask import Flask, Response

def stream_agent_response(thread_id, agent_id):
    # Use AI Foundry's streaming API
    with client.agents.create_stream(
        thread_id=thread_id,
        agent_id=agent_id
    ) as stream:
        for event in stream:
            if event.type == "content_delta":
                yield f"data: {event.delta.text}\n\n"
        yield "data: [DONE]\n\n"

@app.route("/api/stream")
def stream():
    return Response(
        stream_agent_response(thread_id, agent_id),
        mimetype="text/event-stream"
    )

Incoming Webhooks — Receiving External Events

Expose an Azure Function as a webhook receiver. When a third-party system (GitHub, Stripe, Salesforce) sends an event, the function validates the payload, extracts the relevant data, and invokes the agent: 

Python
import azure.functions as func
import hmac, hashlib, json

def main(req: func.HttpRequest) -> func.HttpResponse:
    # 1. Validate webhook signature
    signature = req.headers.get("X-Hub-Signature-256")
    payload = req.get_body()
    expected = "sha256=" + hmac.new(
        WEBHOOK_SECRET.encode(), payload, hashlib.sha256
    ).hexdigest()

    if not hmac.compare_digest(signature, expected):
        return func.HttpResponse(status_code=401)

    # 2. Parse event and invoke agent
    event = json.loads(payload)
    prompt = f"Analyze this event: {json.dumps(event)}"

    # 3. Call AI Foundry agent (same pattern as before)
    result = invoke_foundry_agent(prompt)

    # 4. Take action based on agent's analysis
    if result["severity"] == "high":
        send_teams_notification(result["summary"])

    return func.HttpResponse(status_code=200)

Event Grid Integration

7. Observability & Monitoring

Every production integration must be observable. Without instrumentation, you are flying blind — unable to detect latency regressions, token budget overruns, or silent failures that degrade user experience.

The Three Pillars of AI Integration Observability

📈

Metrics

Track quantitative signals: request count, latency (p50/p95/p99), token usage per invocation, error rate, and active thread count.

📋

Logs

Structured logs with correlation IDs that trace a request from the client through the gateway, to the Function, and into the AI Foundry agent run.

🔗

Traces

Distributed traces using Application Insights that show end-to-end latency breakdown across all services in the integration chain.

Application Insights Integration

Python
import logging
from opencensus.ext.azure.log_exporter import AzureLogHandler
from opencensus.ext.azure.trace_exporter import AzureExporter
from opencensus.trace.tracer import Tracer

# Configure Application Insights
logger = logging.getLogger(__name__)
logger.addHandler(AzureLogHandler(
    connection_string=os.environ["APPLICATIONINSIGHTS_CONNECTION_STRING"]
))

tracer = Tracer(exporter=AzureExporter(
    connection_string=os.environ["APPLICATIONINSIGHTS_CONNECTION_STRING"]
))

async def invoke_agent_with_telemetry(
    client, thread_id, agent_id, message
):
    with tracer.span(name="agent_invocation") as span:
        span.add_attribute("agent_id", agent_id)
        span.add_attribute("thread_id", thread_id)

        # Track timing
        start = time.time()

        try:
            client.agents.create_message(
                thread_id=thread_id, role="user", content=message
            )
            run = client.agents.create_and_process_run(
                thread_id=thread_id, agent_id=agent_id
            )

            duration = time.time() - start
            span.add_attribute("duration_seconds", duration)
            span.add_attribute("run_status", run.status)

            # Log custom metrics
            logger.info("Agent invocation completed", extra={
                "custom_dimensions": {
                    "agent_id": agent_id,
                    "duration_ms": int(duration * 1000),
                    "status": run.status,
                    "prompt_tokens": run.usage.prompt_tokens,
                    "completion_tokens": run.usage.completion_tokens,
                }
            })

            return run

        except Exception as e:
            span.add_attribute("error", str(e))
            logger.exception("Agent invocation failed")
            raise

8. Error Handling & Resilience Patterns

AI agent integrations face unique failure modes: model timeouts, token limit exhaustion, content filter triggers, and transient service outages. Robust error handling is not optional — it is a requirement for production readiness.

Error Categories

CategoryExamplesHandling Strategy
Transient HTTP 429 (rate limit), 503 (service unavailable) Retry with exponential backoff + jitter
Client Error HTTP 400 (bad request), invalid message format Return descriptive error to caller; do not retry
Agent Failure Run status = "failed", content filter triggered Log details, return fallback message, alert on repeat failures
Timeout Agent processing exceeds allowed duration Cancel run, notify user, offer retry option
Quota Exhausted Token budget exceeded, TPM/RPM limits hit Queue request for later processing, notify user of delay

Retry Pattern with Exponential Backoff

Python
import time, random

def invoke_with_retry(func, max_retries=3, base_delay=1.0):
    """Invoke a function with exponential backoff and jitter."""
    for attempt in range(max_retries + 1):
        try:
            return func()
        except RateLimitError as e:
            if attempt == max_retries:
                raise
            # Exponential backoff: 1s, 2s, 4s + random jitter
            delay = base_delay * (2 ** attempt) + random.uniform(0, 0.5)
            logger.warning(
                f"Rate limited, retrying in {delay:.1f}s "
                f"(attempt {attempt + 1}/{max_retries})"
            )
            time.sleep(delay)
        except ContentFilterError:
            # Content filter is not retryable
            logger.error("Content filter triggered — not retrying")
            raise

Circuit Breaker Pattern

When the AI Foundry service is experiencing sustained failures, a circuit breaker prevents your application from wasting resources on calls that will fail. The circuit "opens" after a threshold of failures, and "closes" again after a cooldown period. 

Python
import time, threading

class CircuitBreaker:
    def __init__(self, failure_threshold=5, reset_timeout=60):
        self.failure_count = 0
        self.failure_threshold = failure_threshold
        self.reset_timeout = reset_timeout
        self.last_failure_time = 0
        self.state = "closed"  # closed, open, half-open
        self.lock = threading.Lock()

    def call(self, func):
        with self.lock:
            if self.state == "open":
                if time.time() - self.last_failure_time > self.reset_timeout:
                    self.state = "half-open"
                else:
                    raise CircuitOpenError(
                        "Circuit breaker is open — agent temporarily unavailable"
                    )

        try:
            result = func()
            with self.lock:
                self.failure_count = 0
                self.state = "closed"
            return result

        except Exception:
            with self.lock:
                self.failure_count += 1
                self.last_failure_time = time.time()
                if self.failure_count >= self.failure_threshold:
                    self.state = "open"
            raise

Shared Infrastructure Pattern

Rather than duplicating agent invocation logic across each channel, centralize it in a shared service layer. Each channel-specific Function calls the same internal API: 

Python
# shared/agent_service.py — centralized agent invocation

class AgentService:
    """Centralized service for invoking AI Foundry agents across all channels."""

    def __init__(self):
        self.client = AIProjectClient(
            credential=DefaultAzureCredential(),
            project=os.environ["FOUNDRY_PROJECT_CONN_STR"]
        )
        self.circuit_breaker = CircuitBreaker()
        self.thread_store = ThreadStore()

    def invoke(
        self,
        message: str,
        channel: str,
        conversation_id: str | None = None,
        agent_id: str | None = None
    ) -> dict:
        """Invoke the agent with channel-agnostic logic."""
        agent_id = agent_id or os.environ["FOUNDRY_AGENT_ID"]

        # Resolve or create thread
        thread_id = None
        if conversation_id:
            thread_id = self.thread_store.get_thread_id(conversation_id)

        if not thread_id:
            thread = self.client.agents.create_thread()
            thread_id = thread.id
            if conversation_id:
                self.thread_store.set_thread_id(conversation_id, thread_id)

        # Invoke with circuit breaker and retry
        def _invoke():
            self.client.agents.create_message(
                thread_id=thread_id, role="user", content=message
            )
            return self.client.agents.create_and_process_run(
                thread_id=thread_id, agent_id=agent_id
            )

        run = self.circuit_breaker.call(
            lambda: invoke_with_retry(_invoke)
        )

        messages = self.client.agents.list_messages(thread_id=thread_id)
        response = messages.data[0].content[0].text.value

        return {
            "response": response,
            "thread_id": thread_id,
            "channel": channel,
            "status": run.status,
            "usage": {
                "prompt_tokens": run.usage.prompt_tokens,
                "completion_tokens": run.usage.completion_tokens
            }
        }
✅ Architecture Best Practice
Centralizing agent invocation in a shared service ensures consistent behavior — retry logic, circuit breaking, telemetry, and thread management — regardless of which channel triggered the request. Update once, benefit everywhere.

10. Production Readiness Checklist

Before deploying any AI agent integration to production, walk through this comprehensive checklist to ensure you have covered security, reliability, observability, and governance.

Security

  • ✅ All secrets stored in Azure Key Vault — zero secrets in application settings or code
  • ✅ Managed Identity used for all Azure service-to-service communication
  • ✅ API endpoints protected with authentication (OAuth 2.0, API Key, or APIM subscription)
  • ✅ Webhook endpoints validate cryptographic signatures before processing
  • ✅ No AI Foundry connection strings or agent IDs exposed to client-side code
  • ✅ Network restrictions: Virtual Network integration or private endpoints where applicable
  • ✅ Data Loss Prevention (DLP) policies configured in Power Platform environments

Reliability

  • ✅ Retry logic with exponential backoff for all agent invocations
  • ✅ Circuit breaker pattern to prevent cascading failures
  • ✅ Dead-letter queues configured for event-driven integrations
  • ✅ Graceful error messages returned to users (never raw exceptions)
  • ✅ Health check endpoints for monitoring service availability
  • ✅ Thread cleanup strategy — archive or delete old threads to manage storage

Observability

  • ✅ Application Insights connected to all Azure Functions and App Services
  • ✅ Custom metrics: latency, token usage, error rate, thread reuse rate
  • ✅ Correlation IDs propagated across all services in the chain
  • ✅ Alerts configured for error rate spikes, latency degradation, and budget overruns
  • ✅ Dashboard showing per-channel usage and performance breakdown

Governance

  • ✅ Content moderation policies enforced (AI Foundry content filters)
  • ✅ User feedback collection mechanism (thumbs up/down) for continuous improvement
  • ✅ Audit logging for all agent interactions (who asked what, when)
  • ✅ Token budget and rate limit policies per user/team/department
  • ✅ Data retention policies aligned with organizational and regulatory requirements
