Module 25 TEAMS BOT — CODE

Bot Handler → AI Foundry Agent

class FoundryBot(ActivityHandler):
    async def on_message_activity(self, ctx: TurnContext):
        # 1. Create or retrieve thread
        thread = client.agents.create_thread()

        # 2. Send user message
        client.agents.create_message(
            thread_id=thread.id,
            role="user",
            content=ctx.activity.text
        )

        # 3. Run the agent
        run = client.agents.create_and_process_run(
            thread_id=thread.id,
            agent_id=AGENT_ID
        )

        # 4. Return response
        msgs = client.agents.list_messages(thread_id=thread.id)
        await ctx.send_activity(msgs.data[0].content[0].text.value)

Module 25 REST APIs

REST API Patterns

PatternResponse TimeClient UXUse Case
Synchronous < 30 seconds Wait for full response Simple Q&A, quick lookups
Async Polling Minutes to hours Submit → Poll → Complete Batch analysis, document processing
SSE Streaming First token < 1s Progressive rendering Chat UIs, real-time assistants
💡 Rule of thumb: Start with synchronous. Add streaming when UX demands it. Use async polling for workloads that exceed HTTP timeout limits.

Module 25 EVENT GRID

Event Grid Integration

Why Event Grid?

  • At-scale routing — millions of events/sec
  • Built-in retry with dead-letter support
  • Native Azure triggers — Blob, Resource, IoT Hub
  • Custom topics — your own event schemas

Production Patterns

  • Idempotent handlers (same event processed twice safely)
  • Dead-letter queue monitoring and alerts
  • Exponential backoff for agent call retries
  • Circuit breaker when AI Foundry is throttled
📋 Pattern: Event Grid → Azure Function → AI Foundry Agent → Teams notification. This is the backbone of most automated AI workflows.
