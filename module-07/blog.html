Introduction: The Power of Choice

One of the most transformative aspects of Azure AI Foundry is the sheer breadth of AI models available at your fingertips. Unlike platforms that lock you into a single provider's ecosystem, AI Foundry's Model Catalog gives you access to over 1,800 models from the world's leading AI labs — all discoverable, testable, and deployable from a unified interface.

Whether you need a frontier-class reasoning model for complex enterprise logic, a lightweight model for edge devices, a multilingual model for global customer support, or a specialized embedding model for semantic search — the catalog has you covered.

What Is the Model Catalog?

The Model Catalog in Azure AI Foundry is a curated, searchable directory of pre-trained AI models from Microsoft and third-party providers. Think of it as an "app store for AI models" — but instead of downloading apps, you're deploying powerful AI capabilities directly into your cloud infrastructure.

Core Capabilities

  • Discover — Browse and search 1,800+ models filtered by task, provider, license type, and more
  • Evaluate — Read detailed model cards with performance benchmarks, capabilities, and limitations
  • Test — Try any model interactively in the built-in Playground before deploying
  • Deploy — Deploy your chosen model as an API endpoint with a few clicks
  • Compare — Run the same prompts against multiple models to find the best fit

Accessing the Catalog

You can access the Model Catalog through multiple paths:

  1. AI Foundry Portal — Navigate to ai.azure.com → Your Project → Model catalog in the left sidebar
  2. Azure AI Foundry SDK — Programmatically query and deploy models via the Python or REST API
  3. Azure CLI — Use az ai commands to list and manage model deployments
💡 Pro Tip

You can filter models by task type (text generation, embeddings, image generation, etc.), provider, license type (open-weight, proprietary), and fine-tuning capability to quickly narrow down your options.

Model Cards & Benchmarks

Every model in the catalog has a detailed Model Card — a standardized document that serves as the model's "nutrition label." Model cards provide the information you need to make informed decisions about which model to use.

Anatomy of a Model Card

Each model card in AI Foundry includes the following sections:

📋
Overview
Model description, version history, provider information, architecture type, parameter count, and context window size.
📊
Benchmarks
Performance scores on standardized tests like MMLU, HumanEval, GSM8K, MATH, HellaSwag, and more.
🎯
Capabilities
Supported tasks (chat, completion, embeddings), input/output modalities, language support, and tool-calling abilities.
⚖️
Licensing
License type (MIT, Apache 2.0, Llama Community, proprietary), usage restrictions, and commercial use terms.
🔒
Safety
Known limitations, bias assessments, content filtering recommendations, and responsible AI considerations.
🚀
Deployment
Available deployment options, supported regions, API compatibility, and integration code samples.

Key Benchmarks Explained

Benchmark What It Measures Why It Matters
MMLU Multitask knowledge (57 subjects) Overall intelligence across academic domains
HumanEval Code generation accuracy Can the model write correct Python code?
GSM8K Grade-school math Step-by-step mathematical reasoning
MATH Advanced math problems Complex mathematical problem-solving
HellaSwag Commonsense reasoning Understanding real-world situations
MT-Bench Multi-turn conversation quality Coherent dialogue over multiple exchanges
MBPP Mostly Basic Python Problems Practical coding ability for common tasks
⚠️ Benchmarks Aren't Everything

Benchmarks provide useful reference points, but they don't tell the whole story. A model scoring 5% higher on MMLU might not perform better for your specific task. Always test models with your own data and prompts in the Playground or via the API before making deployment decisions.

Choosing the Right Model

With 1,800+ models to choose from, how do you pick the right one? Use this decision framework to systematically narrow down your options:

🎯 Model Selection Decision Framework
1

Define Your Task

What exactly do you need the model to do? Text generation, code completion, embeddings, image analysis, translation, summarization, classification? Start with the task type filter in the catalog.

2

Set Quality Requirements

Do you need frontier-class accuracy (GPT-4o, o3, Llama 405B) or is "good enough" acceptable? Higher quality generally means higher cost and latency. Define your minimum acceptable quality bar.

3

Evaluate Latency Needs

Is this user-facing (needs <2 second response) or batch processing (latency doesn't matter)? Smaller models like Phi-4-mini or GPT-4o mini excel in low-latency scenarios.

4

Consider Budget Constraints

What's your per-token budget? Open-weight models on Managed Compute can be cheaper at scale. Serverless APIs are cheaper for low-volume. Use the pricing calculator to estimate costs.

5

Check Data & Compliance Requirements

Do you need data to stay in your subscription? Do you need to fine-tune? Is there a specific license your legal team requires? These constraints may eliminate some options.

6

Test in the Playground

Shortlist 2–3 candidates and test them with your actual prompts and data in the AI Foundry Playground. Compare quality, speed, and cost side by side before committing.

✅ The "Cascade" Strategy

Many production systems use a model cascade: route simple queries to a cheap, fast model (GPT-4o mini) and only escalate complex queries to an expensive model (o3). This can reduce costs by 60–80% while maintaining quality where it matters most.

Licensing & Cost Considerations

Understanding licensing is crucial before deploying any model in production. The Model Catalog includes models with different license types, each with different implications for your business.

License Types in the Catalog

License Type Examples Commercial Use Modify / Fine-tune Key Restrictions
MIT / Apache 2.0 Phi-4, Mixtral ✅ Yes ✅ Yes Minimal; attribution may be required
Llama Community Llama 3.1, 3.2 ✅ Yes (under 700M MAU) ✅ Yes Extra license needed if >700M monthly users
Proprietary GPT-4o, o3, Command R+ ✅ Via API terms ⚠️ Limited No weight access; usage governed by API ToS

Cost Optimization Strategies

  1. Right-size your model — Don't use GPT-4o for tasks that GPT-4o mini handles just as well
  2. Use Provisioned Throughput — For predictable workloads, PTUs can be 40–60% cheaper than pay-per-token
  3. Implement caching — Cache common responses to reduce redundant API calls
  4. Optimize prompts — Shorter, well-crafted prompts reduce input tokens without sacrificing quality
  5. Consider open-weight models — Managed Compute deployments of Llama/Phi can be more cost-effective at high volumes
  6. Monitor usage — Use AI Foundry's built-in monitoring to track token consumption and costs per deployment
💡 Cost Estimation

Use the Azure Pricing Calculator to estimate costs before deploying. Factor in both input tokens (your prompts) and output tokens (model responses) — output tokens are typically 2–4× more expensive than input tokens.

Playground: Testing Models Side by Side

The AI Foundry Playground is your interactive testing ground. It allows you to send prompts to deployed models and see responses in real time — no code required.

Playground Features

  • Chat mode — Multi-turn conversation interface; test system prompts, user messages, and assistant behavior
  • Completions mode — Single-shot text generation; useful for classification, extraction, and summarization
  • Parameter tuning — Adjust temperature, top-p, max tokens, frequency penalty, and presence penalty in real time
  • System prompt editor — Define the model's persona, instructions, and guardrails
  • Multiple deployments — Switch between different model deployments to compare outputs

Side-by-Side Comparison Workflow

To effectively compare models, follow this workflow:

  1. Deploy 2–3 candidate models from the Model Catalog (e.g., GPT-4o, Llama 3.1 70B, Mistral Large)
  2. Prepare a test suite — Create 5–10 representative prompts that reflect your actual use case
  3. Use identical prompts — Send the exact same system prompt and user message to each model
  4. Evaluate dimensions — Score each response on accuracy, relevance, coherence, formatting, and latency
  5. Record results — Document your findings in a comparison matrix for stakeholder review
EXAMPLE
# Sample comparison prompt for customer support classification

System: You are a customer support classifier. Classify the 
following message into exactly one category: billing, 
technical, account, feedback, or other. Respond with only 
the category name.

User: I've been charged twice for my subscription this month 
and I need a refund for the duplicate payment.

# Expected output: billing
# Compare: Which model consistently gets this right?
# Compare: Which model responds fastest?
# Compare: Which model follows the "category only" instruction best?

To use the Playground, you need at least one model deployed in your AI Foundry project. Serverless API deployments are the fastest way to get started — most models can be deployed and tested within minutes.

Mini Hack Exercise

🔬 Mini Hack Compare 3 Models on the Same Task

Objective: Deploy three different models and compare their performance on a real-world task using the AI Foundry Playground.

Time: 30–45 minutes

Steps:

  1. Open the Model Catalog
    Navigate to ai.azure.com → Your Project → Model catalog
  2. Deploy 3 Models
    Choose three models from different families. Suggested combination:
    • GPT-4o mini (OpenAI — Serverless API)
    • Phi-4 (Microsoft — Serverless API or Managed Compute)
    • Mistral Small (Mistral — Serverless API)
  3. Open the Playground
    Go to the Playground and select your first deployed model
  4. Set a System Prompt
    Use this system prompt for all three models: 
    You are a helpful assistant that explains technical concepts 
in simple terms. When explaining, use an analogy that a 
10-year-old would understand. Keep responses under 150 words.
  5. Send Test Prompts
    Test each model with these prompts:
    • "What is an API?"
    • "Explain how encryption works"
    • "What is the difference between AI and machine learning?"
  6. Evaluate & Compare
    For each response, score on a 1–5 scale:
    • Accuracy — Is the explanation correct?
    • Simplicity — Would a 10-year-old understand it?
    • Analogy quality — Is the analogy creative and fitting?
    • Conciseness — Is it under 150 words?
  7. Document Results
    Record your comparison in a table and identify which model performed best overall
✅ Success Criteria

You've successfully completed the Mini Hack when you can explain why you would choose one model over the others for this specific task — and what trade-offs (cost, speed, quality) influenced your decision.

Key Takeaways

  • Azure AI Foundry's Model Catalog provides access to 1,800+ models from 25+ providers, all in one place
  • Major model families include OpenAI GPT/o-series, Meta Llama, Mistral, Microsoft Phi, and Cohere — each with distinct strengths
  • Three deployment types — Serverless API, Managed Compute, and Provisioned Throughput — serve different needs from prototyping to production
  • Model Cards provide standardized information about capabilities, benchmarks, licensing, and safety for every model
  • Use the 6-step decision framework to systematically choose the right model: task → quality → latency → budget → compliance → test
  • Always test models with your actual prompts and data in the Playground — benchmarks alone don't predict real-world performance
  • Understand licensing implications before deploying: MIT/Apache 2.0, Llama Community License, and proprietary API terms have different restrictions
  • The model cascade strategy — routing simple queries to cheap models and complex queries to premium models — can reduce costs by 60–80%
