1. Introduction: AI as a Managed Service

Cloud Solution Providers (CSPs) and Managed Service Providers (MSPs) occupy a unique position in the Azure ecosystem. They don't just consume AI services — they resell, customize, and operate AI solutions on behalf of dozens or hundreds of end customers. Azure AI Foundry provides the platform foundation, but orchestrating it across multiple tenants at scale demands deliberate architecture decisions around governance, isolation, billing, and support.

ℹ️ Azure AI Foundry, Not Azure ML

This module focuses exclusively on Azure AI Foundry — the unified platform for building, deploying, and managing AI agents and applications. Azure AI Foundry consolidates model catalog access, prompt engineering, evaluation, and agent orchestration into a single control plane. All references in this module pertain to Azure AI Foundry projects, hubs, and deployments.

The CSP model in Azure enables partners to provision, manage, and bill Azure subscriptions on behalf of their customers. When combined with AI Foundry, this creates powerful opportunities — and unique challenges. This module addresses both.

2. Cloud Solution Provider (CSP) Integration

2.1 Understanding the CSP Model

The Azure CSP program allows partners to act as the customer's primary point of contact for Azure services. Partners own the billing relationship, provide first-line support, and manage subscriptions through Partner Center or the Partner Center API.

🔗
CSP Subscription Model for AI Foundry

In the CSP model, the partner provisions Azure subscriptions in the customer's tenant. AI Foundry resources — hubs, projects, model deployments, and agent configurations — live within these subscriptions. The CSP partner retains Admin on Behalf Of (AOBO) access by default.

  • Direct CSP (Tier 1): Partner transacts directly with Microsoft, bills customers, provides full support
  • Indirect CSP (Tier 2): Partner works through an indirect provider for billing; manages the customer relationship
  • Both tiers can provision and manage AI Foundry resources in customer subscriptions

2.2 Provisioning AI Foundry Resources via CSP

Azure CLI
# Step 1: Create customer subscription via Partner Center API
# (Handled through Partner Center portal or API)

# Step 2: As CSP admin, create AI Foundry Hub in customer subscription
az ai foundry hub create \
  --name "customer-acme-ai-hub" \
  --resource-group "rg-acme-ai" \
  --subscription "<customer-subscription-id>" \
  --location "eastus2" \
  --display-name "Acme Corp AI Hub"

# Step 3: Create project within the hub
az ai foundry project create \
  --name "acme-customer-service-bot" \
  --hub-name "customer-acme-ai-hub" \
  --resource-group "rg-acme-ai" \
  --subscription "<customer-subscription-id>"

# Step 4: Deploy model for the customer
az ai foundry model deployment create \
  --project-name "acme-customer-service-bot" \
  --name "gpt-4o-acme" \
  --model-id "azureai://openai/gpt-4o/2026-02" \
  --sku-name "GlobalStandard" \
  --sku-capacity 50

⚠️ AOBO Permissions Best Practice

Admin on Behalf Of (AOBO) grants the CSP partner Owner role on customer subscriptions by default. For AI Foundry workloads handling sensitive data, consider implementing Azure Lighthouse for more granular, auditable delegated access instead of broad AOBO permissions. This follows the principle of least privilege.

3. Managing AI Workloads for Multiple Customers

3.1 Deployment Topology Options

When managing AI Foundry workloads for multiple customers, the deployment topology defines how resources are isolated, shared, and managed. There are three primary patterns:

Pattern Isolation Level Cost Efficiency Complexity Best For
Dedicated Hub per Customer 🔒 Full 💰 Low 📈 High Regulated industries, large enterprise
Shared Hub, Separate Projects 🔐 Moderate 💰💰 Medium 📊 Medium SMB customers, standard workloads
Fully Shared (Logical Isolation) 🔑 Basic 💰💰💰 High 📉 Low Dev/test, non-sensitive workloads

3.2 Centralized Operations Dashboard

CSP partners need visibility across all customer AI Foundry deployments. Build a centralized monitoring solution using Azure Lighthouse combined with Azure Monitor and Azure Resource Graph. 

KQL — Azure Resource Graph
// Query all AI Foundry hubs across managed customer subscriptions
resources
| where type == "microsoft.machinelearningservices/workspaces"
| where kind == "Hub"
| project name, resourceGroup, subscriptionId, location,
          properties.provisioningState, properties.friendlyName,
          tags["csp:customer"], tags["csp:tier"]
| order by subscriptionId
KQL — Azure Resource Graph
// Monitor model deployment health across all customers
resources
| where type == "microsoft.machinelearningservices/workspaces/onlineendpoints/deployments"
| project name, resourceGroup, subscriptionId,
          properties.provisioningState, properties.scaleSettings,
          tags["csp:customer"]
| where properties.provisioningState != "Succeeded"

3.3 Automated Customer Onboarding

Automate the creation of AI Foundry resources for new CSP customers using infrastructure as code. The following Bicep template creates a complete AI Foundry environment for a new customer: 

Bicep
@description('CSP Customer AI Foundry Onboarding Template')
param customerName string
param location string = resourceGroup().location
param cspTier string = 'silver'  // bronze | silver | gold | platinum
param modelSku string = 'GlobalStandard'
param tpuCapacity int = 30

// AI Services account (shared backing for the hub)
resource aiServices 'Microsoft.CognitiveServices/accounts@2024-10-01' = {
  name: 'ais-${customerName}'
  location: location
  kind: 'AIServices'
  sku: { name: 'S0' }
  properties: {
    customSubDomainName: 'ais-${customerName}'
    publicNetworkAccess: 'Disabled'
    networkAcls: { defaultAction: 'Deny' }
  }
  tags: {
    'csp:customer': customerName
    'csp:tier': cspTier
    'csp:managed': 'true'
  }
}

// AI Foundry Hub
resource aiHub 'Microsoft.MachineLearningServices/workspaces@2024-10-01' = {
  name: 'hub-${customerName}'
  location: location
  kind: 'Hub'
  sku: { name: 'Basic'; tier: 'Basic' }
  identity: { type: 'SystemAssigned' }
  properties: {
    friendlyName: '${customerName} AI Hub'
    publicNetworkAccess: 'Disabled'
  }
  tags: {
    'csp:customer': customerName
    'csp:tier': cspTier
    'csp:managed': 'true'
  }
}

// Default AI Foundry Project
resource aiProject 'Microsoft.MachineLearningServices/workspaces@2024-10-01' = {
  name: 'proj-${customerName}-default'
  location: location
  kind: 'Project'
  sku: { name: 'Basic'; tier: 'Basic' }
  identity: { type: 'SystemAssigned' }
  properties: {
    friendlyName: '${customerName} Default Project'
    hubResourceId: aiHub.id
  }
  tags: {
    'csp:customer': customerName
    'csp:tier': cspTier
  }
}

4. Cross-Tenant Governance & Isolation

4.1 Azure Lighthouse for Delegated Management

Azure Lighthouse is the recommended mechanism for CSP partners to manage customer AI Foundry resources with fine-grained, auditable access. Unlike AOBO, Lighthouse provides:

  • Granular RBAC: Assign specific roles (e.g., Cognitive Services Contributor) rather than Owner
  • Just-in-Time elevation: Combine with Eligible Authorizations for time-bound privileged access
  • Audit trail: All actions are logged in the customer's activity log with the partner's identity
  • Customer visibility: Customers can see and revoke delegations at any time
JSON — Lighthouse Delegation Template
{
  "$schema": "https://schema.management.azure.com/schemas/2019-08-01/...",
  "contentVersion": "1.0.0.0",
  "parameters": {},
  "resources": [{
    "type": "Microsoft.ManagedServices/registrationDefinitions",
    "apiVersion": "2022-10-01",
    "name": "[guid('CSP-AIFoundry-Delegation')]",
    "properties": {
      "registrationDefinitionName": "CSP AI Foundry Management",
      "managedByTenantId": "<csp-partner-tenant-id>",
      "authorizations": [
        {
          "principalId": "<ai-ops-group-id>",
          "roleDefinitionId": "b24988ac-6180-42a0-ab88-20f7382dd24c",
          // Contributor — manage AI Foundry resources
          "principalIdDisplayName": "CSP AI Operations Team"
        },
        {
          "principalId": "<ai-reader-group-id>",
          "roleDefinitionId": "acdd72a7-3385-48ef-bd42-f606fba81ae7",
          // Reader — monitoring and audit
          "principalIdDisplayName": "CSP Monitoring Team"
        }
      ]
    }
  }]
}

4.2 Data Isolation Strategies

🛡️
Network-Level Isolation
  • Private Endpoints: Each customer AI Foundry hub uses dedicated private endpoints within their VNet
  • NSG Rules: Network Security Groups restrict traffic between customer environments
  • Private DNS Zones: Separate DNS zones prevent cross-tenant name resolution leaks
  • Hub-Spoke VNet Topology: CSP management VNet peers with customer VNets via controlled gateways
🗝️
Identity & Access Isolation
  • Managed Identity per Customer: Each AI Foundry hub uses its own system-assigned identity
  • Customer-Managed Keys (CMK): Customers retain control of encryption keys via their own Key Vault
  • Conditional Access: Enforce MFA and compliant device policies on CSP admin access
  • Cross-tenant Access Policies: Explicitly define which partner identities can access which resources
💾
Data Residency & Storage Isolation
  • Separate Storage Accounts: Each customer's AI Foundry data resides in their own storage account
  • Geo-Restriction: Enforce Azure Policy to keep resources within required regions
  • Data Classification Tags: Automated tagging for data sensitivity levels
  • Purge Protection: Enable soft-delete and purge protection on all customer key vaults

4.3 Governance Policies at Scale

Use Azure Policy to enforce governance rules across all managed customer subscriptions: 

JSON — Azure Policy Definition
{
  "displayName": "Require CSP tags on AI Foundry resources",
  "policyType": "Custom",
  "mode": "Indexed",
  "parameters": {},
  "policyRule": {
    "if": {
      "allOf": [
        {
          "field": "type",
          "equals": "Microsoft.MachineLearningServices/workspaces"
        },
        {
          "anyOf": [
            { "field": "tags['csp:customer']", "exists": false },
            { "field": "tags['csp:tier']",     "exists": false },
            { "field": "tags['csp:managed']",  "exists": false }
          ]
        }
      ]
    },
    "then": { "effect": "deny" }
  }
}

5. SLA Management & Support Tiers

5.1 Defining CSP Support Tiers for AI Services

CSP partners typically offer tiered support to their customers. For AI Foundry managed services, structure your SLA tiers around availability, response times, model performance, and capacity guarantees.

Dimension 🥉 Bronze 🥈 Silver 🥇 Gold 💎 Platinum
Availability SLA 99.5% 99.9% 99.95% 99.99%
Incident Response (Sev-1) 4 hours 1 hour 30 min 15 min
Model Deployment Shared PTU Standard PTU Dedicated PTU Reserved PTU + Burst
Throughput Guarantee Best effort 30 TPM baseline 100 TPM guaranteed 500 TPM dedicated
Backup & DR Daily backup 6-hour RPO 1-hour RPO Near-zero RPO, hot standby
Dedicated Support Engineer ❌ ❌ ✅ Shared ✅ Named
Prompt/Agent Optimization ❌ Quarterly review Monthly review Continuous optimization

5.2 SLA Monitoring & Enforcement

KQL — Log Analytics
// SLA compliance dashboard query: Measure AI Foundry endpoint availability
let sla_window = 30d;
AzureDiagnostics
| where TimeGenerated > ago(sla_window)
| where ResourceProvider == "MICROSOFT.MACHINELEARNINGSERVICES"
| where OperationName contains "inference"
| summarize
    TotalRequests   = count(),
    SuccessCount    = countif(ResultType == "Success"),
    FailedCount     = countif(ResultType != "Success"),
    AvgLatencyMs    = avg(DurationMs),
    P99LatencyMs    = percentile(DurationMs, 99)
    by Resource, bin(TimeGenerated, 1h)
| extend AvailabilityPct = round(toreal(SuccessCount) / TotalRequests * 100, 3)
| where AvailabilityPct < 99.9
| order by TimeGenerated desc

5.3 Escalation & Incident Management Flow

Detection & Alert
Azure Monitor detects SLA breach → fires alert rule → creates incident in ITSM tool (e.g., ServiceNow)
Triage & Classification
CSP L1 team classifies severity based on impact to customer AI workloads and their SLA tier
Investigation
L2 AI engineers use AI Foundry diagnostics, Azure Monitor logs, and Resource Health to identify root cause
Resolution & Communication
Apply fix, validate via automated health checks, notify customer with RCA within SLA timeframe
Post-Incident Review
Conduct blameless post-mortem, update runbooks, apply preventative measures across all customer environments

✅ When to Use Each Pattern

Pattern A — Use when customers have regulatory requirements (HIPAA, PCI-DSS, SOC2), need dedicated model capacity, or require customer-managed encryption keys.

7. Cost Management & Chargeback Models

7.1 Cost Allocation Strategies

CSP partners need accurate cost attribution to bill customers correctly and maintain healthy margins. AI Foundry costs span compute, model inference (tokens), storage, and networking.

🏷️
Tag-Based Allocation
Enforce csp:customer tags on all resources; use Azure Cost Management tag filtering
📊
Subscription-Level Billing
Dedicated subscriptions per customer for the cleanest cost separation
🔢
Token Metering
Track inference tokens per project/agent using AI Foundry metrics and custom dashboards
💹
Margin Management
Apply partner margin on top of Azure consumption using Partner Center billing APIs
Python
# Token usage tracking per customer for chargeback
from azure.monitor.query import LogsQueryClient
from azure.identity import DefaultAzureCredential
from datetime import timedelta

credential = DefaultAzureCredential()
client = LogsQueryClient(credential)

# Query token usage across all managed AI Foundry projects
query = """
AzureDiagnostics
| where ResourceProvider == "MICROSOFT.MACHINELEARNINGSERVICES"
| where Category == "RequestResponse"
| extend CustomerTag = tostring(tags_s["csp:customer"])
| extend PromptTokens = toint(properties_s["promptTokens"])
| extend CompletionTokens = toint(properties_s["completionTokens"])
| summarize
    TotalPromptTokens = sum(PromptTokens),
    TotalCompletionTokens = sum(CompletionTokens),
    TotalRequests = count()
    by CustomerTag, bin(TimeGenerated, 1d)
| order by CustomerTag, TimeGenerated
"""

result = client.query_workspace(
    workspace_id="<log-analytics-workspace-id>",
    query=query,
    timespan=timedelta(days=30)
)

for row in result.tables[0].rows:
    customer, date, prompt_tk, completion_tk, requests = row
    cost = (prompt_tk * 0.000005) + (completion_tk * 0.000015)
    print(f"{customer} | {date} | ${cost:.4f} | {requests} requests")

8. Security & Compliance Across Tenants

8.1 Security Baseline for CSP AI Foundry Deployments

🔐
CSP Security Checklist
  1. Network isolation: All AI Foundry hubs behind private endpoints; no public network access
  2. Identity: Managed identities for all service-to-service auth; no credential storage
  3. Encryption: Customer-managed keys (CMK) for Gold/Platinum tiers; platform keys for Bronze/Silver
  4. Monitoring: Microsoft Sentinel workspace with cross-tenant data connectors
  5. Compliance: Azure Policy initiative enforcing CIS benchmarks on all managed subscriptions
  6. Content Safety: Azure AI Content Safety enabled on all model deployments
  7. Audit: Diagnostic settings forwarding to CSP's central Log Analytics workspace
  8. DR: Geo-redundant deployments for Gold/Platinum tiers with automated failover

11. Summary & Key Takeaways

✅ Key Takeaways
  1. CSP + AI Foundry enables partners to deliver managed AI services at scale, leveraging Partner Center for provisioning and billing
  2. Azure Lighthouse is preferred over AOBO for granular, auditable cross-tenant management of AI resources
  3. Deployment topology (dedicated vs. shared) should match customer compliance, capacity, and cost requirements
  4. Tiered SLAs differentiate your offering — map Bronze through Platinum to specific availability, throughput, and support commitments
  5. Tag-based cost allocation and token metering enable accurate chargeback and margin management
  6. Security baselines must be enforced uniformly via Azure Policy, with enhanced controls (CMK, dedicated PTU) for premium tiers
  7. Compliance is shared — understand the Microsoft vs. CSP vs. customer responsibility boundaries for each framework
