Why Fine-Tune?

Sometimes even the best prompt can't reliably produce what you need:

🎯 The Problem

  • Long system prompts → high cost & latency
  • Inconsistent output format
  • Domain terminology not understood
  • Few-shot examples consume context window

🔧 The Solution: Fine-Tuning

  • Model internalizes your requirements
  • Consistent outputs without prompting
  • 50–80% reduction in prompt tokens
  • 10–30% accuracy improvement on domain tasks

The Decision Framework

🎯 Prompt Engineering

  • Quick adjustments
  • Format/style changes
  • Zero cost
  • Start here first

📚 RAG

  • Dynamic data needed
  • Source citations
  • Large knowledge bases
  • Data freshness

🔧 Fine-Tuning

  • Domain expertise
  • Specific tone/style
  • Reduce latency
  • Scale consistency

💡 These are complementary — Fine-Tune for style + RAG for data = production-grade

Decision Flowchart

1
Can a better prompt solve it? → Start with prompt engineering (free & fast)
2
Does the model need access to specific/changing data? → Add RAG for grounded answers
3
Need specific output style or domain expertise? → Fine-tune the model
4
Need all of the above? → Fine-tune + RAG = best of both worlds

Preparing Training Data

AI Foundry uses JSONL (JSON Lines) format — one training example per line:

{"messages": [
  {"role": "system", "content": "You are a medical coding assistant..."},
  {"role": "user", "content": "Patient has acute bronchitis."},
  {"role": "assistant", "content": "J20.9 — Acute bronchitis, unspecified"}
]}

✅ Quality Checklist

  • 50+ examples minimum (100+ recommended)
  • Consistent structure throughout
  • Diverse, representative inputs
  • Correct assistant responses

⚠️ Avoid

  • Fabricated / synthetic data
  • PII or sensitive information
  • Duplicate examples
  • Conflicting instructions

Hyperparameter Configuration

n_epochs

Number of full passes through training data.

Default: auto (3–5 typically)

⚠️ Too many = overfitting

learning_rate_multiplier

Scales the base learning rate.

Default: auto (1.0–2.0)

⚠️ Too high = instability

batch_size

Examples processed together per step.

Default: auto (1–4)

Larger = more stable gradients

seed

Controls reproducibility.

Default: random

Set fixed seed for consistent comparisons

Reading the Loss Curves

PatternTraining LossValidation LossAction
✅ Healthy↓ Decreasing↓ DecreasingContinue training
⚠️ Overfitting↓ Decreasing↑ IncreasingReduce epochs, add data
⚠️ Underfitting→ Flat / High→ Flat / HighIncrease epochs/LR
✅ Converged→ Plateaued→ PlateauedTraining complete

🔑 Key insight: Watch the gap between training and validation loss. A widening gap signals overfitting.

Validate & Deploy

Validation Steps

  1. Review final training metrics
  2. Test in Playground with new inputs
  3. Run batch evaluation against test set
  4. Compare with base model side-by-side

Deployment

  1. Go to Deployments in AI Foundry
  2. Select your fine-tuned model
  3. Set deployment name & TPM limit
  4. Call via Azure OpenAI API
response = client.chat.completions.create(
    model="my-custom-model",
    messages=[{"role": "user",
     "content": "Your query here"}]
)

Cost & Performance

Cost Components

  • Training: One-time (model × tokens × epochs)
  • Hosting: Per hour while deployed
  • Inference: Per token per request

💰 Cost Strategy

Start with gpt-4o-mini — cheaper to train and often sufficient when fine-tuned.

Performance Gains

⚡ Latency: Up to 50% faster (shorter prompts)

🎯 Accuracy: 10–30% improvement on domain tasks

📉 Tokens: 60–80% reduction in prompt tokens

Key Takeaways

  • Start with prompts → add RAG → fine-tune for behavior & style
  • JSONL chat format with 50–100+ high-quality, diverse examples
  • AI Foundry manages all training infrastructure — no GPUs to provision
  • Watch loss curves: converging = good, diverging = overfitting
  • Validate against test set, compare with base model
  • Fine-tune + RAG = production-grade solution
