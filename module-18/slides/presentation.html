Threads in Azure AI Agent Service

The foundation of multi-turn conversations

Short-Term Memory

In-context memory within the current thread

The model sees all messages in the thread (up to the token limit) as its context. This is the simplest form of agent memory — no extra infrastructure needed.

✓ Advantages
  • Simple — built into the Agent Service
  • Automatic — no extra code required
  • Contextual — full conversation fidelity
⚠ Limitations
  • Bounded by context window size
  • Ephemeral — lost when thread ends
  • Cost grows with conversation length

Truncation Strategies

Gracefully managing context overflow

🔄 type: "auto"

System automatically manages truncation. Oldest messages are dropped as needed to fit the context window.

📋 type: "last_messages"

Keep only the last N messages. Gives precise control over how much history the model sees.

Python — Truncation & token limits
# Run with truncation strategy run = client.agents.create_and_process_run( thread_id=thread.id, agent_id=agent.id, truncation_strategy={ "type": "last_messages", "last_messages": 20 # Keep last 20 messages }, max_prompt_tokens=80000, # Limit input tokens max_completion_tokens=4096 # Limit output tokens )

Key Takeaways

What to remember from this module

  • Threads are the foundation of multi-turn conversations — every message is persisted and ordered
  • Truncation strategies prevent context overflow — use auto or last_messages to stay within limits
  • Long-term memory requires external storage — Cosmos DB for structure, AI Search for semantics
  • Combine short-term + long-term for best results — threads for conversation, stores for persistence
  • Always design for memory boundaries — plan for token limits, stale data, and memory eviction
