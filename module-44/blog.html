RBAC & Managed Identity

Azure AI Foundry leverages Microsoft Entra ID (formerly Azure Active Directory) for identity and access management. The two key mechanisms are Role-Based Access Control (RBAC) for authorization and Managed Identity for credential-free authentication between Azure services.

Understanding the RBAC Model

RBAC in Azure follows a simple formula: Who (security principal) + What (role definition) + Where (scope) = Role Assignment. Every action in Azure AI Foundry — from deploying a model to reading a dataset — is governed by this model.

┌─────────────────────────────────────────────────────────────┐ │ RBAC Assignment Model │ ├─────────────────────────────────────────────────────────────┤ │ │ │ Security Principal Role Definition Scope │ │ ┌──────────────┐ ┌───────────────┐ ┌────────────┐ │ │ │ User │ │ Contributor │ │ Subscription│ │ │ │ Group │ ──▶ │ Reader │ ──▶│ Res. Group │ │ │ │ Service Prin.│ │ Custom Role │ │ Resource │ │ │ │ Managed ID │ │ Owner │ │ AI Project │ │ │ └──────────────┘ └───────────────┘ └────────────┘ │ │ │ │ Result: Allow or Deny specific actions at the given scope │ └─────────────────────────────────────────────────────────────┘
⚠️ Least Privilege Principle

Always assign the most restrictive role that still allows the user or service to perform their tasks. Avoid using Owner or Contributor at the subscription level for AI workloads. Scope down to the resource group or individual resource whenever possible.

Assigning Roles via Azure CLI

Use the Azure CLI to assign RBAC roles to users, groups, or managed identities:

Azure CLI # Assign Cognitive Services User to a user at the AI Foundry project scope az role assignment create \ --assignee "user@contoso.com" \ --role "Cognitive Services User" \ --scope "/subscriptions/{sub-id}/resourceGroups/{rg}/providers/Microsoft.CognitiveServices/accounts/{resource}" # Assign Azure AI Developer role at the resource group level az role assignment create \ --assignee "dev-team-group-id" \ --role "Azure AI Developer" \ --scope "/subscriptions/{sub-id}/resourceGroups/{rg}" # List current role assignments for an AI Foundry resource az role assignment list \ --scope "/subscriptions/{sub-id}/resourceGroups/{rg}" \ --output table

✅ Best Practice — Role Assignment Strategy

Create Microsoft Entra ID security groups for each team function (e.g., ai-developers, ai-operators, ai-auditors) and assign roles to groups rather than individuals. This simplifies access management as team members rotate.

Managed Identity Patterns

Managed Identity is Azure's mechanism for providing services with automatically managed credentials in Microsoft Entra ID. This eliminates the need to store connection strings, API keys, or other secrets in your code or configuration.

System-Assigned vs. User-Assigned

🔐

System-Assigned

Created automatically with the Azure resource. Tied to the resource lifecycle — deleted when the resource is deleted. One-to-one relationship. Best for single-purpose workloads.

🪪

User-Assigned

Created as a standalone Azure resource. Can be shared across multiple resources. Lifecycle independent of any single resource. Best for multi-service architectures and shared access patterns.

Configuring Managed Identity for AI Foundry

Azure CLI # Enable system-assigned managed identity on an AI Services resource az cognitiveservices account identity assign \ --name "my-ai-services" \ --resource-group "rg-ai-foundry-prod" # Create a user-assigned managed identity az identity create \ --name "uami-ai-foundry" \ --resource-group "rg-ai-foundry-prod" \ --location "eastus2" # Grant the managed identity access to Key Vault secrets az role assignment create \ --assignee "$(az identity show -n uami-ai-foundry -g rg-ai-foundry-prod --query principalId -o tsv)" \ --role "Key Vault Secrets User" \ --scope "/subscriptions/{sub-id}/resourceGroups/{rg}/providers/Microsoft.KeyVault/vaults/{vault}" # Grant managed identity access to Storage az role assignment create \ --assignee "$(az identity show -n uami-ai-foundry -g rg-ai-foundry-prod --query principalId -o tsv)" \ --role "Storage Blob Data Contributor" \ --scope "/subscriptions/{sub-id}/resourceGroups/{rg}/providers/Microsoft.Storage/storageAccounts/{account}"

Managed Identity in Python Code

Python from azure.identity import DefaultAzureCredential from azure.ai.projects import AIProjectClient # DefaultAzureCredential automatically uses Managed Identity in Azure # and falls back to CLI / VS Code credentials in local dev credential = DefaultAzureCredential() # No API keys, no connection strings — just identity-based auth client = AIProjectClient( credential=credential, subscription_id="your-subscription-id", resource_group_name="rg-ai-foundry-prod", project_name="my-ai-project", ) # All downstream calls use the managed identity token response = client.agents.create_agent( model="gpt-4o", name="secure-agent", instructions="You are a helpful assistant.", )
🚫 Anti-Pattern — Hardcoded Keys

Never embed API keys, connection strings, or secrets directly in source code. Use DefaultAzureCredential for Azure resources and Key Vault references for external secrets. Hardcoded credentials are the #1 cause of production security incidents.

Monitoring Dashboards & Alerts

Production AI systems require comprehensive monitoring across multiple dimensions: infrastructure health, model performance, API usage, and cost tracking. Azure Monitor and Application Insights provide the observability stack for Azure AI Foundry.

Key Metrics to Monitor

📊

API Latency & Throughput

Track P50/P95/P99 response times and requests per second for all model endpoints. Alert on latency spikes above your SLA thresholds.

⚠️

Error Rates & HTTP Codes

Monitor 4xx and 5xx error rates. A sudden spike in 429 (throttled) responses indicates capacity issues. Track 401/403 for auth failures.

🔢

Token Consumption

Track prompt tokens, completion tokens, and total tokens per deployment. Essential for cost management and quota planning.

💰

Cost & Budget

Set up Azure Cost Management budgets with alerts at 50%, 80%, and 100% thresholds. Break down costs by deployment and team.

Setting Up a Monitoring Dashboard

Use Azure Monitor workbooks or the Azure portal's built-in dashboard editor to create a unified view of your AI operations:

KQL — Azure Monitor // Query: API latency percentiles over the last 24 hours AzureDiagnostics | where ResourceProvider == "MICROSOFT.COGNITIVESERVICES" | where Category == "RequestResponse" | where TimeGenerated > ago(24h) | summarize P50 = percentile(DurationMs, 50), P95 = percentile(DurationMs, 95), P99 = percentile(DurationMs, 99), TotalRequests = count() by bin(TimeGenerated, 1h) | order by TimeGenerated asc // Query: Token usage by model deployment AzureDiagnostics | where ResourceProvider == "MICROSOFT.COGNITIVESERVICES" | where Category == "RequestResponse" | extend modelDeployment = tostring(properties_s) | summarize TotalPromptTokens = sum(toint(promptTokens_d)), TotalCompletionTokens = sum(toint(completionTokens_d)) by modelDeployment, bin(TimeGenerated, 1h) // Query: Error rate breakdown AzureDiagnostics | where ResourceProvider == "MICROSOFT.COGNITIVESERVICES" | where TimeGenerated > ago(24h) | summarize Total = count(), Errors4xx = countif(httpStatusCode_d >= 400 and httpStatusCode_d < 500), Errors5xx = countif(httpStatusCode_d >= 500) by bin(TimeGenerated, 15m)

Content Safety & Compliance

Azure AI Content Safety provides built-in protection against harmful content in your AI applications. It works as a multi-layered filtering system that can detect and block harmful text and images across categories including violence, hate speech, sexual content, and self-harm.

Content Safety Architecture

User Request │ ▼ ┌─────────────────────┐ │ Input Filters │ ← Prompt Shield (jailbreak detection) │ Content Safety │ ← Category filters (hate, violence, etc.) │ Blocklists │ ← Custom blocked terms └──────────┬──────────┘ │ Passes filters ▼ ┌─────────────────────┐ │ AI Model │ │ (GPT-4o, etc.) │ └──────────┬──────────┘ │ ▼ ┌─────────────────────┐ │ Output Filters │ ← Category filters on response │ Content Safety │ ← Groundedness detection │ Protected Material│ ← Copyright detection └──────────┬──────────┘ │ Passes filters ▼ User Response

Configuring Content Safety Filters

Python from azure.ai.contentsafety import ContentSafetyClient from azure.ai.contentsafety.models import AnalyzeTextOptions, TextCategory from azure.identity import DefaultAzureCredential credential = DefaultAzureCredential() client = ContentSafetyClient( endpoint="https://my-content-safety.cognitiveservices.azure.com", credential=credential, ) # Analyze text for harmful content across all categories request = AnalyzeTextOptions( text="Text to analyze for safety...", categories=[ TextCategory.HATE, TextCategory.VIOLENCE, TextCategory.SELF_HARM, TextCategory.SEXUAL, ], ) response = client.analyze_text(request) for category_result in response.categories_analysis: print(f"Category: {category_result.category}, " f"Severity: {category_result.severity}") # Severity levels: 0 (safe) → 2 (low) → 4 (medium) → 6 (high) # Configure your threshold based on your application's risk tolerance

Custom Blocklists

For domain-specific content filtering, create custom blocklists that supplement the built-in safety categories:

Azure CLI # Create a custom blocklist for your organization az rest --method PUT \ --url "https://my-content-safety.cognitiveservices.azure.com/contentsafety/text/blocklists/enterprise-blocklist?api-version=2024-09-01" \ --body '{"description": "Enterprise-specific blocked terms"}' # Add terms to the blocklist az rest --method POST \ --url "https://my-content-safety.cognitiveservices.azure.com/contentsafety/text/blocklists/enterprise-blocklist:addOrUpdateBlocklistItems?api-version=2024-09-01" \ --body '{"blocklistItems": [{"description": "Competitor name", "text": "competitor-term"}]}'

Compliance & Regulatory Considerations

Enterprise AI deployments must comply with a range of regulatory frameworks. Azure AI Foundry provides tools and certifications to help you meet these requirements:

🏛️

Azure Policy

Enforce organizational rules at scale: require specific SKUs, mandate private endpoints, restrict regions, and ensure diagnostic settings are enabled on all AI resources.

📋

Microsoft Purview

Discover, classify, and govern data across your AI pipeline. Track data lineage from source through embeddings to model responses.

🔍

Audit Logging

Azure Activity Logs and diagnostic settings capture every control-plane and data-plane operation. Route logs to Log Analytics for compliance reporting.

🌍

Data Residency

Deploy AI Foundry resources in specific Azure regions to meet data residency requirements (GDPR, sovereign cloud). Use Azure Policy to enforce region constraints.

Enforcing Policies with Azure Policy

Azure CLI # Assign a built-in policy: require diagnostic settings on Cognitive Services az policy assignment create \ --name "require-diagnostics-ai" \ --display-name "Require diagnostic settings on AI Services" \ --policy "b889a06c-ec72-4b03-910a-cb169ee18721" \ --scope "/subscriptions/{sub-id}/resourceGroups/rg-ai-foundry-prod" \ --params '{"logAnalyticsWorkspaceId": {"value": "/subscriptions/{sub-id}/resourceGroups/{rg}/providers/Microsoft.OperationalInsights/workspaces/{workspace}"}}' # Assign policy: deny Cognitive Services resources without private endpoints az policy assignment create \ --name "deny-public-cognitive-services" \ --display-name "Deny public network access on Cognitive Services" \ --policy "0725b4dd-7e76-479c-a735-68e7ee23d5ca" \ --scope "/subscriptions/{sub-id}"

Security Hardening Checklist

Use this comprehensive checklist to verify your Azure AI Foundry deployment meets production security standards. Each item represents a critical security control:

Identity & Access

  • ✓Managed Identity enabled — All Azure AI resources use system-assigned or user-assigned managed identities. No API keys stored in code or configuration files.
  • ✓RBAC configured at minimum scope — Role assignments scoped to the resource group or individual resource level, not the subscription. Owner role restricted to infrastructure administrators only.
  • ✓Entra ID Conditional Access — Multi-factor authentication (MFA) enforced for all human users accessing AI Foundry resources. Location-based and device-compliance policies applied.
  • ✓Service principal rotation — For any remaining service principals, credentials are rotated on a 90-day schedule. Certificate-based authentication preferred over client secrets.
  • ✓Just-in-Time access — Privileged Identity Management (PIM) configured for elevated roles. Time-bound access for operations tasks.

Network Security

  • ✓Private endpoints enabled — AI Services, Storage, Key Vault, and AI Search accessible only via private endpoints. Public network access disabled.
  • ✓NSG rules tightened — Network Security Groups restrict traffic to only required ports and source IP ranges. Deny-all default with explicit allow rules.
  • ✓DNS resolution configured — Private DNS zones linked to VNet for privatelink resolution. No split-horizon DNS issues.
  • ✓Firewall / WAF in place — Azure Firewall or Application Gateway with WAF protects public-facing endpoints. TLS 1.2+ enforced on all connections.

Data Protection

  • ✓Encryption at rest — All storage uses Microsoft-managed or customer-managed keys (CMK) via Azure Key Vault. CMK recommended for regulated workloads.
  • ✓Encryption in transit — TLS 1.2 minimum enforced on all service connections. Certificate pinning implemented for critical service-to-service calls.
  • ✓Key Vault for secrets — All secrets, certificates, and encryption keys stored in Azure Key Vault with soft delete and purge protection enabled.
  • ✓Data classification applied — All data assets classified using Microsoft Purview sensitivity labels. PII detection and masking enabled in ingestion pipelines.

Monitoring & Response

  • ✓Diagnostic settings enabled — All AI resources send logs and metrics to a central Log Analytics workspace. Retention set to ≥90 days for compliance.
  • ✓Microsoft Defender for Cloud — Defender plans enabled for all AI-related resource types. Security recommendations reviewed and remediated weekly.
  • ✓Alerts configured — Action groups set up for critical, warning, and informational alerts. Runbooks defined for each alert type.
  • ✓Incident response plan — Documented procedures for security incidents including breach notification timelines, escalation paths, and containment steps.

Network Security Deep Dive

Network isolation is a fundamental security control for AI workloads. Azure AI Foundry supports private endpoints, virtual network integration, and managed network configurations:

Key Takeaways

  1. RBAC is non-negotiable — Assign the minimum role at the narrowest scope. Use Entra ID security groups, not individual assignments.
  2. Managed Identity eliminates secrets — DefaultAzureCredential provides seamless, credential-free authentication for all Azure AI Foundry resources.
  3. Monitor everything — API latency, error rates, token consumption, and cost must be visible in a single dashboard with proactive alerts.
  4. Content safety is multi-layered — Input filters, output filters, Prompt Shield, custom blocklists, and groundedness detection work together to protect your application.
  5. Network isolation is essential — Private endpoints with disabled public access, combined with NSGs and DNS configuration, create a zero-trust network perimeter.
  6. Compliance requires automation — Azure Policy enforces guardrails at scale; Microsoft Purview tracks data lineage; audit logs provide the evidence trail.
  7. Security is continuous — The hardening checklist is not a one-time exercise. Build security reviews into your CI/CD pipeline and sprint ceremonies.
