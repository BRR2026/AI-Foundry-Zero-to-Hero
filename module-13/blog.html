Introduction

Large language models (LLMs) are remarkably capable — but they only know what was in their training data. When you need answers grounded in your documents, databases, or knowledge bases, you need Retrieval-Augmented Generation (RAG).

RAG is the most widely adopted pattern for building production AI applications that combine the reasoning power of an LLM with the factual accuracy of a retrieval system. In this module you will walk through every stage of a RAG pipeline — from raw documents to grounded, cited answers — using Azure AI Foundry and Azure AI Search.

💡 Key Insight

RAG doesn't replace the model — it augments it. The model does the reasoning; retrieval supplies the facts. This separation is what makes RAG so powerful and composable.

By the end of this module you will understand the four phases of a RAG pipeline, be able to implement each phase with Azure services, and connect them end-to-end into a working question-answering system.

1 · RAG Architecture Overview

Every RAG system, regardless of complexity, follows the same four-phase loop:

📥 Ingestion
→
🔍 Retrieval
→
🧩 Augmentation
→
✨ Generation

Phase 1 — Ingestion

Raw data (PDFs, HTML, Markdown, databases) is loaded, split into semantically meaningful chunks, converted to vector embeddings, and stored in a search index. This phase runs offline — usually as a batch job or scheduled pipeline — and only needs to be repeated when the source data changes.

Phase 2 — Retrieval

When a user asks a question, the query is converted to a vector and used to search the index. The top-k most relevant chunks are returned. Retrieval can use pure vector search, keyword search, or — most commonly — hybrid search that combines both.

Phase 3 — Augmentation

The retrieved chunks are inserted into the model's prompt alongside the user's original question and a system instruction. This step is sometimes called grounding because it grounds the model in factual evidence rather than relying on parametric memory.

Phase 4 — Generation

The augmented prompt is sent to the LLM, which synthesizes a coherent answer. Because the model can see the retrieved text, it can cite sources, quote passages, and acknowledge when the context does not contain an answer.

✅ Why RAG Beats Fine-Tuning for Most Use Cases

Fine-tuning bakes knowledge into model weights — it's expensive, hard to update, and difficult to audit. RAG keeps knowledge in a searchable index that you can update in minutes, version, and inspect. Use fine-tuning for style or format; use RAG for facts and recency.

2 · Document Ingestion Pipeline

Ingestion transforms raw, unstructured data into a searchable index. It consists of four sub-steps: loading, chunking, embedding, and indexing.

2.1 Document Loading

The first step is reading documents into a normalized text format. Azure AI Foundry supports loading from multiple sources:

  • Azure Blob Storage — PDFs, Word documents, images (with OCR)
  • SharePoint / OneLake — enterprise content libraries
  • Web crawlers — HTML pages via URL
  • Databases — SQL or Cosmos DB rows

For PDF extraction, Azure AI Document Intelligence provides layout-aware parsing that preserves tables, headers, and reading order — far superior to naive text extraction.

Python from azure.ai.documentintelligence import DocumentIntelligenceClient from azure.identity import DefaultAzureCredential # Initialize the Document Intelligence client client = DocumentIntelligenceClient( endpoint="https://<your-resource>.cognitiveservices.azure.com", credential=DefaultAzureCredential() ) # Analyze a PDF using the prebuilt layout model with open("company-handbook.pdf", "rb") as f: poller = client.begin_analyze_document( model_id="prebuilt-layout", body=f, content_type="application/pdf" ) result = poller.result() # result.content contains the full extracted text # result.tables contains structured table data print(f"Extracted {len(result.pages)} pages")

2.2 Chunking Strategies

Raw documents must be split into smaller segments — chunks — that fit within the model's context window and carry self-contained meaning. The choice of chunking strategy dramatically impacts retrieval quality.

Strategy Description Best For
Fixed-size Split every N tokens with overlap Simple documents, logs
Sentence-based Split on sentence boundaries Prose, articles
Semantic Use embeddings to detect topic shifts Mixed-format documents
Document-aware Split on headings, sections, pages Structured reports, manuals
Recursive Hierarchical split: section → paragraph → sentence Complex, nested documents
⚠️ Chunk Size Matters

Chunks that are too large dilute relevance — the model sees noise alongside the answer. Chunks that are too small lose context — the answer fragment is meaningless in isolation. A common starting point is 512–1024 tokens with 10–15% overlap.

Python from azure.ai.generative import split_documents # Fixed-size chunking with overlap chunks = split_documents( documents=documents, chunk_size=1024, # tokens per chunk chunk_overlap=128, # overlapping tokens between chunks use_token_count=True ) print(f"Created {len(chunks)} chunks from {len(documents)} documents") # Each chunk retains metadata: source file, page, section for chunk in chunks[:3]: print(f" Source: {chunk.metadata['source']} | Tokens: {chunk.token_count}")

2.3 Embedding Generation

Each chunk is converted to a dense vector — a numerical fingerprint that captures its semantic meaning. Azure AI Foundry provides access to embedding models like text-embedding-3-large (3072 dimensions) and text-embedding-3-small (1536 dimensions).

Python from azure.ai.projects import AIProjectClient from azure.identity import DefaultAzureCredential project = AIProjectClient( credential=DefaultAzureCredential(), endpoint="https://<your-project>.services.ai.azure.com" ) # Generate embeddings for a batch of chunks response = project.inference.embed( model="text-embedding-3-large", input=[chunk.text for chunk in chunks] ) # Each embedding is a list of floats for i, item in enumerate(response.data): chunks[i].embedding = item.embedding print(f"Chunk {i}: {len(item.embedding)}-dim vector")

2.4 Indexing in Azure AI Search

The final ingestion step pushes the chunks and their embeddings into an Azure AI Search index. The index stores the text, metadata, and vector alongside keyword-searchable fields.

Python from azure.search.documents import SearchClient from azure.search.documents.indexes import SearchIndexClient from azure.search.documents.indexes.models import ( SearchIndex, SearchField, SearchFieldDataType, VectorSearch, HnswAlgorithmConfiguration, VectorSearchProfile, SearchableField, SimpleField ) # Define the index schema index = SearchIndex( name="rag-knowledge-base", fields=[ SimpleField(name="id", type=SearchFieldDataType.String, key=True), SearchableField(name="content", type=SearchFieldDataType.String), SearchableField(name="title", type=SearchFieldDataType.String), SimpleField(name="source", type=SearchFieldDataType.String, filterable=True), SearchField( name="content_vector", type=SearchFieldDataType.Collection(SearchFieldDataType.Single), searchable=True, vector_search_dimensions=3072, vector_search_profile_name="hnsw-profile" ), ], vector_search=VectorSearch( algorithms=[HnswAlgorithmConfiguration(name="hnsw-config")], profiles=[VectorSearchProfile( name="hnsw-profile", algorithm_configuration_name="hnsw-config" )] ) ) # Create the index index_client = SearchIndexClient( endpoint="https://<search-service>.search.windows.net", credential=DefaultAzureCredential() ) index_client.create_or_update_index(index) # Upload documents search_client = SearchClient( endpoint="https://<search-service>.search.windows.net", index_name="rag-knowledge-base", credential=DefaultAzureCredential() ) docs = [ { "id": chunk.id, "content": chunk.text, "title": chunk.metadata["title"], "source": chunk.metadata["source"], "content_vector": chunk.embedding } for chunk in chunks ] result = search_client.upload_documents(documents=docs) print(f"Indexed {len(docs)} chunks")

3 · Retrieval Strategies

Once documents are indexed, you need an effective retrieval strategy. The goal is to find the most relevant chunks for a given user query with high precision and recall.

3.1 Vector Search

Vector search converts the user query into an embedding and finds chunks with the most similar vectors using cosine similarity or dot product. It excels at finding semantically related content even when exact keywords don't match.

Python from azure.search.documents.models import VectorizedQuery # Embed the user's question query_embedding = project.inference.embed( model="text-embedding-3-large", input=["What is the company's parental leave policy?"] ).data[0].embedding # Pure vector search results = search_client.search( search_text=None, vector_queries=[ VectorizedQuery( vector=query_embedding, k_nearest_neighbors=5, fields="content_vector" ) ] ) for r in results: print(f"Score: {r['@search.score']:.4f} | {r['content'][:80]}...")

3.2 Hybrid Search

Hybrid search combines keyword (BM25) and vector results using Reciprocal Rank Fusion (RRF). This approach captures both exact keyword matches and semantic similarity — consistently outperforming either method alone.

Python # Hybrid search: keyword + vector combined results = search_client.search( search_text="parental leave policy", # keyword component vector_queries=[ VectorizedQuery( vector=query_embedding, # vector component k_nearest_neighbors=5, fields="content_vector" ) ], top=5 ) for r in results: print(f"[Hybrid] Score: {r['@search.score']:.4f} | {r['title']}")

3.3 Semantic Ranking

Azure AI Search offers a built-in semantic ranker — a cross-encoder model that re-scores the initial results for deeper relevance. Semantic ranking is applied as a second pass after hybrid retrieval and dramatically improves precision.

Python from azure.search.documents.models import QueryType # Hybrid search with semantic ranking results = search_client.search( search_text="parental leave policy", vector_queries=[ VectorizedQuery( vector=query_embedding, k_nearest_neighbors=50, fields="content_vector" ) ], query_type=QueryType.SEMANTIC, semantic_configuration_name="my-semantic-config", top=5 ) for r in results: print(f"Reranker Score: {r['@search.reranker_score']:.4f}") print(f" Caption: {r['@search.captions'][0].text}")
🔑

Keyword Search (BM25)

Statistical term matching. Great for exact terms, product codes, and proper nouns that embed poorly.

🧠

Vector Search

Semantic similarity via embeddings. Finds conceptually related content even without keyword overlap.

⚡

Hybrid Search

Combines BM25 + vector via RRF fusion. Best of both worlds — the recommended default approach.

🎯

Semantic Ranking

Cross-encoder re-ranking on top of hybrid results. Highest precision for complex, nuanced queries.

Strategy Precision Latency When to Use
Keyword only Medium ~5 ms Exact term lookups, codes, IDs
Vector only High ~15 ms Conceptual / natural language queries
Hybrid (keyword + vector) Higher ~20 ms General-purpose retrieval (recommended default)
Hybrid + Semantic Ranking Highest ~120 ms Complex queries needing deep relevance

4 · Grounding & Prompt Construction

Retrieval gives you the right chunks — but how you present them to the model matters just as much. Grounding is the practice of injecting retrieved context into the prompt so the model can reason over real data instead of hallucinating answers.

4.1 The Grounded Prompt Pattern

A well-structured RAG prompt has three components arranged in a specific order:

  1. System Prompt — defines the assistant's role, behavior constraints, and citation rules
  2. Retrieved Context — the top-k chunks from the search index, clearly delimited
  3. User Query — the original question from the user
Prompt Template ## System Prompt You are an AI assistant for Contoso Corporation. Answer questions using ONLY the provided context. Rules: - If the context does not contain the answer, say "I don't have enough information to answer that question." - Cite your sources using [Source: filename] notation. - Be concise and factual. - Never fabricate information. ## Retrieved Context <context> [Source: handbook-section-4.pdf, Page 12] Contoso provides 16 weeks of paid parental leave for all full-time employees, regardless of gender. Leave can be taken in a single block or split across the first year... [Source: benefits-faq.pdf, Page 3] Parental leave is available to employees who have completed at least 90 days of employment. Both birth and adoptive parents qualify... </context> ## User Query What is the parental leave policy at Contoso?

4.2 Prompt Construction Best Practices

  • Delimiter retrieved context — use clear markers (<context> tags, markdown headers, or XML) so the model knows where context begins and ends
  • Include source metadata — filenames, page numbers, and section titles enable accurate citations
  • Limit context size — include only the top 3–5 most relevant chunks; more isn't always better
  • Instruct the model to refuse — always include a fallback instruction for when the context is insufficient
  • Order matters — place the most relevant chunks first; models pay more attention to early context
🚨 Anti-Pattern: Stuffing the Context Window

Dumping 50 chunks into the prompt is counterproductive. Models suffer from the "lost in the middle" problem — they attend strongly to the beginning and end of the context but can miss information in the middle. Stick to 3–5 highly relevant chunks and let the retrieval system do the filtering.

4.3 Building the Prompt in Code

Python def build_grounded_prompt(query: str, search_results) -> list: """Construct a grounded prompt from search results.""" # Format retrieved chunks with source attribution context_parts = [] for i, result in enumerate(search_results): source = result["source"] content = result["content"] context_parts.append( f"[Source {i+1}: {source}]\n{content}" ) context_block = "\n\n".join(context_parts) return [ { "role": "system", "content": """You are an AI assistant for Contoso Corporation. Answer questions using ONLY the provided context. If the context doesn't contain the answer, say so. Cite sources using [Source N] notation.""" }, { "role": "user", "content": f"""<context> {context_block} </context> Question: {query}""" } ]

5 · End-to-End RAG with Azure AI Foundry

Azure AI Foundry provides two streamlined paths to build production RAG systems: the Agents API with built-in file search, and a manual pipeline using AI Search + chat completions. Both approaches converge on the same architecture but differ in how much the platform handles for you.

5.1 Option A — Agents API with File Search

The Azure AI Foundry Agents API provides a fully managed RAG experience. You upload files, the platform handles chunking, embedding, and indexing automatically, and the agent performs retrieval-augmented generation at query time.

Python from azure.ai.projects import AIProjectClient from azure.ai.projects.models import ( FileSearchTool, FilePurpose, VectorStoreDataSource, VectorStoreDataSourceAssetType, ) from azure.identity import DefaultAzureCredential # Connect to your AI Foundry project project = AIProjectClient( credential=DefaultAzureCredential(), endpoint="https://<your-project>.services.ai.azure.com" ) # Step 1 — Upload the PDF to the project file = project.agents.upload_file_and_poll( file_path="company-handbook.pdf", purpose=FilePurpose.AGENTS ) print(f"Uploaded file ID: {file.id}") # Step 2 — Create a vector store from the uploaded file vector_store = project.agents.create_vector_store_and_poll( file_ids=[file.id], name="company-knowledge-base" ) print(f"Vector store ID: {vector_store.id}") # Step 3 — Create an agent with file search enabled agent = project.agents.create_agent( model="GPT-5.1", name="handbook-assistant", instructions="""You are a helpful assistant that answers questions about the company handbook. Always cite the specific section or page when answering. If you cannot find the answer in the provided documents, say so clearly.""", tools=[FileSearchTool()], tool_resources={ "file_search": { "vector_store_ids": [vector_store.id] } } ) print(f"Agent ID: {agent.id}")
Python — Querying the Agent # Step 4 — Create a thread and ask a question thread = project.agents.create_thread() project.agents.create_message( thread_id=thread.id, role="user", content="What is the parental leave policy?" ) # Step 5 — Run the agent (retrieval + generation) run = project.agents.create_and_process_run( thread_id=thread.id, agent_id=agent.id ) # Step 6 — Read the grounded response messages = project.agents.list_messages(thread_id=thread.id) for msg in messages: if msg.role == "assistant": for block in msg.content: print(block.text.value) # Citations are included as annotations for ann in block.text.annotations: print(f" 📎 {ann.file_citation.file_id}")
✅ When to Use the Agents API for RAG

The Agents API is ideal when you want minimal infrastructure — the platform handles chunking, embedding, indexing, and retrieval. It's the fastest path to a working RAG prototype. For production systems with custom retrieval logic, large-scale indexes, or hybrid search, use Option B with Azure AI Search directly.

5.2 Option B — AI Search + Chat Completions

For full control over the retrieval pipeline, use Azure AI Search for indexing and retrieval, then pass the results into a chat completion call. This approach lets you tune every parameter: chunk size, embedding model, search strategy, re-ranking, and prompt template.

Python from azure.ai.projects import AIProjectClient from azure.search.documents import SearchClient from azure.search.documents.models import VectorizedQuery, QueryType from azure.identity import DefaultAzureCredential credential = DefaultAzureCredential() # Initialize clients project = AIProjectClient( credential=credential, endpoint="https://<your-project>.services.ai.azure.com" ) search_client = SearchClient( endpoint="https://<search-service>.search.windows.net", index_name="rag-knowledge-base", credential=credential ) # Step 1 — Embed the user query user_query = "What is the parental leave policy?" query_vector = project.inference.embed( model="text-embedding-3-large", input=[user_query] ).data[0].embedding # Step 2 — Hybrid search with semantic ranking results = search_client.search( search_text=user_query, vector_queries=[VectorizedQuery( vector=query_vector, k_nearest_neighbors=50, fields="content_vector" )], query_type=QueryType.SEMANTIC, semantic_configuration_name="default", top=5 ) # Step 3 — Build the grounded prompt messages = build_grounded_prompt(user_query, list(results)) # Step 4 — Generate the answer response = project.inference.complete( model="GPT-5.1", messages=messages, temperature=0.3, max_tokens=800 ) print(response.choices[0].message.content)

Best Practices

Building a RAG pipeline that works in demos is easy. Building one that works in production requires attention to these details:

Ingestion Best Practices

  • Preserve document structure — use layout-aware extractors (Document Intelligence) instead of raw text extraction; tables, headers, and lists carry crucial context
  • Enrich chunks with metadata — store source filename, page number, section heading, and creation date alongside each chunk for filtering and citations
  • Use consistent chunk sizes — mixing 100-token and 5000-token chunks in the same index leads to uneven retrieval quality
  • Add overlap — 10–15% token overlap between chunks prevents sentences from being split across boundaries

Retrieval Best Practices

  • Start with hybrid search — pure vector or pure keyword search each have blind spots; hybrid search with RRF is the recommended baseline
  • Use semantic ranking for complex queries — the cross-encoder re-ranker is slower but significantly more accurate for ambiguous or multi-part questions
  • Retrieve more, present fewer — retrieve top-50, re-rank, then feed top-5 to the model
  • Apply metadata filters — if the user specifies a date range, department, or document type, filter at the index level before ranking

Generation Best Practices

  • Use low temperature — set temperature=0.1–0.3 for factual Q&A to reduce hallucination
  • Instruct the model to cite — explicitly ask for citations in the system prompt and provide source metadata in the context
  • Handle "no answer" gracefully — the system prompt must instruct the model to decline when context is insufficient, rather than guessing
  • Test with adversarial queries — ask questions the documents don't cover; verify the model correctly refuses to answer
⚠️ Common Pitfall: Ignoring Evaluation

A RAG pipeline is only as good as its retrieval quality. Track metrics like retrieval recall@k (does the right chunk appear in the top-k?), answer faithfulness (does the answer match the retrieved context?), and groundedness (does the answer avoid hallucination?). Azure AI Foundry's evaluation SDK makes this straightforward.
