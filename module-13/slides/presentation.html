🤔 Why RAG?

  • Hallucination Problem: LLMs generate plausible but incorrect answers from training data alone
  • Stale Knowledge: Model training data has a cutoff date — your company data changes daily
  • No Source of Truth: Without grounding, users can't verify where answers come from
  • RAG Solution: Retrieve real documents at query time → ground every answer in facts

💡 RAG doesn't change the model — it changes what the model sees at inference time.

🏗️ RAG Architecture

🔍

1. Retrieve

Query Azure AI Search to find relevant document chunks using vector/hybrid search

🔗

2. Augment

Inject retrieved context into the prompt alongside the user's question

✨

3. Generate

Model produces a grounded response with citations from retrieved documents

✅ The key insight: the model only answers from your data, not its training knowledge.

🔎 Setting Up Azure AI Search

  • Create Search Resource: Basic tier or above (supports vector search)
  • Document Processing: Chunk documents into 500-token overlapping segments
  • Vectorize Content: Generate embeddings with text-embedding-ada-002
  • Index Schema: Content field + vector field + metadata (source, page number)
  • Search Strategy: Hybrid search = keyword (BM25) + vector similarity
# Index schema fields fields = [ SimpleField(name="id", type=String, key=True), SearchableField(name="content", type=String), SimpleField(name="source_file", type=String), SearchField(name="content_vector", vector_search_dimensions=1536) ]

📡 "On Your Data" in AI Foundry

  • No-code RAG setup directly in the AI Foundry portal
  • Connect a deployed model → Azure AI Search index in clicks
  • Supports hybrid search, semantic ranking, and vector search
  • Built-in citation extraction and source attribution
  • Great for prototyping before building custom pipelines

📚 Citation Handling & Source Attribution

What You Get

  • Inline citation markers: [doc1], [doc2]
  • Source file paths and titles
  • Content snippets from matched chunks
  • Relevance scores per document

Best Practices

  • Parse the context.citations array
  • Map indices to original documents
  • Display clickable source links
  • Verify citation accuracy in testing

⚠️ Always verify citations — the model may occasionally mis-attribute content to the wrong source.

💬 System Prompt Design for RAG

SYSTEM_PROMPT = """You are a helpful company assistant. Answer questions based ONLY on the provided documents. Rules: 1. Only use information from retrieved documents. 2. If documents don't contain the answer, say: "I don't have enough information to answer." 3. Always cite sources using [doc1], [doc2] notation. 4. Synthesize from multiple sources when relevant. 5. Never fabricate information."""

✅ A strong system prompt is your first line of defense against hallucination in RAG apps.

⚠️ Error Handling & Edge Cases

Common Edge Cases

  • No relevant documents found
  • Conflicting information across sources
  • Query too vague for meaningful retrieval
  • Documents in multiple languages

Mitigation Strategies

  • Set in_scope: true to limit to indexed data
  • Tune top_n_documents (3–5 optimal)
  • Add retry logic with exponential backoff
  • Log retrieval scores for monitoring

⚖️ RAG vs Fine-Tuning

Aspect RAG Fine-Tuning
Knowledge Update Update search index (minutes) Retrain model (hours/days)
Data Freshness Real-time with index updates Frozen at training time
Citations Built-in source attribution Not natively supported
Best For Dynamic, document-based Q&A Style, tone, or format changes
Cost Search + inference costs Training + inference costs

🎯 Key Takeaways

  • RAG = Retrieve → Augment → Generate — the most important pattern for enterprise AI
  • Azure AI Search provides vector/hybrid retrieval for RAG pipelines
  • "On Your Data" enables no-code RAG prototyping in AI Foundry
  • Citations matter — parse, display, and verify source attribution
  • System prompts are your first defense against hallucination
  • RAG ≠ Fine-tuning — choose based on your data update frequency
