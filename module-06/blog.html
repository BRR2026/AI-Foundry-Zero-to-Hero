Introduction: Why Prompt Engineering Matters

Welcome back to Azure AI Foundry: Zero to Hero. In Module 5, you explored the model catalogue and learned how to deploy foundation models inside your AI Foundry project. Now that you have a model running, the obvious question is: how do you get it to do what you want?

The answer is prompt engineering — the discipline of crafting inputs (prompts) that guide a large language model (LLM) to produce accurate, consistent, and useful outputs. If the model is the engine, the prompt is the steering wheel, the accelerator, and the brakes all in one.

Prompt engineering is not merely “asking a question.” It is a structured practice that involves:

  • Defining the model’s persona and constraints through system messages
  • Selecting the right technique — zero-shot, few-shot, or chain-of-thought — for the complexity of the task
  • Building reusable templates that separate logic from data
  • Tuning generation parameters like temperature, top_p, and penalties to control output behaviour
  • Iterating and testing systematically to handle edge cases
ℹ️ Why This Matters for Production

In a prototype, a mediocre prompt might seem “good enough.” In production, that same prompt will fail on 15% of inputs, generate inconsistent formats, waste tokens, and confuse downstream systems. Investing in prompt engineering upfront saves debugging time, reduces API costs, and builds user trust.

In this module, you will build a working customer feedback classifier in the AI Foundry Playground — step by step, technique by technique. By the end, you will have a robust prompt that handles ambiguity, edge cases, and multi-language input with confidence.

Let’s begin with the foundation: understanding how messages are structured in the chat-completion API.

Understanding Message Roles

Every call to a chat-completion model (GPT-5.1, GPT-4.1, and similar models deployed in AI Foundry) uses an array of messages. Each message has a role and content. There are three roles you need to know:

The System Message

The system message is the most important part of your prompt. It is processed first and establishes the model’s persona, ground rules, output format, and behavioural constraints. Think of it as the “operating manual” you hand to the model before it sees any user input.

{
  "role": "system",
  "content": "You are a senior customer support agent for Contoso Electronics.\nYou respond in a professional, empathetic tone.\nAlways suggest a concrete next step.\nIf you don't know the answer, say so — never fabricate information.\nRespond in valid JSON with keys: response, next_step, confidence."
}
💡 Tip — System Message Best Practices

Be explicit. Don’t assume the model will infer your constraints. If you want JSON output, say “respond in valid JSON.” If you want short answers, say “respond in 2–3 sentences maximum.” Ambiguity in the system prompt leads to ambiguity in the output.

The User Message

The user message contains the end user’s input — their question, the data they want processed, or the task they need completed. In a multi-turn conversation, each new user turn adds another user message to the array.

{
  "role": "user",
  "content": "My order #12345 arrived with a broken screen. I need a replacement."
}

The Assistant Message

The assistant message represents the model’s previous responses. It serves two critical purposes:

  1. Conversation continuity: In multi-turn conversations, assistant messages preserve context so the model “remembers” what it said earlier.
  2. Few-shot examples: You can insert synthetic assistant messages to show the model the exact output format you expect (more on this in the Few-Shot section).

How Messages Flow Together

Here is a complete multi-turn conversation showing all three roles:

// Complete messages array for a chat-completion API call
{
  "messages": [
    {
      "role": "system",
      "content": "You are a helpful travel assistant. Respond in JSON."
    },
    {
      "role": "user",
      "content": "Plan a 3-day trip to Tokyo for a food lover."
    },
    {
      "role": "assistant",
      "content": "{\"destination\": \"Tokyo\", \"itinerary\": [...]}"
    },
    {
      "role": "user",
      "content": "Add a day trip to Nikko and increase the budget."
    }
  ],
  "temperature": 0.7,
  "max_tokens": 800
}
ℹ️ Message Order Matters

The model processes messages sequentially. The system message is always first and carries the most weight in shaping behaviour. User and assistant messages alternate to create a conversation history. The model’s new response is based on all messages in the array — not just the last one.

The Anatomy of a Great System Prompt

A production-grade system prompt typically contains these components, in order:

  1. Role definition: “You are a [specific role] for [company/context].”
  2. Task description: Exactly what the model should do.
  3. Constraints: What the model must NOT do.
  4. Output format: The exact structure of the expected response (JSON schema, markdown, bullets).
  5. Tone/style: Professional, casual, technical, empathetic.
  6. Edge case handling: What to do with unknown inputs, empty data, or off-topic queries.
// Production-grade system prompt template
"You are a senior medical triage assistant for HealthFirst Clinic.

TASK:
Given a patient's symptom description, assess urgency and recommend
the appropriate care pathway.

CONSTRAINTS:
- Never diagnose specific conditions.
- Never recommend medications.
- Always advise seeing a medical professional for serious symptoms.
- If symptoms suggest an emergency, respond with urgency 'EMERGENCY'.

OUTPUT FORMAT:
Respond ONLY with valid JSON:
{
  \"urgency\": \"EMERGENCY | URGENT | ROUTINE | SELF_CARE\",
  \"care_pathway\": \"string\",
  \"reasoning\": \"one sentence\",
  \"disclaimer\": \"This is not a medical diagnosis.\"
}

TONE: Professional, calm, empathetic.

EDGE CASES:
- Empty input: urgency = 'UNKNOWN'.
- Non-medical queries: urgency = 'NOT_APPLICABLE'."

Zero-Shot Prompting

Zero-shot prompting means giving the model a task description without any examples. You rely entirely on the model’s pre-training to understand what you want. This is the simplest technique and works surprisingly well for straightforward tasks.

When Zero-Shot Works Well

  • Simple classification: Sentiment analysis, spam detection, topic categorisation
  • Translation: “Translate the following English text to French.”
  • Summarisation: “Summarise this article in 3 bullet points.”
  • Extraction: “Extract all email addresses from this text.”
  • Q&A: “What is the capital of Norway?”

Zero-Shot Examples

Example 1: Sentiment Classification

// System message
"Classify the sentiment of the user's text as exactly one of:
positive, negative, or neutral. Respond with a single word."

// User message
"The battery life on this phone is absolutely incredible.
I can go two full days without charging!"

// Expected response
"positive"

Example 2: Entity Extraction

// System message
"Extract all named entities (people, places, organisations) from the
user's text. Return as a JSON array of objects with 'entity' and 'type'."

// User message
"Satya Nadella announced at Microsoft Build that Azure AI Foundry
will expand to three new regions in Europe by Q3 2026."

// Expected response
[
  {"entity": "Satya Nadella", "type": "person"},
  {"entity": "Microsoft Build", "type": "event"},
  {"entity": "Azure AI Foundry", "type": "product"},
  {"entity": "Europe", "type": "location"}
]

Example 3: Summarisation

// System message
"You are a news editor. Summarise the user's article in exactly
3 bullet points. Each bullet must be one sentence, max 20 words."

// User message
"[Long article about renewable energy policy changes in the EU...]"

// Expected response
"- The EU approved a 45% renewable energy target by 2030.
- Solar and wind subsidies will increase by €12 billion annually.
- Member states must submit updated energy plans by March 2027."

When Zero-Shot Falls Short

Zero-shot prompting struggles when:

  • The output format is complex or domain-specific
  • The task requires nuanced judgement across many categories
  • You need high consistency across many different inputs
  • The model hasn’t seen enough similar tasks in pre-training

When zero-shot doesn’t produce reliable results, it’s time to bring in examples. That’s few-shot prompting.

Few-Shot Prompting

Few-shot prompting provides the model with 2–5 examples of the desired input→output pattern before presenting the actual task. The examples act as a “teaching” mechanism, showing the model exactly what format, style, and level of detail you expect.

Why Few-Shot Works So Well

LLMs are exceptional pattern matchers. When you provide examples, the model identifies the pattern — the structure, the vocabulary, the formatting — and replicates it for new inputs. This is called in-context learning, and it is one of the most powerful capabilities of modern language models.

✅ Key Benefit

Few-shot prompting can increase output consistency from ~70% to ~95% for classification and formatting tasks — without any model fine-tuning.

Building Effective Few-Shot Prompts

Step 1: Choose Diverse Examples

Your examples should cover the range of expected inputs — not just the easy cases. Include at least one example from each category or output type.

Step 2: Use Consistent Formatting

Every example must follow the exact same format. If your output is JSON, every example must be valid JSON. If it’s a single word, every example must be a single word.

Step 3: Place Examples in the Right Location

Few-shot examples can be placed in the system message or as alternating user/assistant message pairs. Both work; the choice depends on your use case:

ApproachProsCons
In system message Clean conversation; invisible to end users Uses system token budget
As user/assistant pairs Natural structure; easy to add/remove Visible in conversation history

Few-Shot Example: Customer Feedback Classifier

// System message
"You are a customer feedback classifier for Contoso Electronics.

Classify each message into ONE category:
- product_quality
- shipping
- customer_service
- pricing
- website_app

Also determine sentiment (positive, negative, neutral)
and confidence (high, medium, low).

Respond ONLY with valid JSON:
{
  \"category\": \"...\",
  \"sentiment\": \"...\",
  \"confidence\": \"...\",
  \"reasoning\": \"One sentence.\"
}"

// Few-shot example 1
{ "role": "user",
  "content": "The headphones broke after one week." }
{ "role": "assistant",
  "content": "{\"category\":\"product_quality\",\"sentiment\":\"negative\",\"confidence\":\"high\",\"reasoning\":\"Product defect reported.\"}" }

// Few-shot example 2
{ "role": "user",
  "content": "Delivery took 3 weeks — way too long!" }
{ "role": "assistant",
  "content": "{\"category\":\"shipping\",\"sentiment\":\"negative\",\"confidence\":\"high\",\"reasoning\":\"Slow delivery complaint.\"}" }

// Few-shot example 3
{ "role": "user",
  "content": "Love the new app update — checkout is so smooth!" }
{ "role": "assistant",
  "content": "{\"category\":\"website_app\",\"sentiment\":\"positive\",\"confidence\":\"high\",\"reasoning\":\"Customer praises app experience.\"}" }

// Actual user input to classify
{ "role": "user",
  "content": "$299 seems steep but the build quality is decent." }
💡 Tip — The Magic Number

Research shows that 3–5 examples is the sweet spot for most tasks. Fewer than 3 may not establish a clear pattern; more than 5 rarely improves performance and wastes tokens. Prioritise diversity over quantity.

Few-Shot for Code Generation

// Show the model your coding style
// Example 1:
User: "Write a function that doubles a number."
Assistant:
def double(n: int) -> int:
    """Return the input multiplied by 2."""
    return n * 2

// Example 2:
User: "Write a function to check if a string is a palindrome."
Assistant:
def is_palindrome(text: str) -> bool:
    """Return True if text reads the same forwards and backwards."""
    cleaned = text.lower().replace(" ", "")
    return cleaned == cleaned[::-1]

// Now the model will follow the same pattern:
// - Type hints
// - Docstring
// - Concise implementation
User: "Write a function to calculate compound interest."

Chain-of-Thought (CoT) Prompting

Chain-of-thought prompting instructs the model to show its reasoning process step by step before arriving at a final answer. Instead of jumping directly to a conclusion, the model “thinks aloud,” which dramatically improves accuracy for complex tasks.

Why CoT Works

LLMs generate text token by token. When you ask the model to output intermediate reasoning steps, those tokens become part of the context for subsequent tokens — effectively giving the model a “scratchpad” to work through the problem. This is particularly effective for:

  • Multi-step arithmetic: “If a shirt costs $85 and is discounted 30%, then taxed at 8.5%...”
  • Logical reasoning: “If all cats are mammals, and some mammals are pets...”
  • Ambiguous classification: Feedback that mentions multiple topics
  • Code debugging: “Trace through this code step by step to find the bug.”

CoT Approach 1: Simple Trigger Phrase

// User message
"A store offers 25% off a $120 jacket. There's an additional
10% loyalty discount applied AFTER the first discount.
What's the final price?

Let's think step by step."

// Model response
"Step 1: Calculate the 25% discount.
  $120 x 0.25 = $30 discount
  Price after first discount: $120 - $30 = $90

Step 2: Calculate the 10% loyalty discount on the new price.
  $90 x 0.10 = $9 discount
  Price after loyalty discount: $90 - $9 = $81

Final price: $81"

CoT Approach 2: Structured Reasoning Framework

// System message with reasoning framework
"You are a decision analysis assistant.

When presented with a problem, follow this framework:
1. IDENTIFY: State the key facts and variables.
2. ANALYZE: Break down into sub-problems.
3. EVALUATE: Consider each sub-problem independently.
4. SYNTHESIZE: Combine evaluations into a coherent answer.
5. CONCLUDE: State your final answer clearly.

Output your reasoning, then your answer in JSON:
{\"answer\": \"...\", \"confidence\": \"high|medium|low\"}"

CoT Approach 3: CoT + Few-Shot (Most Powerful)

Combine chain-of-thought with few-shot examples for maximum effectiveness. Show the model how to reason by including reasoning traces in your examples:

// Few-shot example with reasoning trace
User: "Product is amazing but shipping was terrible. Classify."

Assistant: "REASONING:
1. The feedback mentions two topics: product (amazing) and shipping (terrible).
2. The primary complaint is about shipping — the negative sentiment is stronger.
3. Product mention is secondary and positive.
4. Primary category: shipping. Sentiment: negative.

{\"category\": \"shipping\", \"sentiment\": \"negative\", \"confidence\": \"medium\",
 \"reasoning\": \"Primary complaint is slow shipping despite positive product mention.\"}"
✅ CoT Accuracy Gains

Studies show that chain-of-thought prompting improves accuracy by 15–30% on reasoning-heavy tasks compared to direct prompting. For math word problems, the improvement can exceed 40%. The key insight: letting the model "show its work" produces better final answers.

When NOT to Use CoT

Chain-of-thought adds tokens (and cost). Skip it when:

  • The task is simple and well-defined (e.g., “translate this sentence”)
  • You need the shortest possible response
  • The model already achieves high accuracy without reasoning
  • Latency is critical and every token counts

Prompt Templates & Best Practices

As your prompt engineering matures, you will find yourself reusing the same prompt structures across different inputs. This is where prompt templates come in — reusable prompt skeletons with variable placeholders that separate logic from data.

Why Templates Matter

  • Consistency: Every invocation uses the same structure, reducing drift
  • Testability: You can unit-test a template with different inputs
  • Collaboration: Teams can share, review, and version templates
  • Maintenance: Update the template once, and all consumers benefit

Template Syntax

Use double-curly-brace placeholders for variables. This is the convention used by most prompt engineering frameworks, including Azure AI Foundry’s Prompt Flow:

// Prompt template with placeholders
"You are a {{role}} assistant for {{company}}.

TASK:
Summarise the following article in {{num_points}} bullet points.
Use {{language}} language. Be concise and factual.

ARTICLE:
{{article_text}}

OUTPUT FORMAT:
Respond in JSON:
{
  \"summary\": [\"bullet 1\", \"bullet 2\", ...],
  \"word_count\": number,
  \"key_topics\": [\"topic1\", \"topic2\"]
}"

Template Variables for the Feedback Classifier

// Parameterised version of our feedback classifier
"You are a customer feedback classifier for {{company_name}}.

Classify each message into ONE of these categories:
{{#each categories}}
- {{this}}
{{/each}}

Determine sentiment ({{sentiment_options}}) and confidence ({{confidence_levels}}).

Respond in JSON:
{
  \"category\": \"...\",
  \"sentiment\": \"...\",
  \"confidence\": \"...\",
  \"reasoning\": \"...\" 
}"

// Variables:
// company_name = "Contoso Electronics"
// categories = ["product_quality", "shipping", "customer_service", "pricing", "website_app"]
// sentiment_options = "positive, negative, neutral"
// confidence_levels = "high, medium, low"

Managing Templates in AI Foundry

Azure AI Foundry provides several ways to manage prompt templates:

  1. Prompt Flow: Build visual workflows with template nodes. Each node can have its own prompt template with variables wired to inputs and outputs.
  2. Playground saved configurations: Save system prompts as named configurations for quick switching and A/B testing.
  3. Code-based templates: Use Python string formatting, Jinja2, or Handlebars syntax in your application code.
# Python example using f-strings
def build_classifier_prompt(company: str, categories: list) -> str:
    cats = "\n".join(f"- {c}" for c in categories)
    return f"""You are a feedback classifier for {company}.

Classify each message into ONE category:
{cats}

Respond in JSON with: category, sentiment, confidence, reasoning."""

# Usage
prompt = build_classifier_prompt(
    company="Contoso Electronics",
    categories=["product_quality", "shipping", "customer_service"]
)
print(prompt)
💡 Tip — Version Your Prompts

Treat prompts like code: store them in version control, tag releases (v1.0, v1.1), write changelogs, and run regression tests when you make changes. A “small tweak” to a prompt can change behaviour across thousands of inputs.

Parameters Deep Dive

Beyond the prompt text itself, you can control model behaviour through generation parameters. These settings determine how the model selects tokens during generation and directly affect the creativity, length, and consistency of outputs.

Temperature

Temperature (range: 0–2) controls the randomness of token selection. It is the single most important parameter to understand.

TemperatureBehaviourUse Case
0.0Deterministic — always picks the highest-probability tokenClassification, extraction, factual Q&A
0.3Low variation — mostly predictable with slight flexibilitySummarisation, code generation
0.7Balanced — good mix of quality and creativityGeneral conversation, content creation
1.0High variation — diverse and creative outputsBrainstorming, creative writing
1.5–2.0Very high randomness — may produce incoherent textRarely recommended
 
// Deterministic classification
{
  "temperature": 0.0,
  "messages": [...]
}
// Same input always produces the same output

// Creative brainstorming
{
  "temperature": 0.9,
  "messages": [...]
}
// Each call produces different creative ideas

Top-P (Nucleus Sampling)

Top-P (range: 0–1) is an alternative to temperature that controls token selection by filtering the probability distribution. Instead of adjusting how random the selection is, top_p defines how many tokens are in the candidate pool.

  • top_p: 0.1 — Only the top 10% of probability mass is considered. Very focused.
  • top_p: 0.5 — Top 50% of tokens are candidates. Moderate diversity.
  • top_p: 0.95 — Nearly all tokens are candidates. High diversity.
  • top_p: 1.0 — All tokens are candidates (default).
⚠️ Warning — Don’t Tune Both

Adjust either temperature or top_p — not both at the same time. They both affect randomness, and tuning both simultaneously creates unpredictable interactions. Most practitioners prefer temperature because it’s more intuitive.

Max Tokens

Max tokens sets an upper limit on how many tokens the model can generate in its response. This does not guarantee the response will be that long — it only prevents it from exceeding the limit.

// Short response for classification
{ "max_tokens": 150 }

// Longer response for detailed analysis
{ "max_tokens": 2000 }

// Very long response for code generation
{ "max_tokens": 4096 }
💡 Tip — Token Budgeting

Remember: input tokens + output tokens = total tokens, and both count toward billing. A long system prompt with many few-shot examples uses input tokens, leaving fewer for the output. Monitor token usage in the AI Foundry Playground’s token counter.

Frequency Penalty

Frequency penalty (range: -2 to 2) penalises tokens that have already appeared in the output, proportional to how many times they’ve appeared. Higher values discourage repetition.

  • 0.0 — No penalty (default). The model may repeat phrases.
  • 0.5–1.0 — Mild to moderate penalty. Reduces obvious repetition.
  • 1.5–2.0 — Strong penalty. May force awkward word choices to avoid repetition.

Presence Penalty

Presence penalty (range: -2 to 2) is similar to frequency penalty but applies a flat penalty to any token that has appeared at all — regardless of how many times. It encourages the model to introduce new topics.

  • 0.0 — No penalty (default). Model may stay on one topic.
  • 0.5–1.0 — Encourages topic diversity.
  • 1.5–2.0 — Strongly pushes the model to explore new topics.

Playground Workflow for Prompt Development

  1. Start with a clear system prompt. Write the role, task, constraints, and output format.
  2. Test with zero-shot first. See if the model understands the task without examples.
  3. Add few-shot examples if zero-shot results are inconsistent.
  4. Tune parameters. Start with temperature 0.0 for classification, 0.7 for general tasks.
  5. Test edge cases. Empty input, long input, multi-language, off-topic queries.
  6. Save your configuration. Name it descriptively (e.g., feedback-classifier-v2).
  7. Export. Use the configuration in Prompt Flow, your application code, or the REST API.

Using the Compare Feature

AI Foundry’s Playground includes a Compare feature that lets you test two different prompts or parameter configurations side by side. This is invaluable for A/B testing:

  • Compare zero-shot vs. few-shot on the same inputs
  • Compare temperature 0.0 vs. 0.3 for classification accuracy
  • Compare two different system prompt versions
  • Compare responses from different model deployments
ℹ️ Token Counter

The Playground displays a token count for each request and response. Use this to monitor costs and ensure your prompts fit within the model’s context window. A GPT-5.1 deployment supports up to 128K tokens of context, but costs scale with usage.

Best Practices

After working through techniques and parameters, let’s consolidate the most important best practices for prompt engineering in production systems.

1. Be Specific and Explicit

Vague prompts produce vague results. Compare these two approaches:

// Bad — vague
"Analyze this customer feedback."

// Good — specific
"Classify this customer feedback into one of five categories
(product_quality, shipping, customer_service, pricing, website_app).
Determine sentiment (positive, negative, neutral).
Respond in JSON with keys: category, sentiment, confidence, reasoning."

2. Specify the Output Format Upfront

Always tell the model exactly what format to use. Include a JSON schema, markdown structure, or explicit formatting rules in your system prompt.

3. Provide Context Before the Task

Structure your prompt as: Context → Task → Format. Give the model background information before asking it to act.

4. Use Delimiters for Input Data

When including user data in your prompt, use clear delimiters to separate instructions from data:

"Classify the customer feedback between the triple backticks.

```
The laptop screen is flickering after the update.
```

Respond in JSON."

5. Iterate Systematically

  1. Write v1 of the prompt
  2. Test with 10–20 diverse inputs
  3. Identify failure patterns
  4. Update the prompt to address failures
  5. Re-test all inputs (regression testing)
  6. Repeat until accuracy meets your threshold

6. Handle Edge Cases Explicitly

Models don’t infer edge case handling. If you don’t tell the model what to do with empty input, it will hallucinate. Add explicit rules for:

  • Empty or whitespace-only input
  • Extremely long input
  • Non-English input (if your system is English-only)
  • Off-topic or adversarial input
  • Input that matches multiple categories

7. Minimise Token Usage

Every token costs money. Write concise system prompts that cover all requirements without unnecessary verbosity. Remove filler words, redundant instructions, and examples that don’t add value.

✅ The 80/20 Rule of Prompting

80% of prompt quality comes from three things: (1) a clear role definition, (2) an explicit output format, and (3) 3 good few-shot examples. Get these right first before optimising everything else.

Common Mistakes & Anti-Patterns

Even experienced developers make these mistakes. Learn to recognise and avoid them.

Anti-Pattern 1: The Vague Prompt

// ❌ Bad
"Do something useful with this data."

// ✅ Good
"Extract all email addresses from the following text and return them
as a JSON array of strings. If no emails are found, return []."

Anti-Pattern 2: No Output Format

// ❌ Bad — no format specified
"Analyze this customer review."

// ✅ Good — explicit format
"Analyze this customer review. Respond in JSON:
{\"sentiment\": \"positive|negative|neutral\",
 \"topics\": [\"array of topics\"],
 \"summary\": \"one sentence\"}"

Anti-Pattern 3: Prompt Stuffing

// ❌ Bad — too much, contradictory
"You are an expert in everything. You know all languages, all
domains, all frameworks. You are the best at everything. Consider
all perspectives, all possibilities, all edge cases, all nuances,
all contexts, all implications..."

// ✅ Good — focused
"You are a Python backend developer specialising in FastAPI.
Generate clean, type-hinted code with docstrings."

Anti-Pattern 4: Ignoring Edge Cases

// ❌ Bad — no edge case handling
"Classify this feedback."
// What happens with empty input? Non-English? Off-topic?

// ✅ Good — explicit guardrails
"Classify this feedback into one of 5 categories.
If empty: return {\"category\": \"unclassifiable\"}.
If off-topic: return {\"category\": \"off_topic\"}.
If non-English: attempt classification on translated meaning."

Anti-Pattern 5: Tuning Temperature AND Top-P

// ❌ Bad — both parameters adjusted
{ "temperature": 0.9, "top_p": 0.3 }

// ✅ Good — one or the other
{ "temperature": 0.9, "top_p": 1.0 }  // Tune temperature only
{ "temperature": 1.0, "top_p": 0.3 }  // Tune top_p only

Anti-Pattern 6: No Regression Testing

You update the prompt to fix one failing case, but don’t re-test previous cases. The fix breaks something that was working before. Always maintain a test suite of 10–20 inputs and run all of them after every prompt change.

Mini Hack: Build a Customer Feedback Classifier

Time to put everything together. In this hands-on exercise, you’ll build a complete customer feedback classifier in the AI Foundry Playground.

Scenario

You work for Contoso Electronics. Your team receives thousands of customer feedback messages daily and needs to classify each one into categories for routing to the right team.

Requirements

  • Classify into 5 categories: product_quality, shipping, customer_service, pricing, website_app
  • Determine sentiment: positive, negative, neutral
  • Assign confidence: high, medium, low
  • Output valid JSON
  • Handle edge cases: empty input, off-topic, multi-language
  • Use temperature 0.0 for deterministic results

Step-by-Step Instructions

Step 1: Write the System Prompt

"You are a customer feedback classifier for Contoso Electronics.

Given a customer feedback message, classify it into exactly ONE
of these categories:
- product_quality: Issues or praise about the physical product
- shipping: Delivery speed, packaging, tracking
- customer_service: Interactions with support agents
- pricing: Cost, discounts, value for money
- website_app: Online experience, bugs, usability

Also determine:
- sentiment: positive, negative, or neutral
- confidence: high, medium, or low

For ambiguous messages, think step by step:
1. Identify the primary topic.
2. Determine if a secondary topic exists.
3. Choose the category matching the PRIMARY concern.
4. Assess sentiment based on overall tone.

Respond ONLY with valid JSON:
{
  \"category\": \"one of the 5 categories\",
  \"sentiment\": \"positive|negative|neutral\",
  \"confidence\": \"high|medium|low\",
  \"reasoning\": \"One sentence explaining your classification.\"
}

Edge case rules:
- Empty/unintelligible input: {\"category\": \"unclassifiable\", ...}
- Off-topic (not feedback): {\"category\": \"off_topic\", ...}
- Non-English: Attempt classification based on content."

Step 2: Add Few-Shot Examples

Add these as user/assistant message pairs in the Playground:

// Example 1: product_quality / negative
User: "The headphones broke after just one week of use."
Assistant: "{\"category\":\"product_quality\",\"sentiment\":\"negative\",\"confidence\":\"high\",\"reasoning\":\"Product defect reported within short timeframe.\"}"

// Example 2: customer_service / negative
User: "Called support 3 times and nobody could help."
Assistant: "{\"category\":\"customer_service\",\"sentiment\":\"negative\",\"confidence\":\"high\",\"reasoning\":\"Repeated unsuccessful support interactions.\"}"

// Example 3: website_app / positive
User: "Love the new app update, checkout is so smooth!"
Assistant: "{\"category\":\"website_app\",\"sentiment\":\"positive\",\"confidence\":\"high\",\"reasoning\":\"Customer praises improved app experience.\"}"

Step 3: Set Parameters

temperature:       0.0   // Deterministic
top_p:             1.0   // Default
max_tokens:        200   // Sufficient for JSON
frequency_penalty: 0.0   // No penalty needed
presence_penalty:  0.0   // No penalty needed

Step 4: Test with These Messages

// Test 1: Clear category
"The laptop arrived with a cracked screen. Very disappointed."
// Expected: product_quality / negative / high

// Test 2: Clear category
"Delivery was super fast, got it in 2 days! Great job."
// Expected: shipping / positive / high

// Test 3: Clear category
"Website keeps crashing when I try to checkout on mobile."
// Expected: website_app / negative / high

// Test 4: Ambiguous — pricing or product_quality?
"$299 is steep for what you get, but the build quality is decent."
// Expected: pricing / negative / medium (primary concern is cost)

// Test 5: Multiple categories
"Product is great but shipping took forever and support was useless."
// Expected: CoT reasoning, then primary category

// Test 6: Edge case — empty
""
// Expected: unclassifiable

// Test 7: Edge case — off-topic
"What time does the store close on Sundays?"
// Expected: off_topic

// Test 8: Edge case — non-English
"Die Lieferung war sehr schnell, danke!"
// Expected: shipping / positive / high

Step 5: Evaluate and Iterate

If any test produces unexpected results, refine your prompt. Common fixes:

  • If the model picks the wrong category for ambiguous input, strengthen the CoT instructions
  • If JSON format is inconsistent, add another few-shot example
  • If edge cases fail, add more explicit rules
  • If responses are too verbose, add “be concise” to the system prompt
✅ Success Criteria

Your classifier passes the Mini Hack when it correctly handles all 8 test messages with the expected categories and sentiments. Bonus points for handling the ambiguous cases (Tests 4 and 5) with clear reasoning.

Key Takeaways

  1. System prompts are the foundation. Invest time crafting a clear role, task, constraints, and output format.
  2. Zero-shot works for simple tasks. Start here and add complexity only when needed.
  3. Few-shot examples (3–5) dramatically improve consistency for classification, formatting, and domain-specific tasks.
  4. Chain-of-thought improves reasoning accuracy by 15–30% on complex, multi-step problems.
  5. Prompt templates separate logic from data, enabling reuse, testing, and collaboration.
  6. Temperature 0.0 for deterministic tasks; 0.7–0.9 for creative tasks. Don’t tune both temperature and top_p.
  7. Always specify output format (JSON schema, markdown, etc.) and handle edge cases explicitly.
  8. The AI Foundry Playground is your laboratory — use Compare, token counting, and saved configurations.
  9. Treat prompts like code: version control, regression test, iterate systematically.
  10. 80% of use cases can be solved with the right prompt — no fine-tuning required.
