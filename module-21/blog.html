1. Introduction — Why Visual Orchestration Matters

Building AI applications rarely involves a single model call. Real-world scenarios demand multi-step pipelines — retrieving context, calling language models, processing outputs, routing between different logic branches, and formatting results. Traditionally, this meant writing complex Python code with tangled dependencies and hard-to-debug data flows.

Prompt Flow in Azure AI Foundry changes this paradigm entirely. It provides a visual, graph-based orchestration layer that lets you design, test, debug, and deploy AI workflows with a drag-and-drop interface — while maintaining full code-level control when you need it.

💡 Key Insight: Prompt Flow is not traditional ML pipelines. It is a purpose-built orchestration tool within Azure AI Foundry for composing LLM-based applications — think of it as the "visual IDE" for AI app logic.

In this module, you'll learn to build flows from scratch, master every node type, debug execution with precision, and deploy production-ready endpoints — all within the AI Foundry experience.

2. What Is Prompt Flow?

Prompt Flow is a development tool and runtime engine for building LLM-based applications using a directed acyclic graph (DAG) paradigm. Each node in the graph performs a specific task — calling a model, running Python code, processing data — and data flows from one node to the next through typed connections.

3. When to Use Prompt Flow

Prompt Flow excels in specific scenarios. Understanding when it's the right tool — and when it isn't — saves time and leads to better architecture decisions.

✅ Use Prompt Flow When

  • Building multi-step LLM pipelines with branching logic
  • Rapidly prototyping different prompt strategies
  • Needing visual debugging of intermediate outputs
  • Running batch evaluations across test datasets
  • Comparing prompt variants side-by-side
  • Deploying LLM apps as managed endpoints
  • Collaborating with non-developers on AI logic

⚠️ Consider Alternatives When

  • Simple single-call LLM apps → direct SDK
  • Heavy data processing → data pipelines
  • Complex agent frameworks → Semantic Kernel / AutoGen
  • Real-time streaming UIs → custom WebSocket code
  • Training or fine-tuning models → model catalog
  • Non-LLM ML workloads → Azure ML pipelines
⚠️ Common Misconception: Prompt Flow is not a replacement for LangChain or Semantic Kernel. It is a complementary tool — you can call LangChain chains inside Python nodes of a Prompt Flow, combining visual orchestration with framework flexibility.

4. Building Flows — Step by Step

Let's walk through building a complete flow from scratch. We'll create a knowledge-grounded Q&A flow that retrieves context, generates an answer, and validates the output.

4.1 Creating a New Flow

1

Open Azure AI Foundry Portal

Navigate to ai.azure.com, select your project, and go to Prompt Flow in the left navigation under Build & Customize.

2

Create from Template or Blank

Click + Create and choose between starting from a blank canvas or using a gallery template. For learning, start with Standard Flow → Blank.

3

Configure Inputs

Define your flow's input schema. Every flow starts with at least one input. Click the Inputs section and add fields with their types.

YAML — flow.dag.yaml
inputs: question: type: string default: "What are the benefits of Azure AI Foundry?" context: type: string default: "" chat_history: type: list default: [] outputs: answer: type: string reference: ${format_output.output} confidence: type: string reference: ${validate_response.output}

4.2 Connecting Nodes

Nodes connect through typed references. Use the ${node_name.output} syntax to pass data between nodes. The visual editor shows these connections as lines between ports.

5. Node Types Deep Dive

5.1 LLM Nodes

LLM nodes call language models using Jinja2 prompt templates with variable substitution. They connect to your Azure OpenAI deployments or other model endpoints configured in your project.

Jinja2 — LLM Prompt Template
system: You are a helpful assistant that answers questions based on provided context. Always cite specific parts of the context in your response. If the context doesn't contain enough information, say so clearly. user: ## Context {{context}} ## Chat History {% for item in chat_history %} - {{item.role}}: {{item.content}} {% endfor %} ## Current Question {{question}} Provide a detailed, accurate answer based on the context above.

🤖 LLM Node Configuration

Connection: Select your Azure OpenAI resource
API: Chat or Completion
Deployment: gpt-4o, gpt-4o-mini, etc.
Parameters: temperature, max_tokens, top_p, presence_penalty
Inputs: Map flow variables to template placeholders

5.2 Python Nodes

Python nodes run custom code for data transformation, API calls, conditional logic, and any processing that isn't an LLM call. They must define a function decorated with @tool.

Python — Preprocessing Node
from promptflow import tool from typing import List @tool def prepare_prompt(question: str, context: str, chat_history: List[dict]) -> str: """Prepare the prompt context by combining retrieved docs and history.""" # Truncate context if too long max_context_chars = 4000 if len(context) > max_context_chars: context = context[:max_context_chars] + "\n... [truncated]" # Format chat history (keep last 5 turns) recent_history = chat_history[-5:] if chat_history else [] formatted = f"Context:\n{context}\n\nRecent History:\n" for turn in recent_history: formatted += f" {turn.get('role', 'user')}: {turn.get('content', '')}\n" formatted += f"\nQuestion: {question}" return formatted
Python — Validation Node
from promptflow import tool import json @tool def validate_response(llm_output: str, context: str) -> str: """Validate the LLM response and assign confidence score.""" if not llm_output or len(llm_output.strip()) < 10: return json.dumps({"confidence": "low", "reason": "Empty or too short"}) # Check if response references context context_words = set(context.lower().split()) response_words = set(llm_output.lower().split()) overlap = len(context_words & response_words) if overlap > 20: confidence = "high" elif overlap > 10: confidence = "medium" else: confidence = "low" return json.dumps({ "confidence": confidence, "word_overlap": overlap, "response_length": len(llm_output) })

6. Testing & Debugging Flows

One of Prompt Flow's strongest advantages is its integrated testing and debugging experience. Unlike traditional code where you might add print statements or breakpoints, Prompt Flow gives you visual inspection of every node's input and output at every execution step.

6.1 Interactive Testing

The Test panel in the flow editor lets you run your flow with sample inputs and inspect results at each node. This is invaluable for iterating on prompt templates.

1

Single Input Test

Click the Run button at the top of the editor. Enter your test input values in the panel that appears. Each node's output appears in real-time as execution progresses through the graph.

2

Inspect Node Outputs

Click any node to view its input/output details. The Trace view shows execution time, token usage, and the complete data that flowed through each connection.

3

Variant Testing

Create multiple variants of LLM nodes (different prompts, parameters) and compare outputs side-by-side. This is essential for prompt optimization.

6.2 Batch Testing

Run your flow against an entire dataset to evaluate quality at scale. Upload a CSV or JSON file with test cases, then Prompt Flow executes against every row and collects metrics.

JSON — test_dataset.jsonl
{"question": "What is Azure AI Foundry?", "context": "Azure AI Foundry is a unified platform...", "expected": "Azure AI Foundry is..."} {"question": "How do I deploy a model?", "context": "Model deployment in AI Foundry...", "expected": "To deploy a model..."} {"question": "What are safety features?", "context": "AI Foundry provides content safety...", "expected": "Safety features include..."}

6.3 Debugging Tips

✅ Pro Debugging Workflow:
  1. Check inputs first — most errors come from malformed input data or missing variables
  2. Run nodes individually — right-click a node and select "Run from here" to isolate issues
  3. Use the Trace view — inspect the complete execution trace including timing and token counts
  4. Compare variants — create a variant with a simplified prompt to rule out prompt-level issues
  5. Check connections — verify API keys and model deployments are active
Common Issue Symptom Solution
Missing connection Node shows red outline Configure the connection in project settings
Type mismatch Serialization error in logs Ensure output types match downstream input types
Token limit exceeded LLM node timeout or error Truncate context or increase max_tokens
Empty LLM response Blank output from LLM node Check prompt template variables are populated
Python node import error ModuleNotFoundError Add dependency to requirements.txt in flow folder

7. Deploying Flows as Endpoints

Once your flow is tested and validated, deploy it as a managed online endpoint that serves real-time predictions via a REST API. Azure AI Foundry handles infrastructure, scaling, and monitoring automatically.

7.1 Deployment Process

1

Prepare for Deployment

Click Deploy in the flow editor toolbar. Select your compute configuration — instance type and instance count determine latency and throughput.

2

Configure Endpoint Settings

Choose endpoint name, authentication type (key or Azure AD), and traffic allocation. For production, enable auto-scaling to handle variable load.

3

Monitor & Iterate

After deployment, monitor latency, error rates, and token usage in the endpoint dashboard. Use the Test tab to validate the deployed endpoint with live requests.

7.2 Calling the Endpoint

Python — Calling a Deployed Flow
import requests import json endpoint_url = "https://<your-endpoint>.<region>.inference.ml.azure.com/score" api_key = "<your-api-key>" headers = { "Content-Type": "application/json", "Authorization": f"Bearer {api_key}", "azureml-model-deployment": "prompt-flow-deployment" } payload = { "question": "What are the benefits of Prompt Flow?", "context": "Prompt Flow provides visual orchestration...", "chat_history": [] } response = requests.post(endpoint_url, headers=headers, json=payload) result = response.json() print(f"Answer: {result['answer']}") print(f"Confidence: {result['confidence']}")
cURL — REST API Call
curl -X POST "https://<endpoint>.inference.ml.azure.com/score" \ -H "Content-Type: application/json" \ -H "Authorization: Bearer <api-key>" \ -d '{ "question": "What are the benefits of Prompt Flow?", "context": "Prompt Flow provides visual orchestration...", "chat_history": [] }'

7.3 Deployment Best Practices

💡 Production Checklist:
  • Authentication: Use Azure AD token-based auth for production; API keys for development only
  • Scaling: Start with 1 instance, enable auto-scale based on request volume
  • Monitoring: Set up alerts for latency spikes (>5s) and error rate (>1%)
  • Versioning: Deploy new versions alongside existing ones with traffic splitting
  • Testing: Run batch tests against the deployed endpoint before shifting production traffic

8. Mini Hack — Customer Support Flow with Routing

🛠️ Mini Hack: Build a Customer Support Flow

Build a flow that classifies incoming customer queries, routes them to specialized handlers, and generates contextual responses — all with visual orchestration.

Architecture

┌────────────────────────────────────────────────────────────────────┐ │ CUSTOMER SUPPORT FLOW — ROUTING ARCHITECTURE │ │ │ │ ┌──────────┐ ┌──────────────┐ ┌──────────────────────────┐ │ │ │ INPUT │───▶│ CLASSIFY │───▶│ ROUTE BY INTENT │ │ │ │ message │ │ INTENT │ │ │ │ │ │ cust_id │ │ (LLM) │ │ billing ──▶ Billing LLM │ │ │ └──────────┘ └──────────────┘ │ technical ─▶ Tech LLM │ │ │ │ general ───▶ General LLM │ │ │ └──────────┬───────────────┘ │ │ │ │ │ ┌───────────▼──────────────┐ │ │ │ FORMAT RESPONSE │ │ │ │ (Python) │ │ │ └───────────┬──────────────┘ │ │ ┌───────────▼──────────────┐ │ │ │ OUTPUT │ │ │ │ response + intent │ │ │ └──────────────────────────┘ │ └────────────────────────────────────────────────────────────────────┘

Step 1 — Intent Classification Node (LLM)

system: You are an intent classifier for a customer support system. Classify the customer message into exactly ONE of these categories: - billing: Payment, invoices, charges, subscriptions, pricing - technical: Bugs, errors, setup, configuration, integration - general: Account info, feedback, general inquiries Respond with ONLY the category name, nothing else. user: Customer message: {{message}}

Step 2 — Routing Logic (Python)

from promptflow import tool @tool def route_by_intent(intent: str, message: str, customer_id: str) -> dict: """Route to specialized handler based on classified intent.""" intent = intent.strip().lower() routing_config = { "billing": { "handler": "billing_specialist", "system_prompt": "You are a billing specialist. Help with " "payment issues, invoice questions, and " "subscription management. Be precise with " "amounts and dates.", "priority": "high" }, "technical": { "handler": "technical_support", "system_prompt": "You are a technical support engineer. Help " "diagnose issues, provide step-by-step " "solutions, and escalate complex problems. " "Include error codes when relevant.", "priority": "medium" }, "general": { "handler": "general_support", "system_prompt": "You are a friendly customer support agent. " "Answer general questions, provide account " "information, and direct customers to the " "right resources.", "priority": "low" } } config = routing_config.get(intent, routing_config["general"]) return { "intent": intent, "system_prompt": config["system_prompt"], "message": message, "customer_id": customer_id, "priority": config["priority"], "handler": config["handler"] }

Step 3 — Specialized Response (LLM)

system: {{system_prompt}} Customer ID: {{customer_id}} Priority: {{priority}} Guidelines: - Be empathetic and professional - Provide actionable next steps - Include relevant reference numbers - Offer to escalate if the issue requires it user: {{message}}

Step 4 — Format Output (Python)

from promptflow import tool import json from datetime import datetime @tool def format_response( llm_response: str, routing_info: dict ) -> str: """Format the final customer support response with metadata.""" result = { "response": llm_response, "metadata": { "intent": routing_info.get("intent", "unknown"), "handler": routing_info.get("handler", "general"), "priority": routing_info.get("priority", "low"), "customer_id": routing_info.get("customer_id", ""), "timestamp": datetime.utcnow().isoformat(), "model": "gpt-4o" } } return json.dumps(result, indent=2)
💡 Challenge Extension: Add a Content Safety node before the response to filter harmful content, and add an evaluation flow that measures response relevance and empathy scores.

9. Best Practices

Flow Design Principles

📐 Keep Flows Focused

Each flow should solve one well-defined task. If your flow exceeds 10-12 nodes, consider splitting it into sub-flows or using a Python node that calls another flow's endpoint.

🔄 Use Variants for Experimentation

Create variants of LLM nodes to test different prompts, models, or parameter settings. Run batch evaluations to compare variants quantitatively before promoting to production.

🛡️ Validate at Every Stage

Add Python validation nodes after LLM calls to check output format, length, content safety, and business rules. Fail gracefully with fallback responses rather than exposing raw errors.

📊 Log Everything

Use the built-in tracing to capture execution details. In production, enable Application Insights integration for long-term monitoring, alerting, and debugging of deployed flows.

Common Anti-Patterns

🛑 Avoid These Mistakes:
  • Monolith flows — 20+ nodes in a single flow are hard to maintain and debug
  • Hardcoded secrets — always use connections, never embed API keys in Python nodes
  • Skipping batch testing — a flow that works for one input might fail at scale
  • Ignoring latency — each node adds latency; minimize sequential LLM calls
  • No error handling — Python nodes should catch exceptions and return structured error responses
