Introduction: Beyond Basic RAG

Retrieval-Augmented Generation (RAG) has become the default architecture for grounding large language models with enterprise data. But if you've built a basic RAG pipeline — chunk documents, generate embeddings, retrieve, stuff into a prompt — you've likely encountered the limits: answers that miss key information, irrelevant context polluting responses, and fragile performance as document complexity grows.

Module 14 takes you beyond basic RAG into advanced patterns that production teams use to build high-quality, reliable AI applications in Azure AI Foundry. We'll explore how chunking strategy fundamentally shapes retrieval quality, why reranking can deliver a 15-35% quality improvement with minimal code changes, and how agentic RAG patterns enable AI systems to dynamically orchestrate their own retrieval.

💡 What You'll Learn

This module covers the full spectrum of RAG optimization: chunking strategies (fixed-size, semantic, sliding window, recursive), reranking and relevance tuning, multi-index RAG for cross-source retrieval, handling complex documents with tables and structured data, and agentic RAG patterns where AI agents dynamically decide their retrieval strategy.

All of these patterns are natively supported by Azure AI Foundry and Azure AI Search. Whether you're building a customer support bot, a knowledge management system, or an internal research assistant, mastering these patterns will dramatically improve the quality and reliability of your AI application.

Chunking Strategies Deep Dive

Chunking is the single most impactful decision in your RAG pipeline. How you split documents determines what your retrieval system can find and what context the LLM receives. A poor chunking strategy undermines everything downstream — embeddings, search, and generation.

Fixed-Size Chunking

The simplest approach: split text into chunks of N tokens (or characters) with a configurable overlap. This is the default in many RAG frameworks and works surprisingly well for uniform, unstructured text like logs, transcripts, and simple articles.

Python
from langchain_text_splitters import CharacterTextSplitter
import tiktoken

def fixed_size_chunk(text: str, chunk_size: int = 512, overlap: int = 50):
    """Split text into fixed-size token chunks with overlap."""
    encoding = tiktoken.encoding_for_model("gpt-4o")
    splitter = CharacterTextSplitter(
        separator="\n",
        chunk_size=chunk_size,
        chunk_overlap=overlap,
        length_function=lambda t: len(encoding.encode(t))
    )
    return splitter.split_text(text)

chunks = fixed_size_chunk(document_text)
print(f"Produced {len(chunks)} chunks")
⚠️ Limitation

Fixed-size chunking can split mid-sentence, mid-paragraph, or even mid-table. This creates chunks that lack context and can mislead both the embedding model and the LLM.

Semantic Chunking

Semantic chunking splits text at natural language boundaries — sentence endings, paragraph breaks, and topic shifts. This preserves the meaning and coherence of each chunk, leading to better embeddings and more relevant retrieval.

Python
from langchain_text_splitters import RecursiveCharacterTextSplitter

def semantic_chunk(text: str, max_chunk_size: int = 800):
    """Split text using semantic boundaries."""
    splitter = RecursiveCharacterTextSplitter(
        separators=[
            "\n## ",      # H2 headers first
            "\n### ",     # Then H3
            "\n\n",       # Paragraph breaks
            "\n",         # Line breaks
            ". ",         # Sentence endings
            " ",          # Words (last resort)
        ],
        chunk_size=max_chunk_size,
        chunk_overlap=100,
        length_function=len,
    )
    return splitter.split_text(text)

chunks = semantic_chunk(document_text)
print(f"Semantic chunking: {len(chunks)} chunks")

The key insight is the separator hierarchy: the splitter tries the first separator (H2 headers) across the entire text, and only falls back to smaller separators when chunks exceed the maximum size. This naturally aligns chunks with document structure.

Sliding Window Chunking

Sliding window creates heavily overlapping chunks, ensuring that information near chunk boundaries is captured in multiple chunks. This is particularly effective for narrative content where context flows continuously across paragraphs.

Configure this by setting a high chunk_overlap relative to chunk_size — typically 30-50% overlap for narrative text. The trade-off is increased storage and index size.

Recursive Chunking

Recursive chunking is a hierarchical strategy that progressively splits text using a cascade of separators. It first tries to split at document section boundaries, then paragraphs, then sentences — only going to smaller units when necessary to stay within size limits.

This is the recommended default for most document types in production RAG applications. It respects document structure while keeping chunks within manageable size limits.

Strategy Comparison

Strategy Context Preservation Best For Trade-off
Fixed-Size Low Uniform content, logs Simple but loses context
Semantic High Articles, documentation Variable chunk sizes
Sliding Window Medium-High Narratives, transcripts Increased storage
Recursive High Structured documents Needs document structure
✅ Best Practice

Use recursive chunking as your default starting point. It handles most document types well and aligns chunks with natural content boundaries. Switch to specialized strategies only when you have specific content types that benefit from them.

Chunk Enrichment

Regardless of your chunking strategy, enriching chunks with contextual metadata significantly improves retrieval quality. Key enrichments include:

  • Section header injection: Prepend the parent section header to each chunk so it retains its place in the document hierarchy
  • Document title and metadata: Add the document title, author, and date to each chunk's metadata fields
  • Summary context: Generate a brief summary of the chunk's surrounding context and store it as metadata
  • Entity extraction: Extract key entities (people, products, dates) and add them as filterable fields
Python
import re

def enrich_chunks(chunks: list[str], document_title: str) -> list[dict]:
    """Enrich chunks with contextual metadata."""
    enriched = []
    current_header = document_title

    for i, chunk in enumerate(chunks):
        header_match = re.match(r'^(#{1,4}\s+.+)', chunk)
        if header_match:
            current_header = header_match.group(1).strip('#').strip()

        enriched.append({
            "content": chunk,
            "title": document_title,
            "section_header": current_header,
            "chunk_index": i,
            "total_chunks": len(chunks),
        })
    return enriched

Embedding Model Selection

The embedding model translates your chunks and queries into vector representations that enable semantic search. Choosing the right model — and configuring it correctly — has a direct impact on retrieval quality.

Azure OpenAI Embedding Models

Model Dimensions Max Tokens Quality Cost
text-embedding-3-small 1536 8,191 Good Lower
text-embedding-3-large 3,072 8,191 Excellent Higher

Dimension Reduction

Azure OpenAI supports native dimension reduction for the text-embedding-3-* models. This allows you to use the higher-quality large model at a reduced dimension (e.g., 1024 or 1536), saving storage and compute while retaining most of the quality.

Python
from openai import AzureOpenAI

client = AzureOpenAI(
    azure_endpoint=ENDPOINT,
    api_key=API_KEY,
    api_version="2024-10-21"
)

# Generate embeddings with reduced dimensions
response = client.embeddings.create(
    input="Azure AI Foundry enables advanced RAG patterns",
    model="text-embedding-3-large",
    dimensions=1024   # Reduce from 3072 to 1024
)

embedding = response.data[0].embedding
print(f"Embedding dimensions: {len(embedding)}")  # 1024
🚫 Critical Rule

Always use the same embedding model and dimensions for both indexing and querying. Mixing models or dimensions will produce meaningless similarity scores and broken retrieval.

Choosing the Right Model

For most production RAG applications, text-embedding-3-large with dimension reduction to 1024 or 1536 offers the best balance of quality, storage efficiency, and cost. If cost is a primary concern and quality requirements are moderate, text-embedding-3-small is a solid choice.

Reranking & Relevance Tuning

Reranking is the highest-ROI optimization you can add to a RAG pipeline. While initial retrieval (using embeddings or keyword search) is fast but approximate, a reranker uses a more sophisticated model to re-score and re-order the retrieved candidates.

How Reranking Works

Query → Initial Retrieval
(top 50, fast) → Cross-Encoder
Reranker → Top 5
(re-scored)

The key distinction is between bi-encoders and cross-encoders:

Bi-Encoder (Embeddings)

Encodes query and document independently. Fast because embeddings can be precomputed, but captures only approximate similarity.

Initial Retrieval

Cross-Encoder (Reranker)

Processes query and document together as a pair. Much more accurate because it sees both texts simultaneously, but slower.

Re-scoring

Azure AI Search Semantic Ranker

Azure AI Search provides a built-in semantic ranker that acts as a cross-encoder reranker. Enable it with a simple configuration change — no additional infrastructure required.

Python
from azure.search.documents.indexes.models import (
    SemanticConfiguration, SemanticSearch,
    SemanticPrioritizedFields, SemanticField
)

# Define semantic configuration
semantic_config = SemanticConfiguration(
    name="my-semantic-config",
    prioritized_fields=SemanticPrioritizedFields(
        content_fields=[SemanticField(field_name="content")]
    )
)

# Add semantic search to your index
index.semantic_search = SemanticSearch(
    configurations=[semantic_config]
)
index_client.create_or_update_index(index)

Querying with Reranking

Python
from azure.search.documents.models import QueryType

results = search_client.search(
    search_text=query,
    vector_queries=[vector_query],
    query_type=QueryType.SEMANTIC,
    semantic_configuration_name="my-semantic-config",
    top=5
)

for result in results:
    print(f"Reranker Score: {result['@search.reranker_score']:.4f}")
    print(f"Content: {result['content'][:100]}...")
    print("---")
✅ Impact

In benchmarks, adding Azure AI Search semantic ranker typically improves top-result relevance by 15-35%. For multi-turn conversations and complex queries, the improvement can be even higher.

Relevance Tuning Parameters

Beyond reranking, fine-tune your retrieval quality with these parameters:

  • Top-K: How many initial candidates to retrieve before reranking. Larger K means better recall but more computation. Start with 50 for initial retrieval, then rerank to top 5.
  • Score threshold: Set a minimum reranker score to filter out low-confidence results rather than always returning exactly K results.
  • Field boosting: Boost certain fields (e.g., title) in keyword search to improve relevance for title-focused queries.
  • Minimum coverage: Require that results cover a minimum percentage of the query terms.

Hybrid Search Optimization

Hybrid search combines keyword search (BM25) with vector search (embeddings) to get the best of both worlds. Keyword search excels at exact matches and specific terms, while vector search understands semantic meaning and handles paraphrases.

Why Hybrid Outperforms Either Alone

🔤 Keyword Search (BM25)

Strengths: exact term matching, proper nouns, product codes, error messages, acronyms.

Weakness: misses semantically similar text with different wording.

🧮 Vector Search

Strengths: semantic understanding, paraphrases, conceptual similarity, multilingual matching.

Weakness: can miss exact matches, especially for rare terms or codes.

Reciprocal Rank Fusion (RRF)

Azure AI Search uses Reciprocal Rank Fusion (RRF) to combine results from keyword and vector search into a single ranked list. RRF works by considering the rank position of each result in each method, not the raw scores — which makes it robust across different scoring scales.

The RRF formula: score(d) = Σ 1 / (k + rank(d)) where k is a constant (typically 60) and the sum is over all retrieval methods that returned document d.

Python
from azure.search.documents.models import VectorizedQuery, QueryType

def hybrid_search_with_reranking(query: str, search_client, embeddings_client):
    """Full hybrid search: keyword + vector + semantic reranking."""

    # Generate query embedding
    query_vector = embeddings_client.embeddings.create(
        input=query,
        model="text-embedding-3-large",
        dimensions=1024
    ).data[0].embedding

    vector_query = VectorizedQuery(
        vector=query_vector,
        k_nearest_neighbors=50,
        fields="content_vector"
    )

    # Hybrid query: keyword + vector + reranking
    results = search_client.search(
        search_text=query,                # Keyword (BM25)
        vector_queries=[vector_query],     # Vector
        query_type=QueryType.SEMANTIC,     # Reranking
        semantic_configuration_name="my-semantic-config",
        top=5
    )

    return list(results)
💡 Pro Tip

The full stack — hybrid search + semantic reranking — consistently outperforms any single retrieval method in Azure AI Search benchmarks. This should be your default production configuration.

Multi-Index RAG

Real-world AI applications rarely have just one type of data. You might need to search across product documentation, support tickets, FAQ articles, API references, and internal knowledge bases — each with different schemas, update frequencies, and content structures.

Multi-index RAG maintains separate Azure AI Search indexes for different data types and queries them in parallel (or selectively) based on the user's intent.

Architecture Pattern

📚 Product Docs 🎫 Support Tickets 📋 FAQ 📖 API Reference

↓ Each with its own index, schema, and chunking strategy ↓

🔀 Query Router → 📊 Result Merger → 🤖 LLM Generation

Query Routing

Not every query needs every index. A query router classifies the user's intent and selects the most relevant index(es):

Python
from openai import AzureOpenAI

def route_query(query: str, client: AzureOpenAI) -> list[str]:
    """Use an LLM to classify query intent and select indexes."""
    response = client.chat.completions.create(
        model="gpt-4o",
        messages=[
            {"role": "system", "content": """Classify the user query and return
            relevant indexes as a JSON array. Available indexes:
            - "product-docs": product documentation and guides
            - "support-tickets": past support tickets and resolutions
            - "faq": frequently asked questions
            - "api-reference": API documentation and code samples"""},
            {"role": "user", "content": query}
        ],
        response_format={"type": "json_object"},
        temperature=0
    )
    import json
    result = json.loads(response.choices[0].message.content)
    return result.get("indexes", ["product-docs"])

Result Merging

When querying multiple indexes, you need to merge and re-score results because scores from different indexes aren't directly comparable. Common approaches include:

  • Score normalization: Normalize scores to [0, 1] range within each index, then merge
  • Reciprocal Rank Fusion: Apply RRF across index results (same approach used in hybrid search)
  • Weighted merging: Assign weights to each index based on query type relevance
  • Reranking after merge: Merge all candidates, then apply a cross-encoder reranker to the combined set
⚠️ Important

Never compare raw search scores across different indexes — they use different scales and distributions. Always normalize or use rank-based fusion.

Cross-Source Retrieval

Cross-source retrieval extends multi-index RAG by blending data from fundamentally different sources — not just separate search indexes, but databases, APIs, file systems, and real-time data streams.

Common Cross-Source Patterns

Source Type Retrieval Method Azure Service
Unstructured docs Vector + keyword search Azure AI Search
Structured data SQL queries Azure SQL / Cosmos DB
Knowledge graphs Graph traversal Azure Cosmos DB (Gremlin)
Real-time data API calls Azure Functions / APIs
Images / PDFs Multimodal embeddings Azure AI Vision

Federation Pattern in Azure AI Foundry

Azure AI Foundry projects can connect multiple data sources through the data connections feature. Each connection can be configured independently with its own indexing pipeline, chunking strategy, and update schedule.

When a query arrives, the system can fan out to multiple data connections, retrieve from each in parallel, and merge results before passing them to the LLM. This is particularly powerful when combined with query routing — sending specific sub-queries to the most relevant source.

✅ AI Foundry Feature

In Azure AI Foundry, you can add data connections to your project under Data + indexes. Each connection supports different source types including Azure Blob Storage, Azure AI Search, Azure Cosmos DB, and uploaded files.

Handling Tables & Structured Data

Tables in PDFs, Word documents, and spreadsheets are notoriously difficult for RAG systems. Standard chunking strategies flatten tables into unreadable text strings, losing the row-column relationships that give the data meaning.

The Problem

Consider a pricing table in a PDF. Fixed-size chunking might produce:

Product A $99 Enterprise Product B $149 Pro
duct C $199 Enterprise Plus...

The column associations are completely lost. A query like "What is the price of Product C?" might return incorrect results because the chunker split the row across chunk boundaries.

Solution: Azure Document Intelligence

Azure Document Intelligence (formerly Form Recognizer) provides specialized models for extracting structured content from documents. It preserves table structure, identifies headers, and maintains cell relationships.

Python
from azure.ai.documentintelligence import DocumentIntelligenceClient
from azure.core.credentials import AzureKeyCredential

client = DocumentIntelligenceClient(
    endpoint=DOC_INTELLIGENCE_ENDPOINT,
    credential=AzureKeyCredential(DOC_INTELLIGENCE_KEY)
)

# Analyze document with layout model
with open("quarterly-report.pdf", "rb") as f:
    poller = client.begin_analyze_document(
        model_id="prebuilt-layout",
        body=f,
        content_type="application/pdf"
    )
    result = poller.result()

# Extract tables with structure preserved
for table in result.tables:
    print(f"Table: {table.row_count} rows x {table.column_count} cols")
    for cell in table.cells:
        print(f"  [{cell.row_index}, {cell.column_index}]: {cell.content}")

Table-to-Text Strategies

Once you've extracted table structure, you need to convert it into text that works well with embeddings and retrieval:

  • Markdown tables: Convert to Markdown format — LLMs understand this well and it preserves structure
  • Row-level chunks: Create one chunk per row, prefixed with column headers as context
  • Natural language summaries: Use an LLM to generate natural language descriptions of table data
  • Hybrid approach: Store both the structured data (for SQL queries) and the text representation (for vector search)
Python
def table_to_row_chunks(table, document_title: str) -> list[dict]:
    """Convert an extracted table into row-level chunks with headers."""
    headers = []
    rows = []

    for cell in table.cells:
        if cell.row_index == 0:
            headers.append(cell.content)
        else:
            while len(rows) <= cell.row_index - 1:
                rows.append({})
            rows[cell.row_index - 1][headers[cell.column_index]] = cell.content

    chunks = []
    for i, row in enumerate(rows):
        text = f"Source: {document_title}\n"
        text += " | ".join(f"{k}: {v}" for k, v in row.items())
        chunks.append({"content": text, "type": "table_row", "row_index": i})

    return chunks
💡 Pro Tip

For financial reports and data-heavy documents, consider a dual-path approach: store structured data in a database for precise queries (e.g., "What was revenue in Q3?") and index text descriptions in Azure AI Search for semantic queries (e.g., "How did revenue trend over the year?").

Agentic RAG Patterns

Traditional RAG follows a fixed pipeline: receive query → retrieve chunks → stuff into prompt → generate response. Agentic RAG breaks this mold by giving an AI agent the autonomy to decide when, what, and how to retrieve information.

This is a paradigm shift. Instead of a pipeline, you have an agent with access to retrieval tools that it invokes dynamically based on its reasoning about the query.

Core Agentic RAG Patterns

1. Query Decomposition

Complex questions often contain multiple sub-questions. An agentic RAG system decomposes the original query into atomic sub-queries, retrieves for each independently, and synthesizes the results.

Example
User: "Compare the pricing and features of Azure AI Foundry
       vs other AI platforms for enterprise deployments."

Agent decomposes into:
  → Sub-query 1: "Azure AI Foundry pricing enterprise"
  → Sub-query 2: "Azure AI Foundry features enterprise"
  → Sub-query 3: "Azure AI Foundry pricing enterprise"
  → Sub-query 4: "Azure AI Foundry features enterprise"
  → Synthesize all results into a comparison

2. Iterative Retrieval

The agent retrieves initial results, evaluates whether they adequately answer the question, and if not, formulates refined queries and retrieves again. This loop continues until the agent is satisfied with the context quality.

🤖 Agent → 🔍 Retrieve → 🤔 Evaluate → ✅ Sufficient?

If not sufficient → refine query → retrieve again (max 3 iterations)

3. Tool-Based Retrieval

The agent has multiple retrieval tools available and decides which to use based on the query type:

Python
tools = [
    {
        "type": "function",
        "function": {
            "name": "search_documentation",
            "description": "Search product documentation for feature info",
            "parameters": {
                "type": "object",
                "properties": {
                    "query": {"type": "string", "description": "Search query"}
                },
                "required": ["query"]
            }
        }
    },
    {
        "type": "function",
        "function": {
            "name": "query_database",
            "description": "Query structured data like pricing and metrics",
            "parameters": {
                "type": "object",
                "properties": {
                    "sql_query": {"type": "string", "description": "SQL query"}
                },
                "required": ["sql_query"]
            }
        }
    },
    {
        "type": "function",
        "function": {
            "name": "search_support_tickets",
            "description": "Search past support tickets for known issues",
            "parameters": {
                "type": "object",
                "properties": {
                    "query": {"type": "string", "description": "Search query"},
                    "status": {"type": "string", "enum": ["resolved", "open"]}
                },
                "required": ["query"]
            }
        }
    }
]

4. Self-Reflection

Before generating the final response, the agent reflects on the retrieved context: Is it relevant? Is it sufficient? Is it contradictory? This self-check prevents hallucination from irrelevant context.

Implementing Agentic RAG in Azure AI Foundry

Azure AI Foundry's Agent Service provides the infrastructure for building agentic RAG systems. You can create agents with tool definitions that include your search indexes, databases, and custom functions. The agent decides which tools to invoke based on each user query.

Python
from azure.ai.projects import AIProjectClient
from azure.identity import DefaultAzureCredential

# Create an AI Foundry project client
project_client = AIProjectClient(
    credential=DefaultAzureCredential(),
    endpoint="https://your-project.services.ai.azure.com"
)

# Create an agent with retrieval tools
agent = project_client.agents.create_agent(
    model="gpt-4o",
    name="rag-agent",
    instructions="""You are a helpful assistant with access to multiple
    data sources. For each query:
    1. Analyze what information is needed
    2. Select the appropriate tool(s)
    3. Retrieve relevant context
    4. Verify the retrieved information is sufficient
    5. Generate a comprehensive response""",
    tools=tools
)
⚠️ Cost Consideration

Agentic RAG uses more LLM calls than traditional RAG (for query decomposition, tool selection, self-reflection). Monitor your token usage and set iteration limits. For most applications, limit to 3 retrieval iterations maximum.

Performance Optimization

Production RAG systems need to balance quality, latency, and cost. Here are the key optimization strategies for Azure AI Foundry RAG applications.

Latency Optimization

Technique Impact How
Embedding caching 50-80% reduction for repeated queries Cache query embeddings in Redis/memory
Dimension reduction 20-40% faster search Use 1024 dims instead of 3072
Limit top-K Linear reduction in reranking time Retrieve 20 instead of 50 candidates
Async parallel retrieval Up to 3x faster for multi-index Query indexes concurrently
Streaming responses Better perceived performance Stream LLM output as it generates

Quality Monitoring

You can't optimize what you don't measure. Azure AI Foundry provides evaluation capabilities to track RAG quality metrics:

  • Groundedness: Does the response cite information from the retrieved context?
  • Relevance: Are the retrieved chunks relevant to the query?
  • Coherence: Is the response well-structured and logically consistent?
  • Completeness: Does the response fully answer the query?
  • Retrieval precision: What fraction of retrieved chunks are actually relevant?

Cost Optimization

  • Tiered retrieval: Use cheaper keyword search first, only invoke vector search and reranking for complex queries
  • Smart caching: Cache LLM responses for identical queries (with TTL for freshness)
  • Chunk size tuning: Larger chunks mean fewer chunks to embed and store, but potentially lower retrieval precision
  • Model selection: Use GPT-4o-mini for query routing and classification; reserve GPT-4o for final generation
