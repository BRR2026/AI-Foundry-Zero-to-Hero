Introduction

Every AI application is only as powerful as the data it can access. Whether you're building a customer support chatbot that needs to reference product documentation, an internal assistant that surfaces HR policies, or an analytics tool that interprets sales data — the foundation of every successful AI project is well-connected, properly prepared data.

In this module, we explore how Azure AI Foundry enables you to connect a wide variety of data sources to your AI projects. We'll cover five key data connectors — Azure Blob Storage, Azure SQL Database, Azure Cosmos DB, SharePoint Online, and Microsoft OneLake — and dive deep into the patterns, strategies, and best practices that make data ingestion, preparation, and indexing effective for Retrieval-Augmented Generation (RAG) workloads.

💡 Why This Matters

RAG (Retrieval-Augmented Generation) is the most common pattern for grounding AI responses in real-world data. Without proper data connectivity and indexing, your AI model operates in a vacuum — unable to provide accurate, contextual, and up-to-date responses. This module gives you the skills to bridge that gap.

By the end of this module, you will be able to connect multiple Azure data sources to AI Foundry, design effective ingestion pipelines, prepare your data for AI consumption, and create vector indexes that power high-quality RAG applications.

Data Sources Overview

Azure AI Foundry supports multiple Azure data services as sources for your AI projects. Each data source has different strengths and is suited for different types of content. Understanding which source to use — and when to combine multiple sources — is critical for building comprehensive knowledge bases.

📦

Azure Blob Storage

Object storage for unstructured data: PDFs, images, text files, Office documents, and more. The most versatile connector for file-based content.

🗃️

Azure SQL Database

Relational database for structured business data. Query specific tables and columns to extract precisely the data your AI needs.

🌍

Azure Cosmos DB

Globally distributed NoSQL database. Ideal for semi-structured document data with flexible schemas and real-time change feeds.

📁

SharePoint Online

Enterprise document management in Microsoft 365. Access content where your teams already collaborate, with built-in permissions.

🏞️

Microsoft OneLake

Unified data lake across Microsoft Fabric. A single data layer for all analytics workloads with built-in governance.

Azure Blob Storage Integration

Azure Blob Storage is the most commonly used data source for AI applications. It provides scalable, cost-effective storage for unstructured content — the kind of content that forms the backbone of most knowledge bases and RAG systems.

Supported File Formats

When connecting Blob Storage to Azure AI Foundry, the following file formats are supported for automatic text extraction and indexing:

  • Documents: PDF, DOCX, PPTX, XLSX, HTML, TXT, MD
  • Data files: JSON, CSV, TSV, XML
  • Images (with OCR): PNG, JPG, TIFF, BMP

Setting Up the Connection

To connect Azure Blob Storage to your AI Foundry project, you need to configure authentication and specify which containers to index. Here's the recommended approach:

  1. Create or identify your storage account — Ensure your storage account is in the same region as your AI Foundry project for optimal performance and to avoid cross-region data transfer charges.
  2. Organize your content in containers — Use separate containers for different content types or domains. For example, product-docs, hr-policies, and technical-manuals.
  3. Configure authentication — Assign the Storage Blob Data Reader RBAC role to your AI Foundry project's managed identity. This is the most secure approach for production.
  4. Create the connection in AI Foundry — Navigate to your project in the AI Foundry portal, go to Data, and add a new connection pointing to your storage account and container.
Azure CLI — Assign RBAC Role bash
# Get the managed identity principal ID of your AI Foundry project PRINCIPAL_ID=$(az resource show \ --resource-group myResourceGroup \ --name myAIFoundryProject \ --resource-type "Microsoft.MachineLearningServices/workspaces" \ --query "identity.principalId" -o tsv) # Assign Storage Blob Data Reader role az role assignment create \ --assignee $PRINCIPAL_ID \ --role "Storage Blob Data Reader" \ --scope "/subscriptions/{sub-id}/resourceGroups/{rg}/providers/Microsoft.Storage/storageAccounts/{account}"

Container Organization Best Practices

✅ Best Practice

Organize your blob containers by content domain rather than file type. This makes it easier to create targeted indexes and manage access permissions. For example, use containers like product-documentation and customer-support-kb rather than all-pdfs and all-docx-files.

Azure SQL & Cosmos DB

While Blob Storage handles unstructured files, Azure SQL Database and Cosmos DB serve as connectors for structured and semi-structured data. These sources are essential when your AI application needs to reference business data, product catalogs, customer records, or any data stored in databases.

Azure SQL Database

Azure SQL Database is ideal when your knowledge base includes structured, relational data. Common scenarios include:

  • Product catalogs — SKU numbers, descriptions, pricing, specifications
  • Customer records — Account details, support history, preferences
  • Knowledge articles — Structured FAQ databases with categories and tags
  • Business metrics — Sales data, KPIs, financial records
SQL Query — Extracting Knowledge Articles sql
-- Extract published knowledge articles for indexing SELECT ka.article_id, ka.title, ka.content, ka.category, ka.last_updated, ka.author FROM knowledge_articles ka WHERE ka.status = 'published' AND ka.last_updated >= '2025-01-01' ORDER BY ka.last_updated DESC;

Azure Cosmos DB

Cosmos DB excels when your data has flexible schemas, requires global distribution, or benefits from the change feed feature for real-time ingestion. It supports multiple APIs (SQL/Core, MongoDB, Cassandra, Gremlin, Table), with the SQL API being most commonly used with AI Foundry.

Cosmos DB Query — Product Catalog sql
-- Query product documents for indexing SELECT c.id, c.productName, c.description, c.category, c.specifications, c._ts AS lastModified FROM c WHERE c.type = "product" AND c.isActive = true ORDER BY c._ts DESC
💡 Tip: Change Feed for Real-time Indexing

Cosmos DB's change feed provides a sorted list of documents that were changed, in the order in which they were modified. You can use this feature with Azure Functions to trigger automatic re-indexing whenever your product catalog, knowledge base, or customer data is updated — keeping your AI application's knowledge always current.

SharePoint & OneLake

SharePoint Online

SharePoint Online is where many enterprises store their critical business documents — from HR policies and compliance manuals to project documentation and departmental wikis. Connecting SharePoint to Azure AI Foundry allows you to unlock this content for AI-powered search and retrieval without requiring users to migrate their files.

Key capabilities of the SharePoint connector include:

  • Site-level and library-level connections — Connect an entire SharePoint site or specific document libraries
  • Automatic metadata extraction — SharePoint document properties (author, created date, custom columns) are preserved as metadata in the index
  • Permission awareness — Respects Microsoft 365 permissions so users only see search results they have access to
  • Version support — Can index the latest version of documents that have version history enabled
⚠️ Authentication Requirement

The SharePoint connector uses Microsoft Entra ID (formerly Azure Active Directory) for authentication. You need to register an application in Entra ID and grant it the Sites.Read.All permission (application-level) to enable the AI Foundry connector to read documents from your SharePoint sites.

Microsoft OneLake

Microsoft OneLake is the unified data lake that underpins all of Microsoft Fabric. If your organization uses Fabric for analytics, data engineering, or data science, OneLake provides a seamless way to make that data available for AI applications without duplication.

Key advantages of OneLake as a data source:

  • One copy of data — No need to duplicate data from your analytics platform into a separate storage account. OneLake serves as a single source of truth.
  • Shortcuts to external data — OneLake shortcuts let you reference data stored in Azure Data Lake Storage Gen2, Amazon S3, or Google Cloud Storage without copying it.
  • Built-in governance — OneLake inherits Fabric's security model, including row-level security, sensitivity labels, and data lineage tracking.
  • Cross-workload access — Data used for Power BI reports, Data Factory pipelines, or Synapse analytics can simultaneously serve as a knowledge base for your AI applications.
✅ When to Use OneLake

Choose OneLake when your organization already uses Microsoft Fabric and you want to avoid duplicating data. OneLake is particularly powerful when the same data is used for both traditional analytics (Power BI dashboards) and AI-powered applications (chatbots, assistants) — it eliminates data silos and ensures consistency.

Data Ingestion Patterns

Once your data sources are connected, the next decision is how to move data from those sources into your AI Foundry index. The ingestion pattern you choose depends on factors like data volume, update frequency, latency requirements, and cost constraints.

Pattern 1: Batch Processing

Batch processing is the simplest and most common ingestion pattern. You process all documents at once (or at scheduled intervals) to build or refresh your index. This pattern is ideal for:

  • Initial index creation from a large document corpus
  • Content that changes infrequently (policies, manuals, reference docs)
  • Scenarios where a few hours of data staleness is acceptable

Pattern 2: Real-time Streaming

Real-time ingestion processes data as soon as it arrives or changes. This pattern uses event-driven architectures to keep your index continuously updated. Use this when:

  • Data changes frequently and users need the latest information
  • You're working with Cosmos DB change feed or Event Grid notifications
  • Your application serves time-sensitive queries (stock data, news, live support)
Event-driven Ingestion with Azure Functions python
import azure.functions as func from azure.search.documents import SearchClient from openai import AzureOpenAI def process_new_document(blob: func.InputStream): """Triggered when a new document is uploaded to Blob Storage.""" # 1. Extract text from the document text = extract_text(blob) # 2. Split into chunks chunks = chunk_text(text, chunk_size=1024, overlap=128) # 3. Generate embeddings for each chunk client = AzureOpenAI(...) embeddings = [ client.embeddings.create( model="text-embedding-3-small", input=chunk ).data[0].embedding for chunk in chunks ] # 4. Upload to search index search_client = SearchClient(...) documents = [ { "id": generate_id(blob.name, i), "content": chunk, "embedding": emb, "source": blob.name, "timestamp": datetime.utcnow().isoformat() } for i, (chunk, emb) in enumerate(zip(chunks, embeddings)) ] search_client.upload_documents(documents)

Pattern 3: Incremental Refresh

Incremental refresh is a hybrid approach that processes only new or modified data since the last sync. This pattern balances freshness with efficiency:

  • Track changes using timestamps, ETags, or change tracking features
  • Process only deltas — new, modified, and deleted documents
  • Schedule regularly — every hour, every few hours, or daily depending on needs
  • Lower cost than full reprocessing since only changed documents generate new embeddings
Pattern Latency Cost Complexity Best For
Batch Hours Low–Medium Low Static content, initial loads
Real-time Seconds Higher High Frequently changing data
Incremental Minutes–Hours Medium Medium Balanced freshness & cost
💡 Recommendation

For most production scenarios, start with batch ingestion to build your initial index, then implement incremental refresh to keep it current. Reserve real-time streaming for use cases where latency is critical and the data source supports event-based notifications (like Cosmos DB change feed or Blob Storage Event Grid).

Data Preparation & Cleaning

Raw data is rarely ready for AI consumption. Before your documents can be effectively indexed and retrieved, they need to go through a preparation pipeline that includes text extraction, cleaning, chunking, and metadata enrichment. The quality of this preparation directly impacts the quality of your AI application's responses.

Text Extraction

The first step in data preparation is extracting clean text from your source documents. Different file formats require different extraction approaches:

  • PDF documents — Use Azure AI Document Intelligence (formerly Form Recognizer) for high-quality text extraction, including tables, headers, and layout preservation
  • Office documents — DOCX and PPTX files can be parsed using their XML-based internal structure
  • HTML content — Strip HTML tags while preserving semantic structure (headings, lists, tables)
  • Scanned documents — Apply OCR (Optical Character Recognition) before indexing; Azure AI Document Intelligence handles this automatically

Chunking Strategies

Chunking is one of the most critical decisions in data preparation. How you split your documents into chunks directly affects retrieval quality. Here are the most common strategies:

📐

Fixed-size Chunking

Split text into equal-sized segments based on token count. Simple but may break sentences or paragraphs mid-thought. Use 512–2048 tokens with 10–15% overlap.

📝

Sentence-based Chunking

Split at sentence boundaries, grouping sentences until reaching the target size. Preserves natural language flow and improves retrieval quality.

📄

Paragraph/Section Chunking

Use document structure (headings, sections, paragraphs) as natural break points. Best for well-structured documents with clear hierarchy.

🧠

Semantic Chunking

Use AI to identify topic boundaries within the text. Most sophisticated but also most compute-intensive. Best for complex, multi-topic documents.

Chunking Configuration in Azure AI Foundry json
{ "chunking_settings": { "strategy": "sentence", "chunk_size": 1024, "chunk_overlap": 128, "max_chunk_size": 2048 }, "metadata_extraction": { "fields": [ "filename", "page_number", "section_heading", "file_path", "last_modified" ] }, "preprocessing": { "remove_headers_footers": true, "remove_watermarks": true, "normalize_whitespace": true, "extract_tables_as_text": true } }

Data Cleaning Checklist

Before indexing, ensure your data passes these quality checks:

  • Remove duplicate documents and near-duplicates
  • Strip headers, footers, and page numbers from PDFs
  • Remove watermarks and irrelevant boilerplate text
  • Normalize whitespace and fix encoding issues (UTF-8)
  • Extract and preserve tables in a readable text format
  • Tag content with relevant metadata (source, date, author, category)
  • Handle multilingual content — identify language per document
  • Validate that OCR output is accurate for scanned documents
  • Remove sensitive or PII data that should not be in the index
  • Verify that extracted text is complete — no truncated content
⛔ Common Mistake

Don't skip the data cleaning step. Indexing poorly prepared data leads to irrelevant search results, hallucinations, and frustrated users. A well-prepared small dataset will outperform a large but messy one every time. Invest the time in quality data preparation — it pays dividends in retrieval accuracy.

Indexing Strategies

Once your data is prepared and chunked, the next step is creating a searchable index. Azure AI Foundry uses Azure AI Search under the hood to provide powerful indexing and retrieval capabilities. Understanding the different indexing strategies helps you optimize for your specific use case.

Vector Index

A vector index stores numerical representations (embeddings) of your text chunks. When a user queries the system, their query is also converted to a vector, and the most similar document vectors are retrieved using cosine similarity or other distance metrics.

  • Embedding models: text-embedding-ada-002, text-embedding-3-small, or text-embedding-3-large
  • Dimensions: 1536 (ada-002), 1536 (3-small), 3072 (3-large)
  • Similarity metric: Cosine similarity (default)
  • Strength: Captures semantic meaning — finds relevant content even when exact keywords don't match

Keyword Index

A keyword index uses traditional full-text search (BM25/TF-IDF) to match query terms against indexed content. This approach is fast and precise for exact term matches.

  • Strength: Exact matches, technical terms, product names, error codes
  • Weakness: Cannot find semantically related content without exact keyword overlap

Hybrid Search (Recommended)

Hybrid search combines vector and keyword retrieval, then uses Reciprocal Rank Fusion (RRF) to merge and re-rank results from both methods. This approach consistently outperforms either method alone.

Index Design Best Practices

✅ Best Practices for Index Design

Always include filterable metadata fields (source file, date, category) — this enables scoped searches that dramatically improve relevance. Use hybrid search (vector + keyword) as your default configuration. Start with HNSW algorithm for vector search — it provides the best balance of speed and recall. Plan for incremental updates — design your index schema with an ID field that maps to source documents so you can update specific chunks without rebuilding the entire index.

Mini Hack: Upload & Index PDF Documents

Now it's time to put theory into practice! In this hands-on mini hack, you'll upload a collection of PDF documents to Azure Blob Storage, connect that storage to Azure AI Foundry, and create a searchable vector index.

🛠️ Lab Details

Duration: 60 minutes  |  Prerequisites: Azure subscription, AI Foundry project, Blob Storage account  |  Detailed instructions: See lab/hands-on-lab.md

Exercise Overview

  1. Prepare Your Data Gather 5–10 PDF documents and upload them to an Azure Blob Storage container called module11-pdf-docs. Organize your files logically and verify they are text-based (not scanned without OCR).
  2. Create a Data Connection In the Azure AI Foundry portal (ai.azure.com), navigate to your project's Data section and add a new Azure Blob Storage connection. Use Managed Identity or account keys for authentication.
  3. Configure Indexing Create a new index with the following settings: chunk size of 1024 tokens, chunk overlap of 128 tokens, sentence-based chunking strategy, and hybrid search (vector + keyword) enabled.
  4. Monitor and Validate Watch the indexing progress in the portal. Once complete, test your index with keyword queries, vector queries, and hybrid queries. Compare the results across all three search types.
  5. Document Your Findings Record which search type returned the most relevant results for different query types. Note the total chunks created and any issues encountered during indexing.

Expected Outcomes

By the end of this mini hack, you should have:

  • A Blob Storage container with 5–10 PDF documents
  • A working data connection in Azure AI Foundry
  • A vector index with hybrid search enabled
  • Validated search results across keyword, vector, and hybrid search modes
  • Understanding of how chunk size affects retrieval quality
💪 Bonus Challenge

If you finish early, try creating a second index with a different chunk size (512 or 2048 tokens) and compare search quality. Then, navigate to the AI Foundry Playground and use your index as a data source for a grounded chat session — ask questions and evaluate how well the AI cites your documents.

Key Takeaways

🎯 What You've Learned in Module 11

  • Five data sources are supported — Azure Blob Storage, Azure SQL, Cosmos DB, SharePoint Online, and Microsoft OneLake each serve different data types and use cases.
  • Authentication matters — Use Managed Identity with RBAC roles for production workloads. Avoid storing keys in code or using anonymous access.
  • Three ingestion patterns — Batch for initial loads, incremental for ongoing updates, real-time for latency-critical scenarios. Start simple and evolve.
  • Data preparation is critical — Quality chunking, cleaning, and metadata enrichment directly determine retrieval quality. Invest time here.
  • Hybrid search is the default choice — Combining vector and keyword search with reciprocal rank fusion delivers the best results for most use cases.
  • Chunk size affects everything — 1024 tokens with 128 overlap is a solid starting point. Smaller chunks increase precision; larger chunks preserve context.
  • Metadata enables filtering — Always include source file, page number, and category metadata. This allows scoped searches that dramatically improve relevance.
  • Azure AI Foundry handles the heavy lifting — The portal provides a guided experience for connecting data sources, configuring indexing, and testing search — no custom code required for basic scenarios.
