1. What Is Foundry Local?

Foundry Local is a lightweight runtime from Microsoft that lets you run AI models entirely on your own device — no internet, no cloud API calls, no data leaving the machine. It is part of the broader Azure AI Foundry ecosystem but operates independently of any Azure subscription.

⚡

Local Inference

Run SLMs and LLMs directly on CPU, GPU, or NPU hardware with near-instant cold starts.

🔒

Data Sovereignty

Your data never leaves the device. Perfect for regulated industries like healthcare and finance.

🌐

Cross-Platform

Works on Windows (x64/ARM), macOS (Apple Silicon), Linux, and even mobile via ONNX Runtime.

🔌

OpenAI-Compatible API

Drop-in replacement for cloud endpoints — use the same OpenAI SDK code you already have.

📌 Key Distinction
Foundry Local is not cloud-based AI services. It is a standalone runtime for on-device inference and is part of the Azure AI Foundry product family. It does not require an Azure subscription to operate.

At its core, Foundry Local bundles ONNX Runtime GenAI with a local REST server that exposes an OpenAI-compatible chat completions endpoint. Models are stored in a local catalog cache and loaded on-demand.

2. When to Use Foundry Local

Foundry Local shines in scenarios where cloud-based inference is impractical, cost-prohibitive, or forbidden by policy:

ScenarioWhy Foundry Local?Example
Offline / Air-Gapped No internet required after model download Military field operations, submarines, remote research stations
Low Latency Eliminates network round-trip; ~10-50 ms first-token on NPU Real-time code completion, in-meeting transcription
Data Privacy Data never leaves the device boundary Patient record summarization in hospitals
Cost Optimization Zero per-token API cost; uses hardware you already own High-volume document classification on edge PCs
IoT / Edge Runs on compact devices with limited connectivity Smart factory quality inspection, agricultural drones
Development & Prototyping Iterate locally without Azure spend Prompt engineering, fine-tune testing, hackathons
💡 Decision Rule
If your model fits in device memory (typically ≤ 14 B parameters for a 16 GB laptop) and you value privacy, latency, or offline capability over raw model size — choose Foundry Local. For 70 B+ models or multi-modal workloads, continue using cloud endpoints.

4. Offline & Low-Connectivity Scenarios

Once a model is downloaded, Foundry Local requires zero internet access for inference. This makes it ideal for disconnected environments.

4.1 Pre-Staging Models for Air-Gapped Deployment

Step 1 — Download on a Connected Machine

Use foundry model download phi-3.5-mini to pull the model to the local cache.

Step 2 — Export the Cache

Copy ~/.foundry/models/ to a USB drive or removable storage.

Step 3 — Transfer to Air-Gapped Device

Copy the model directory to the same path on the target device.

Step 4 — Run Offline

Start the server with foundry service start — it discovers cached models automatically.

# On connected machine
foundry model download phi-3.5-mini
foundry cache export --output /mnt/usb/foundry-models/

# On air-gapped machine
foundry cache import --from /mnt/usb/foundry-models/
foundry service start
# API available at http://localhost:5272/v1/chat/completions
⚠️ License Compliance
Even in air-gapped scenarios, ensure you comply with the model license (e.g., MIT for Phi models). Keep a copy of the license file alongside exported models.

7.1 API Compatibility

Because Foundry Local exposes an OpenAI-compatible REST API, migrating between local and cloud is a one-line change:

from openai import OpenAI

# ── Cloud endpoint ──
# client = OpenAI(api_key="sk-...", base_url="https://api.openai.com/v1")

# ── Foundry Local endpoint ──
client = OpenAI(
    base_url="http://localhost:5272/v1",
    api_key="not-needed"  # local — no auth required
)

response = client.chat.completions.create(
    model="phi-3.5-mini",
    messages=[{"role": "user", "content": "Explain edge AI in 3 sentences."}],
    temperature=0.7,
    max_tokens=200
)

print(response.choices[0].message.content)

8.5 Understanding the Local Model Catalog

Foundry Local maintains a local model catalog — a registry of all models available for download and those already cached on disk. Unlike cloud-based model registries, this catalog is designed for disconnected operation after initial sync.

8.5.1 Catalog Operations

# List all available models (requires internet on first sync)
foundry model list

# Show only downloaded models (works offline)
foundry model list --downloaded

# Get detailed model information
foundry model info phi-3.5-mini

# Show cache location and disk usage
foundry cache info

# Clean unused model files
foundry cache clean --keep-latest 2

8.5.2 Model Versioning

In production edge deployments, pinning to a specific model version prevents unexpected behavior changes:

# Download a specific version
foundry model download phi-3.5-mini@1.2.0

# Run a pinned version
foundry model run phi-3.5-mini@1.2.0

# List available versions
foundry model versions phi-3.5-mini
# v1.0.0  (2025-09-15)  2.3 GB  Stable
# v1.1.0  (2025-12-01)  2.4 GB  Stable
# v1.2.0  (2026-03-10)  2.4 GB  Latest

8.5.3 Custom Model Import

You can import custom ONNX models (e.g., fine-tuned variants) into the Foundry Local catalog:

# Import a custom ONNX model
foundry model import --name my-custom-phi \
    --path ./exported-model/ \
    --format onnx \
    --tokenizer ./tokenizer/

# The model is now available in the local catalog
foundry model run my-custom-phi
💡 Pro Tip
Use Azure AI Foundry (cloud) to fine-tune a model, export it to ONNX format using the optimum library, then import it into Foundry Local for edge deployment. This gives you the best of both worlds: cloud-scale training with local inference.

8.6 Security Considerations for Local AI

Running AI models locally introduces a different security profile than cloud deployments. Consider these critical aspects:

8.6.1 Network Exposure

🚨 Critical Security Warning
By default, Foundry Local binds to localhost (127.0.0.1), which is only accessible from the local machine. Never expose Foundry Local on 0.0.0.0 without authentication in production. If you need network access (e.g., from other containers), use a reverse proxy with TLS and authentication. 
# ✅ Safe — localhost only (default)
foundry service start

# ⚠️ Use with caution — accessible on all interfaces
foundry service start --host 0.0.0.0

# ✅ Recommended for network access — use with reverse proxy
# nginx.conf or Caddy with TLS + basic auth in front of port 5272

8.6.2 Model Supply Chain Security

  • Verify model integrity: Foundry Local validates SHA-256 checksums on download. Never skip verification.
  • Use official catalog: Only download models from the official Foundry model catalog or trusted internal registries.
  • Audit model licenses: Keep license files alongside models, especially in air-gapped deployments.
  • Scan custom models: If importing custom ONNX models, ensure they come from a trusted training pipeline.

9. Best Practices & Troubleshooting

9.1 Best Practices

  • Right-size your model: Use INT4 quantized variants (e.g., Phi-3.5-mini) for edge devices. Only use FP16/FP32 when accuracy is critical and hardware supports it.
  • Pin model versions: Use foundry model run phi-3.5-mini@1.2.0 in production to avoid unexpected behavior from model updates.
  • Monitor resource usage: Use foundry status to track memory and GPU utilization. Set --max-tokens to cap memory consumption.
  • Implement graceful fallback: If the local model fails to load, fallback to a cloud endpoint automatically in your app logic.
  • Use containers for edge: Package Foundry Local + model in a Docker container for reproducible edge deployments.
  • Pre-warm on boot: Add foundry service start to your system startup to minimize first-request latency.
