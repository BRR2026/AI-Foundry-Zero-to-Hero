What Is Foundry Local?

A lightweight local runtime for AI inference directly on your device — laptop, edge appliance, or IoT gateway.

  • Part of the Azure AI Foundry ecosystem
  • Not Azure Machine Learning
  • No Azure subscription required
  • OpenAI-compatible REST API
  • Powered by ONNX Runtime GenAI
⚡

Key Value Proposition

Same code, same SDK — just change the base URL from a cloud endpoint to localhost:5272.

🔒

Data Never Leaves

All processing stays on-device. Perfect for HIPAA, GDPR, and classified workloads.

OpenAI-Compatible API

Switch between cloud and local with one line:

from openai import OpenAI

# Cloud
# client = OpenAI(api_key="sk-...", base_url="https://...")

# Foundry Local — just change the URL
client = OpenAI(
    base_url="http://localhost:5272/v1",
    api_key="not-needed"
)

response = client.chat.completions.create(
    model="phi-3.5-mini",
    messages=[{"role": "user",
               "content": "Explain edge AI"}],
    temperature=0.7
)
print(response.choices[0].message.content)

Offline & Air-Gapped Deployment

Workflow

  • 1. Download on connected machine
  • 2. Export cache to USB / media
  • 3. Transfer to air-gapped device
  • 4. Import & start service offline
Zero internet after model download. Verify by disconnecting Wi-Fi! 
# Connected machine
foundry model download phi-3.5-mini
foundry cache export \
  --output /mnt/usb/models/

# Air-gapped machine
foundry cache import \
  --from /mnt/usb/models/
foundry service start
# Ready at localhost:5272

Best Practices

Do ✅

  • Use INT4 for edge devices
  • Pin model versions in production
  • Containerize for reproducibility
  • Pre-warm on boot for low latency
  • Implement cloud fallback logic
  • Monitor with foundry status

Avoid ❌

  • Running FP32 on memory-constrained devices
  • Ignoring model license compliance
  • Exposing the API on 0.0.0.0 without auth
  • Skipping hardware capability checks
  • Using Foundry Local for 70B+ models

Key Takeaways

  • Foundry Local = on-device AI runtime (not Azure ML)
  • Hardware priority: NPU → GPU → CPU
  • OpenAI-compatible API — one-line migration
  • Zero internet after model download
  • Containers for edge & IoT at scale
  • INT4 quantization for resource-constrained devices
