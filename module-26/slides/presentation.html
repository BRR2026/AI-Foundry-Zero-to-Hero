Three Deployment Types

🖥️

Managed Compute

Dedicated VMs (GPU/CPU). Full control over SKU, autoscaling, and networking. Best for fine-tuned or custom models.

⚡

Serverless API

Pay-per-token, zero infrastructure. Instant availability for Model Catalog models. Ideal for prototyping and variable loads.

🔒

Provisioned Throughput

Reserved PTU capacity. Guaranteed latency and throughput for mission-critical production SLA workloads.

Deploy via SDK

# Deploy a serverless model via Azure AI Foundry SDK
from azure.ai.projects import AIProjectClient

deployment = project_client.deployments.begin_create_or_update(
    name="gpt-4o-production",
    model="gpt-4o",
    model_version="2024-11-20",
    sku_name="GlobalStandard",
    sku_capacity=50,  # 50K tokens/min
).result()

Also available via Azure CLI, REST API, Bicep, and Terraform.

Scaling Strategies

Serverless

Adjust sku_capacity (TPM). No instance management. Scaling is transparent.

Managed Compute

Set min_instances, max_instances, and target_utilization. Scale-to-zero possible.

Provisioned (PTU)

Fixed PTU allocation. Use the Capacity Calculator for sizing. ~10 PTUs ≈ 6 req/min (GPT-4o).

Blue/Green Deployments

Run two model versions behind a single endpoint URL. Gradually shift traffic from blue (stable) to green (new).

  • Zero-downtime model upgrades
  • Instant rollback capability
  • A/B comparison of model quality
  • Risk-managed production releases

Traffic Splitting — Code

# Configure 90/10 traffic split
endpoint = project_client.endpoints.begin_create_or_update(
    name="gpt4o-endpoint",
    traffic={
        "gpt4o-blue": 90,   # current stable version
        "gpt4o-green": 10,  # new canary version
    },
).result()

# Instant rollback — shift 100% back to blue
project_client.endpoints.begin_create_or_update(
    name="gpt4o-endpoint",
    traffic={"gpt4o-blue": 100, "gpt4o-green": 0},
).result()

⚠️ Traffic percentages must sum to 100. Changes propagate in seconds.

Authentication Methods

🔑

API Key

  • Simple api-key header
  • Primary + secondary keys
  • Zero-downtime rotation
  • Store in Key Vault
🛡️

Microsoft Entra ID (Recommended)

  • Managed identity — no secrets
  • Fine-grained RBAC
  • Full audit trail
  • Works with DefaultAzureCredential
Azure AI Foundry Training Module 26

API Key Rotation Strategy

  • Clients use primary key
  • Regenerate primary key
  • Update all clients to new primary
  • Regenerate secondary key

Secondary key stays active during rotation → zero downtime

Security Best Practices

  • ✅ Store keys in Azure Key Vault
  • ✅ Automate rotation every 90 days
  • ✅ Use separate keys per environment
  • ❌ Never commit to source control
  • ❌ Never share across dev/staging/prod

Deployment Best Practices

📊 Monitor Everything

Latency P50/P95/P99, token usage, error rates, content filter triggers.

🔄 Always Traffic Split

Never deploy new models to 100%. Start at 5-10% canary.

📋 Pin Model Versions

Use NoAutoUpgrade. Explicit versioning prevents surprises.

🏗️ Infrastructure as Code

Define deployments in Bicep/Terraform. Version alongside app code.

Blue/Green Traffic Splitting
