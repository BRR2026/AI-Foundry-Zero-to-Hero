1. Deployment Overview

Once you've built, fine-tuned, or selected a foundation model inside Azure AI Foundry, the next critical step is deploying it as a production-grade endpoint. An endpoint transforms a static model artifact into a live, scalable REST API that applications can consume. Without proper deployment, even the most sophisticated model remains a lab experiment.

Azure AI Foundry provides a unified deployment experience across multiple compute paradigms—each tailored to different workload profiles, latency requirements, and cost constraints. Whether you're spinning up a quick prototype or deploying a mission-critical production system, the platform gives you the tools to match infrastructure to your exact requirements.

What is an Endpoint?

An endpoint in Azure AI Foundry is a durable, versioned REST API resource that accepts inference requests and returns model predictions. Each endpoint has:

  • A unique URL — a stable address clients use to send requests, independent of the underlying model version.
  • Authentication — API keys or Microsoft Entra ID tokens that secure access to the endpoint.
  • One or more deployments — each deployment represents a specific model version and configuration running behind the endpoint.
  • Traffic rules — percentage-based routing that determines how requests are distributed across deployments.
  • Monitoring hooks — automatic integration with Azure Monitor for latency, throughput, and error tracking.
💡

Key Distinction: An endpoint is the stable URL your application calls. A deployment is a specific model instance running behind that endpoint. One endpoint can host multiple deployments with traffic split between them.

2. Deployment Types

Azure AI Foundry supports three primary deployment types, each optimized for different production scenarios:

🖥️

Managed Compute

Dedicated VM-based infrastructure for models that require custom containers, fine-tuned weights, or GPU acceleration. Full control over SKU, scaling rules, and networking.

⚡

Serverless API

Pay-per-token deployment with no infrastructure management. Ideal for foundation models from the Model Catalog. Instant availability, usage-based billing.

🔒

Provisioned Throughput

Reserved capacity measured in Provisioned Throughput Units (PTUs). Guaranteed throughput with predictable latency for mission-critical workloads.

Decision Framework

Choosing the right deployment type depends on your specific requirements. Use this decision framework:

🤔

Quick Decision Guide:

→ Custom / fine-tuned model? Use Managed Compute.

→ Catalog model + variable load? Use Serverless API.

→ Production SLA + guaranteed latency? Use Provisioned Throughput.

→ Prototyping or experimentation? Start with Serverless API, migrate later.

Comparison Matrix

Dimension Managed Compute Serverless API Provisioned Throughput
Infrastructure Dedicated VMs (GPU/CPU) Fully managed (no VMs) Reserved capacity (PTUs)
Billing Model Per-VM-hour Per-token (input/output) Per-PTU-hour (reserved)
Scaling Manual or auto (instance count) Automatic (transparent) Fixed PTU allocation
Latency Low (dedicated resources) Variable (shared infra) Predictable & consistent
Custom Models ✅ Full support ❌ Catalog models only ✅ Select models
Best For Fine-tuned / custom models Rapid prototyping, variable load Production SLA workloads

Deploying via the Azure AI Foundry SDK

The azure-ai-projects SDK provides a unified deployment API. Here's how to deploy a model from the catalog using a serverless (pay-per-token) deployment:

# Deploy a model using the Azure AI Foundry SDK
from azure.identity import DefaultAzureCredential
from azure.ai.projects import AIProjectClient

# Initialize the Foundry project client
credential = DefaultAzureCredential()
project_client = AIProjectClient(
    credential=credential,
    subscription_id="<subscription-id>",
    resource_group_name="<resource-group>",
    project_name="<project-name>"
)

# Deploy a serverless (pay-per-token) model
deployment = project_client.deployments.begin_create_or_update(
    name="gpt-4o-production",
    model="gpt-4o",
    model_version="2024-11-20",
    sku_name="GlobalStandard",    # Serverless API tier
    sku_capacity=50,              # Rate limit: 50K tokens/min
).result()

print(f"Endpoint: {deployment.properties.endpoint_url}")
print(f"Status:   {deployment.properties.provisioning_state}")
Python

Azure CLI Deployment

3. Endpoint Configuration & Scaling

Every deployment creates an endpoint—a unique URL that accepts inference requests. Configuring the endpoint correctly ensures reliability, performance, and cost efficiency.

Key Configuration Parameters

Parameter Description Example
SKU Name Deployment tier (GlobalStandard, Standard, ProvisionedManaged) GlobalStandard
SKU Capacity Rate limit in thousands of tokens/min (TPM) or PTUs 80 (80K TPM)
Version Policy How model version upgrades are handled NoAutoUpgrade
Content Filter Content safety policy applied to inputs/outputs DefaultV2
Network Isolation Private endpoint or public access restrictions PrivateEndpoint

Scaling Strategies

Serverless deployments scale automatically. You control throughput via rate limits (tokens per minute):

# Update rate limit for a serverless deployment
updated = project_client.deployments.begin_create_or_update(
    name="gpt-4o-production",
    model="gpt-4o",
    sku_name="GlobalStandard",
    sku_capacity=120,  # Increase to 120K TPM
).result()
Python

Managed compute deployments scale by instance count. Enable autoscaling to handle variable load:

# Configure autoscaling for managed compute
from azure.ai.projects.models import (
    ManagedComputeDeployment,
    AutoScaleSettings
)

deployment = ManagedComputeDeployment(
    name="my-custom-model",
    model="my-fine-tuned-llm",
    instance_type="Standard_NC24ads_A100_v4",
    instance_count=2,
    scale_settings=AutoScaleSettings(
        min_instances=1,
        max_instances=5,
        polling_interval=10,       # seconds
        target_utilization=70,   # percent
    ),
)
Python

Provisioned Throughput provides fixed, guaranteed capacity measured in PTUs. Capacity is reserved and billed hourly:

# Deploy with Provisioned Throughput Units (PTUs)
deployment = project_client.deployments.begin_create_or_update(
    name="gpt-4o-ptu-prod",
    model="gpt-4o",
    sku_name="ProvisionedManaged",
    sku_capacity=100,  # 100 PTUs
).result()

# PTU Sizing Guide:
# ~10 PTUs ≈ 6 requests/min for GPT-4o
# Use the Capacity Calculator in the Foundry Portal
Python
💡

Cost Optimization Tip: Start with a Serverless API deployment during development and testing. Once you understand your token consumption patterns, evaluate whether Provisioned Throughput (PTU) offers better economics for your steady-state production workload.

4. Blue/Green Deployments & Traffic Splitting

Blue/green deployment is a release strategy that lets you run two versions of a model simultaneously, gradually shifting traffic from the old version (blue) to the new version (green). This minimizes risk and enables instant rollback.

How It Works in Azure AI Foundry

Azure AI Foundry endpoints support multiple deployments behind a single endpoint URL. You control what percentage of traffic goes to each deployment:

Traffic Split Visualization — Gradual Rollout

Phase 1: Canary (10% to green)

Blue — 90%
10%

Phase 2: Validation (50/50)

Blue — 50%
Green — 50%

Phase 3: Full Cutover

Green — 100%
Blue (current stable) Green (new version)

Implementing Traffic Splitting

# Step 1: Create the "blue" deployment (current production model)
blue = project_client.deployments.begin_create_or_update(
    name="gpt4o-blue",
    model="gpt-4o",
    model_version="2024-08-06",
    sku_name="GlobalStandard",
    sku_capacity=80,
).result()

# Step 2: Create the "green" deployment (new model version)
green = project_client.deployments.begin_create_or_update(
    name="gpt4o-green",
    model="gpt-4o",
    model_version="2024-11-20",
    sku_name="GlobalStandard",
    sku_capacity=30,
).result()

# Step 3: Configure traffic splitting on the endpoint
endpoint = project_client.endpoints.begin_create_or_update(
    name="gpt4o-endpoint",
    traffic={
        "gpt4o-blue": 90,    # 90% to current version
        "gpt4o-green": 10,   # 10% canary to new version
    },
).result()

print(f"Endpoint URL: {endpoint.properties.scoring_uri}")
print(f"Traffic:      {endpoint.properties.traffic}")
Python

Gradual Rollout Strategy

Deploy Green (Canary)

Deploy the new model version alongside the existing blue deployment. Route 10% of traffic to green for initial validation.

Monitor & Validate

Compare latency, error rates, and output quality between blue and green using Azure Monitor metrics and custom evaluation scripts.

Increase Traffic

If metrics are healthy, increase green traffic to 50%, then 75%, then 100%. At each phase, validate for at least 15–30 minutes.

Full Cutover or Rollback

Once green is at 100% with healthy metrics, decommission the blue deployment. If issues arise at any stage, set blue back to 100% instantly.

Rollback in Seconds

One of the most powerful aspects of traffic splitting is the ability to perform an instant rollback. If the green deployment exhibits degraded performance, elevated error rates, or unexpected behavior, you can revert all traffic to the stable blue deployment in seconds:

# Instant rollback — shift all traffic back to blue
project_client.endpoints.begin_create_or_update(
    name="gpt4o-endpoint",
    traffic={
        "gpt4o-blue": 100,
        "gpt4o-green": 0,
    },
).result()
print("⚠️ Rollback complete — 100% traffic on blue")
Python

Rollback via Azure CLI

# Rollback using Azure CLI — equally fast
az ai model deployment update-traffic \
  --endpoint-name gpt4o-endpoint \
  --project-name my-ai-project \
  --resource-group my-rg \
  --traffic "gpt4o-blue=100,gpt4o-green=0"

# Verify the rollback
az ai model deployment show-traffic \
  --endpoint-name gpt4o-endpoint \
  --project-name my-ai-project \
  --resource-group my-rg
Bash

Monitoring During Traffic Splits

Effective blue/green deployment requires continuous monitoring. Key metrics to track at each phase:

Metric Blue Baseline Green Threshold Action if Exceeded
Error Rate < 0.5% < 2% Rollback immediately
P95 Latency Baseline ± 10% < 3x baseline Investigate; rollback if sustained
Token Usage Known average < 1.5x baseline Check for prompt issues
Content Filter Triggers Known rate < 2x baseline Investigate model behavior
⚠️

Important: Traffic percentages must always sum to 100. When you update the traffic split, the change propagates within seconds but existing in-flight requests will complete on their original deployment.

5. API Key Management & Authentication

Securing your deployed endpoints is critical. Azure AI Foundry supports two authentication methods:

🔑

API Key Authentication

Simple key-based auth using the api-key header. Each deployment has a primary and secondary key for zero-downtime rotation.

🛡️

Microsoft Entra ID (OAuth)

Token-based auth using managed identities and RBAC. No secrets to manage — the recommended approach for production workloads.

API Key Operations

# Retrieve API keys for a deployment
keys = project_client.deployments.list_keys(
    name="gpt-4o-production"
)

print(f"Primary Key:   {keys.primary_key[:8]}...")
print(f"Secondary Key: {keys.secondary_key[:8]}...")

# Rotate the primary key (secondary stays active for zero-downtime)
project_client.deployments.regenerate_key(
    name="gpt-4o-production",
    key_type="primary"
)
print("✅ Primary key rotated. Update your clients to use the new key.")
Python

Calling an Endpoint with API Key

import requests

endpoint_url = "https://<project>.services.ai.azure.com/models/chat/completions"
api_key = "<your-api-key>"

response = requests.post(
    endpoint_url,
    headers={
        "Content-Type": "application/json",
        "api-key": api_key,
    },
    json={
        "model": "gpt-4o-production",
        "messages": [
            {"role": "user", "content": "Explain blue/green deployments."}
        ],
        "max_tokens": 500,
    }
)

print(response.json()["choices"][0]["message"]["content"])
Python

Using Microsoft Entra ID (Recommended for Production)

from azure.identity import DefaultAzureCredential
from openai import AzureOpenAI

credential = DefaultAzureCredential()
token = credential.get_token("https://cognitiveservices.azure.com/.default")

client = AzureOpenAI(
    azure_endpoint="https://<project>.services.ai.azure.com",
    azure_ad_token=token.token,
    api_version="2024-12-01-preview",
)

completion = client.chat.completions.create(
    model="gpt-4o-production",
    messages=[{"role": "user", "content": "Hello!"}],
)
print(completion.choices[0].message.content)
Python

Key Rotation Best Practices

Use Dual Keys for Zero-Downtime Rotation

Always configure clients with the primary key. When rotating, regenerate primary → update clients → regenerate secondary. The secondary key remains valid during the transition window, ensuring no requests fail.

Store Keys in Azure Key Vault

Never hardcode API keys in source code or configuration files. Use Key Vault references and managed identity to securely access keys at runtime. This centralizes secret management and provides audit logging.

Automate Rotation with Azure Functions

Set up a scheduled Azure Function that rotates keys on a regular cadence (e.g., every 90 days) and updates Key Vault automatically. This eliminates manual rotation tasks and reduces the risk of key compromise.

Implement Key Vault Event-Driven Rotation

Use Azure Event Grid to trigger automatic key rotation when keys approach their expiration date. Combine with Azure Functions for a fully automated rotation pipeline.

Network Security for Endpoints

Beyond authentication, secure your endpoints at the network level:

  • Private Endpoints: Deploy endpoints within a VNet using Azure Private Link. Traffic never traverses the public internet.
  • IP Allowlisting: Restrict access to known IP ranges for public endpoints.
  • Service Tags: Use Azure service tags in NSG rules to allow traffic only from trusted Azure services.
  • Content Filters: Apply content safety policies to prevent prompt injection, harmful content generation, and data exfiltration through model outputs.
# Example: Configure a deployment with private endpoint
deployment = project_client.deployments.begin_create_or_update(
    name="gpt-4o-secure",
    model="gpt-4o",
    sku_name="GlobalStandard",
    sku_capacity=50,
    properties={
        "networkIsolation": {
            "publicNetworkAccess": "Disabled",
        },
        "contentFilter": "DefaultV2",
    },
).result()
Python
🔐

Security Warning: Never commit API keys to source control. Use environment variables, Key Vault, or managed identity. Consider enabling Microsoft Entra ID authentication to eliminate API keys entirely for your production endpoints.

8. Best Practices

📊

Monitor Everything

Enable Azure Monitor for latency (P50/P95/P99), token usage, error rates, and model-specific quality metrics on every deployment.

🔄

Always Use Traffic Splitting

Never deploy a new model version directly to 100% traffic. Start with 5-10% canary and validate metrics before each increment.

💰

Right-Size Your Deployment

Use the PTU Capacity Calculator for provisioned deployments. For serverless, set rate limits based on observed peak TPM + 20% headroom.

🛡️

Secure by Default

Prefer Entra ID over API keys. Enable private endpoints for network isolation. Use content filters to prevent prompt injection and harmful outputs.

📋

Pin Model Versions

Always specify exact model versions. Set NoAutoUpgrade for version policy to prevent unexpected model changes in production.

🏗️

Infrastructure as Code

Define deployments in Bicep or Terraform for reproducibility. Store endpoint configs alongside your application code in source control.

Common Anti-Patterns to Avoid

❌

Anti-Pattern 1: Big-Bang Deployments. Deploying a new model version to 100% traffic without canary validation. Always use traffic splitting with gradual rollout.

❌

Anti-Pattern 2: Auto-Upgrading in Production. Setting AutoUpgrade version policy on production endpoints. Model behavior can change unexpectedly when a new version is released.

❌

Anti-Pattern 3: Shared API Keys. Using the same API key across development, staging, and production environments. Each environment should have independent keys with separate rotation schedules.

❌

Anti-Pattern 4: No Monitoring. Deploying an endpoint without configuring alerts. You won't know about degraded performance until users complain. Set up proactive monitoring from day one.
