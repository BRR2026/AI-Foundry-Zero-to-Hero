2. The M365 Copilot Extensibility Model

Microsoft 365 Copilot supports three primary extensibility patterns. Understanding which pattern to use is critical for building effective integrations with Azure AI Foundry.

Three Extension Types

Extension TypeDescriptionBest ForComplexity
Declarative Agents Custom Copilot agents with instructions, knowledge, and actions defined via manifest Domain-specific Q&A, scoped assistants Low–Medium
API Plugins OpenAPI-described REST endpoints that Copilot can invoke during conversations Real-time data, transactional operations Medium
Message Extensions Teams-based search/action extensions surfaced in the Copilot compose experience Teams-centric search, card-based results Medium–High
⚠️ Important Distinction
Declarative Agents are the newest and most powerful extensibility pattern. They allow you to define custom instructions, ground knowledge in external data (like an AI Foundry index), and configure specific actions — all without writing backend code. API Plugins provide more flexibility for custom logic but require a hosted service.

Extensibility Decision Framework

📋

Declarative Agent

Choose when you need a scoped Copilot persona with access to specific data sources and pre-defined conversation instructions. Ideal for connecting to AI Foundry knowledge bases.

⚡

API Plugin

Choose when Copilot needs to call your custom API for real-time data retrieval, execute business logic, or perform transactional operations via AI Foundry agents.

💬

Message Extension

Choose when the primary interaction model is Teams-based and you need rich Adaptive Card responses with search and action capabilities.

🔗

Graph Connectors

Choose when you want to ingest external data into the Microsoft Graph so Copilot can reason over it natively alongside M365 content.

The Plugin Manifest

Every Copilot extension starts with a manifest file that describes its capabilities. For declarative agents, this includes instructions, knowledge sources, and actions:

3. Building Message Extensions and Plugins

Message extensions are a powerful way to surface AI Foundry intelligence in Microsoft Teams and M365 Copilot. They enable search-based and action-based interactions directly in the compose box.

Search-Based Message Extension

A search-based message extension lets users query your AI Foundry knowledge base from within Teams or Copilot. The flow is:

1

User Invokes the Extension

The user types a query in the Copilot compose box or selects the extension from the app flyout.

2

Query Routed to Your Backend

Teams sends the query to your bot endpoint hosted on Azure (App Service, Functions, or Container Apps).

3

AI Foundry Processes the Query

Your backend calls the AI Foundry endpoint — either a deployed RAG pipeline, prompt flow, or AI agent — to generate results.

4

Results Returned as Adaptive Cards

Formatted results are sent back as Adaptive Cards that render inline in Teams or Copilot.

Bot Framework Implementation

TypeScript
import { TeamsActivityHandler, CardFactory, TurnContext } from 'botbuilder';
import { AzureAIFoundryClient } from '@azure/ai-foundry';

export class KnowledgeSearchBot extends TeamsActivityHandler {
  private foundryClient: AzureAIFoundryClient;

  constructor() {
    super();
    this.foundryClient = new AzureAIFoundryClient({
      endpoint: process.env.AI_FOUNDRY_ENDPOINT!,
      credential: new DefaultAzureCredential()
    });
  }

  async handleTeamsMessagingExtensionQuery(
    context: TurnContext,
    query: any
  ) {
    const searchQuery = query.parameters[0]?.value || '';

    // Call AI Foundry RAG endpoint
    const response = await this.foundryClient.agents.run({
      agentId: process.env.KNOWLEDGE_AGENT_ID!,
      messages: [{ role: 'user', content: searchQuery }],
      tools: [{ type: 'azure_ai_search' }]
    });

    // Transform to Adaptive Cards
    const attachments = response.choices.map(choice =>
      CardFactory.heroCard(
        choice.title,
        choice.content.substring(0, 200),
        [],
        [{ type: 'openUrl', title: 'View Source', value: choice.sourceUrl }]
      )
    );

    return {
      composeExtension: {
        type: 'result',
        attachmentLayout: 'list',
        attachments
      }
    };
  }
}

Connecting via Prompt Flow

One of the cleanest integration patterns uses Prompt Flow as the orchestration layer between AI Foundry and M365 Copilot:

Python
# prompt_flow/knowledge_search/flow.py
from promptflow import tool
from azure.search.documents import SearchClient
from azure.identity import DefaultAzureCredential
from openai import AzureOpenAI

@tool
def search_and_respond(query: str, top_k: int = 5) -> dict:
    """Search knowledge base and generate grounded response."""

    # 1. Vector search in Azure AI Search
    search_client = SearchClient(
        endpoint=os.environ["SEARCH_ENDPOINT"],
        index_name="knowledge-index",
        credential=DefaultAzureCredential()
    )

    results = search_client.search(
        search_text=query,
        query_type="semantic",
        semantic_configuration_name="default",
        top=top_k,
        select=["title", "content", "source_url"]
    )

    # 2. Build context from search results
    context_parts = []
    sources = []
    for r in results:
        context_parts.append(f"[{r['title']}]: {r['content']}")
        sources.append({"title": r["title"], "url": r["source_url"]})

    context = "\n\n".join(context_parts)

    # 3. Generate grounded response with Azure OpenAI
    client = AzureOpenAI(
        azure_endpoint=os.environ["AOAI_ENDPOINT"],
        azure_deployment="GPT-5.1",
        api_version="2025-01-01-preview"
    )

    completion = client.chat.completions.create(
        model="GPT-5.1",
        messages=[
            {"role": "system", "content": f"""Answer using ONLY the context below.
            Cite sources using [Source Title] notation.

            Context:
            {context}"""},
            {"role": "user", "content": query}
        ],
        temperature=0.3
    )

    return {
        "answer": completion.choices[0].message.content,
        "sources": sources,
        "confidence": results.get_count() / top_k if results.get_count() else 0
    }

6. Copilot Studio Integration

Microsoft Copilot Studio provides a low-code/no-code environment for building custom copilots and extending M365 Copilot. It can orchestrate multiple AI agents — including those powered by Azure AI Foundry.

Multi-Agent Orchestration

Copilot Studio enables multi-agent orchestration where different specialized agents handle different aspects of a user's request:

🧠

Orchestrator Agent

The central Copilot Studio agent that receives user queries and routes them to specialized sub-agents based on intent.

📚

Knowledge Agent

An AI Foundry-powered RAG agent that searches indexed documents and returns grounded answers with citations.

📊

Analytics Agent

An AI Foundry agent that queries structured data, runs calculations, and generates charts or summaries.

🔧

Action Agent

An agent that performs transactional operations — creating tickets, updating records, or triggering workflows.

Connecting AI Foundry to Copilot Studio

1

Deploy Your AI Foundry Agent

Create and deploy a RAG pipeline or AI agent in Azure AI Foundry. Note the endpoint URL and API key.

2

Create a Custom Connector in Copilot Studio

In Copilot Studio, create a custom connector that points to your AI Foundry endpoint using the OpenAPI spec.

3

Add as a Plugin Action

Register the connector as a plugin action in your copilot's topic flow, mapping input/output parameters.

4

Publish to M365 Copilot

Publish the copilot to Microsoft 365 Copilot so it appears as an available extension for licensed users.

Key Compliance Areas

AreaRequirementImplementation
Data Residency Data must stay within geographic boundaries Deploy AI Foundry and Search in matching Azure regions
Data Classification Sensitivity labels must be honored Use Microsoft Purview integration for label-aware filtering
Access Control Users can only access data they're authorized to see Implement OBO (On-Behalf-Of) token flow with Azure AD
Audit Logging All plugin invocations must be logged Enable Application Insights and M365 Unified Audit Log
Data Loss Prevention Prevent sensitive data from leaking via AI responses Configure DLP policies in Microsoft Purview
Responsible AI Content filtering and safety guardrails Use AI Foundry content filters and system message controls

EU Data Boundary Compliance

🔒 Critical for EU Customers
If your M365 tenant is in the EU Data Boundary, ensure your AI Foundry project is deployed in an EU Azure region (e.g., westeurope, northeurope, francecentral). Cross-region API calls that transfer personal data outside the EU boundary can violate GDPR and the EU Data Boundary commitment.

Access Control with On-Behalf-Of Flow

The OBO (On-Behalf-Of) token flow ensures that your AI Foundry endpoint respects the calling user's permissions — not a service account's permissions:

TypeScript
import { ConfidentialClientApplication } from '@azure/msal-node';

async function getOBOToken(userToken: string): Promise<string> {
  const msalClient = new ConfidentialClientApplication({
    auth: {
      clientId: process.env.AAD_APP_CLIENT_ID!,
      clientSecret: process.env.AAD_APP_CLIENT_SECRET!,
      authority: `https://login.microsoftonline.com/${process.env.TENANT_ID}`
    }
  });

  const result = await msalClient.acquireTokenOnBehalfOf({
    oboAssertion: userToken,
    scopes: ['https://cognitiveservices.azure.com/.default']
  });

  return result!.accessToken;
}

// Use OBO token to call AI Foundry with user's identity
async function queryWithUserContext(userToken: string, query: string) {
  const oboToken = await getOBOToken(userToken);

  const response = await fetch(process.env.AI_FOUNDRY_ENDPOINT!, {
    method: 'POST',
    headers: {
      'Authorization': `Bearer ${oboToken}`,
      'Content-Type': 'application/json'
    },
    body: JSON.stringify({
      messages: [{ role: 'user', content: query }]
    })
  });

  return response.json();
}

8. Security and Authentication Model

Securing the integration between M365 Copilot and AI Foundry involves multiple layers:

🔑

Azure AD SSO

Single sign-on ensures users authenticate once via their M365 identity. The bot/API receives tokens automatically.

🛡️

Managed Identity

Use system-assigned managed identity for your App Service to call AI Foundry — no secrets in code.

🔐

Private Endpoints

Deploy AI Foundry behind private endpoints so data never traverses the public internet.

📋

App Consent

Admin consent workflows ensure IT approves which Copilot extensions can access organizational data.

Security Checklist

✅ Do

  • Use managed identity for service-to-service calls
  • Implement OBO flow for user-context queries
  • Store secrets in Azure Key Vault
  • Enable content filtering in AI Foundry
  • Validate user permissions before returning results
  • Log all plugin invocations to Application Insights

❌ Don't

  • Hardcode API keys in manifest or code
  • Use service account tokens for user queries
  • Expose AI Foundry endpoints publicly without auth
  • Skip input validation on user queries
  • Return raw AI Foundry responses without filtering
  • Ignore sensitivity labels on indexed documents

10. Mini Hack: Create a M365 Copilot Plugin

Step-by-Step Overview

1

Scaffold the Project

Use Teams Toolkit to create a new "Declarative Agent" project with an API plugin action.

2

Configure AI Foundry Backend

Create an Azure Functions API that proxies queries to your AI Foundry RAG endpoint using managed identity.

3

Define the OpenAPI Specification

Write the OpenAPI spec for your search endpoint, including parameter descriptions for Copilot's orchestrator.

4

Create the Declarative Agent Manifest

5

Key Code: Azure Function Backend

TypeScript
// api/src/functions/knowledgeSearch.ts
import { app, HttpRequest, HttpResponseInit } from '@azure/functions';
import { DefaultAzureCredential } from '@azure/identity';

app.http('knowledgeSearch', {
  methods: ['POST'],
  authLevel: 'anonymous', // Auth handled by EasyAuth / AAD
  handler: async (request: HttpRequest): Promise<HttpResponseInit> => {
    const { query, top_k = 5 } = await request.json() as any;

    // Get token via managed identity
    const credential = new DefaultAzureCredential();
    const token = await credential.getToken(
      'https://cognitiveservices.azure.com/.default'
    );

    // Call AI Foundry endpoint
    const aiResponse = await fetch(process.env.AI_FOUNDRY_ENDPOINT!, {
      method: 'POST',
      headers: {
        'Authorization': `Bearer ${token.token}`,
        'Content-Type': 'application/json'
      },
      body: JSON.stringify({
        messages: [
          {
            role: 'system',
            content: `You are a knowledge base assistant. Answer questions
              using only the provided context. Cite sources.`
          },
          { role: 'user', content: query }
        ],
        data_sources: [{
          type: 'azure_search',
          parameters: {
            endpoint: process.env.SEARCH_ENDPOINT,
            index_name: 'knowledge-index',
            authentication: { type: 'system_assigned_managed_identity' },
            top_n_documents: top_k
          }
        }]
      })
    });

    const result = await aiResponse.json();

    return {
      status: 200,
      jsonBody: {
        answer: result.choices[0].message.content,
        citations: result.choices[0].message.context?.citations || [],
        query_used: query
      }
    };
  }
});

11. Best Practices

Plugin Design Guidelines

✅ Effective Plugin Design

  • Write clear, descriptive operationId names — Copilot uses these for intent matching
  • Provide detailed parameter descriptions — they guide Copilot's parameter extraction
  • Return structured JSON with consistent schemas
  • Include source citations in every response
  • Handle errors gracefully with user-friendly messages
  • Keep response times under 10 seconds

❌ Common Mistakes

  • Vague operation descriptions that confuse Copilot's routing
  • Returning unstructured text blobs instead of JSON
  • No error handling — plugin silently fails
  • Ignoring Adaptive Card rendering guidelines
  • Not testing with Copilot's orchestrator (only testing the API)
  • Skipping admin consent configuration

Performance Optimization

TechniqueImpactImplementation
Response Caching Reduce latency for repeated queries Azure Redis Cache with 5-minute TTL for common queries
Async Processing Handle long-running operations Return partial results immediately, complete via proactive messages
Connection Pooling Reduce overhead on AI Foundry calls Reuse HTTP clients and SDK instances across requests
Index Optimization Faster search results Use semantic ranking + vector search in Azure AI Search
