Introduction

Welcome to Module 29 of the Azure AI Foundry: Zero to Hero training series. In Modules 27 and 28 you packaged your AI agents into containers and set up CI/CD pipelines. Now it's time to tackle one of the most critical challenges in production AI: scaling deployments across multiple environments.

Every enterprise follows a promotion path — Dev → Staging → Production — but AI workloads introduce unique complexities. Model endpoints, API keys, connection strings, compute SKUs, and safety configurations all differ between environments. Getting this wrong leads to outages, data leaks, and failed deployments.

💡 Why This Module Matters

Production AI systems need environment isolation, safe promotion gates, and multi-region resilience. This module teaches you the patterns used by enterprises running AI workloads at scale on Azure AI Foundry — including the infrastructure-as-code, CI/CD pipelines, and configuration management strategies that make it repeatable.

By the end of this module you will be able to design a multi-environment deployment topology, implement promotion gates between Dev/Staging/Prod, deploy agents to multiple Azure regions, configure load balancing and failover, and manage environment-specific settings without code changes. Let's build for scale.

The promotion model defines how your AI agents move from development through testing to production. In Azure AI Foundry, each environment maps to a separate AI Foundry project (or hub), ensuring full isolation of resources, permissions, and data.

Promotion Gate Model

Each environment transition requires passing a set of quality gates. These gates are enforced by your CI/CD pipeline and are non-negotiable for production deployments:

  1. Dev → Staging Gate: All unit tests pass, linting clean, container image builds successfully, basic smoke tests pass against dev endpoint.
  2. Staging → Prod Gate: Integration tests pass, evaluation benchmarks meet thresholds (groundedness ≥ 0.85, relevance ≥ 0.80), load testing passes at 2× expected traffic, security scan clean, manual approval from release manager.
⚠️ Important: Never Skip Staging

It's tempting to promote directly from Dev to Prod for "small changes." Don't. AI systems exhibit emergent behaviors that only surface under realistic conditions. Staging is where you catch prompt regressions, unexpected model responses, and integration failures before they affect real users.

Environment Architecture in Azure AI Foundry

Azure AI Foundry uses a Hub → Project hierarchy. For multi-environment deployments, you have two main architectural patterns:

Pattern 1: Shared Hub, Separate Projects

A single AI Foundry Hub hosts multiple projects — one per environment. This is simpler to manage but offers less isolation:

┌─────────────────────────────────────┐ │ AI Foundry Hub │ │ (Shared region) │ │ │ │ ┌───────────┐ ┌───────────────┐ │ │ │ Project: │ │ Project: │ │ │ │ agent-dev │ │ agent-staging │ │ │ └───────────┘ └───────────────┘ │ │ │ │ ┌───────────────┐ │ │ │ Project: │ │ │ │ agent-prod │ │ │ └───────────────┘ │ └─────────────────────────────────────┘

Pattern 2: Separate Hubs per Environment (Recommended)

Each environment gets its own Hub and Project. This provides complete isolation of networking, identity, and resources:

┌──────────────┐ ┌──────────────┐ ┌──────────────┐ │ Hub: DEV │ │ Hub: STAGING │ │ Hub: PROD │ │ Region: EUS │ │ Region: EUS │ │ Region: EUS │ │ │ │ │ │ │ │ ┌──────────┐ │ │ ┌──────────┐ │ │ ┌──────────┐ │ │ │ Project: │ │ │ │ Project: │ │ │ │ Project: │ │ │ │ agent │ │──▶│ │ agent │ │──▶│ │ agent │ │ │ └──────────┘ │ │ └──────────┘ │ │ └──────────┘ │ └──────────────┘ └──────────────┘ └──────────────┘
✅ Best Practice

Use separate Hubs per environment for production workloads. This enables independent networking (VNet integration), separate managed identities, environment-specific RBAC, and independent scaling — all critical for compliance and security.

Infrastructure as Code for Environments

Every environment should be defined in code using Bicep or Terraform. This ensures reproducibility, auditability, and eliminates configuration drift between environments.

Parameterized Bicep Template

Use a single Bicep template with environment-specific parameter files:

Bicep // main.bicep — Multi-environment AI Foundry deployment param environmentName string // 'dev', 'staging', 'prod' param location string param aiHubName string param aiProjectName string param skuName string param modelDeploymentCapacity int resource aiHub 'Microsoft.MachineLearningServices/workspaces@2024-10-01' = { name: aiHubName location: location kind: 'hub' sku: { name: skuName } properties: { friendlyName: 'AI Hub - ${environmentName}' publicNetworkAccess: environmentName == 'prod' ? 'Disabled' : 'Enabled' } tags: { environment: environmentName managedBy: 'bicep' } } resource aiProject 'Microsoft.MachineLearningServices/workspaces@2024-10-01' = { name: aiProjectName location: location kind: 'project' sku: { name: skuName } properties: { friendlyName: 'Agent Project - ${environmentName}' hubResourceId: aiHub.id } }

Multi-Region Deployment Strategies

For high-availability and disaster recovery, production AI workloads should be deployed across multiple Azure regions. Azure AI Foundry supports several multi-region patterns:

Active-Passive

One primary region handles all traffic while a secondary region remains on standby. Failover is triggered manually or by health probes.

✅

Pros

Lower cost (standby resources), simpler configuration, easier data consistency. Good for workloads with relaxed RTO requirements.

❌

Cons

Cold start delay during failover, potential data loss if replication lag exists, higher RTO than active-active.

Active-Active

Both regions serve live traffic simultaneously. Azure Front Door or Traffic Manager distributes requests based on latency, geography, or weighted routing.

┌──────────────────┐ │ Azure Front Door │ │ (Global LB) │ └────────┬─────────┘ ┌──────┴──────┐ ▼ ▼ ┌──────────────┐ ┌──────────────┐ │ East US │ │ West Europe │ │ │ │ │ │ AI Foundry │ │ AI Foundry │ │ Hub + Project│ │ Hub + Project│ │ │ │ │ │ Agent API │ │ Agent API │ │ GPT-4o │ │ GPT-4o │ └──────────────┘ └──────────────┘

Region Selection Considerations

Factor Guidance
Model Availability Not all models are available in every region. Check the model availability matrix before choosing regions.
Data Residency Compliance requirements (GDPR, sovereignty) may restrict which regions you can use.
Latency Choose regions close to your users. Use Azure Front Door for latency-based routing.
Quota & Capacity TPM (tokens per minute) quotas are per-region. Spreading across regions increases total throughput.
Paired Regions Use Azure paired regions (e.g., East US / West US) for best cross-region replication guarantees.

Load Balancing and Failover

Once you deploy across multiple regions, you need intelligent traffic distribution and automatic failover. Azure provides several services for this:

Azure Front Door for AI Workloads

Azure Front Door is the recommended global load balancer for AI agent APIs. It provides:

  • Latency-based routing — Sends users to the fastest responding region
  • Health probes — Automatically detects endpoint failures and reroutes traffic
  • WAF integration — Protects your AI endpoints from attacks
  • SSL termination — Handles TLS certificates at the edge
  • Session affinity — Keeps multi-turn conversations routed to the same backend
Bicep // Azure Front Door for multi-region AI agent routing resource frontDoor 'Microsoft.Cdn/profiles@2024-02-01' = { name: 'fd-ai-agent-prod' location: 'global' sku: { name: 'Premium_AzureFrontDoor' } } resource originGroup 'Microsoft.Cdn/profiles/originGroups@2024-02-01' = { parent: frontDoor name: 'agent-backends' properties: { loadBalancingSettings: { sampleSize: 4 successfulSamplesRequired: 3 additionalLatencyInMilliseconds: 50 } healthProbeSettings: { probePath: '/health' probeRequestType: 'GET' probeProtocol: 'Https' probeIntervalInSeconds: 30 } } }

API Management as an AI Gateway

Azure API Management (APIM) sits between your clients and AI Foundry endpoints, providing:

  • Token rate limiting — Control TPM consumption per client or subscription
  • Retry with failover — Automatically retry against a secondary region on 429/503 errors
  • Request/response logging — Capture prompts and completions for monitoring
  • Caching — Cache identical prompts to reduce model calls and cost
XML — APIM retry policy <retry count="3" interval="2" delta="1" first-fast-retry="true" condition="@(context.Response.StatusCode == 429 || context.Response.StatusCode == 503)"> <set-backend-service base-url="@{ var backends = new[] { "https://agent-eastus.azurewebsites.net", "https://agent-westeurope.azurewebsites.net" }; return backends[context.Retry.Count % backends.Length]; }" /> </retry>

Failover Patterns for AI Agents

🔄

Circuit Breaker

After N consecutive failures, stop routing to the failed region for a cooldown period. Prevents cascading failures and gives the region time to recover.

📊

Weighted Distribution

Gradually shift traffic from a failing region (90/10 → 50/50 → 10/90) rather than instant cutover. Reduces blast radius of failover events.

🏥

Health Endpoint

Implement a /health endpoint that checks model connectivity, database access, and downstream dependencies before returning healthy.

Implementing a Health Check Endpoint

Python from fastapi import FastAPI, Response from azure.ai.projects import AIProjectClient from azure.identity import DefaultAzureCredential import os, time app = FastAPI() @app.get("/health") async def health_check(): checks = {} try: # Check AI Foundry project connectivity client = AIProjectClient( credential=DefaultAzureCredential(), project_connection_string=os.environ["PROJECT_CONNECTION"] ) start = time.time() client.agents.list_agents(limit=1) checks["ai_foundry"] = { "status": "healthy", "latency_ms": round((time.time() - start) * 1000) } except Exception as e: checks["ai_foundry"] = {"status": "unhealthy", "error": str(e)} all_healthy = all(c["status"] == "healthy" for c in checks.values()) status_code = 200 if all_healthy else 503 return Response( content=json.dumps({"status": "healthy" if all_healthy else "unhealthy", "checks": checks}), status_code=status_code, media_type="application/json" )

Environment-Specific Configuration Management

Managing configuration across environments is one of the most error-prone aspects of multi-environment deployments. The key principle: configuration should never be baked into your container image.

The Configuration Hierarchy

  1. Azure App Configuration — Centralized key-value store with labels per environment
  2. Azure Key Vault — Secrets, connection strings, and API keys
  3. Environment Variables — Injected at container runtime from App Configuration / Key Vault
  4. Feature Flags — Toggle capabilities without redeployment

Azure App Configuration with Environment Labels

Azure CLI # Set config values with environment labels az appconfig kv set --name appconfig-ai-agent \ --key "Agent:ModelDeployment" \ --value "gpt-4o-dev" \ --label "dev" --yes az appconfig kv set --name appconfig-ai-agent \ --key "Agent:ModelDeployment" \ --value "gpt-4o-prod" \ --label "prod" --yes # Set Key Vault reference for secrets az appconfig kv set-keyvault --name appconfig-ai-agent \ --key "Agent:ConnectionString" \ --secret-identifier "https://kv-agent-prod.vault.azure.net/secrets/project-conn" \ --label "prod" --yes

Loading Configuration by Environment

Python from azure.appconfiguration import AzureAppConfigurationClient from azure.identity import DefaultAzureCredential import os def load_config(environment: str) -> dict: """Load environment-specific configuration from Azure App Configuration.""" client = AzureAppConfigurationClient( base_url=os.environ["APP_CONFIG_ENDPOINT"], credential=DefaultAzureCredential() ) config = {} # Fetch all settings for this environment label settings = client.list_configuration_settings( label_filter=environment ) for setting in settings: config[setting.key] = setting.value return config # Usage — environment is injected via ENVIRONMENT env var env = os.environ.get("ENVIRONMENT", "dev") config = load_config(env) model_deployment = config["Agent:ModelDeployment"]
🚨 Anti-Pattern: Hardcoded Configuration

Never hardcode connection strings, endpoints, or model deployment names in your application code or Dockerfiles. These values must come from external configuration. A single container image should be promotable across all environments without modification.

CI/CD Promotion Pipelines

Your CI/CD pipeline orchestrates the entire promotion flow. Here's a GitHub Actions workflow that deploys through all three environments with approval gates:

YAML — .github/workflows/deploy-agent.yml name: Deploy AI Agent on: push: branches: [main] env: ACR_NAME: acragentprod IMAGE_NAME: ai-agent jobs: build: runs-on: ubuntu-latest outputs: image-tag: ${{ steps.meta.outputs.version }} steps: - uses: actions/checkout@v4 - name: Build & push to ACR run: | az acr build --registry $ACR_NAME \ --image $IMAGE_NAME:${{ github.sha }} . deploy-dev: needs: build environment: development runs-on: ubuntu-latest steps: - name: Deploy to Dev run: | az containerapp update \ --name agent-dev \ --resource-group rg-ai-dev \ --image $ACR_NAME.azurecr.io/$IMAGE_NAME:${{ github.sha }} - name: Run smoke tests run: | python tests/smoke_test.py \ --endpoint https://agent-dev.azurecontainerapps.io deploy-staging: needs: deploy-dev environment: staging # Requires approval in GitHub runs-on: ubuntu-latest steps: - name: Deploy to Staging run: | az containerapp update \ --name agent-staging \ --resource-group rg-ai-staging \ --image $ACR_NAME.azurecr.io/$IMAGE_NAME:${{ github.sha }} - name: Run evaluation benchmarks run: | python evaluations/run_eval.py \ --endpoint https://agent-staging.azurecontainerapps.io \ --min-groundedness 0.85 \ --min-relevance 0.80 deploy-prod: needs: deploy-staging environment: production # Requires manager approval runs-on: ubuntu-latest steps: - name: Deploy to Prod (Blue-Green) run: | # Deploy to inactive revision az containerapp revision copy \ --name agent-prod \ --resource-group rg-ai-prod \ --image $ACR_NAME.azurecr.io/$IMAGE_NAME:${{ github.sha }} # Shift 10% traffic to new revision az containerapp ingress traffic set \ --name agent-prod \ --resource-group rg-ai-prod \ --revision-weight latest=10 - name: Validate & promote to 100% run: | python tests/canary_validation.py az containerapp ingress traffic set \ --name agent-prod \ --resource-group rg-ai-prod \ --revision-weight latest=100
💡 Blue-Green & Canary Deployments

Azure Container Apps supports traffic splitting across revisions. Use canary deployments (10% → 50% → 100%) in production to gradually validate new versions. If the canary fails evaluation, roll back by shifting all traffic to the previous revision — zero downtime.

Monitoring & Observability Across Environments

Each environment should send telemetry to a shared monitoring stack so you can compare metrics across Dev, Staging, and Prod:

Observability Stack

📈

Application Insights

Distributed tracing, request logging, and custom metrics for each environment. Tag telemetry with cloud_RoleName to filter by environment.

📊

Azure Monitor Workbooks

Cross-environment dashboards comparing latency, error rates, token consumption, and evaluation scores side by side.

🔔

Alert Rules

Environment-specific thresholds. Prod alerts go to PagerDuty; staging alerts go to Slack; dev alerts are informational only.

📋

Azure AI Foundry Tracing

Built-in tracing in AI Foundry captures agent runs, tool calls, and model interactions with full prompt/completion logging.

Key Takeaways

🔀

Dev → Staging → Prod is Non-Negotiable

Every AI agent must traverse environments with quality gates. No exceptions — AI systems have emergent behaviors that only appear under realistic conditions.

🏗️

Separate Hubs for Isolation

Use separate AI Foundry Hubs per environment for complete isolation of identity, networking, data, and compute resources.

🌍

Multi-Region = Resilience

Deploy to multiple Azure regions with Azure Front Door for latency-based routing and automatic failover. Active-active for critical workloads.

⚙️

Externalize Configuration

Use Azure App Configuration and Key Vault. One container image, many environments. Never bake config into your Docker image.

🚦

Canary Deployments for Safety

Use traffic splitting (10% → 50% → 100%) in production. Validate each step with automated evaluation before promoting.

📊

Observe Everything

Cross-environment dashboards, environment-tagged telemetry, and differentiated alert thresholds are essential for operating at scale.
