Introduction

Modern AI applications rarely rely on a single model. Different models offer different strengths — GPT-4o excels at complex reasoning and code generation, Phi-4 delivers blazing-fast responses for simple tasks at a fraction of the cost, and Meta-Llama-3.1-8B provides strong open-weight performance for summarization and extraction workloads.

The challenge isn't choosing one model — it's building a system that intelligently selects the right model for each request. This is multi-model orchestration: a design pattern where a routing layer analyzes incoming requests and dispatches them to the most appropriate model based on task complexity, cost constraints, latency requirements, and capability needs.

💡 Key Insight

Organizations that implement multi-model routing in Azure AI Foundry typically see 60–80% cost reductions compared to routing all requests through a single premium model, while maintaining 95%+ quality on their output benchmarks.

In this module, you'll learn the three core routing patterns, understand how to orchestrate multiple models deployed in Azure AI Foundry's model catalog, and build a working smart router in the hands-on Mini Hack.

When to Use Multiple Models

Not every application needs multi-model orchestration. A simple chatbot with consistent behavior requirements may be perfectly served by a single model. However, as your application grows in complexity, traffic volume, or cost sensitivity, multi-model architectures become increasingly valuable.

Single-Model vs Multi-Model

🎯

Single Model

Simpler architecture, consistent behavior, uniform latency. Best for early-stage apps, uniform workloads, and when operational simplicity matters most.

🔀

Multi-Model

Cost-optimized, specialized quality per task, built-in redundancy. Best for production apps with diverse workloads, high traffic, and cost sensitivity.

Decision Matrix: When to Go Multi-Model

Factor Single Model Multi-Model
Monthly API spend < $500 > $500 — routing pays for itself
Request diversity Uniform task types Mix of simple & complex tasks
Latency requirements Consistent is fine Some tasks need sub-300ms
Availability needs Standard SLA sufficient Need 99.9%+ with fallbacks
Team capacity Small team, minimize ops Can invest in routing logic
⚠️ Important

Multi-model routing adds architectural complexity. Start with a single model, measure your workload distribution, and add routing when the data justifies the investment. Don't prematurely optimize.

Common Multi-Model Use Cases

  • Customer support platforms — Route FAQ answers to Phi-4, complex troubleshooting to GPT-4o
  • Document processing pipelines — Use Llama for extraction and summarization, GPT-4o for analysis
  • Code assistants — Phi-4 for autocomplete suggestions, GPT-4o for complex refactoring
  • Content platforms — Llama for content classification, GPT-4o for content generation
  • Search applications — Phi-4 for query understanding, GPT-4o for answer synthesis

Router Patterns

A model router is the decision-making layer that sits between your application and the model pool. It examines each incoming request and determines which model should handle it. There are three fundamental routing patterns, each optimized for different priorities.

Pattern 1: Cost-Based Routing

Cost-based routing is the simplest and most impactful pattern. The router always selects the cheapest model that meets a minimum quality threshold for the given request. This requires you to define quality tiers and map them to model capabilities.

How It Works

  1. Classify the incoming request by complexity (simple, moderate, complex)
  2. Look up which models can handle that complexity level
  3. Sort qualifying models by cost (ascending)
  4. Route to the cheapest qualifying model
cost_router.py Python
from azure.ai.inference import ChatCompletionsClient from azure.identity import DefaultAzureCredential # Define model capabilities and costs ROUTING_CONFIG = { "simple_qa": { "models": ["Phi-4"], "max_tokens": 256 }, "summarization": { "models": ["Meta-Llama-3.1-8B", "Phi-4"], "max_tokens": 1024 }, "complex_reasoning": { "models": ["gpt-4o"], "max_tokens": 4096 }, "code_generation": { "models": ["gpt-4o", "Meta-Llama-3.1-8B"], "max_tokens": 2048 }, }

Pattern 2: Capability-Based Routing

Capability-based routing selects models based on what they're best at, not just what they cost. Each model has a capability profile, and the router matches the task type to the model with the strongest track record for that category.

Task Type Primary Model Fallback Rationale
Complex reasoning GPT-4o — Strongest chain-of-thought capability
Classification Phi-4 Llama 3.1 Fast, accurate on structured tasks
Summarization Llama 3.1 8B Phi-4 Strong extraction with low cost
Code generation GPT-4o Llama 3.1 Best code quality and debugging
Translation GPT-4o Llama 3.1 Broadest language support
Entity extraction Phi-4 Llama 3.1 Low-latency structured output
capability_router.py Python
class CapabilityRouter: def __init__(self): self.capability_map = { "classification": ["Phi-4", "Meta-Llama-3.1-8B"], "summarization": ["Meta-Llama-3.1-8B", "Phi-4"], "reasoning": ["gpt-4o"], "code_gen": ["gpt-4o", "Meta-Llama-3.1-8B"], "extraction": ["Phi-4", "Meta-Llama-3.1-8B"], } def classify_task(self, prompt): # Simple keyword-based classifier prompt_lower = prompt.lower() if any(w in prompt_lower for w in ["classify", "categorize", "label"]): return "classification" if any(w in prompt_lower for w in ["summarize", "summary", "tldr"]): return "summarization" if any(w in prompt_lower for w in ["code", "function", "implement", "debug"]): return "code_gen" return "reasoning" # default to strongest def route(self, prompt): task = self.classify_task(prompt) models = self.capability_map.get(task, ["gpt-4o"]) return models[0] # Return primary model
✅ Best Practice

Start with a simple keyword-based classifier, then graduate to using a small model (like Phi-4 itself) as the classifier. This "model-as-router" pattern can achieve 90%+ classification accuracy with minimal added latency.

Pattern 3: Fallback & Cascade Routing

The cascade pattern is the most resilient routing strategy. It attempts the cheapest model first, validates the response quality, and escalates to the next model if the result doesn't meet the quality threshold. This provides both cost optimization and reliability.

cascade_router.py Python
import asyncio from azure.ai.inference import ChatCompletionsClient from azure.identity import DefaultAzureCredential async def cascade_completion(prompt, endpoint, credential): """Try models cheapest-first, escalate on failure.""" models = ["Phi-4", "Meta-Llama-3.1-8B", "gpt-4o"] for model_name in models: try: client = ChatCompletionsClient( endpoint=endpoint, credential=credential ) response = client.complete( model=model_name, messages=[{ "role": "user", "content": prompt }] ) # Validate response quality if response.choices[0].finish_reason == "stop": return { "model": model_name, "response": response, "escalated": model_name != models[0] } except Exception as e: print(f"Model {model_name} failed: {e}") continue raise Exception("All models in cascade failed")
⚠️ Important Consideration

Cascade routing adds latency when escalation occurs. In the worst case (all escalations), a request hits every model in the chain. Set appropriate timeouts and monitor your escalation rate — if it exceeds 30%, your initial model selection may need tuning.

The SmartModelRouter Class

Here's a complete implementation of a smart router that combines cost-based and capability-based routing with fallback support:

smart_router.py Python
from azure.ai.inference import ChatCompletionsClient from azure.identity import DefaultAzureCredential import logging logger = logging.getLogger(__name__) class SmartModelRouter: """Routes requests to optimal models in Azure AI Foundry.""" def __init__(self, endpoint): self.endpoint = endpoint self.credential = DefaultAzureCredential() self.models = { "gpt-4o": { "cost_per_1k_input": 0.005, "cost_per_1k_output": 0.015, "capability": "high", "latency_ms": 800, "tasks": ["reasoning", "code", "creative", "analysis"], }, "Phi-4": { "cost_per_1k_input": 0.0002, "cost_per_1k_output": 0.0002, "capability": "medium", "latency_ms": 200, "tasks": ["classification", "qa", "extraction"], }, "Meta-Llama-3.1-8B": { "cost_per_1k_input": 0.0003, "cost_per_1k_output": 0.0003, "capability": "medium", "latency_ms": 300, "tasks": ["summarization", "extraction", "translation"], }, } self.stats = {"total": 0, "by_model": {}} def route(self, task_complexity, max_cost=None): """Select optimal model based on task and cost.""" cap_levels = { "simple": ["medium", "high"], "moderate": ["medium", "high"], "complex": ["high"], } required = cap_levels.get(task_complexity, ["high"]) candidates = [ name for name, cfg in self.models.items() if cfg["capability"] in required ] if max_cost: candidates = [ m for m in candidates if self.models[m]["cost_per_1k_input"] <= max_cost ] if not candidates: return "gpt-4o" # Always fall back return sorted( candidates, key=lambda m: self.models[m]["cost_per_1k_input"] )[0] def complete(self, prompt, task_complexity="moderate"): """Route and execute a completion request.""" model = self.route(task_complexity) logger.info(f"Routing to {model} for {task_complexity}") client = ChatCompletionsClient( endpoint=self.endpoint, credential=self.credential ) response = client.complete( model=model, messages=[{"role": "user", "content": prompt}] ) # Track usage stats self.stats["total"] += 1 self.stats["by_model"][model] = \ self.stats["by_model"].get(model, 0) + 1 return { "model": model, "content": response.choices[0].message.content, "usage": response.usage, }

The azure-ai-inference SDK provides a unified ChatCompletionsClient that works identically across all model providers. Whether you're calling GPT-4o, Phi-4, or Llama, the API interface is the same — only the model parameter changes.

Implementation Guide

Follow these steps to implement multi-model orchestration in your Azure AI Foundry project:

  1. Deploy Models from the Model Catalog Navigate to ai.azure.com, open your project, and deploy GPT-4o, Phi-4, and Meta-Llama-3.1-8B as serverless API endpoints from the model catalog.
  2. Install the Azure AI Inference SDK Run pip install azure-ai-inference azure-identity to install the unified SDK that works with all model providers in Azure AI Foundry.
  3. Build the Task Classifier Create a lightweight classifier that categorizes incoming prompts by task type and complexity. Start with keyword heuristics, then consider using Phi-4 as a meta-classifier.
  4. Implement the Router Logic Build your routing engine using the patterns covered in this module — cost-based, capability-based, or cascade. Store routing configs externally (JSON or environment variables) for easy tuning.
  5. Add Observability & Logging Log every routing decision: which model was selected, why, the response latency, and token usage. This data drives future optimizations and helps catch routing errors.
  6. Test with Mixed Workloads Create a test suite with a realistic mix of simple, moderate, and complex prompts. Measure cost, latency, and quality across the full workload to validate your routing rules.
  7. Monitor and Iterate Use Azure Monitor and Application Insights to track routing patterns in production. Adjust thresholds based on real-world escalation rates and quality scores.
⚠️ Externalize Configuration

Never hardcode routing rules. Store your model registry, capability maps, and cost thresholds in a configuration file or Azure App Configuration. This allows you to update routing logic without redeploying your application.

Configuration-Driven Routing

router_config.json JSON
{ "models": { "gpt-4o": { "endpoint": "https://<project>.services.ai.azure.com/models", "cost_input_1k": 0.005, "cost_output_1k": 0.015, "capability": "high", "max_tokens": 4096 }, "Phi-4": { "endpoint": "https://<project>.services.ai.azure.com/models", "cost_input_1k": 0.0002, "cost_output_1k": 0.0002, "capability": "medium", "max_tokens": 1024 }, "Meta-Llama-3.1-8B": { "endpoint": "https://<project>.services.ai.azure.com/models", "cost_input_1k": 0.0003, "cost_output_1k": 0.0003, "capability": "medium", "max_tokens": 2048 } }, "routing_rules": { "simple": ["Phi-4", "Meta-Llama-3.1-8B"], "moderate": ["Meta-Llama-3.1-8B", "Phi-4", "gpt-4o"], "complex": ["gpt-4o"] } }

Key Takeaways

🎯 Module 22 — Key Takeaways

  • No single model rules them all — GPT-4o, Phi-4, and Llama each excel at different task categories.
  • Cost-based routing sends each request to the cheapest model that meets the quality bar, saving 60–80% on typical workloads.
  • Capability-based routing matches task types to models optimized for that specific capability.
  • Fallback cascade tries the cheapest model first and escalates only when needed — providing both cost savings and resilience.
  • Azure AI Foundry unifies access — the azure-ai-inference SDK provides a single interface for all models in the catalog.
  • Externalize routing configuration — keep model registries and routing rules in config files, not code.
  • Measure everything — log routing decisions, track escalation rates, and monitor cost-per-request to continuously optimize.
  • Start simple, iterate — begin with cost-based routing on two models, then add capability routing and cascades as your workload grows.
