Empower your AI agents with the ability to call external functions, query databases, invoke APIs, and take real-world actions — turning conversational AI into powerful, autonomous systems with Azure AI Foundry.

Overview

Function calling is the mechanism that transforms a large language model from a passive text generator into an active agent capable of performing real-world tasks. In Azure AI Foundry, custom tools and function calling allow your agents to retrieve live data, execute business logic, interact with databases, and orchestrate multi-step workflows — all while maintaining natural conversational flow.

This module covers the complete lifecycle of building function-enabled agents: defining tool schemas with JSON Schema, implementing function calling patterns, connecting to external services, and building robust error-handling strategies that keep your agents reliable in production.

Key Concept

Function calling does not mean the model executes code directly. Instead, the model outputs a structured JSON request indicating which function to call and with what arguments. Your application code then executes the function and returns the result to the model for further reasoning.

  • Connect agents to live REST APIs, databases, and internal microservices

Defining Custom Functions for Agents

Custom functions are the building blocks of tool-enabled agents. Each function is described by a tool definition that tells the model what the function does, what parameters it accepts, and what format those parameters should take. Azure AI Foundry uses the OpenAI-compatible tool specification format, built on JSON Schema.

Anatomy of a Tool Definition

Every tool definition consists of three key components:

Component Purpose Best Practice
name Unique identifier for the function Use snake_case, be descriptive (e.g., get_weather_forecast)
description Tells the model when and why to use this function Be specific about capabilities and limitations
parameters JSON Schema defining input shape and validation Use required fields, add enum constraints where possible

Example: Weather Lookup Tool

# Define a weather lookup tool with JSON Schema parameters
weather_tool = {
    "type": "function",
    "function": {
        "name": "get_weather",
        "description": "Get current weather conditions and forecast for a specific city. Returns temperature, humidity, wind speed, and a short text forecast.",
        "parameters": {
            "type": "object",
            "properties": {
                "city": {
                    "type": "string",
                    "description": "City name, e.g., 'Seattle' or 'London'"
                },
                "units": {
                    "type": "string",
                    "enum": ["celsius", "fahrenheit"],
                    "description": "Temperature unit preference"
                }
            },
            "required": ["city"],
            "additionalProperties": false
        }
    }
}

Example: Meeting Booking Tool

# Define a meeting booking tool
meeting_tool = {
    "type": "function",
    "function": {
        "name": "book_meeting",
        "description": "Schedule a meeting in the user's calendar. Checks for conflicts and sends invitations to all attendees.",
        "parameters": {
            "type": "object",
            "properties": {
                "title": {
                    "type": "string",
                    "description": "Meeting title/subject"
                },
                "date": {
                    "type": "string",
                    "description": "Meeting date in ISO 8601 format (YYYY-MM-DD)",
                    "pattern": "^\\d{4}-\\d{2}-\\d{2}$"
                },
                "start_time": {
                    "type": "string",
                    "description": "Start time in 24-hour format (HH:MM)"
                },
                "duration_minutes": {
                    "type": "integer",
                    "minimum": 15,
                    "maximum": 480,
                    "description": "Meeting duration in minutes"
                },
                "attendees": {
                    "type": "array",
                    "items": { "type": "string", "format": "email" },
                    "description": "List of attendee email addresses"
                }
            },
            "required": ["title", "date", "start_time", "duration_minutes"],
            "additionalProperties": false
        }
    }
}
Pro Tip

Write descriptions as if explaining the function to a new team member. The model uses the description field to decide when to invoke a tool, so clarity directly impacts accuracy. Include what the function returns and any constraints.

Schema Design Best Practices

  • Use enums: Constrain values to valid options (e.g., "enum": ["celsius", "fahrenheit"])
  • Add patterns: Use regex patterns for dates, IDs, and formatted strings
  • Set boundaries: Use minimum/maximum for numbers, minLength/maxLength for strings
  • Mark required fields: Always specify which parameters are mandatory
  • Disable extras: Set "additionalProperties": false to prevent hallucinated fields
  • Nest wisely: Keep schemas flat where possible; deep nesting reduces model accuracy

Function Calling Patterns & Schemas

Azure AI Foundry supports several function calling patterns, each suited to different scenarios. Understanding these patterns lets you design agents that can handle simple lookups, complex multi-step workflows, and everything in between.

⚙️ Your Code Executes function

Pattern 1: Sequential Function Calling

The model calls one function at a time, waiting for results before deciding whether to call another function or generate a final response. 

from azure.ai.projects import AIProjectClient
from azure.identity import DefaultAzureCredential
import json

# Initialize the AI Foundry client
client = AIProjectClient(
    credential=DefaultAzureCredential(),
    project_endpoint="https://<your-project>.services.ai.azure.com"
)

# Register tools with the agent
tools = [weather_tool, meeting_tool]

# Create the agent with function calling capabilities
agent = client.agents.create_agent(
    model="gpt-4o",
    name="scheduling-assistant",
    instructions="""You are a scheduling assistant that can check weather 
    and book meetings. Always check weather before suggesting 
    outdoor meeting locations.""",
    tools=tools
)

# Create a conversation thread
thread = client.agents.create_thread()
client.agents.create_message(
    thread_id=thread.id,
    role="user",
    content="What's the weather in Seattle? If it's nice, book a lunch meeting."
)

# Run the agent — it will call functions as needed
run = client.agents.create_and_process_run(
    thread_id=thread.id,
    agent_id=agent.id
)

Pattern 2: Parallel Function Calling

When the model determines that multiple independent functions are needed, it can request them all at once. This reduces latency and is ideal for dashboard-style queries. 

# The model may request multiple function calls simultaneously
# Example: "What's the weather in Seattle, Portland, and Vancouver?"
# → Model emits 3 parallel get_weather calls

import asyncio

async def handle_tool_calls(tool_calls):
    """Execute multiple tool calls concurrently."""
    tasks = []
    for call in tool_calls:
        func_name = call.function.name
        args = json.loads(call.function.arguments)
        
        if func_name == "get_weather":
            tasks.append(fetch_weather_async(args["city"]))
        elif func_name == "book_meeting":
            tasks.append(create_meeting_async(**args))
    
    results = await asyncio.gather(*tasks, return_exceptions=True)
    return results

Pattern 3: Chained (Multi-Turn) Function Calling

The result of one function call informs the next. The model reasons over intermediate results and decides on subsequent actions, creating sophisticated multi-step workflows.

Chained Function Calling — Weather → Schedule → Book
get_weather Step 1: Check conditions
→
LLM Decides "Weather is good"
→
check_availability Step 2: Find open slot
→
book_meeting Step 3: Confirm booking
Important

Azure AI Foundry agents handle the function calling loop automatically when you use create_and_process_run(). For manual control over each step (e.g., to add custom logging or approval gates), use create_run() and process tool calls in your own loop.

JSON Schema Deep Dive

JSON Schema is the backbone of function calling. Well-crafted schemas reduce hallucinated parameters and improve tool selection accuracy. Key schema features for function calling:

Schema Feature Use Case Example
enum Constrain to known values "enum": ["high", "medium", "low"]
pattern Validate format with regex "pattern": "^\\d{4}-\\d{2}-\\d{2}$"
minimum/maximum Bound numeric ranges "minimum": 1, "maximum": 100
items Define array element types "items": {"type": "string", "format": "email"}
$defs Reusable sub-schemas Shared address or contact structures
oneOf/anyOf Conditional parameters Different schemas based on action type

Connecting to a REST API

import httpx
from azure.identity import DefaultAzureCredential

async def get_weather(city: str, units: str = "celsius") -> dict:
    """Fetch live weather data from an external API."""
    api_key = get_secret_from_keyvault("weather-api-key")
    base_url = "https://api.weatherservice.com/v2/current"
    
    async with httpx.AsyncClient(timeout=10.0) as client:
        response = await client.get(
            base_url,
            params={
                "q": city,
                "units": "metric" if units == "celsius" else "imperial",
                "appid": api_key
            }
        )
        response.raise_for_status()
        data = response.json()
    
    return {
        "city": city,
        "temperature": data["main"]["temp"],
        "humidity": data["main"]["humidity"],
        "conditions": data["weather"][0]["description"],
        "units": units
    }

Connecting to a Database

import aioodbc

async def query_customer_orders(
    customer_id: str,
    status: str = "all"
) -> dict:
    """Query customer order history from Azure SQL Database."""
    conn_str = get_secret_from_keyvault("sql-connection-string")
    
    query = """
        SELECT order_id, product_name, quantity, total_price, status
        FROM orders 
        WHERE customer_id = ? 
    """
    params = [customer_id]
    
    if status != "all":
        query += " AND status = ?"
        params.append(status)
    
    query += " ORDER BY order_date DESC LIMIT 20"
    
    async with aioodbc.connect(dsn=conn_str) as conn:
        async with conn.cursor() as cursor:
            await cursor.execute(query, params)
            rows = await cursor.fetchall()
    
    return {
        "customer_id": customer_id,
        "order_count": len(rows),
        "orders": [
            {"id": r[0], "product": r[1], "qty": r[2], "total": float(r[3]), "status": r[4]}
            for r in rows
        ]
    }
Security Warning

Never expose raw SQL access to AI agents. Always use parameterized queries, restrict database permissions to read-only where possible, and implement row-level security. Store credentials in Azure Key Vault — never hard-code API keys or connection strings.

Implementing a Function Router

A function router maps tool call names to your implementation functions. This pattern keeps your code organized and makes it easy to add new tools: 

from typing import Callable, Any

class FunctionRouter:
    """Routes agent tool calls to implementation functions."""
    
    def __init__(self):
        self._registry: dict[str, Callable] = {}
    
    def register(self, name: str, func: Callable):
        """Register a function handler by tool name."""
        self._registry[name] = func
    
    async def dispatch(self, tool_call) -> str:
        """Execute the requested function and return JSON result."""
        func_name = tool_call.function.name
        args = json.loads(tool_call.function.arguments)
        
        if func_name not in self._registry:
            return json.dumps({"error": f"Unknown function: {func_name}"})
        
        try:
            result = await self._registry[func_name](**args)
            return json.dumps(result)
        except Exception as e:
            return json.dumps({"error": str(e), "function": func_name})

# Usage
router = FunctionRouter()
router.register("get_weather", get_weather)
router.register("book_meeting", book_meeting)
router.register("query_customer_orders", query_customer_orders)

Implementing Retry with Exponential Backoff

import asyncio
import random
from functools import wraps

def retry_with_backoff(max_retries=3, base_delay=1.0, max_delay=30.0):
    """Decorator: retry async functions with exponential backoff + jitter."""
    def decorator(func):
        @wraps(func)
        async def wrapper(*args, **kwargs):
            for attempt in range(max_retries + 1):
                try:
                    return await func(*args, **kwargs)
                except (httpx.TimeoutException, httpx.HTTPStatusError) as e:
                    if attempt == max_retries:
                        raise
                    delay = min(base_delay * (2 ** attempt) + random.uniform(0, 1), max_delay)
                    await asyncio.sleep(delay)
        return wrapper
    return decorator

@retry_with_backoff(max_retries=3, base_delay=1.0)
async def get_weather(city: str, units: str = "celsius") -> dict:
    # ... implementation with automatic retry

Circuit Breaker Pattern

import time

class CircuitBreaker:
    """Prevents repeated calls to a failing service."""
    
    def __init__(self, failure_threshold=5, recovery_timeout=60):
        self.failure_threshold = failure_threshold
        self.recovery_timeout = recovery_timeout
        self.failure_count = 0
        self.last_failure_time = None
        self.state = "closed"  # closed = healthy, open = failing
    
    def can_execute(self) -> bool:
        if self.state == "closed":
            return True
        if time.time() - self.last_failure_time > self.recovery_timeout:
            self.state = "half-open"
            return True
        return False
    
    def record_success(self):
        self.failure_count = 0
        self.state = "closed"
    
    def record_failure(self):
        self.failure_count += 1
        self.last_failure_time = time.time()
        if self.failure_count >= self.failure_threshold:
            self.state = "open"

# Usage in function router
weather_breaker = CircuitBreaker(failure_threshold=3, recovery_timeout=30)

async def safe_get_weather(city, units="celsius"):
    if not weather_breaker.can_execute():
        return {"error": "Weather service temporarily unavailable",
                "fallback": "Please try again in a minute"}
    try:
        result = await get_weather(city, units)
        weather_breaker.record_success()
        return result
    except Exception:
        weather_breaker.record_failure()
        return {"error": "Weather service failed",
                "fallback": "Unable to retrieve weather data"}

Structured Error Responses

When a function fails, return a structured error so the model can reason about the failure and communicate it to the user appropriately: 

# ✅ Good: Structured error with context
{
    "error": "service_unavailable",
    "message": "Weather API returned HTTP 503",
    "retry_after_seconds": 30,
    "fallback_data": {
        "last_known_temperature": 18,
        "data_age_minutes": 45
    }
}

# ❌ Bad: Raw exception string
"ConnectionError: HTTPSConnectionPool(host='api.weather.com', port=443): Max retries exceeded"
Pro Tip

Include a fallback_data field with cached or approximate data. This lets the model still provide a useful (if partial) response. Users prefer an approximate answer to no answer at all.
